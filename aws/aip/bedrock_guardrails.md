# Bedrock ガードレール / プロンプト管理 / モニタリング

## ガードレール多層防御・監査・耐障害性（AIP-18）

### 正解：Bedrockガードレール（プロンプト攻撃フィルター）+ ガードレールプロファイル（クロスリージョン）+ CloudWatch Logs

3つの要件と対応するコンポーネント：

| 要件 | 正解の対応 |
|---|---|
| プロンプトインジェクション・ジェイルブレイクへの多層防御 | ガードレールの**プロンプト攻撃フィルター**（MLベース）+ コンテンツフィルター「高」 |
| 介入理由・ポリシー違反の詳細を監査ログに記録 | **モデル呼び出しログ記録** → CloudWatch Logs |
| リージョン障害時もガードレール評価が継続 | **ガードレールプロファイル**でクロスリージョンガードレール推論を有効化 |

---

### 怪しかったポイント①：プロンプト攻撃フィルター

通常のコンテンツフィルター（暴力・ヘイトなど）とは別に、**プロンプトインジェクション専用のMLベースフィルター**が存在する。

```
コンテンツフィルター（カテゴリ）
  → 暴力・ヘイト・性的・差別的表現の検出
  → NONE/LOW/MEDIUM/HIGH の4段階

プロンプト攻撃フィルター（← これが重要）
  → プロンプトインジェクション・ジェイルブレイクをMLで検出
  → 正規表現や単語リストでは検出できない高度な攻撃に対応
  → コンテンツフィルターと組み合わせることで多層防御を実現
```

**引っかけ：Lambda + 正規表現**はルールベースのため、巧妙に偽装されたプロンプトインジェクションを検出できない。

---

### 怪しかったポイント②：CloudTrail vs CloudWatch Logs（モデル呼び出しログ）

```
CloudTrail
  = AWS APIの操作ログ（誰がいつどのAPIを呼んだか）
  = 「InvokeModel APIが呼ばれた」という事実のみ
  = ガードレールが"なぜ"ブロックしたかの詳細は含まれない

モデル呼び出しログ記録 → CloudWatch Logs
  = 入力プロンプト・出力・ガードレールの介入詳細をフルで記録
  = ブロック理由・違反したポリシー・編集内容まで追跡可能
  = コンプライアンス監査に必要な情報はこちら
```

→ 「判定理由・ポリシー違反の詳細」が要件なら**CloudWatch Logs（モデル呼び出しログ）**が正解。CloudTrailは不十分。

---

### 怪しかったポイント③：クロスリージョンガードレール推論（ガードレールプロファイル）

```
ガードレールプロファイル + クロスリージョンガードレール推論
  = ガードレール評価そのものを複数リージョンで分散処理
  = リージョン障害時も別リージョンでガードレール評価が継続
  = Bedrockネイティブの耐障害性機能

Route 53 フェイルオーバー / Global Accelerator
  = DNSやネットワークレベルでトラフィックを別リージョンに転送
  = ガードレール評価は転送先でゼロから動かす必要がある（準備が必要）
  = ガードレール評価の継続を「保証」する仕組みではない
```

→ 「ガードレール評価が継続できる」という要件は、ネットワーク層のフェイルオーバーではなく**ガードレール推論自体のクロスリージョン分散**で解決する。

---

### 不正解パターン整理

| 選択肢 | 不正解の理由 |
|---|---|
| Lambda + 正規表現 + ガードレール + CloudTrail + Global Accelerator | 正規表現は高度な攻撃を検出できない。CloudTrailはガードレール介入詳細を記録しない |
| ガードレール（拒否トピック・単語フィルターのみ）+ CloudTrail + Route 53 | プロンプト攻撃フィルター未使用。CloudTrailでは介入理由不十分。Route 53はガードレール継続を保証しない |
| ガードレール + SageMakerカスタム分類 + 各リージョンSMエンドポイント + S3 | 過剰なカスタム実装。Bedrockネイティブ機能で要件を満たせるのにオーバーエンジニアリング |

### 正解：OpenSearch Service + k-NN

**「表現は異なるが同じ意図」の問い合わせを検出するには、意味ベクトルの類似検索が必要。**

```
新しい問い合わせ → 埋め込みモデルでベクトル化
  ↓
OpenSearch k-NN で近似最近傍探索
  ├─ 類似度が閾値以上 → キャッシュから応答を返す（モデル呼び出しなし）
  └─ 類似度が閾値未満 → Bedrockを呼び出し → 応答を新規キャッシュに追加
```

### 各選択肢の比較と間違えた理由

| 選択肢 | 結果 | 理由 |
|---|---|---|
| **OpenSearch k-NN** | ✅ 正解 | ANN専用アルゴリズム（HNSW・IVF）でミリ秒検索。意味ベクトル検索に最適化 |
| **Aurora pgvector** | ❌ 不正解（選んだもの） | ベクトル検索はできるが、RDB上でのSQL ORDER BYは大規模ANNに最適化されていない。「応答時間の短縮」要件でOpenSearchに劣る |
| **ElastiCache Redis** | ❌ 不正解（選びそうになった） | ハッシュ値の完全一致のみ。「表現が違う同じ意図」を検出できない |
| **DynamoDB + Comprehend** | ❌ 不正解 | Comprehendのキーフレーズ抽出は構文レベル。意味全体をベクトルで捉えるのとは別物 |

### pgvector vs OpenSearch k-NNの核心的な違い

```
pgvector（Aurora PostgreSQL）
  → RDBのクエリエンジン上で動作
  → SQL ORDER BY + 距離関数 = 全件スキャンに近い処理になりやすい
  → 既存PostgreSQLに追加する場合に便利

OpenSearch k-NN
  → HNSW・IVFなどANN専用アルゴリズムをネイティブサポート
  → 数百万件規模でもミリ秒単位の検索
  → 「応答時間を可能な限り短縮」要件に最適
```

### AIP出題での識別ポイント

- 「意味的に近い」「表現は違うが同じ意図」→ ベクトル検索が必要
- 「完全一致」「ハッシュ」→ Redis/ElastiCacheの引っかけ
- 「キーフレーズ」「キーワード」→ Comprehendの引っかけ（構文レベル≠意味レベル）
- 「低レイテンシー」「大規模」→ pgvectorよりOpenSearch k-NN

### 正解構成

**S3（データセット） + EventBridge + Lambda + Bedrock評価ジョブ + CloudWatch**

```
カスタムデータセット（JSONL形式）→ S3に格納
  ↓
EventBridge（定期スケジュール）→ Lambda起動
  ↓
Lambda → CreateEvaluationJob API呼び出し（S3のURI指定）
  ↓
Bedrock自動モデル評価ジョブ実行
  ↓
別Lambda（ジョブ完了検知）→ CloudWatchにメトリクス送信
  ↓
CloudWatch Logs Insightsダッシュボードで可視化
```

### 重要ポイント

| 項目 | 内容 |
|---|---|
| データ形式 | **JSONL形式**でS3に格納（DynamoDBは不可） |
| データアクセス | 評価ジョブ実行ロールに**S3読み取り権限のIAMポリシー**が必要 |
| CORS | WebUI（ブラウザ）からS3オブジェクトを直接参照する場合に必要 |
| テキスト生成の評価指標 | **RWK（Real-World Knowledge）スコア**：事実の正確性を評価 |
| 要約の評価指標 | **BERTスコア**：意味的な類似度を評価（同義語・言い換えにも対応） |

### CORSとは

ブラウザがデフォルトで**異なるオリジンへのリクエストをブロック**する。S3側でCORS設定を入れることで、特定のWebアプリのオリジンからのアクセスを許可できる。

```
IAMポリシー = AWS（サーバー）レベルのアクセス制御
CORS設定    = ブラウザレベルのアクセス制御（別オリジンからの通信許可）
→ 両方必要（役割が違う）
```

### 評価指標の使い分け（引っかけ対策）

| 指標 | 用途 | AIPでの出題 |
|---|---|---|
| **RWK** | テキスト生成の事実正確性 | ✓ 正解 |
| **BERTスコア** | 要約の意味的類似度 | ✓ 正解 |
| BLEU | 機械翻訳評価（テキスト生成正確性には不向き） | ✗ 不正解選択肢 |
| ROUGE | 要約評価に使えるがBERTより粗い | ✗ 不正解選択肢 |

### 不正解パターン

| 選択肢 | 不正解の理由 |
|---|---|
| DynamoDB格納 | Bedrock評価ジョブはS3のJSONLのみ対応。DynamoDBから直接読めない |
| SageMaker Processing + カスタムコンテナ + OpenSearch | インフラ管理コストが大幅増。「運用負荷最小化」に反する |
| Step Functions + InvokeModel + Comprehend + QuickSight | Comprehendは品質スコアリングツールではない。QuickSightはCWより重い |

### 概要
Amazon Bedrock のネイティブ機能として、プロンプトテンプレートを一元管理する仕組み。コード変更なしにプロンプトを更新・バージョン管理できる。

### バリアント（Variant）とは

**共通のベーステンプレートから、条件に応じて異なるバージョンのプロンプトを定義する機能。**

```
共通テンプレート（全部門共通フォーマット）
  ├─ バリアント：営業部門用（親しみやすいトーン）
  ├─ バリアント：コンプライアンス部門用（厳格・法的トーン）
  └─ バリアント：経理部門用（数値重視・簡潔なトーン）
```

- 各バリアントはトーン・言い回し・制約条件などを個別に定義できる
- 部門IDなどのコンテキストを渡すことで、対応するバリアントが選択される
- テンプレート自体はBedrock側で管理されるため、コード変更不要で更新可能

### AIP-15の正解構成

**Bedrock プロンプト管理（バリアント） + Bedrock ガードレール**

| 要件 | 対応する機能 |
|---|---|
| 部門ごとに異なるトーン | プロンプト管理の**バリアント**で各部門向けを定義 |
| 統一フォーマット（要点・リスク評価・注意事項） | ベーステンプレートに統一フォーマットを定義 |
| 暴力的・差別的表現の排除 | ガードレールの**コンテンツカテゴリフィルター** |
| PIIが回答に含まれない | ガードレールの**機密情報フィルター** |
| 運用中に柔軟に変更できる | ガードレールのポリシーはコンソールから随時更新可能 |
| 運用チームの負担を最小化 | Bedrockネイティブ機能のみで完結（外部サービス不要） |

### ガードレールのフィルタータイプ整理

| フィルター | 用途 |
|---|---|
| **コンテンツカテゴリフィルター** | 暴力・ヘイト・性的表現などの有害カテゴリを強度設定でブロック |
| **単語フィルター** | 特定の禁止ワードリストでブロック |
| **機密情報フィルター（PIIフィルター）** | 氏名・電話番号・クレジットカード番号などのPIIをマスクまたはブロック |
| **トピック拒否ポリシー** | ビジネス上で扱いたくないトピック（例：競合他社の話題）を拒否 |
| **グラウンディングチェック** | ハルシネーション検出（ソース文書との一致度チェック） |

### 引っかけ選択肢との比較

| 選択肢 | 不正解の理由 |
|---|---|
| S3にJSONテンプレート + Lambda + Comprehend | 応答生成**後**にフィルタリング → PIIや有害表現が一度生成される。Comprehendは汎用NLPで専用フィルタではない |
| SageMaker部門別エンドポイント + Lambda + Macie | Macieはデータレイク・S3上のPII検出用。リアルタイムのLLM出力検査には向かない。インフラ管理コスト大 |
| プロンプト管理 + Step Functions + Comprehend併用 | Step Functionsは不要な複雑さを追加。ComprehendはBedrockガードレールと比べて機能が限定的 |

# Bedrock ガードレール / モニタリング

## ハルシネーション検出・監査構成

### 全体像

```
入力コンテキスト（ソース文書）
        ↓
  Bedrock ガードレール（文脈的グラウンディングチェック）
        ↓ グラウンディングスコアがしきい値未満 → ブロック or フラグ付け
  モデル呼び出し
        ↓
  出力ログ → S3（監査・事後分析）
        ↓
  CloudWatch メトリクス（InputTokenCount / OutputTokenCount）
        ↓
  CloudWatch 異常検出アラーム
```

---

## 各コンポーネント詳細

### 1. モデル呼び出しログ（S3保存）

- すべての入出力を監査・事後分析できる基盤
- **テキスト入出力の記録はオプション設定**で、明示的に有効化が必要
- 保存先は S3 または CloudWatch Logs

---

### 2. Bedrock ガードレール：文脈的グラウンディングチェック

ハルシネーション（根拠のない生成）をリアルタイムで検出・対処する機能。

- モデルへの入力コンテキスト（ソース文書）と生成された出力を比較し、出力がソースに根拠を持つかどうかを評価
- **グラウンディングスコア**（0〜1）を設定し、しきい値を下回ると応答をブロックまたはフラグ付け
- リクエスト処理パイプラインに組み込まれるためリアルタイムで機能する

#### ガードレールで設定できるポリシータイプ（組み合わせ可）

| ポリシー | 内容 |
|---|---|
| 文脈的グラウンディング | ハルシネーション検出 |
| コンテンツフィルタリング | 有害コンテンツのブロック |
| トピック拒否 | 特定トピックへの応答を拒否 |
| PII検出 | 個人情報の検出・マスク |
| 機密情報フィルター | PII等の検出・ブロック・マスク |

---

### 3. CloudWatch 異常検出アラーム（トークン使用量）

Bedrock は以下のメトリクスを CloudWatch に自動発行する：

- `InputTokenCount`
- `OutputTokenCount`

**異常検出アラーム**の特徴：
- 過去のメトリクスデータに機械学習アルゴリズムを適用
- 時間帯・曜日・トレンドを考慮した「正常な変動帯」を自動学習
- 変動帯から逸脱した場合にアラームが発火
- 静的しきい値では捉えられない**動的な異常パターン**も検出できる

#### 注意点

- 異常検出は**最低2週間分のメトリクスデータ**があると精度が向上
- 導入直後はモデルの学習が不十分なため誤検知が発生する可能性あり
- Bedrockの料金はトークン数ベースのため、トークン監視は**コスト管理にも直結**

## マルチモーダルデータ × RAG + ガードレール構成（AIP-10）

### 要件

- 異なる形式のデータ（映像・音響・テキスト）を活用した自然言語QA
- 観測事実に基づかない推測的な回答を防止
- 操作履歴を監査目的で記録

### 正解構成

```
映像 / 音響 / 観察記録
  ↓
SageMaker Processing（データ前処理・正規化・構造化）
  ↓
Bedrock ナレッジベース（RAG）
  ↓ ベクトル検索 → 関連情報をコンテキストに挿入
Bedrock FM（回答生成）
  ↓
Bedrock ガードレール（コンテキストグラウンディングチェック）

監査：CloudTrail
設定管理：AWS AppConfig（プロンプトテンプレート）
```

### 各コンポーネントの役割

| コンポーネント | 役割 |
|---|---|
| **SageMaker Processing** | カスタムスクリプト/コンテナで大規模データ前処理を実行するマネージドジョブ環境 |
| **Bedrock ナレッジベース（RAG）** | 観測データをベクトル化し、クエリに意味的に近い情報を検索してコンテキストとしてFMに渡す |
| **ガードレール（コンテキストグラウンディングチェック）** | 回答が提供されたコンテキスト（検索結果）に基づいているかを検証。根拠のない推論・憶測をリアルタイムで抑止 |
| **AWS AppConfig** | アプリ設定の管理・デプロイサービス。プロンプトテンプレートのバージョン管理・段階的ロールアウトに使う |
| **CloudTrail** | `InvokeModel` / `Retrieve` / `RetrieveAndGenerate` などのAPI呼び出しを記録。誰がいつどのモデルにリクエストを送ったかを追跡 |

### ハルシネーション抑制の二重構造

RAGだけでは完全に防げないため、ガードレールのグラウンディングチェックを併用するのが推奨。

### 試験頻出の引っかけ：SageMaker Model Monitor との混同

| サービス | 用途 |
|---|---|
| **SageMaker Model Monitor** | SageMakerにデプロイされたMLモデルのデータドリフト・品質劣化を**統計的に監視**する |
| **Bedrock ガードレール** | 生成AIの回答が推測的かどうかを**リアルタイムに判定・制御**する |

→ 「生成AIの出力が推測的かどうかを制御したい」ならガードレール。Model Monitor は生成AIの出力フィルタリングには使えない。

### CloudTrail vs Bedrock モデル呼び出しログの使い分け

| | CloudTrail | モデル呼び出しログ |
|---|---|---|
| 記録内容 | 誰がいつどのAPIを呼んだか（メタデータ） | 入力プロンプトと出力レスポンスの全文 |
| 用途 | 操作監査・ガバナンス | 詳細な監査・デバッグ |
| 保存先 | S3 / CloudWatch Logs | S3 / CloudWatch Logs |
