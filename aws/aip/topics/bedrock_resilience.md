# Bedrock 耐障害性・ルーティング・キャッシュ

## クロスリージョン推論（Cross-region inference）

### 概要

**同一地理的エリア内の複数リージョンに、同じモデルへのリクエストを自動分散する機能。**

```
単一リージョンの問題：
  us-east-1 だけ使用
  → ピーク時にスロットリング（Too many requests）発生

クロスリージョン推論：
  us-east-1 → 上限に近づくと...
  us-west-2 にも自動的にルーティング
  us-east-2 にも自動的にルーティング
  → 米国内の複数リージョンのクォータを束ねて使える
  → 同じモデル（Claude等）を使用するため応答品質は変わらない
```

**ポイント：「地理的エリア内」での分散**
- `us.*` プレフィックス → 米国内複数リージョン
- `eu.*` プレフィックス → EU内複数リージョン
- `ap.*` プレフィックス → アジアパシフィック内複数リージョン
- 地域をまたいだルーティングはしない（データレジデンシー考慮）

### 特徴

| 項目 | 内容 |
|---|---|
| 設定方法 | 推論プロファイルを作成するだけ。アプリコードの変更は最小限 |
| 応答品質 | 全リージョンで**同一のFM**を使用するためリージョン間で差異なし |
| 料金体系 | オンデマンド推論のまま維持（従量課金）。リクエストがなければ課金なし |
| スループット | 単一リージョンのクォータ上限を実質的に拡大できる |

### 監視メトリクス（CloudWatch）

モデル呼び出しログを有効化して以下を監視する：

| メトリクス | 用途 |
|---|---|
| `Invocations` | 呼び出し回数 |
| `InputTokenCount` | 入力トークン消費量 |
| `OutputTokenCount` | 出力トークン消費量 |
| `InvocationThrottles` | スロットリング発生状況 |

→ この情報をもとにリージョン配分の最適化やクォータ引き上げリクエストの判断が可能

### 注意点

- **データレジデンシー**：データが別リージョンに送信されるため、データ保管場所に関する法的要件がある場合は対象リージョンの制限に注意
- **クォータ確認・引き上げ**：リージョンごとのデフォルトクォータは Service Quotas コンソールから確認可能

### 推論プロファイル（Inference Profile）とは

モデルIDの代わりに指定することで、リクエストを複数リージョンに自動分散させるルーティング設定のラッパー。

**命名規則**
```
eu.amazon.nova-pro-v1:0
│    │       │
│    │       └─ モデル名・バージョン
│    └─────── 提供元（amazon / anthropic など）
└──────────── 地理プレフィックス（分散先リージョン群）
```

| プレフィックス | 分散先 |
|---|---|
| `us.*` | 米国内の複数リージョン |
| `eu.*` | EU内（eu-west-1、eu-west-3、eu-north-1、eu-central-1など） |
| `ap.*` | アジアパシフィック内の複数リージョン |

**通常のモデルIDとの違い**
```
# 通常（単一リージョン固定）
model_id = "amazon.nova-pro-v1:0"
→ 呼び出したリージョンのみで処理

# 推論プロファイル（マルチリージョン分散）
model_id = "eu.amazon.nova-pro-v1:0"
→ EU内の複数リージョンに自動ルーティング
```

### システム定義 vs アプリケーション推論プロファイル

| | システム定義推論プロファイル | アプリケーション推論プロファイル |
|---|---|---|
| 作成者 | AWSが事前設定 | ユーザーが作成 |
| ルーティング | AWSが自動決定（ラウンドロビン等） | プライマリ/セカンダリを明示指定 |
| フェイルオーバー | あり（キャパシティ不足時） | あり（より細かく制御可能） |
| 用途 | 通常のクロスリージョン推論 | 可用性・パフォーマンス最適化 |

```
★★★ キャパシティ不足時のフェイルオーバー（主目的）
★★★ 可用性の向上
★★  レイテンシ改善（おまけ程度）
```

→ アプリケーション推論プロファイル（プライマリ/セカンダリ設定）が正解
→ Route 53・Global Accelerator は追加コスト発生 → 不正解

- ラウンドロビン = 均等分散（パフォーマンス無視）→ 不正解パターン
- 正解 = レイテンシ・可用性に基づくスマートルーティング

### アプリケーション推論プロファイルの別用途：マルチテナントのコスト追跡

可用性・フェイルオーバー以外に、**テナント（顧客/診療所/部署等）ごとにBedrockの利用コストを分解して追跡する**という全く別の使い道がある。

```
テナントIDごとに推論プロファイルを作成
  → S3キープレフィックス等のテナント識別子に基づいて、
    要約用Lambdaが対応するプロファイルを選択して呼び出す
  → Bedrock呼び出しがプロファイル単位でタグ付けされる
  → Cost Explorer / 請求側でテナント別のコストを分解できる
```

**判断ルール**：「複数テナント/顧客/拠点ごとのFM利用コストを追跡したい」→ **アプリケーション推論プロファイル**（テナントIDごとに作成）。追加のインフラ（DB・ダッシュボード等）を用意する必要がなく、設定ロジックも最小限で済む。

**混同注意（EventBridge/CloudWatchのメトリクスでは追跡不可）**：EventBridgeのCloudWatchメトリクス（`Invocations`/`FailedInvocations`等）は**イベント配信の成否・レイテンシーのみ**を計測し、ターゲット（Step Functions等）が起動した後のBedrock呼び出し回数・トークン数・コストは一切追跡しない。EventBridgeはターゲット呼び出し後に何が起きたかを知らない。

**混同注意（S3 Storage Lensでは追跡不可）**：S3 Storage Lensはオブジェクト数・サイズ・リクエストアクティビティなど**ストレージの使用状況メトリクスのみ**。Bedrock/Lambdaの推論コストは対象外で、テナント別コスト追跡の要件を満たさない。

### BedrockがEventBridgeに送信するイベントの範囲

BedrockがEventBridgeに送信するイベントは**ジョブ単位の状態変化のみ**（`Model Customization Job State Change`、`Batch Inference Job State Change`、Bedrock Data Automationジョブの状態変化）。オンデマンドのリアルタイム推論（`InvokeModel`の同期呼び出し）1回ごとにイベントは発生しない。

### プロビジョンドスループットの購入オプション

| オプション | 特徴 |
|---|---|
| **コミットメントなし（時間課金）** | 柔軟だが単価が最も高い |
| **1ヶ月コミットメント** | 中程度の割引 |
| **6ヶ月コミットメント** | 最も割引率が高い |

その他のポイント：
- ファインチューニング済みのカスタムモデルにも適用可能
- Bedrockはフルマネージドなのでインフラ管理（GPU・パッチ等）は不要
- **ネイティブなオートスケーリング機能は現時点で非対応** → ピーク負荷に合わせた事前のキャパシティ計画が重要
- コスト管理：時間単位の固定料金のため予算が予測しやすい

CloudWatchで監視できる主なメトリクス：
- `InvocationLatency`：推論の応答時間
- `ThrottledCount`：スロットリング発生回数

### 比較：プロビジョニングされたスループット vs クロスリージョン推論

| 項目 | Provisioned Throughput | Cross-region inference |
|---|---|---|
| キャパシティ | **専用キャパシティを確保（保証）** | 複数リージョンのオンデマンドを束ねる（ベストエフォート） |
| 料金 | 1時間単位 or 1〜6ヶ月の固定コミットメント | 従量課金のまま（リクエストなければ課金なし） |
| スロットリング解消 | ◯ **確実に防げる** | △ 全リージョン同時逼迫時は防げない |
| 向いているケース | ピーク負荷が予測可能・厳格なレイテンシ要件 | トラフィックが変動・非ピーク時のコストを抑えたい |
| 非ピーク時のコスト | **課金継続** | **課金なし** |
| 注意 | **クロスリージョン推論とは併用不可** | Provisionedとは併用不可 |

**使い分けの判断基準：**
- ピーク負荷が予測可能で厳格なSLAがある → **Provisioned**（専用キャパシティで保証）
- トラフィックが時間帯で変動・固定コストを避けたい → **Cross-region**（ベストエフォートで分散）
- Cross-regionはあくまで共有キャパシティの利用効率を高めるもの。ピーク保証はできない

**エクスポネンシャルバックオフ（AWSベストプラクティス）：**
- `ThrottlingException` などの一時的なエラーに対して待機時間を段階的に増やしながら再試行
- AWS SDKにデフォルトで組み込まれている
- Provisionedと組み合わせることで万が一の一時的エラーにも対応できる

## モデル更新の自動カナリアデプロイ

### 構成

**Provisioned Throughput（新旧） + EventBridge + Step Functions + Lambda + CloudWatch**

```
モデル更新イベント
  ↓ 検知
EventBridge
  ↓ 起動
Step Functions ステートマシン
  ├─ Task：Lambdaでリクエスト配分比率を変更
  ├─ Wait：一定時間待機（メトリクス蓄積）
  ├─ Task：LambdaでCloudWatchメトリクス取得・判定
  └─ Choice：
       ├─ 正常 → 比率引き上げ（次の段階へ）
       └─ 異常 → 旧モデルへ自動切り戻し
```

### ポイント

- Provisioned Throughputで新旧モデルそれぞれに専用キャパシティを確保
- 新旧モデルそれぞれに固有のARNが割り当てられるため、API層での振り分け先の切り替えが容易
- EventBridgeがモデル更新イベントを検知してステートマシンを自動起動 → エンドツーエンドで自動化

### コントロールプレーン（Step Functions）とデータプレーン（実リクエスト）の分離（盲点になりやすい）

Step Functionsは**リクエスト毎に割り込む場所（データプレーン）にはいない**。決済APIのようなリアルタイム処理で、1リクエストごとにStep Functions実行を挟むのは低レイテンシー要件に反するため非現実的。実際の役割分担：

```
① コントロールプレーン（Step Functions + Lambda）
   ワークフロー実行の状態として trafficPercentage（例：10→30→60→100）を持ち回す
   Task状態のLambdaが、この比率を Parameter Store / DynamoDB / AppConfig 等の
   外部ストアに書き込む（ワークフロー自体が実リクエストを処理するわけではない）
       ↓
② データプレーン（実際の決済リクエストを処理するアプリケーション層）
   リクエストが来るたびに、①で書き込まれた比率を読み取り、
   重み付きランダム判定（例：random() < 比率 → 新モデルARN、それ以外 → 旧モデルARN）で
   Bedrock InvokeModel の呼び出し先モデルARNを決定する
```

→ Step Functionsのステートマシンが持つ「比率」は**あくまでワークフロー内部の制御変数**であり、それ自体がネットワークレベルでトラフィックを物理的に振り分けているわけではない。振り分けの実行は別レイヤー（アプリケーションのルーティングロジック）が担う

### Step Functions のワークフロータイプの選択

| タイプ | 最大実行時間 | カナリアデプロイへの適性 |
|---|---|---|
| **Standard** | 最大1年 | ◯ 数時間〜数日にわたるデプロイに対応 |
| Express | 最大5分 | ✕ 不向き |

### CloudWatch メトリクス取得

- Lambda から `GetMetricData API` を呼び出すことで、指定期間のメトリクスをプログラム的に取得
- レイテンシー・エラー率・トークン使用量などをしきい値と比較して判定ロジックを実装

## デプロイ戦略の比較（Domain 5 Task 5.1）

モデル更新を安全にデプロイ・検証する戦略。問われ方で使い分ける。

| 戦略 | 仕組み | ユーザー影響 | 特徴・使いどころ |
|---|---|---|---|
| **シャドウデプロイ** | 本番トラフィックの**コピー**を新モデルにも流すが、**結果はユーザーに返さず記録のみ** | **ゼロ**（裏で評価） | 本番負荷で新モデルを検証したいが一切リスクを取りたくない。新旧比較に最適 |
| **カナリアデプロイ** | トラフィックを**5%→25%→100%** と段階的に新モデルへ | 一部ユーザーに限定 | 段階展開でリスク最小化。異常検知で自動切り戻し（上記カナリアデプロイ参照） |
| **段階的公開（Progressive rollout）** | カナリアと同義系。トラフィックを徐々に増やす | 段階的に拡大 | カナリアの一般名 |
| **ブルー/グリーン** | 新旧2環境を用意し**一気に全切替**。問題あれば即旧環境へ戻す | 切替時に全員 | 安全な即時切替＋即時ロールバック。ダウンタイム最小 |

```
シャドウ：    本番 ─┬→ 旧（ユーザーに返す）
                    └→ 新（記録のみ・返さない）
カナリア：    5% → 25% → 100% と段階的に新へ
ブルー/グリーン： Blue(旧稼働) / Green(新待機) → 検証OKで一括切替（問題ならBlueへ即戻し）
```

- **フォールバック（自動ロールバック）**：問題検出時に以前のバージョンへ自動で戻す仕組み（カナリアの自動切り戻し・ブルー/グリーンの即時切替がこれに該当）

## プロンプトキャッシング（AIP-33・公式確認済みで訂正あり）

### 概要

```
Bedrock ネイティブの機能。
リクエスト間で共通するプロンプトのプレフィックス部分（システムプロンプト・ツール定義・
長いドキュメント等）を「キャッシュチェックポイント」として明示的に指定し、
後続リクエストでの再計算をスキップする。

共通部分 → 初回リクエストでキャッシュ書き込み（cache write）
        → 次回以降、TTL内に同一プレフィックスが来ればキャッシュ読み取り（cache read）
        → キャッシュ読み取り分は標準入力レートより低いレートで課金
```

⚠️ **訂正（過去の記述の誤り）**：
- ~~「レイテンシー最適化推論(`latency optimized`)の有効化が前提」~~ → **誤り**。プロンプトキャッシングは独立した機能で、レイテンシー最適化推論を有効にする必要はない
- ~~「Bedrockが自動管理。開発者が明示的に作成する必要なし」~~ → **誤り（Converse/InvokeModel APIでは不正確）**。**Converse APIの`system`/`messages`/`tools`フィールドに`cachePoint`ブロックを明示的に配置する必要がある**（InvokeModelでは`cache_control`）。自動なのはBedrockコンソールのプレイグラウンドや、Amazon Novaの自動プロンプトキャッシュ、Claude向けの「簡略化キャッシュ管理（末尾に1つブレークポイントを置くだけで直近約20コンテンツブロックを自動探索）」など**一部の限定的なケースのみ**

### 適用条件（公式確認済み）

| 条件 | 内容 |
|---|---|
| チェックポイント配置 | Converse APIは`system`/`messages`/`tools`の各フィールドに`cachePoint`を配置（**1リクエスト最大4箇所**）。処理順は`tools`→`system`→`messages`で、前段の内容変更は後段のキャッシュも無効化する |
| 最小トークン数（チェックポイントごと） | モデル依存。例：Claude 3.7 Sonnet・Claude Opus 4 = 1,024トークン／Claude Opus 4.5・Haiku 4.5・Sonnet 4.5 = 4,096トークン |
| TTL | 多くのモデルで**5分**（キャッシュヒットのたびにリセット）。Claude Opus 4.5・Haiku 4.5・Sonnet 4.5は**1時間**オプションも選択可（`cachePoint`に`"ttl":"1h"`を指定） |
| 対応API | Converse/ConverseStream・InvokeModel/InvokeModelWithResponseStream。**Prompt Managementでもプロンプト作成・編集時にキャッシュ有効化を選択可能**（ただしそれはこの機能自体を使うという意味） |
| 非対応 | **バッチ推論APIでは利用不可**（オンデマンド推論エンドポイントのみ対応） |
| 料金 | キャッシュ読み取り＝標準入力より低レート／キャッシュ書き込み＝モデルによっては標準入力より**高い**レート。高頻度リクエストではヒット率が上がり書き込みコストを上回る割引効果が出る |
| レスポンスの確認項目 | `cacheReadInputTokens` / `cacheWriteInputTokens`（Converse API）。合計入力トークン＝`inputTokens + cacheReadInputTokens + cacheWriteInputTokens` |

### プロンプトキャッシュ vs Prompt Management（混同注意・誤答しやすい）

| | **プロンプトキャッシュ** | **Prompt Management** |
|---|---|---|
| 目的 | **入力トークンの再計算を省いてコスト・レイテンシーを削減** | プロンプトの**保存・バージョン管理・バリアント比較・承認ワークフロー** |
| 何を解決するか | 「同じ8,000トークンのプレフィックスを毎回課金・再計算している」問題 | 「プロンプトの変更履歴を追跡したい・環境間で一貫させたい」問題 |
| 入力トークン課金への影響 | **削減する**（cachePoint配置で直接効く） | **削減しない**（リソース識別子で参照する構成に変えても、推論時は毎回全トークンを処理する。プロンプトキャッシュ自体を明示的に有効化しない限り効果なし） |

→ 「リクエスト間で変わらない静的なプロンプトの**コスト削減**」が問われたら**プロンプトキャッシュ**、「プロンプトの**バージョン管理・ガバナンス**」が問われたら**Prompt Management**。名前が似ているだけで解決する課題が全く違う

## インテリジェントプロンプトルーティング（AIP-24）

### 機能概要

**Amazon Bedrock のインテリジェントプロンプトルーティング**は、受信したプロンプトの複雑度を自動判定し、最適なモデルへ自動的に振り分けるマネージド機能（2024年末に追加）。

**混同注意**：これは「複雑度に基づく自動振り分け」という**Bedrock内蔵の機能**。一方「タスクタイプ（要約/分類/翻訳）や顧客設定に基づいて動的にFMへルーティングし、ランタイムで再デプロイなしにルーティングロジックを更新したい」という**業務ロジック駆動のルーティング**は組み込み機能では対応できず、**AWS AppConfig（動的設定）+ Step Functions（オーケストレーション+サーキットブレーカー）を自前で組む**のが正解構成になる（AIP公式模試で確認済み）。

```
ユーザーからのメッセージ
  ↓ Bedrock がプロンプトの複雑度を自動判定
  ├─ 単純（定型的な問い合わせ）→ 小型・高速・低コストモデルへ
  └─ 複雑（高度な推論が必要） → 大型・高精度モデルへ
  ↓
応答を返す
```

### CRI vs プロンプトルーティング 比較（公式模擬Q1）

スロットリング（Too many requests）の問題で両方が選択肢に出てくる場面の比較。

| | CRI（クロスリージョン推論） | Intelligent Prompt Routing |
|---|---|---|
| **何を分散するか** | **同一モデル**を複数リージョンへ | **同一ファミリーの異なるモデル**へ |
| **使うモデル** | 完全に同じ（Claude Sonnet → Claude Sonnet） | 変わる（Claude Haiku / Sonnet / Opus） |
| **解決する問題** | スロットリング・スループット不足 | コスト最適化・応答速度 |
| **「同じFMを使う」要件** | ✅ 満たす | ❌ 違反する |

```
問題文の見分けポイント：
  「同じ FM を使用する必要がある」→ CRI が正解
  「コストを抑えて品質を保ちたい」「難易度に応じて振り分けたい」→ Prompt Routing が正解
```

## AWS サービスクォータ（Service Quotas）とは

### 概要

AWS の各サービスには**利用上限値（クォータ）**が設定されている。デフォルト値はサービスごとに決まっており、必要に応じて**AWS Service Quotas コンソール**から引き上げをリクエストできる。

### Bedrock のクォータ例

| クォータ | 内容 |
|---|---|
| オンデマンド同時リクエスト数 | 同時に処理できる InvokeModel のリクエスト数上限 |
| トークン/分（TPM） | 1分間に処理できるトークンの合計数 |
| リクエスト/分（RPM） | 1分間に送信できるリクエスト数 |

### クォータ超過時のエラー

クォータを超えるとリクエストが **`ThrottlingException`** で拒否される。  
→ リトライ（指数バックオフ）で対処、または引き上げをリクエスト。

## Bedrock API 障害耐性パターン（AIP-47）

### 問題のシナリオ
ストリーミング API のピーク時スロットリング・5xx エラー・ストリーム中断・トークン超過が同時発生。

---

### サンダリングハード問題（Thundering Herd Problem）

```
固定間隔リトライの場合：
  障害発生 → 100クライアントが全員 t+2秒後にリトライ
  → 同時刻に100リクエストが集中 → さらに負荷増大 → 障害悪化

ジッター付き指数バックオフの場合：
  障害発生 → 各クライアントのリトライタイミングがランダムに分散
  → 負荷が時間的に分散 → サービスが回復しやすくなる
```

---

### ジッター付き指数バックオフ

```
リトライ間隔の計算式：
  待機時間 = min(cap, base * 2^n) + random(0, jitter)

  n     = リトライ回数
  base  = 基本待機時間（例: 0.5秒）
  cap   = 最大待機時間（例: 30秒）
  jitter= ランダム揺らぎ（タイミングを分散させる）

例（base=1秒, cap=30秒）：
  1回目: 1秒 + ランダム
  2回目: 2秒 + ランダム
  3回目: 4秒 + ランダム
  4回目: 8秒 + ランダム  ← 指数的に増加
  5回目: 16秒 + ランダム
```

---

### サーキットブレーカーパターン（Circuit Breaker Pattern）

```
3つの状態：

Closed（正常）
  → エラー率が閾値未満：通常どおりリクエスト送信
  → エラー率が閾値超過 → Open に遷移

Open（遮断）
  → リクエスト送信を一時停止（即座に失敗を返す）
  → 一定時間後 → Half-Open に遷移

Half-Open（試験中）
  → 少量のリクエストを試行
  → 成功 → Closed に戻る
  → 失敗 → Open に戻る
```

---

### ストリーミング中断時のチャンクバッファリング

```
❌ ダメな方法：
  中断を検知 → 受信済みチャンクを破棄 → 最初からリクエスト再送
  → 無駄なAPI呼び出し、レイテンシー増大

✅ 正しい方法：
  受信したチャンクをインメモリバッファに蓄積
  → 各チャンクのシーケンス番号・受信時刻を記録
  → 中断検知 → 最終受信位置をコンテキストに含めて新規ストリーム開始
  → 重複チャンクはスキップ
  → ユーザーには途切れのない応答として表示
```

| アプローチ | リトライ | 過負荷停止 | 中断復旧 | トークン管理 |

## 地理的クロスリージョン推論プロファイル vs 通常（AIP-65）

### 2種類のクロスリージョン推論プロファイルの違い

| 項目 | 通常のクロスリージョン推論プロファイル | 地理的クロスリージョン推論プロファイル |
|---|---|---|
| **目的** | スループット確保（需要ピーク対応） | データレジデンシー準拠 |
| **ルーティング先** | グローバル任意リージョン（制限なし） | 特定地理的境界内のみ（US/EU/APAC等） |
| **プロンプトデータの越境** | 発生しうる | 境界内に封じ込め保証 |
| **主なユースケース** | 高可用性・低レイテンシー確保 | GDPR・医療規制・データ主権要件 |

### RAGにおける盲点（AIP-65 のポイント）
```
「S3やベクトルDBはEU内に置いてある → データ保存はOK」
  ↑ だけでは不十分！

RAGパイプラインの流れ:
  検索クエリ → ベクトルDB → 検索結果（患者データ等）→ プロンプトに埋め込む → FMで推論

= プロンプト自体に機密データが含まれる
= 推論が行われるリージョンも規制対象

→ 地理的クロスリージョン推論プロファイルで推論先リージョンを地理的に制限する必要がある
```

### 補足：Amazon Kendra の性質
- **MLベースのセマンティック＋キーワードハイブリッド**検索サービス（純粋なキーワード検索ではない）
- 自然言語理解・意図解釈を持つエンタープライズ検索
- RAGパイプラインとしてのカスタマイズ性はOpenSearch Serverlessより低い
- データレジデンシー要件（推論場所の制御）を解決するサービスではない

## モデルカスケード

**「簡単なクエリは軽いモデル、難しいクエリは重いモデルに順番に流す」仕組み。**

```
クエリ
  ↓ Nova Micro（安い・速い）で試す
  十分な品質？
    ├─ Yes → そのまま返す（コスト最小）
    └─ No  → Nova Lite へ
                十分な品質？
                  ├─ Yes → 返す
                  └─ No  → Nova Pro へ
```

### 実装パターン3つ

| パターン | 内容 | 向いている場面 |
|---|---|---|
| **ルールベース** | Lambdaでキーワード・長さで判断 | シンプル・制御しやすい |
| **Intelligent Prompt Routing** | AWSのAIが自動判断 | マネージド・コード不要 |
| **分類モデル** | 軽量モデルで複雑さを分類してから振り分け | 高精度なルーティング |

### Bedrock Intelligent Prompt Routing

**AWSのAIがクエリを分析してモデルを自動選択する機能。**
- コスト・品質のバランスを自動最適化
- コード不要
- 試験での「インテリジェントルーティング」の正解パターン
- **重要な制約**：ルーティングできるのは**同一モデルファミリー内のみ**（例：Anthropic同士、Meta同士）。**異なるファミリー間（Llama⇔Claude等）はサポート対象外** → 「小規模Llama⇔大規模Claude」のような要件にはIntelligent Prompt Routingは使えない（罠）。1ルーターにつき同一ファミリー内で**厳密に2モデル**（fallbackモデル＋比較対象モデル）を指定する仕様

### カスケード vs Bedrock Agent

```
モデルカスケード：
  どのモデルに投げるか判断するだけ（1ステップ）
  → Lambda分岐で十分

Bedrock Agent：
  複数ツールを自律的に使いながら複数ステップを実行
  → 複雑なタスクのオーケストレーション向け
  → カスケードには過剰・レイテンシ増
```

## A/Bテスト（モデル評価）（Task 2.4）

```
通常トラフィック 90% → 現行モデル（Claude Sonnet）
残り          10% → 候補モデル（Claude Opus）

→ 一定期間後にレイテンシー・品質スコアを比較
→ 有意に改善していれば全トラフィックを切り替え
```

- AppConfig / Parameter Store の機能フラグでコードデプロイ不要で比率変更可能
- 結果は DynamoDB / Timestream に記録して統計分析

### CloudWatch Evidently

GenAI アプリケーションの A/B テストを統計的に管理するマネージドサービス。

**主な機能：**

| 機能 | 内容 |
|---|---|
| **トラフィック分割** | 変数（プロンプトバリアントなど）ごとにユーザーを自動振り分け |
| **統計的有意性の判定** | 収集したメトリクスをベイズ統計または頻度統計で評価し、有意差を自動判定 |
| **カスタムメトリクス** | タスク完了率・ユーザー満足度スコアなど任意の業務指標を評価対象に設定可能 |
| **セーフガード** | 負のインパクトを検出した場合にトラフィックを自動的に元のバリアントに戻す |

**GenAI での利用パターン：**

```
プロンプトバリアント A（現行）
プロンプトバリアント B（候補）
         ↓
  CloudWatch Evidently
  ・トラフィックを 50/50 で分割
  ・タスク完了率・レイテンシーを収集
  ・統計的有意性を自動判定
         ↓
  有意に改善 → バリアント B に全切替
  有意差なし → バリアント A を維持
```

- 評価メトリクスの選択が重要：プロンプトの**有効性**は「タスク完了率」で測る（レイテンシーやトークン数は有効性指標でない）

## セマンティックキャッシュ（アプリ層）（Task 4.1）

Bedrockのプロンプトキャッシュ（KVキャッシュ）とは別に、**アプリ層で入力→回答のペアをキャッシュ**する仕組み。

### キャッシュ種別の比較

| 種別 | 判定方法 | FM呼ぶか | 実装場所 |
|---|---|---|---|
| **決定論的ハッシュ** | SHA-256で完全一致 | 呼ばない | CloudFront / ElastiCache |
| **セマンティックキャッシュ** | embeddingの類似度 | 呼ばない | OpenSearch / ElastiCache |
| **プロンプトキャッシュ（KV）** | 固定プレフィックス | 呼ぶ（計算省略） | Bedrockネイティブ |

### セマンティックキャッシュ フロー

```
【MISS時（初回）】
クエリ → embedding変換 → OpenSearchでk-NN検索 → 類似なし
  → Bedrock FM呼び出し → 回答生成
  → (クエリのembedding, 回答テキスト) をOpenSearchに保存

【HIT時（2回目以降）】
クエリ → embedding変換 → k-NN検索 → 類似度 > 閾値（例: 0.95）
  → DBから回答テキストを返却（FMを呼ばない）
```

### 結果フィンガープリンティング

**出力（回答テキスト）をSHA-256でハッシュ化し、保存フェーズで重複エントリを統合する**手法。

- **HIT判定はしない**（セマンティックキャッシュが担う）
- **主目的：重複排除**。別の入力から同じ回答が生成された場合にDBを統合してストレージ節約

```
FM回答テキスト → SHA-256 → 出力フィンガープリント
  └─ 同じハッシュがDBに存在？
      YES → 重複統合（例：「EC2とは？」と「EC2を説明して」が同一回答）
      NO  → 新エントリとして保存（入力embedding + 回答テキスト + 出力hash）
```

### 階層型キャッシュアーキテクチャ

```
[リクエスト]
      ↓
L1: CloudFront エッジ（決定論的ハッシュ・完全一致のみ・超低レイテンシ）
      ↓ MISS
L2: ElastiCache（インメモリ・決定論的ハッシュ・ms単位）
      ↓ MISS
L3: OpenSearch（セマンティック類似検索・数十ms）
      ↓ MISS
Bedrock FM呼び出し → 結果をL1〜L3に書き戻す
```

### キャッシュ無効化ポリシー

| 戦略 | 内容 | 適するケース |
|---|---|---|
| **TTL** | 一定時間後に自動削除 | FAQ（24h）・株価（60秒）など鮮度要件に応じて設定 |
| **イベントベース** | モデル更新・ドキュメント更新時に関連エントリを明示削除 | モデル切替時の全キャッシュ無効化 |
| **LRU** | アクセスが少ないエントリから自動削除 | ElastiCacheのデフォルト動作 |

### CloudWatchキャッシュ分析

```python
# Lambdaでカスタムメトリクスを送信
cloudwatch.put_metric_data(
    Namespace='CacheMetrics',
    MetricData=[
        {'MetricName': 'CacheHit',  'Value': 1},  # HIT時
        {'MetricName': 'CacheMiss', 'Value': 1},  # MISS時
    ]
)
# ヒット率 = CacheHit / (CacheHit + CacheMiss) × 100
# ヒット率低下 → 閾値・チャンキング戦略を見直す
```

## コスト最適化パターン（Task 4.1）

→ コスト配分タグ・AWS Budgets・AWS Cost Anomaly Detectionは [cost_management.md](cost_management.md) に分離済み

### モデルフォールバックメカニズム

カスケード（品質不足で上位モデルへ）とは異なり、**エラー・タイムアウト時に下位モデルへ切り替えてサービス継続**するのが主目的。

```
Fable 5 呼び出し
  ├─ 成功 → そのまま返す
  ├─ ThrottlingError / Timeout
  │     → Sonnet にフォールバック（サービス継続）
  └─ Sonnet も失敗 → Haiku にフォールバック（最低限の応答を保証）
```

可用性優先の設計。コスト削減は副次的効果。

### 停止条件（暴走防止）

GenAIプロセス（エージェントループ等）が無限実行してコストが膨らむのを防ぐ。

| 実装場所 | 設定 |
|---|---|
| **Bedrockエージェント** | `maxSessionDuration`・最大オーケストレーションステップ数 |
| **Step Functions** | `TimeoutSeconds`（ワークフロー全体）・`HeartbeatSeconds`（各State） |
| **CloudWatch + Lambda** | InvocationCountが急増→アラーム→Lambdaで強制停止 |

### CloudFormation + キャパシティプランニング

インフラをコード化して環境（dev/stg/prod）間で一貫した構成を再現する。

```yaml
Resources:
  BedrockProvisionedThroughput:
    Type: AWS::Bedrock::ProvisionedModelThroughput
    Properties:
      ModelUnits: 5      # 予測トラフィックから算出
  LambdaFunction:
    Type: AWS::Lambda::Function
    Properties:
      ReservedConcurrentExecutions: 100  # Bedrock過剰リクエスト防止
```

### Step Functions プロンプト最適化ワークフロー

本番ではなく**チューニングフェーズで**複数プロンプト変種を自動テストする用途。

```
Step Functions ステートマシン（チューニング専用）：
  ① 変種A/B/Cを並列でBedrock呼び出し
  ② トークン数・品質スコアを集計（Lambda）
  ③ コスト効率 = 品質 / トークン数 で最良変種を選定
  ④ Parameter Storeに書き込む → 本番Lambdaはここから読む
```

## 事前計算とキャッシュ戦略（Task 4.2）

「予測可能なクエリ」を事前にFMで処理し、低レイテンシーストアに保存しておくパターン。

```
【事前計算フロー】
EventBridge（夜間スケジュール）→ Lambda → Bedrock FM
  → 結果をDynamoDB + ElastiCacheに保存

【本番リクエスト】
ユーザー → ElastiCache（HIT→即返却）
           → DynamoDB（HIT→返却 + Redisに書き戻し）
           → FM呼び出し（MISS時のみ）
```

### レイテンシー比較

| サービス | レイテンシー | 特徴 |
|---|---|---|
| **DynamoDB + DAX** | **マイクロ秒** | DynamoDB専用インメモリキャッシュ。DynamoDB API互換 |
| **ElastiCache（Redis）** | ミリ秒未満 | 汎用インメモリ。セマンティックキャッシュにも使える |
| **DynamoDB単体** | 1桁ms | 永続KVストア |
| **S3** | 数十ms | オブジェクトストレージ |

**試験の判断軸：**
- 「最小レイテンシー」+「DynamoDB」→ **DAX**（マイクロ秒）
- 「汎用の超高速キャッシュ」→ **ElastiCache（Redis）**
- RedshiftはDWH（分析用）≠ Redis（キャッシュ）：読み間違い注意

## オーケストレーション vs スケーリング（Task 4.2）

```
【オーケストレーション】= 指揮者（処理の流れ・順序・分岐を制御）
Step Functions:
  A → [B, C, D]（並列）→ E（マージ）
  エラー時はFにフォールバック
  → 「何を・どの順で・どう組み合わせるか」を管理

【スケーリング】= 楽団員の人数調整（処理能力の増減）
SQS + Lambda:
  リクエスト増 → Lambdaを自動増加
  → 「同じ処理をどれだけこなすか」を制御
```

**試験の判断軸：**
- 「複雑なワークフロー」「並列＋マージ」「分岐・エラーハンドリング」→ **Step Functions**
- 「大量リクエストの流量制御」「キューイング」「非同期処理」→ **SQS**

---
