# Amazon Bedrock Agents

## 概要

基盤モデル（FM）を中核として、ユーザーの自然言語リクエストを理解し、マルチステップのタスクを自律的に実行するフルマネージドサービス。
リクエストを分析し、ナレッジベース検索やアクショングループ呼び出しを自動的にオーケストレーションする。

---

## オーケストレーション戦略

- デフォルトは **ReAct（Reason and Action）**：FMのツールユースパターンを使って計画→実行を繰り返す
- Lambdaで独自ロジックに差し替えることも可能（Custom Orchestration）

---

## 構成要素

| 要素 | 説明 |
|---|---|
| **Foundation Model** | 推論・計画のエンジン |
| **Instructions** | エージェントの役割・目的を定義するプロンプト |
| **Action Groups** | エージェントが実行できるアクションの定義（OpenAPIスキーマ + Lambda） |
| **Knowledge Bases** | RAGソース。クエリして追加コンテキストを取得 |
| **Prompt Templates** | 前処理・オーケストレーション・後処理の各ステップのベースプロンプト（カスタマイズ可） |
| **Guardrails** | 有害コンテンツのブロック・フィルター |

---

## Amazon Bedrock Prompt Management（公式模擬Q6）

プロンプトをコードの外でマネージドに管理する機能。

- **バージョン管理**：プロンプトの変更履歴を管理
- **テンプレート化**：変数を埋め込んで再利用可能なプロンプトを作成
- **複数モデル対応**：同じプロンプトテンプレートを異なるモデルに適用
- **ユースケース**：スタイルガイドや業務ルールをプロンプトとして一元管理

Q6の構成パターン（リアルタイムストリーミング）：
```
記者が「分析」ボタンをクリック
  → API Gateway WebSocket API（リアルタイム双方向）
  → Lambda（メタデータからカテゴリー読み取り → Bedrock にルーティング）
  → Bedrock Prompt Management（スタイルガイドプロンプト管理）
  → Bedrock Streaming API（修正候補をリアルタイムで返す）
```

## Bedrock Agents 固有の機能

以下はすべて **Bedrock Agents に紐づく機能**であり、Bedrock 全体の機能ではない点に注意。

### アクショングループ

- OpenAPIスキーマ（JSON/YAML）で外部APIの操作を定義
- エージェントがユーザーの意図に基づいて適切なAPI操作を自動選択
- Lambda関数経由で外部APIを呼び出す

### セッション管理

- 同一の `sessionId` 内で複数ターンの対話コンテキストを自動維持
- **Memory機能**を使えば異なるセッション間でもユーザーの過去のやり取りを記憶できる

### トレース機能（enableTrace）

オーケストレーションの各ステップを詳細に記録できる。エージェントの処理パイプラインは前処理→オーケストレーション（推論ループ）→後処理の3段階からなる。

| トレース種別 | 内容 | パイプライン上の位置 |
|---|---|---|
| `PreProcessingTrace` | 入力の分類・妥当性検証（ユーザー入力をどう解釈したか） | 前処理段階 |
| `OrchestrationTrace` | 推論・ツール選択・アクション呼び出しとナレッジベースクエリのI/O（ReActループの思考過程） | オーケストレーション段階（推論ループそのもの） |
| `PostProcessingTrace` | 最終応答の整形・検証（出力をどう組み立てたか） | 後処理段階 |
| `CustomOrchestrationTrace` | カスタムオーケストレーション（独自制御ロジック）実装時の補助トレース |
| `GuardrailTrace` | ガードレールによるコンテンツ介入・ポリシー制御の記録 |
| `RoutingClassifierTrace` | マルチエージェントのルーティング判断の記録 |
| `FailureTrace` | ステップ障害（失敗）情報の記録 |
| `ModelInvocationInput` | 特定ステップでモデルに渡された入力の中身 |

## ナレッジベース

- Bedrock Agents 経由でも使えるが、**単体（Knowledge Bases for Amazon Bedrock）でも独立して利用可能**
- S3などのデータソースからドキュメントを取り込み、チャンキング・ベクトル化・ベクトルストアへの格納を自動実施
- RAGパイプラインがフルマネージドで提供される
- モデルの再トレーニングなしに最新情報が反映される（データソース同期のみでOK）

### ベクトルストアの選択肢

- Amazon OpenSearch Serverless
- Amazon Aurora PostgreSQL
- など（いずれもマネージド運用）

### 運用ポイント

- データソース同期は **Amazon EventBridge** と組み合わせてスケジュール実行が可能
- チャンキング戦略の調整で検索精度を最適化できる
- 商品カタログなど頻繁に更新されるデータとの相性が良い（同期のみで反映・再学習不要）

### 検索パラメータ

| パラメータ | 内容 |
|---|---|
| `top-k` | 検索結果の取得件数。少なくするとレイテンシー改善、多くすると網羅性向上 |

### チャンク戦略の種類

| 戦略 | 特徴 |
|---|---|
| 固定サイズ | シンプル。文字数・トークン数で均等に分割 |
| セマンティック | 意味のまとまりで分割。検索精度が高い |
| 階層的 | 粗い粒度と細かい粒度を組み合わせて管理 |

### 対応フォーマット

PDF・テキスト・HTML・CSV など多岐にわたる

## Human-in-the-Loop（HITL）

AI の処理フローに人間の判断・確認を組み込む設計パターン。
AWS では主に **Bedrock Agent の HITL** と **Step Functions の waitForTaskToken** の 2 つのアプローチがある。

### Bedrock Agent の Human-in-the-Loop

エージェントのオーケストレーション内で処理を一時停止し、人間の入力を待つ仕組み。
対話セッション内でリアルタイムに確認・修正をやり取りするユースケースに向く。

```
ユーザー ← → Bedrock Agent（ReAct ループ）
                  ↓ 判断が必要なステップで処理を中断
               人間に確認・入力を要求
                  ↓ 入力を受けて処理を再開
               次のアクションへ
```

**典型的な用途**
- エージェントがアクションを実行する前に「本当に実行しますか？」と確認する
- 中間結果を人間にレビューさせてから次のステップに進む
- 情報が不足したときに追加質問をして対話を継続する

### Step Functions waitForTaskToken との使い分け

| | Bedrock Agent HITL | Step Functions waitForTaskToken |
|---|---|---|
| 待機時間 | **秒〜分**（対話セッション内） | **分〜日**（外部の承認フロー） |
| 状態管理 | エージェントのセッション内で管理 | ステートマシンが明示的に管理 |
| 監査証跡 | 対話ログ（CloudWatch 等） | **DynamoDB 等への明示的な記録が必要** |
| 向いているケース | リアルタイムの確認・修正 | コンプライアンス上の承認フロー・長時間待機 |
| 不向きなケース | 数時間〜数日の承認待ち | 単純な対話内の確認 |

## マルチエージェント・コラボレーション（AIP-78）

### 基本アーキテクチャ

```
スーパーバイザーエージェント（Supervisor）
  ├─ 役割：ユーザーの問い合わせを受け取り、意図を自然言語で分類
  │         適切なコラボレーターにタスクを委譲する「指揮官」
  │
  ├─ コラボレーターエージェント A（口座管理専用）
  │     └─ 専用ナレッジベース A（口座管理データ）
  │
  ├─ コラボレーターエージェント B（ローン審査専用）
  │     └─ 専用ナレッジベース B（ローン審査データ）
  │
  ├─ コラボレーターエージェント C（投資アドバイス専用）
  │     └─ 専用ナレッジベース C（投資データ）
  │
  └─ コラボレーターエージェント D（カスタマーサポート専用）
        └─ 専用ナレッジベース D（サポートFAQ・規約）
```

### スーパーバイザーエージェント（Supervisor）

- ユーザーの問い合わせを**自然言語で意図分類**してコラボレーターに委譲
- 複数のコラボレーターエージェントを「知っている」状態で管理
- 直接回答の生成はしない（オーケストレーション専任）
- 新しい業務領域を追加する際は「コラボレーターを登録するだけ」で済む → 拡張容易

### コラボレーターエージェント（Collaborator）

- 特定の業務領域に特化したエージェント
- 担当領域の**専用ナレッジベース**を持ち、RAG で回答を生成
- スーパーバイザーから委譲されたタスクのみを処理

### なぜコラボレーターには専用ナレッジベースが必要か

```
統合ナレッジベース（全領域混在）の問題：
  「金利」でKNN検索
    → 口座管理の金利、ローンの金利、投資商品の金利、FX金利... が全部ヒット
    → ノイズ大 → 回答精度が低下

専用ナレッジベース（領域分離）の利点：
  「金利」でKNN検索（ローン審査KBのみ）
    → ローン金利に関する文書だけがヒット
    → ノイズなし → 高精度な回答

KNN（k近傍法）で類似検索する以上、
データが混在するほど無関係な文書がヒットする確率が上がる。
→ 業務領域ごとの完全な分離が精度の前提条件になる。
```

### 2つの動作モード

| | スーパーバイザーモード | スーパーバイザー・ルーターモード |
|---|---|---|
| **動作** | コラボレーターの応答を受け取り、統合・要約してから返す | 最も適切なコラボレーターに直接ルーティングし、その応答をそのまま返す |
| **向き** | 複数コラボレーターの結果を統合する必要がある場合 | 単一コラボレーターで完結できる場合（シンプル・高速） |
| **例** | 「ローン審査と投資アドバイスを両方考慮した回答が必要」 | 「口座残高を教えて」→ 口座管理エージェントだけで完結 |

### 拡張性の比較

| 構成 | 新業務領域追加時の作業 |
|---|---|
| **SV + 専用KB付きコラボレーターN** | コラボレーター1個 + KB1個を作成してSVに登録するだけ |
| 単一エージェント + 統合KB | システムプロンプト全体の見直し + 統合KBへのデータ追加 |
| 業務領域別SV + SQS + Lambda | SQS設定変更 + Lambdaのルーティングロジック変更 |

- 2025年3月にGAになりスーパーバイザー型の分業構成が可能
- まずシングルエージェントで検証し、段階的に拡張するのが推奨

## Amazon Q Developer カスタマイズ（AIP-23）

### Amazon Q Developer とは

IDE（VS Code / JetBrains 等）に統合されるAIコーディングアシスタント。

| ティア | 特徴 |
|---|---|
| Q Developer（Free） | 個人利用。汎用的な提案 |
| **Q Developer Pro** | 組織管理・ユーザー割り当て・**カスタマイズ機能**が使える |

### カスタマイズ（Customization）機能

**社内のコードリポジトリをデータソースとして読み込み、Q Developer の提案をカスタマイズする機能。**

```
管理者が設定するもの
  ├─ データソース：社内フレームワーク / コーディング規約 / ユーティリティライブラリの Git リポジトリ
  ├─ カスタマイズを割り当てるチーム（Pro ユーザー単位）
  └─ プロジェクトリポジトリ自体は変更不要
         ↓
開発者が IDE で使うとき
  → Q Developer の提案に社内規約・フレームワークが反映されたコードが出てくる
```

### データソースの種類と運用詳細

| 項目 | 内容 |
|---|---|
| 接続できるデータソース | **Amazon S3 バケット**、**AWS CodeCommit リポジトリ** 等 |
| 組織あたりの上限 | カスタマイズは最大 **10 個**まで作成可能（2024年時点） |
| 割り当て管理 | **IAM Identity Center** のグループ・ユーザー単位で割り当て |
| データソース更新時 | 管理者が**再同期**を実行すれば反映（各プロジェクトへの追加作業は不要） |
| 基本性能への影響 | カスタマイズ適用後も汎用的な提案能力は維持される。**組織固有のパターンが追加的に反映**される仕組み |

| | Amazon Q Developer | Amazon Q Business |
|---|---|---|
| 用途 | **コーディング支援**（IDE に統合） | **社内チャットボット**（ドキュメント検索・Q&A） |

## Lambda Function URL と MCP サーバー（AIP-25）

### Model Context Protocol（MCP）とは

AI エージェントがツール（外部 API・データソース等）を呼び出すための**標準プロトコル**。
MCP サーバーは HTTPS エンドポイントとして公開し、クライアント（AI エージェント）がツールを呼び出す。

### Lambda Function URL とは

Lambda 関数に**直接 HTTPS エンドポイントを付与する機能**。API Gateway 等の追加サービス不要。

```
Lambda 関数
  ↓ Function URL を有効化（設定1回のみ）
https://xxxx.lambda-url.ap-northeast-1.on.aws/
  ← 直接 HTTPS で呼び出せる
```

### Lambda Function URL（レスポンスストリーミング + AWS_IAM 認証）

```
MCP クライアント（AI エージェント）
  ↓ SigV4 署名付きリクエスト
Lambda Function URL
  ├─ auth_type: AWS_IAM（SigV4 署名を検証）
  └─ invoke_mode: RESPONSE_STREAM（ストリーマブル HTTP）
  ↓
Lambda 関数（MCP サーバーのロジック）
  ↓ ストリーミングレスポンス
クライアントに逐次返却
```

### 各設定項目の意味

| 設定 | 値 | 意味 |
|---|---|---|
| 認証タイプ | `AWS_IAM` | 呼び出し元に SigV4 署名を要求。IAM ポリシーで `lambda:InvokeFunctionUrl` 権限を付与 |
| 呼び出しモード | `RESPONSE_STREAM` | レスポンスを逐次ストリーミング。ストリーマブル HTTP に対応 |
| 外部パートナーのアクセス | クロスアカウント IAM ロール + `lambda:InvokeFunctionUrl` 権限 | 追加サービス不要で外部からの強固な認証を実現 |

### Lambda Function URL vs API Gateway

| 観点 | Lambda Function URL | API Gateway |
|---|---|---|
| 管理コンポーネント数 | **少ない**（Lambda だけ） | 多い（API Gateway + Lambda） |
| レスポンスストリーミング | **対応**（RESPONSE_STREAM モード） | REST API は対応・HTTP API は非対応（後述） |
| 認証 | AWS_IAM または NONE | Lambda オーソライザー / Cognito / IAM |
| 用途 | 単純な HTTP エンドポイント・MCPサーバー | 高度なルーティング・複数Lambda統合・スロットリング |

### API Gateway とストリーミング

| API Gateway オプション | ストリーミング | 備考 |
|---|---|---|
| **REST API** | ✅ **対応** | `InvokeWithResponseStreaming` で Lambda からストリーミング可能。タイムアウト最大 15 分 |
| **HTTP API** | ❌ 非対応 | Lambda Response Streaming に非対応。全レスポンスをバッファリングしてから返す |
| **WebSocket API** | ⚠️ **別プロトコル** | HTTP ストリーミングではない（後述） |

#### REST API のストリーミングの仕組み

Lambda ハンドラーが以下の順でレスポンスを返す：

```
① JSON レスポンスメタデータ
② 8 バイトの null 区切り文字
③ レスポンスペイロード（ストリーミング）
```

**制約**：
- 最初の 10 MB は帯域無制限。10 MB 超は 2 MB/秒に制限
- アイドルタイムアウト：リージョナル/プライベート = 5 分、エッジ最適化 = 30 秒
- VTL によるレスポンス変換・キャッシングは非対応

#### WebSocket API は「別プロトコル」

```
【HTTP ストリーミング（Lambda Function URL / API Gateway REST API）】
  HTTP(S) プロトコルのまま chunked transfer encoding でレスポンスを逐次返す
  → MCP の「ストリーマブル HTTP トランスポート」要件を満たす

【API Gateway WebSocket API】
  ws:// / wss:// という別プロトコルで持続接続を確立
  サーバーから postToConnection でメッセージをプッシュ
  → HTTP ではない。MCP の要件を満たさない
```

WebSocket は「リアルタイム双方向通信」（チャット UI のトークン逐次表示など）には有効だが、MCP の「HTTP 上でのストリーミング」要件には対応できない。

#### AIP-25 で Lambda Function URL が正解な理由（REST API が使えても）

API Gateway REST API でもストリーミング自体は実現できる。それでも Lambda Function URL が正解なのは：

- **管理コンポーネントの最小化**：API Gateway を追加すると設定・監視・バージョン管理のオーバーヘッドが増える
- **Lambda Function URL はゼロ追加コスト**：Lambda の組み込み機能。追加サービス不要

### AWS SigV4 署名とは

AWS へのリクエストを認証するための署名方式。**トークンを事前発行して送るのではなく、リクエストそのものに毎回署名する**のが本質。

#### トークン認証との違い

```
【一般的なトークン認証（JWT / OAuth 等）】
サーバー → トークンを発行 → クライアントが保存
クライアント → Authorization: Bearer eyJhbG... （毎回同じトークンを送る）
問題点：盗まれたら有効期限まで悪用される。改ざん検知できない。

【AWS SigV4】
トークンを事前発行しない。リクエストごとに署名値を計算して送る。
AWS 側も同じ計算を行い、結果が一致すれば認証成功。
```

#### 署名の材料

```
① HTTP メソッド（GET / POST 等）
② URL（パス + クエリストリング）
③ リクエストボディのハッシュ値（SHA-256）
④ タイムスタンプ（年月日時分）
⑤ リージョン + サービス名（例：ap-northeast-1 / lambda）
⑥ AWS シークレットアクセスキー
  → 全部を組み合わせて HMAC-SHA256 で署名値を計算
```

実際のリクエストヘッダー：

```http
Authorization: AWS4-HMAC-SHA256
  Credential=AKIAXXXXX/20250525/ap-northeast-1/lambda/aws4_request,
  SignedHeaders=content-type;host;x-amz-date,
  Signature=a3f9b2c1...（計算された署名値）

X-Amz-Date: 20250525T120000Z
```

#### SigV4 が防ぐ攻撃

| 攻撃 | 防御できる理由 |
|---|---|
| リクエストの改ざん | ボディのハッシュが署名に含まれる。改ざんすると署名不一致 |
| リプレイ攻撃（後で同じリクエストを再送） | タイムスタンプが署名に含まれる。**15 分以上古いリクエストは AWS が拒否** |
| 認証情報の盗難 | シークレットキー自体はネットワークに流れない（署名値だけ送る） |

### Amazon Cognito とは何か（混同注意）

Cognito には 2 つの機能があり、目的が異なる。

| 機能 | 目的 | 典型的な使い方 |
|---|---|---|
| **User Pools（ユーザープール）** | **認証**（誰であるかを確認） | サービスへのサインアップ・ログイン。Google / Facebook / SAML でのフェデレーション（ソーシャルログイン）も可能 |
| **Identity Pools（ID プール）** | **認可**（AWS リソースへのアクセス権付与） | User Pool の JWT トークンや Google 認証情報を AWS 一時認証情報（IAM ロール）に変換 |

**Cognito が向いているケース**（AIP試験での識別）

| 要件 | 適切か |
|---|---|
| 「エンドユーザーがウェブ/モバイルアプリにログインする」 | ✅ User Pools |
| 「Google/Facebook アカウントでログインさせたい（ソーシャルログイン）」 | ✅ User Pools のフェデレーション |
| 「クライアントアプリから S3/DynamoDB に直接アクセスさせたい」 | ✅ Identity Pools |
| **「サービス間（M2M）の API 認証を IAM で強固に行いたい」** | ❌ **Cognito は不要。IAM ロール + SigV4 が正解** |

## Step Functions によるエージェントオーケストレーション（AIP-38）

### Amazon States Language (ASL)

Step Functions のステートマシン定義に使う **JSON ベースの言語**。  
Task / Choice / Parallel / Wait / Map などのステートを組み合わせてワークフローを記述する。

```json
{
  "Comment": "購買支援フロー",
  "StartAt": "商品検索",
  "States": {
    "商品検索": {
      "Type": "Task",
      "Resource": "arn:aws:bedrock:...",
      "Retry": [...],
      "Catch": [...],
      "Next": "パーソナライズ推薦"
    }
  }
}
```

コードを書かず**宣言的**にワークフローを定義できる点が最大のメリット。

---

### Retry フィールド

一時的な障害を自動で再試行する設定。

| パラメータ | 意味 |
|---|---|
| `ErrorEquals` | 対象エラータイプ（例: `["States.TaskFailed"]`）|
| `IntervalSeconds` | 初回リトライまでの待機秒数 |
| `MaxAttempts` | 最大試行回数（0 = リトライなし）|
| `BackoffRate` | 指数バックオフの倍率（例: 2.0 なら毎回2倍に延びる）|

```json
"Retry": [
  {
    "ErrorEquals": ["States.TaskFailed"],
    "IntervalSeconds": 2,
    "MaxAttempts": 3,
    "BackoffRate": 2.0
  }
]
```

---

### Catch フィールド

リトライ上限を超えた場合（または対象エラー発生時）に**代替パスへ遷移**する設定。

```json
"Catch": [
  {
    "ErrorEquals": ["States.ALL"],
    "Next": "代替フロー"
  }
]
```

- `ErrorEquals: ["States.ALL"]` = すべてのエラーをキャッチ
- `Next` で指定したステートに遷移 → フォールバック処理を実行

## MCP トランスポートと認証（AIP-39）

### Streamable HTTP（MCPトランスポート）

| 項目 | 内容 |
|---|---|
| 用途 | **リモート MCP サーバー**との通信（ネットワーク越し） |
| 対比 | STDIO = ローカルプロセス間通信（リモート不可） |
| プロトコル | 通常の HTTP リクエスト/レスポンス（ストリーミングは任意、必須ではない） |
| AWSとの組み合わせ | **API Gateway 互換**（Lambda + API GW でサーバーレス化可能） |
| AIP試験ポイント | サーバーレスでMCPサーバーを公開する → Streamable HTTP を選ぶ |

### OIDC（OpenID Connect）

- **OAuth 2.0 の拡張**：OAuth 2.0 は「認可」のみ → OIDC は「認証（誰であるか）」を追加
- JWT（ID トークン）でユーザー情報（`sub`, `email`, `groups` 等）を伝達
- **Amazon Cognito** は OIDC プロバイダーとして動作
- API Gateway の Cognito オーソライザーが JWT を自動検証 → Lambda にクレームを渡す

### Cognito vs IAM 使い分け（再整理）

| シナリオ | 正解 |
|---|---|
| 外部エンドユーザーの認証・認可（OAuth 2.0/OIDC） | **Cognito オーソライザー** |
| AWS ユーザー/サービスが Lambda・Bedrock を呼ぶ | **IAM ロール + SigV4** |
| MCPクライアント ↔ Lambda（サービス間） | IAM ロール |
| モバイル/Webアプリユーザーがバックエンドに接続 | Cognito |

## Amazon Q Business の Data Accessor パターン（AIP-40）

### Amazon Q Business とは
- エンタープライズ向けの**フルマネージド生成AIアシスタント**
- S3、SharePoint、Salesforceなど40以上のデータソースコネクタ
- **セマンティック検索（意味的類似性）対応**のインデックスを自動構築

### Data Accessor パターン（マルチテナントSaaS）

```
顧客Aのアカウント              SaaS企業のアカウント
┌─────────────────────┐        ┌──────────────────┐
│  Amazon Q Business  │←──────│  Data Accessor   │

```

| 項目 | 内容 |
|---|---|
| データ所有権 | 顧客アカウント内に完全に保持 |
| SaaS側が受け取るもの | 検索結果（スニペット・要約）のみ。生データにはアクセスしない |
| クロスアカウント | Data Accessor ロールで最小権限を委譲 |
| インフラ管理 | フルマネージド。クラスター管理不要 |

## Amazon Bedrock AgentCore（AIP-45）

2025年発表。AIエージェントを**本番環境で大規模運用するための統合インフラプラットフォーム**。  
従来の Bedrock Agents が「推論・ツール呼び出し」に特化するのに対し、AgentCore は**運用・インフラ面の機能を一括提供**する。

### 主要機能と対応要件

| 機能 | 内容 | 対応する要件 |
|---|---|---|
| **Memory & Session Management** | セッションIDベースで会話履歴・コンテキストをマネージドに永続保持 | セッション間の文脈維持 |
| **AgentCore Identity** | IAM + OAuth 2.0 / SAML でセッション単位の認証・認可 | ユーザーごとの認証・認可 |
| **Event Processing** | 同期（リアルタイム）と非同期（バックグラウンド）を両方ネイティブサポート | リアルタイム＋バックグラウンド処理 |
| **Observability** | **AgentCore Observability**（専用の本番監視基盤）。CloudWatch・X-Ray統合で推論ステップを継続監視 ※`enableTrace`（Agents for Bedrock のデバッグ機能）とは別物 | 本番モニタリング |
| **サーバーレス自動スケール** | キャパシティプランニング不要 | 運用負荷最小 |

### ツール登録アーキテクチャ

```
Lambda 関数    → API Gateway  (同期)   ┐
外部 REST API  → API Gateway  (同期)   ├→ AgentCore が OpenAPI スキーマを
バックグラウンド → EventBridge (非同期) ┘   もとにツールを自律選択・実行
```

- **OpenAPI スキーマ**でツールのインターフェースを定義する
- エージェントが引数・呼び出し先を自律判断 → **オーケストレーションのカスタムコードが不要**

### Bedrock Agents vs AgentCore

| | Bedrock Agents | Bedrock AgentCore |
|---|---|---|
| 主な役割 | 推論・ツール呼び出し・RAG連携 | エージェントの**インフラ・運用基盤** |
| セッション管理 | 限定的（永続メモリは別途実装が必要） | **マネージドで永続管理** |
| 認証・認可 | 別途実装が必要 | **ビルトイン（IAM + OAuth）** |
| 同期・非同期 | 基本同期 | **両方ネイティブサポート** |
| スケーリング | Lambda依存 | **サーバーレス自動スケール** |

## プロンプトチェーンとは（AIP-67 補足）

### 概念

```
プロンプトチェーン = 複数のプロンプトを順番に実行し、
                    前のステップの出力を次のステップの入力に使う処理パターン

例（AIP-67 の3段階チェーン）：
  Step 1: 症状の分類
          「以下の症状を分類してください: {症状テキスト}」
                    ↓ 分類結果
  Step 2: 関連栄養データの検索
          「{分類結果} に関連する栄養情報を検索する」
                    ↓ 検索結果
  Step 3: パーソナライズ食事プランの生成
          「{症状分類} と {栄養データ} をもとに食事プランを生成してください」
                    ↓ 最終回答
```

### プロンプトチェーンでの再現性問題と解決策

```
問題：同じ症状入力でも毎回違う食事プランが返される

原因：
  ① temperature が高い → 出力がランダム化される
  ② プロンプトテンプレートが管理されていない → 誰かが手動で変えてしまう
  ③ 各ステップのパラメータが統一されていない

解決：Bedrock プロンプト管理 で temperature ≈ 0 + プロンプトをバージョン固定
  → チェーン全体の出力が安定化 → 再現性99.5%以上の要件を満たす
```

### 実装方法の選択肢

| 方法 | 向いている場面 |
|---|---|
| **Bedrock Flows** | ビジュアルにノードで設計したい・RAGパイプライン・プロンプトチェーン |
| **Bedrock Agents** | 自律的にツールを選んでタスク実行（エージェント的・動的判断が必要） |
| **Step Functions + Bedrock SDK** | 決定論的処理・厳密なフロー制御・SQS連携が必要 |
| **Lambda（自前実装）** | シンプルな数段階なら直接実装も可 |

## Amazon Bedrock Flows（Bedrockフロー）（AIP-73）

**ビジュアルなノードベースのワークフロービルダー。** コードを書かずに「入力→処理→出力」の流れをグラフィカルに設計できる。

### フロー全体のイメージ

```
入力ノード（ユーザー入力受け取り）
  ↓
プロンプトノード（プロンプト管理テンプレート + ガードレールを適用してFM呼び出し）
  ↓
条件ノード（値や内容で分岐）
  ↓
出力ノード（ユーザーへ返却）
```

### ノードの種類

| ノード | 役割 |
|---|---|
| **入力ノード** | フロー全体の入口。ユーザーの入力を受け取る |
| **出力ノード** | フロー全体の出口。結果を返す |
| **プロンプトノード** | プロンプト管理のテンプレートを指定してFMを呼び出す。**ガードレールも同ノードに関連付け可** |
| **条件ノード** | 値や条件によって次ノードへの分岐を制御 |
| **エージェントノード** | Bedrock Agentを呼び出す |
| **ナレッジベースノード** | Knowledge Basesを検索する（RAG） |
| **Lambdaノード** | Lambda関数を呼び出してカスタム処理 |

### Bedrock Flows vs Step Functions

| | Bedrock Flows | Step Functions |
|---|---|---|
| 向き | AI処理チェーン・RAGパイプライン | 決定論的処理・外部サービス統合 |
| 設計方法 | ビジュアルノード（ノーコード寄り） | JSONの状態定義（コード寄り） |
| AI特化機能 | ✅ プロンプトノード・エージェントノード内蔵 | ❌ 別途Bedrock SDK呼び出しが必要 |
| waitForTaskToken | ❌ 非対応 | ✅ 対応（人間承認フロー） |

```
Bedrock Flows が向くケース（Bedrock 内で完結）
  FM呼び出し → 条件分岐 → KB検索 → FM呼び出し

Step Functions が向くケース（外部をまたぐ）
  FM呼び出し → DynamoDB書き込み → 人間承認待ち → 外部API呼び出し
```

### CoT（Chain-of-Thought）との組み合わせ

各推論ステップをプロンプトノードとして定義し、ノードの順番がそのまま思考の順番になる。

```
[Input: 論文テキスト]
  ↓ [Prompt ノード①: 「まず前提条件を整理してください」]
  ↓ [Prompt ノード②: 「次に仮説を立ててください」]
  ↓ [Prompt ノード③: 「証拠と照合して結論を出してください」]
  ↓ [Output: 構造化された推論結果]
```

→「CoT テンプレート」「推論ステップを体系的に管理」が出たら **Bedrock Flows** が正解候補。

### 試験での識別キーワード

| キーワード | 正解 |
|---|---|
| 「コードなしで AI パイプライン」「ノードでつなぐ」「条件分岐を含む生成AI」 | **Bedrock Flows** |
| 「CoT テンプレート」「推論ステップを順番に管理」 | **Bedrock Flows** |
| 「自律的にツールを選んで実行」 | **Bedrock Agents** |
| 「人間承認待ち・外部サービスをまたぐ」 | **Step Functions** |

## プロンプト管理のテンプレート変数（AIP-73・74）

`{{変数名}}` の形式でテンプレート内に埋め込む。実行時に値を動的に差し込む仕組み。

```
【テンプレート例：医療アドバイスアプリ（AIP-73）】

システムプロンプト:
あなたは医療アドバイザーです。以下の形式で必ず回答してください。
1. 症状の概要
2. 考えられる原因
3. 推奨する対処法
4. 受診の目安

ユーザープロンプト:
患者情報:
- 年齢: {{age}}歳
- 症状: {{symptoms}}
- 既往歴: {{medical_history}}

上記の患者に適切な健康アドバイスを提供してください。
```

実行時に `age=45`、`symptoms=頭痛・発熱`、`medical_history=高血圧` を渡すと自動差し込み。

- **必須変数が渡されなかった場合はエラー**（必須チェックの役割も果たす）
- 変数・FM選択・推論パラメータ・バージョン番号を一括でバージョン管理できる
- フローのプロンプトノードに設定することで、フロー実行時に変数値が渡される

## インストラクションベースのファインチューニング（公式模擬試験メモ）

### 通常のファインチューニングとの違い

| 種類 | 学習データ形式 | 目的 |
|---|---|---|
| **継続事前学習（Continued Pre-training）** | 大量のテキスト（ラベルなし） | ドメイン知識を追加学習 |
| **通常のファインチューニング** | 入力→出力のペア | 特定タスクへの特化 |
| **インストラクションベースのファインチューニング** | 指示→応答のペア | **指示に従う能力を強化** |

### インストラクションチューニングとは

```
学習データの形式：
  {
    "instruction": "次の文章を要約してください",
    "input": "（元の長文テキスト）",
    "output": "（期待する要約結果）"
  }

→ 「指示（instruction）+ 入力 → 期待する応答」という形式で学習させる
→ モデルが「指示を読んで適切に行動する」能力を習得する
```

### Bedrock でのファインチューニング

- Bedrock のカスタムモデル機能（Fine-tuning）で対応
- 学習データは S3 に置いた JSONL 形式
- **インストラクションチューニング** = 指示追従能力の強化
- **継続事前学習** = ドメイン固有知識の注入（医療・法律など）

## Amazon Bedrock AgentCore Runtime（公式模擬 Q15）

### AgentCore Runtime とは
- **既存の Python エージェントコードを最小オーバーヘッドでデプロイ**するためのマネージドランタイム
- HTTPサーバー設定・エンドポイントルーティング・運用状態モニタリングを**自動管理**
- JSON応答（低レイテンシー）とストリーミング応答（長時間）の両方をサポート

### デプロイ方法2種類

| 方法 | 概要 | オーバーヘッド |
|---|---|---|
| **AgentCore SDK** (`@app.entrypoint`) | デコレーターを付けるだけでHTTPサーバー・エンドポイント自動生成 | 最小 |
| **AgentCore スターターツールキット** | パッケージング・コンテナ化・ECRプッシュ・デプロイを自動化 | 最小 |

### AgentCore SDK の使い方

```python
from bedrock_agentcore import AgentCore

app = AgentCore()

@app.entrypoint  # ← これだけで以下が自動設定される
def handler(payload):
    # エージェントロジック
    return response

# 自動生成されるもの:
# - ポート8080のHTTPサーバー
# - /invocations エンドポイント
# - /ping エンドポイント（ヘルスチェック）
```

### セッション分離とプロトコルサポート（公式模擬 Q43）

- **フレームワーク非依存**：Strands・LangGraph・CrewAI等、既存のローカルエージェントコードをほぼそのままクラウドにデプロイできる
- **セッション単位のmicroVM分離**：ユーザーセッションごとに専用CPU/メモリ/ファイルシステムを持つ独立したmicroVMが起動し、セッション終了後は完全破棄・メモリサニタイズされる（セッション間のデータ汚染を防止）
- **MCP / A2A（Agent to Agent）をネイティブサポート**：AgentCore RuntimeはMCPサーバーのホスティング環境も標準機能として提供する。事前構築済みMCPサーバーを設定するだけでツール接続が完了し、カスタムコンテナやAPI管理が不要になる
- 最大8時間の長時間実行、100MBペイロード、双方向ストリーミングにも対応

## Bedrock Agent セッションメモリ（sessionId）

### 通常のBedrock API vs Bedrock Agent のコンテキスト管理

```
【通常のBedrock Converse / InvokeModel API】
  開発者がmessages[]を自前で積み上げてAPIに毎回送る
  → コンテキスト管理は開発者の責任

【Bedrock Agent の InvokeAgent API】
  sessionId を渡すだけで自動的にコンテキストを維持
  request = {
    agentId: "xxx",
    sessionId: "user-123-session",  ← これだけ
    inputText: "前の話の続きで..."
  }
  → AWS側がセッション内の会話履歴を自動管理
```

### sessionIdの仕組み

```
1回目：sessionId: "abc123" + "商品Aについて教えて"
  ↓ Agentが内部でセッション状態を保存

2回目：sessionId: "abc123" + "その価格は?"
  ↓ AgentがsessionIdから前の文脈を自動復元
  ↓ 「その」= 商品Aのこととして理解
```

### なぜDynamoDB案が「追加実装が必要」か

```
DynamoDB案：
  毎回 DynamoDB から履歴取得 → messages[]組み立て → Bedrock呼び出し
  → 自前でセッション管理実装 → 工数増 → ❌

Bedrock Agent（sessionId）案：
  sessionIdを渡すだけ → AWS側が全部やる → ✅（Bedrock Way）
```

## AWS Strands Agents / Agent Squad

### AWS Strands Agents とは

**AWSが提供するオープンソースのエージェントSDK（Python）。**

```
Bedrock Agentsとの違い：
  Bedrock Agents → フルマネージド・GUIで設定・コード最小
  AWS Strands   → コードベース・柔軟なカスタマイズ・オープンソース

Strandsが向いている場面：
  ・複雑なカスタムロジックが必要
  ・Bedrock以外のFMも使いたい（マルチプロバイダ）
  ・エージェントの挙動を細かく制御したい
```

### AWS Agent Squad（旧: Multi-Agent Orchestrator）とは

**複数のエージェントを束ねるオーケストレーション層。**

```
複数エージェント構成例：
  Agent Squad（オーケストレーター）
    ├─ カスタマーサポートエージェント（FAQ回答担当）
    ├─ 注文処理エージェント（在庫・配送担当）
    └─ エスカレーションエージェント（人間に繋ぐ担当）

→ ユーザーの質問を判断して適切なエージェントに振り分ける
```

### 3つの関係

| サービス | 位置づけ | 特徴 |
|---|---|---|
| **Bedrock Agents** | AWSマネージド | GUIで設定・コード最小・フルマネージド |
| **AWS Strands** | SDK（コードベース） | カスタマイズ性高・オープンソース |
| **Agent Squad** | オーケストレーター | 複数エージェントの振り分け・管理 |

## MCP（Model Context Protocol）

### MCPとは

**異なるFM・ツールを共通のプロトコルで接続する標準規格（Anthropic発）。**

```
MCPなしの世界：
  Claude用のツール接続コード
  Nova用のツール接続コード    ← FMごとに別々実装
  Llama用のツール接続コード

MCPありの世界：
  MCP対応なら全FMで同じコード → 相互運用性◎
  FMを変えてもツール接続コードを書き直し不要
```

### MCPの構造

```
MCPサーバー（ツール・データ提供側）：
  「私はこんな機能を提供できます」と宣言
  例：Figma MCP Server → デザインデータを提供
      Bedrock KB MCP Server → ドキュメント検索を提供

MCPクライアント（FM/Agent側）：
  MCPサーバーを呼び出す側
  例：Claude Desktop、Bedrock Agent、AWS Strands
```

### MCPクライアントライブラリとは

```
MCPプロトコルの通信規約を「開発者が意識しなくていいように包んだコード」

ライブラリなし：JSON-RPCのフォーマット、ハンドシェイク、エラーハンドリングを全部自前実装
ライブラリあり：mcp_client.call_tool("search", query="...") で1行で呼べる

= 「プロトコルの複雑さを抽象化」= 相互運用性が向上する
```

### Figmaとの対応（理解の補助）

```
FigmaがMCPサーバーを提供
  ↓
Claude DesktopがMCPクライアントとして接続
  ↓
「このデザインファイルを読んで」と指示するだけ
→ プロトコルの複雑さを意識しない

AWSでも同様：
  Bedrock KB・S3・Lambda等がMCPサーバーを提供
  Bedrock Agent / Strandsが接続 → 同じI/Fで複数ツールを扱える
```

### MCPサーバーによるレガシーAPIゲートウェイパターン

レガシーAPI（例：CMS）の前段にMCPサーバーを立て、関数呼び出しスキーマ（semantic・AI向けインターフェース）として公開するパターン。個々のAPIエンドポイントの挙動の不整合はMCPサーバー内部で吸収する。Bedrock AgentCoreをMCPクライアント（エージェント実行環境）として構成すると、AgentCore経由の各エージェントワークフローが同じMCP関数群を呼び出せる。
