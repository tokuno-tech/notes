# SageMaker / MLOps パターン

## MLモデル再学習 → 自動デプロイ構成（AIP-13）

### 要件
- S3にデータが追加されるたびに再学習・デプロイを自動化
- モデル学習に数時間かかる
- 運用オーバーヘッドを最小化

### 構成

**Step Functions Standard + SageMaker Pipelines**

```
S3にバッチファイルがアップロード
  ↓ S3イベント通知
Lambda（ファイル情報を取得・整形）
  ↓
Step Functions Standard ワークフロー
  ↓ StartPipelineExecution（.syncパターンで完了を待機）
SageMaker Pipelines
  ├─ データ前処理
  ├─ モデルトレーニング
  ├─ モデル評価
  ├─ モデル登録
  └─ エンドポイントへのデプロイ
```

### 各コンポーネントの役割

| コンポーネント | 役割 |
|---|---|
| **Step Functions Standard** | 最大1年間実行可能。状態管理・エラーハンドリング・リトライを組み込みで提供。数時間かかるML学習のオーケストレーションに最適 |
| **SageMaker Pipelines** | ML専用のオーケストレーションサービス。前処理→学習→評価→登録→デプロイをDAGとして定義。パラメータ管理・成果物管理・再現性を実現 |
| `.sync` パターン | Step FunctionsがSageMaker Pipelinesの完了を待機できる同期統合パターン |

### SageMaker Pipelinesの利点
- パイプライン定義をバージョン管理できる
- 各実行のパラメータ・成果物・メトリクスを追跡できる
- SageMaker Studio と統合されGUI でパイプラインを可視化・モニタリング可能
- MLOps のベストプラクティスに沿った構成を実現

## Provisioned Throughput（バッチ安定性） + SageMaker Neo（オンプレ推論）（AIP-26）

### 要件

- 大量の医療画像を定期バッチ処理。**安定したパフォーマンス**が必要（→ Provisioned Throughput）
- 患者データの規制によりオンプレミスで推論を実行（→ SageMaker Neo）
- クラウドとオンプレミスの**ハイブリッド構成**

### 構成（2つ）

#### ① Amazon Bedrock Provisioned Throughput（バッチ安定性）

```
オンデマンド推論（共有リソース）
  → スロットリング発生リスクあり。ピーク時にレイテンシが変動する

Provisioned Throughput（専用キャパシティ）
  → 予約した処理能力が常に確保される
  → 大規模バッチジョブで「安定したパフォーマンス」要件に合致
```

| 項目 | Provisioned Throughput | オンデマンド |
|---|---|---|
| 処理能力 | 専用確保（MU単位で予約） | 共有リソース |
| スロットリングリスク | **なし**（予約分まで保証） | あり |
| 費用モデル | 時間課金（使用量に関わらず） | リクエスト数 × トークン数 |
| 適用シーン | 大量バッチ・低レイテンシ保証 | 小〜中規模・散発的なリクエスト |

#### ② Amazon SageMaker Neo（オンプレ推論）

```
クラウド上のモデル（PyTorch / TensorFlow 等）
  ↓ SageMaker Neo でコンパイル
        ├─ 対象ハードウェアを指定（ARM / x86 / NVIDIA GPU / Inferentia 等）
        └─ そのハードウェア向けに最適化された実行バイナリを生成
  ↓
オンプレミスサーバー / エッジデバイスにデプロイ
  → クラウドAPIを呼ばずにローカルで推論実行
  → 患者データがクラウドに送信されない
```

- **コンパイル = モデルを特定のハードウェア用に変換**
- AWS IoT Greengrass と組み合わせることでエッジデバイス管理も可能

## SageMaker 推論エンドポイント比較（公式模擬 Q11）

### 4種類の推論方式

| 方式 | レイテンシー | 最大ペイロード | GPU対応 | オンデマンド | 用途 |
|---|---|---|---|---|---|
| **リアルタイム推論** | ミリ秒 | **25 MB** | ✅ | ✅ | 低レイテンシー・継続的推論 |
| **非同期推論** | **最大15分** | **1 GB** | ✅ | ✅ | 大ペイロード・長時間処理 |
| **サーバーレス推論** | 秒 | 少量 | **❌ GPU非対応** | ✅ | トラフィック不定・コスト最適化 |
| **バッチ変換** | 非リアルタイム | 大規模 | ✅ | **❌ オンデマンド不可** | オフライン大量一括処理 |

### 選択の判断フロー

```
GPU必須？
  → No → サーバーレス推論も候補
  → Yes ↓

オンデマンド（個別リクエスト）？
  → No → バッチ変換
  → Yes ↓

ペイロード > 25MB または 処理 > 数秒？
  → Yes → 非同期推論（最大1GB、最大15分）
  → No  → リアルタイム推論
```

## 分散推論の並列化3種類

大規模モデルが1枚のGPUに収まらない場合、複数GPUに分散する。方式が3種類ある。

```
【問題】70Bパラメータのモデル → 1枚のGPU (80GB VRAM) に収まらない
        → 複数GPUに分割する必要がある
```

### ① テンソル並列化（Tensor Parallelism）← 最重要

**各層の重み行列（テンソル）を複数GPUに横断的に分割する。**

```
重み行列 W（巨大）を縦に分割：
  GPU-0: W の左半分を担当 → 部分計算結果 out_0
  GPU-1: W の右半分を担当 → 部分計算結果 out_1
  → 結果を AllReduce（合算）して最終出力

実行イメージ（行列積の例）：
  全体: y = x @ W      （W が巨大で1GPUに収まらない）
  分割: y = x @ W_left + x @ W_right
         GPU-0が左側   GPU-1が右側を同時計算
```

**特徴：**
- 1層の計算を複数GPUで**同時に**実行 → レイテンシーが下がる
- GPU間で頻繁に中間結果を通信（AllReduce）が必要 → 通信オーバーヘッドが発生
- `tensor_parallel_degree` で分割GPU数を指定

**向いている場面：** 1層が大きすぎて収まらない超大規模モデル、低レイテンシーが要件

---

### ② パイプライン並列化（Pipeline Parallelism）

**モデルを「層のグループ」に分けて複数GPUに縦断的に割り当てる。**

```
Transformer 48層 を GPU 4枚で分担：
  GPU-0: 層 1〜12
  GPU-1: 層 13〜24
  GPU-2: 層 25〜36
  GPU-3: 層 37〜48

前の段の出力が次の段の入力になる（水道管のように流れる）
```

**特徴：**
- 各GPUは自分の層の計算だけ担当 → GPU間通信は層の境界のみ
- バブル（待ち時間）が発生しやすい（前の段が終わらないと次が動けない）

**向いている場面：** 層数が多くメモリが足りない場合

---

### ③ データ並列化（Data Parallelism）

**各GPUがモデルのフルコピーを持ち、異なるデータを処理する。**

```
GPU-0: モデル全体のコピー → バッチ A を処理
GPU-1: モデル全体のコピー → バッチ B を処理
GPU-2: モデル全体のコピー → バッチ C を処理
→ 勾配を同期して重みを更新（学習時）
```

**特徴：**
- モデル自体は分割しない → 通信コストが比較的少ない
- **学習時に多用**。推論での主な用途はスループット向上（同時リクエスト数を増やす）

**向いている場面：** スループット重視・モデルが1GPUに収まる場合

---

### 3種類の比較表

| 方式 | 分割対象 | 主な効果 | デメリット |
|------|---------|---------|-----------|
| **テンソル並列化** | 各層の重み行列（横断） | **レイテンシー低下**・メモリ分散 | GPU間通信が多い |
| パイプライン並列化 | 層グループ（縦断） | メモリ分散 | バブル（待ち時間）発生 |
| データ並列化 | なし（モデルをコピー） | **スループット向上** | モデルが1GPUに収まる必要あり |

## SageMaker AI + DJL Serving のリソース最適化（公式模擬 Q8）

### DJL (Deep Java Library) Serving とは
- SageMaker AI 上で LLM をサービングするためのフレームワーク
- **連続バッチ処理（continuous batching）** をサポート
- `serving.properties` でチューニング可能

### 主要チューニングパラメータ

| パラメータ | 意味 | 効果 |
|---|---|---|
| `max_seq_length` | 1リクエストの最大I/Oシーケンス長 | 短くすると **バッチサイズ↑、同時実行数↑、スループット↑** |
| `tensor_parallel_degree` | 1モデルを何GPUに分割するか | モデルが収まる最小GPU数に設定 → 残りGPUで**追加レプリカ**を動かす |
| `rolling_batch_size` | ローリングバッチサイズ | メモリが空けば増やせる |

### 最適化の考え方
1. **実トラフィックの最大シーケンス長を計測** → 設定値が過大なら短縮
2. **モデルの実メモリフットプリント測定** → tensor_parallel_degree を最小化
3. **空いたGPUにレプリカを追加** → 1インスタンスのスループット最大化

### 例（公式模擬 Q8）
- 8 GPU インスタンス、モデルは 4 GPU に収まる
- 実シーケンス長は設定の 1/10
- **正解の組み合わせ**:
  - `max_seq_length` を短縮 → バッチサイズ拡大
  - `tensor_parallel_degree=4` + **インスタンスごとに2レプリカ** デプロイ

## LoRA（Low-Rank Adaptation）

**モデル全体を再学習せず、小さなアダプター層だけを学習する効率的なファインチューニング手法。**

```
Full Fine-tuning：元モデルの全パラメータを更新（高コスト）
LoRA：元モデルを凍結 → 小さなアダプター行列だけ学習（低コスト）
```

| | Full Fine-tuning | LoRA |
|---|---|---|
| 学習対象 | 全パラメータ | アダプター層のみ |
| コスト | 非常に高い | 安い |
| 精度 | 最高 | Full比で遜色ない場合が多い |
| 切り替え | 1モデル1用途 | アダプターを差し替えるだけ |

**アダプターの使い回し：**
```
ベースモデル（共通・凍結）
  ├── アダプターA（医療用）
  ├── アダプターB（法律用）
  └── アダプターC（カスタマーサポート用）
```

関連：**QLoRA** = LoRAをさらに量子化して省メモリ化

## SageMaker デプロイパターン

### マルチモデルエンドポイント

**1つのエンドポイント（GPU）に複数モデルを同居させる。**

```
従来：モデル1つ = エンドポイント1つ = GPU1台 → コスト比例増

マルチモデルエンドポイント：
  モデルA ┐
  モデルB ├ → 同じGPUエンドポイントに同居
  モデルC ┘
  → GPU1台で複数モデルを捌く → コスト大幅削減
```

### 推論パイプライン（Inference Pipeline）

**複数コンテナを直列に繋いで推論エンドポイントを構成する機能。**

```
リクエスト
  ↓ [コンテナ1：前処理]（正規化・特徴量変換）
  ↓ [コンテナ2：モデル推論]
  ↓ [コンテナ3：後処理]（結果変換・スコアリング）
  ↓ レスポンス
```

| | カスタムコンテナ（1つ） | 推論パイプライン |
|---|---|---|
| 構成 | 1コンテナに全部書く | 複数コンテナを直列接続 |
| 向いている場面 | ロジックが一体不可分 | 前処理・推論・後処理を独立して管理・更新したい |

### Processing Job との使い分け

| | Processing Job | 推論エンドポイント内の前後処理 |
|---|---|---|
| タイミング | **バッチ・事前処理** | **リアルタイム・推論のたびに実行** |
| 用途 | **学習データの前処理**・特徴量エンジニアリング | 入力変換・出力加工（金融規制フォーマット等） |
| 実装 | 独立したジョブ | カスタムコンテナ or 推論パイプライン |

### SageMakerのスケーリング

```
Bedrock：クロスリージョン推論プロファイル（AWS管理）
SageMaker：ELB（ロードバランサー）+ EC2 Auto Scaling（自前設定）
```

### 本番トラフィックを止めずにエンドポイント設定を更新する（UpdateEndpoint）

エンドポイントを削除して作り直す（新規エンドポイント作成→全トラフィック移行→旧エンドポイント削除）と、その間トラフィックが途切れる。
**同一エンドポイントに対して新しいエンドポイント設定（EndpointConfig）を作成し、`UpdateEndpoint` APIで設定を差し替える**と、SageMaker側が新設定のインスタンスを裏で立ち上げてから旧インスタンスを退役させるため、可用性を維持したまま（ブルー/グリーン的に）切り替えられる。データキャプチャの有効化もこの新しいエンドポイント設定に対して行う。

### 新モデルの運用パフォーマンスを本番トラフィックで検証する：シャドウテスト vs A/Bテスト

新バージョンのモデルの**運用メトリクス**（レイテンシー・エラー率・リソース使用率）を、実際の本番トラフィックの下でエンドユーザーへの影響なしに評価したい場合は **SageMaker AI シャドウテスト** を使う。同一エンドポイント上で本番バリアントと一緒にシャドウバリアントをデプロイし、本番トラフィックのコピーをシャドウバリアントにも流すが、その応答はユーザーには返さない。トラフィックの複製・メトリクス収集はSageMaker側が自動処理するため追加インフラ不要。

| 選択肢 | 何のための手法か | この要件（運用メトリクス検証・ユーザー非公開）に合わない理由 |
|---|---|---|
| シャドウテスト | 運用メトリクスをユーザー非公開で検証 | ○（正解） |
| A/Bテスト（本番バリアントの重み） | ユーザーフィードバック・ビジネスメトリクスの検証 | 新モデルの応答がユーザーに返る＝先に公開してしまう |
| 別エンドポイント＋カスタムルーティング | 同様の分離テストは可能 | ルーティングロジックを自作・管理する必要があり運用オーバーヘッド増 |
| Model Monitor | データドリフト・品質低下の検出 | デプロイ済みモデルの監視用。異なるモデルバージョン間の運用メトリクス比較はしない。デプロイ前段階の評価には使えない |

## SageMaker Model Monitor：組み込み4種で足りないケース（NLP/埋め込みドリフト）

[security_governance.md](./security_governance.md) の「Model Monitor 4種類」（データ品質・モデル品質・バイアスドリフト・Feature Attributionドリフト）はいずれも**表形式の数値特徴量の統計分布**を前提にした組み込み監視。テキストの埋め込みベクトル（意味的な文体・トピックのドリフト）はこの4種のどれにも当てはまらず、組み込みモニターでは検出できない。

この場合は**カスタムAmazon ECRイメージを使ってModel Monitorの監視スケジュールを自作**し、エンドポイントでキャプチャした推論データ（埋め込みベクトル）とモデルのベースライントレーニングデータセットを比較する独自のドリフトメトリクス計算ロジックをコンテナ内に実装する。Model Monitor自体はスケジュール実行・S3への違反レポート保存などのマネージド基盤を提供し続けるので、「メトリクス計算だけカスタム」という位置づけ。

**Clarifyと混同しない**：ClarifyはバイアスドリフトとFeature Attributionドリフト（＝表形式データにおける特徴量重要度の変化）を検出するツールで、NLPの埋め込み意味的ドリフトの検出には使わない。

## モデル最適化技術

### 量子化（Quantization）

**モデルのパラメータの数値精度を落として軽量化する技術。**

```
65Bパラメータの場合：
  FP32（4バイト）→ 約260GB
  FP16（2バイト）→ 約130GB（1/2）
  INT8（1バイト）→ 約65GB（1/4）
  INT4（0.5バイト）→ 約32GB（1/8）
```

| 精度 | メモリ | 速度 | 品質 | 主な用途 |
|---|---|---|---|---|
| FP32 | 大 | 遅い | 最高 | 学習時 |
| FP16 | 1/2 | 速い | ほぼ同等 | 推論の標準 |
| INT8 | 1/4 | 速い | 少し落ちる | コスト重視の推論 |
| INT4 | 1/8 | 最速 | 落ちる | エッジ・軽量化 |

### 遅延ロード（Lazy Loading）

最初から全部メモリに載せない → 必要な時点でロード → 起動時間短縮・メモリ節約

### モデルシャーディング

モデルを複数GPUに分割して分散 → 1台のGPUに載らない大規模モデルを動かせる → アクセラレーターインターコネクトが必要

### 動的バッチ処理

**リクエスト数・GPU使用率を見てバッチサイズを自動調整する。**

```
固定バッチ（問題あり）：
  常にバッチサイズ=8 → リクエスト少ない時はGPU余る（無駄）

動的バッチ（正解）：
  リクエスト多い → バッチサイズ大きく → GPUフル活用
  リクエスト少ない → バッチサイズ小さく → 無駄なし
```

## KVキャッシュ

**FMが過去のトークンの計算結果を再利用するための仕組み。**

```
キャッシュなし：新しいトークン生成のたびに全過去トークンを再計算
キャッシュあり：前のトークンの計算結果（Key・Value）をメモリに保存して再利用
```

### KVキャッシュ vs Bedrockプロンプトキャッシュ

| | KVキャッシュ | Bedrockプロンプトキャッシュ |
|---|---|---|
| スコープ | **1リクエスト内** | **複数リクエスト間** |
| ヒット判定 | なし（自動蓄積） | プレフィックスのハッシュ一致 |
| TTL | リクエスト終了で消える | 5分 |
| 目的 | トークン生成の高速化 | APIコスト・レイテンシ削減 |

## SageMaker Processing Job の用途

**学習専用ではなく、汎用スクリプト実行基盤。**

```
Training Job    = モデルを学習させる（学習専用）
Processing Job  = 汎用処理（前処理・後処理・データ検証・評価など何でも）
Endpoint        = リアルタイム推論を提供
```

Processing Jobでできること：
1. **前処理**：生データ → クリーニング → 特徴量エンジニアリング
2. **後処理**：モデル出力 → 整形 → ビジネスロジック適用
3. **モデル評価**：精度指標計算・レポート生成
4. **データ検証**：入力データのバリデーション・整形

## SageMaker ML Lineage Tracking（Task 3.3）

ML の学習から推論まで全工程を DAG（有向非巡回グラフ）構造で記録する仕組み。

| エンティティ | 意味 | 例 |
|---|---|---|
| **Artifact** | 成果物の実体 | S3 データ・モデルファイル |
| **Action** | 操作・イベント | 学習・評価・デプロイ |
| **Context** | グループ・文脈 | 実験・パイプライン |

```
S3生データ(Artifact) → 学習(Action) → モデルv1(Artifact)
                                            ↓
                                  評価(Action) → メトリクス(Artifact)
                                            ↓
                                  デプロイ(Action) → エンドポイント(Artifact)
```

- `create_artifact()` / `create_action()` / `create_context()` でカスタムエンティティを追加
- `add_association()` でクロスアカウントの関連付けも可能（適切な IAM ロール設定が必要）

## SageMaker Model Cards（Task 3.3）

モデルの特性・想定ユースケース・制約・リスク・評価結果を構造化文書として記録する機能。

- `create_model_card()` / `update_model_card()` / `export_model_card()` API で自動化可能
- CI/CD パイプラインに組み込んでモデル登録時に自動生成・更新
- 規制当局への説明責任・監査対応に使う

**⚠️ Model Cards はデータリネージュ追跡ではなく「モデルの文書化」ツール**

SageMaker Model Registryはモデルパッケージの登録・承認状態が変化すると`SageMaker Model Package State Change`イベントをEventBridgeに送信する。このイベントをトリガーにLambdaを呼び出し、モデル登録時のポリシー検証を自動化できる。

## SageMaker Data Wrangler（Task 4.3）

**GUIベースのデータ準備・変換ツール。コードなしでデータのクリーニング・変換・品質検証ができる。**

### Glue Data Quality との使い分け

| | AWS Glue Data Quality | SageMaker Data Wrangler |
|---|---|---|
| **操作方法** | コード（DQDL言語） | **GUI（ノーコード）** |
| **得意** | 大規模ETL・自動化・定期実行 | 探索的データ分析・視覚的な品質確認 |
| **主な用途** | 本番パイプラインの品質ゲート | 初期データ調査・前処理フロー設計 |

### ベクトルストア用データ準備での役割

```
S3（生データ）
    ↓
SageMaker Data Wrangler（GUI）
    ├─ 欠損値・外れ値・分布の可視化
    ├─ テキストクリーニング（特殊文字除去・正規化）
    └─ 変換フローを .flow ファイルとして保存・再利用
    ↓
検証済みデータを S3 へ出力 → Bedrock Knowledge Base への取り込み
```

## SageMaker Clarify バイアス指標（Task 3.3）

| 指標 | 略称 | 意味 | 値の解釈 |
|---|---|---|---|
| ラベルにおける正の比率の差 | **DPPL** | 属性間で陽性判定率がどれだけ違うか | 0に近いほど公平 |
| 差別的効果 | **DI** | マイノリティ ÷ 多数派の陽性率 | 1に近いほど公平（0.8未満で要注意） |
| 条件付き拒否の差 | **DCR** | 実際は陽性なのに拒否された割合の差 | 0に近いほど公平 |
