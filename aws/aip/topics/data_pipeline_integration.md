# データパイプライン・統合サービス

## Glue の実態

**「データ処理に特化したパイプライン基盤」。ETLだけではない。**

```
Step Functions：汎用ワークフロー（何でもオーケストレート）
Glue：データ処理に特化したワークフロー
```

### Glue でできること

| 機能 | 内容 |
|---|---|
| ETLジョブ | 生データ → 変換 → 出力（Spark/Python） |
| ワークフロー | ジョブの依存関係・連鎖管理 |
| トリガー | スケジュール / イベント / オンデマンド |
| Data Catalog | データの場所・スキーマのメタデータ管理 |
| クローラー | S3・RDSのスキーマを自動検出 |
| **Job Bookmarks** | **増分処理（前回処理済みの続きから実行）** |

### 増分処理（Job Bookmarks）の仕組み

```
初回：全ファイルを処理 → 「ここまで処理した」を記録
2回目以降：前回記録以降の差分だけ処理
```

S3のタイムスタンプ・ファイルサイズの変化で差分を検知。

### Glueパイプラインでのベクトル化

```
Glue ETLジョブ（Python）
  → Bedrock Embeddings APIを呼び出すコードを書く
  → ベクトル生成 → OpenSearch / Knowledge Basesに書き込み
```

## Lake Formation の実態

**「データのセキュリティ・ガバナンスを一元管理する層」。**

```
Lake Formation = 鍵と入館証の管理
Data Catalog   = 建物の地図
Glue ETL       = 実際に作業する人
```

### Lake Formation × Glue の連携

```
Lake Formation
  ↓ Data Catalogを内包・管理
Glue Data Catalog（Lake Formationが管理）
  ↓ 「どこに何のデータがあるか」を把握
Glue ETLジョブ
  ↓ Data Catalogを参照して処理対象を把握
  ↓ Lake Formationの権限チェックを経てアクセス
処理実行
```

### LF-Tags（タグベースアクセス制御）

```
データ側にタグ：S3パス → tag: department=legal
ロール側にタグ：GlueジョブロールA → tag: department=legal
Lake Formation：同じタグを持つロールだけアクセス可
```

## データ統合の試験パターン

### Step Functions vs Glue の使い分け

| | Glue | Step Functions |
|---|---|---|
| 得意 | データ処理・ETLの連鎖 | 汎用ワークフロー |
| 処理の主体 | データ変換・集計 | Lambda・ECS等の呼び出し |
| 試験の文脈 | 「データパイプライン・同期」 | 「HITL・複雑なビジネスフロー」 |

## ベクトルストア更新パターン

| 要件 | 正解 |
|---|---|
| 「増分更新・変更検出」 | **Knowledge Bases**（組み込み機能） |
| 「自動同期・ワークフロー・データ系統管理」 | **Glue** |
| 「リアルタイム変更検知・即時反映」 | **Webhook / EventBridge + Lambda** |
| 「完全更新・バッチ処理」 | Batch（ただし非効率） |
| 「ドキュメント変更 → 即ベクトルストア更新」 | **DynamoDB Streams / S3イベント → Lambda** |

## S3メタデータ vs DynamoDBの使い分け

| 要件 | 正解 | 理由 |
|---|---|---|
| 「バージョニング・変更履歴が必要」 | **S3メタデータ** | S3バージョニングが自動対応 |
| 「低レイテンシー・高頻度アクセス」 | **DynamoDB** | ミリ秒レイテンシー |
| 「GSI複数・属性が広範」 | DynamoDB注意 | GSI上限20個/テーブル |

## DynamoDB Streams

**DynamoDBテーブルへの変更をリアルタイムに検知して流す仕組み。**

```
DynamoDBでデータ変更（追加・更新・削除）
  ↓
DynamoDB Streamsがイベント記録
  ↓
Lambda がトリガーされて処理
```

| | S3イベント通知 | DynamoDB Streams |
|---|---|---|
| 監視対象 | S3ファイル操作 | DynamoDBレコード変更 |
| 典型用途 | ファイルアップロード検知 | DBデータ変更の伝搬 |

## S3 オブジェクトメタデータの3種類（AIP-44）

RAG パイプラインでファイル本体を読み込まずに背景情報を取得したい場合、S3 のネイティブメタデータ機能だけで要件を満たせる。

| 種類 | 付与方法 | 用途 | 変更 |
|---|---|---|---|
| **システム定義メタデータ** | S3 が自動管理 | `Last-Modified`（投稿日時）など | 変更不可 |
| **ユーザー定義メタデータ** | `x-amz-meta-` プレフィックス付きヘッダー | 執筆者名・ライセンス等の自由テキスト | オブジェクトのコピー操作が必要 |
| **オブジェクトタグ** | キーバリューペア（最大10個） | 学問領域・カテゴリ等の分類情報 | `PutObjectTagging API` でいつでも変更可 |

### 取得方法

- **`HEAD Object API`**：オブジェクト本体をダウンロードせずにシステム/ユーザー定義メタデータを取得
- **`GetObjectTagging API`**：タグのみ取得

→ ファイル本体の取得コスト・時間なしに背景情報にアクセスできる。

### オブジェクトタグの階層構造（AIP-83）

タグは最大10個のキーバリューペアだが、キー自体を`domain.subfield.method`のように階層的に設計すれば、1個のタグで多階層の分類（科学分野→細分野→研究手法）を表現できる。

### メタデータの保存場所とアーキテクチャ上のトレードオフ（AIP-83）

| 設計 | 特徴 |
|---|---|
| S3ネイティブメタデータ（システム定義＋ユーザー定義＋タグ）に集約 | 本体とメタデータが同一オブジェクトに紐づくため同期問題が起きない |
| DynamoDB/RDS等の別DBに全メタデータを分離保存 | 本体（S3）とメタデータ（DB）が別システムになるため、S3イベント通知やStreams等での同期維持が別途必要になる |

- **SageMaker Feature Store**はML特徴量（モデルの学習・推論用の数値化された特徴量）の管理に特化したサービスで、汎用的なドキュメントメタデータの準備・格納には最適化されていない（評価データセット文脈での使用例は[model_evaluation.md](./model_evaluation.md#sagemaker-feature-store-と-bedrock-評価の組み合わせaip-79)を参照）

## AWS Glue：RAG 前処理パイプラインへの活用（AIP-61）

### AWS Glue の3大コンポーネント

| コンポーネント | 役割 | AIP-61 要件との対応 |
|---|---|---|
| **Glue クローラー** | S3などのデータソースを自動スキャン → Glue Data Catalog にスキーマ・パーティション・更新履歴を自動登録 | 一元的なメタデータ管理と追跡性 |
| **Glue ETL ジョブ** | Python Shell / PySpark ベースのカスタム変換スクリプトを実行（サーバーレス、DPU 従量課金） | テキスト分割ロジックのカスタマイズ |
| **Glue Data Quality** | DQDL（Data Quality Definition Language）で宣言的に品質ルールを定義、ETLジョブ内に組み込み可能、CloudWatch 連携でアラート | データ品質ルール定義と継続的監視 |

### Glue Data Catalog の特徴
- Hive メタストア互換のリポジトリ
- テーブル定義・カラム情報・データ型・ロケーション・更新履歴などを一元管理
- **Athena / Redshift Spectrum / EMR** からも参照可能（メタデータの再利用性が高い）
- S3 オブジェクトタグ（簡易キーバリューのみ）と比べてはるかにリッチなメタデータ管理が可能

### DQDL ルールの例
```
Rules = [
  Completeness "content" >= 0.99,    # 欠損値チェック
  Uniqueness "doc_id" >= 1.0,        # 重複チェック
  ColumnLength "chunk_text" >= 50    # チャンク長チェック
]
```

### Glue ETL ジョブが特に有効なユースケース
- テキストチャンキング（段落・見出し・トークン長考慮の分割）
- 医療記録のPII マスキング
- 文書メタデータ（著者・日付・カテゴリ）の抽出と付与
- 重複文書の除去
- エンコーディング統一・文字化け修正

## Kinesis ファミリーの使い分け（Domain 5 Task 5.1 で再確認）

| サービス | 役割 | 処理能力 | 典型ユースケース |
|---|---|---|---|
| **Kinesis Data Streams (KDS)** | 高スループットなストリーム取り込み・転送（**配管役**）。順序保証・リプレイ（最大365日保持）・複数コンシューマ | **変換なし**。処理は下流（Lambda/KCL）が担当 | 全インタラクションを高スループットで流し込み横断分析、取引/センサーデータ |
| **Kinesis Data Firehose** | S3 / Redshift / OpenSearch 等への配信（バッファリングのニアリアルタイム） | **Lambdaで軽度な変換**（整形・フィルタ）可 | ログをS3へ配信して Athena 分析 |
| **Managed Service for Apache Flink**（旧 Kinesis Data Analytics）| リアルタイムのストリーム分析 | **集計・ウィンドウ処理・データ整形**（本格処理）| リアルタイム集計・異常検知 |

### フィードバック収集：DynamoDB vs Kinesis の境界（重要）

| やりたいこと | 適切な構成 |
|---|---|
| **離散的なフィードバック**（👍/👎・星評価を受け取って保存）| **API Gateway + Lambda + DynamoDB** |
| **高ボリュームの連続的なやり取りストリームをキャプチャして“パターン分析”** | **Kinesis Data Streams** |

- 「ユーザー評価を保存するだけ」なら **Kinesis は過剰（オーバーエンジニアリング）**。DynamoDB で十分
- 「全インタラクションのパターンをリアルタイムで横断分析・複数の下流で消費」なら Kinesis が妥当
- ※ KDS 自体はデータ整形をしない。「ストリームの整形・集計」は Flink（旧 Analytics）の領域＝**別レイヤー**
