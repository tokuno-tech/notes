# Bedrock ガードレール / プロンプト管理 / モニタリング

## ガードレール多層防御・監査・耐障害性（AIP-18）

### 正解：Bedrockガードレール（プロンプト攻撃フィルター）+ ガードレールプロファイル（クロスリージョン）+ CloudWatch Logs

3つの要件と対応するコンポーネント：

| 要件 | 正解の対応 |
|---|---|
| プロンプトインジェクション・ジェイルブレイクへの多層防御 | ガードレールの**プロンプト攻撃フィルター**（MLベース）+ コンテンツフィルター「高」 |
| 介入理由・ポリシー違反の詳細を監査ログに記録 | **モデル呼び出しログ記録** → CloudWatch Logs |
| リージョン障害時もガードレール評価が継続 | **ガードレールプロファイル**でクロスリージョンガードレール推論を有効化 |

---

### 怪しかったポイント①：プロンプト攻撃フィルター

通常のコンテンツフィルター（暴力・ヘイトなど）とは別に、**プロンプトインジェクション専用のMLベースフィルター**が存在する。

```
コンテンツフィルター（カテゴリ）
  → 暴力・ヘイト・性的・差別的表現の検出
  → NONE/LOW/MEDIUM/HIGH の4段階

プロンプト攻撃フィルター（← これが重要）
  → プロンプトインジェクション・ジェイルブレイクをMLで検出
  → 正規表現や単語リストでは検出できない高度な攻撃に対応
  → コンテンツフィルターと組み合わせることで多層防御を実現
```

**引っかけ：Lambda + 正規表現**はルールベースのため、巧妙に偽装されたプロンプトインジェクションを検出できない。

---

### 怪しかったポイント②：CloudTrail vs CloudWatch Logs（モデル呼び出しログ）

```
CloudTrail
  = AWS APIの操作ログ（誰がいつどのAPIを呼んだか）
  = 「InvokeModel APIが呼ばれた」という事実のみ
  = ガードレールが"なぜ"ブロックしたかの詳細は含まれない

モデル呼び出しログ記録 → CloudWatch Logs
  = 入力プロンプト・出力・ガードレールの介入詳細をフルで記録
  = ブロック理由・違反したポリシー・編集内容まで追跡可能
  = コンプライアンス監査に必要な情報はこちら
```

→ 「判定理由・ポリシー違反の詳細」が要件なら**CloudWatch Logs（モデル呼び出しログ）**が正解。CloudTrailは不十分。

---

### 怪しかったポイント③：クロスリージョンガードレール推論（ガードレールプロファイル）

```
ガードレールプロファイル + クロスリージョンガードレール推論
  = ガードレール評価そのものを複数リージョンで分散処理
  = リージョン障害時も別リージョンでガードレール評価が継続
  = Bedrockネイティブの耐障害性機能

Route 53 フェイルオーバー / Global Accelerator
  = DNSやネットワークレベルでトラフィックを別リージョンに転送
  = ガードレール評価は転送先でゼロから動かす必要がある（準備が必要）
  = ガードレール評価の継続を「保証」する仕組みではない
```

→ 「ガードレール評価が継続できる」という要件は、ネットワーク層のフェイルオーバーではなく**ガードレール推論自体のクロスリージョン分散**で解決する。

### 不正解パターン整理

| 選択肢 | 不正解の理由 |
|---|---|
| Lambda + 正規表現 + ガードレール + CloudTrail + Global Accelerator | 正規表現は高度な攻撃を検出できない。CloudTrailはガードレール介入詳細を記録しない |
| ガードレール（拒否トピック・単語フィルターのみ）+ CloudTrail + Route 53 | プロンプト攻撃フィルター未使用。CloudTrailでは介入理由不十分。Route 53はガードレール継続を保証しない |
| ガードレール + SageMakerカスタム分類 + 各リージョンSMエンドポイント + S3 | 過剰なカスタム実装。Bedrockネイティブ機能で要件を満たせるのにオーバーエンジニアリング |

### 正解：OpenSearch Service + k-NN

**「表現は異なるが同じ意図」の問い合わせを検出するには、意味ベクトルの類似検索が必要。**

```
新しい問い合わせ → 埋め込みモデルでベクトル化
  ↓
OpenSearch k-NN で近似最近傍探索
  ├─ 類似度が閾値以上 → キャッシュから応答を返す（モデル呼び出しなし）
  └─ 類似度が閾値未満 → Bedrockを呼び出し → 応答を新規キャッシュに追加
```

### 各選択肢の比較と間違えた理由

| 選択肢 | 結果 | 理由 |
|---|---|---|
| **OpenSearch k-NN** | ✅ 正解 | ANN専用アルゴリズム（HNSW・IVF）でミリ秒検索。意味ベクトル検索に最適化 |
| **Aurora pgvector** | ❌ 不正解（選んだもの） | ベクトル検索はできるが、RDB上でのSQL ORDER BYは大規模ANNに最適化されていない。「応答時間の短縮」要件でOpenSearchに劣る |
| **ElastiCache Redis** | ❌ 不正解（選びそうになった） | ハッシュ値の完全一致のみ。「表現が違う同じ意図」を検出できない |
| **DynamoDB + Comprehend** | ❌ 不正解 | Comprehendのキーフレーズ抽出は構文レベル。意味全体をベクトルで捉えるのとは別物 |

### pgvector vs OpenSearch k-NNの核心的な違い

```
pgvector（Aurora PostgreSQL）
  → RDBのクエリエンジン上で動作
  → SQL ORDER BY + 距離関数 = 全件スキャンに近い処理になりやすい
  → 既存PostgreSQLに追加する場合に便利

OpenSearch k-NN
  → HNSW・IVFなどANN専用アルゴリズムをネイティブサポート
  → 数百万件規模でもミリ秒単位の検索
  → 「応答時間を可能な限り短縮」要件に最適
```

### AIP出題での識別ポイント

- 「意味的に近い」「表現は違うが同じ意図」→ ベクトル検索が必要
- 「完全一致」「ハッシュ」→ Redis/ElastiCacheの引っかけ
- 「キーフレーズ」「キーワード」→ Comprehendの引っかけ（構文レベル≠意味レベル）
- 「低レイテンシー」「大規模」→ pgvectorよりOpenSearch k-NN

### 正解構成

**S3（データセット） + EventBridge + Lambda + Bedrock評価ジョブ + CloudWatch**

```
カスタムデータセット（JSONL形式）→ S3に格納
  ↓
EventBridge（定期スケジュール）→ Lambda起動
  ↓
Lambda → CreateEvaluationJob API呼び出し（S3のURI指定）
  ↓
Bedrock自動モデル評価ジョブ実行
  ↓
別Lambda（ジョブ完了検知）→ CloudWatchにメトリクス送信
  ↓
CloudWatch Logs Insightsダッシュボードで可視化
```

### 重要ポイント

| 項目 | 内容 |
|---|---|
| データ形式 | **JSONL形式**でS3に格納（DynamoDBは不可） |
| データアクセス | 評価ジョブ実行ロールに**S3読み取り権限のIAMポリシー**が必要 |
| CORS | WebUI（ブラウザ）からS3オブジェクトを直接参照する場合に必要 |
| テキスト生成の評価指標 | **RWK（Real-World Knowledge）スコア**：事実の正確性を評価 |
| 要約の評価指標 | **BERTスコア**：意味的な類似度を評価（同義語・言い換えにも対応） |

### CORSとは

ブラウザがデフォルトで**異なるオリジンへのリクエストをブロック**する。S3側でCORS設定を入れることで、特定のWebアプリのオリジンからのアクセスを許可できる。

```
IAMポリシー = AWS（サーバー）レベルのアクセス制御
CORS設定    = ブラウザレベルのアクセス制御（別オリジンからの通信許可）
→ 両方必要（役割が違う）
```

### 評価指標の使い分け（引っかけ対策）

| 指標 | 用途 | AIPでの出題 |
|---|---|---|
| **RWK** | テキスト生成の事実正確性 | ✓ 正解 |
| **BERTスコア** | 要約の意味的類似度 | ✓ 正解 |
| BLEU | 機械翻訳評価（テキスト生成正確性には不向き） | ✗ 不正解選択肢 |
| ROUGE | 要約評価に使えるがBERTより粗い | ✗ 不正解選択肢 |

### 不正解パターン

| 選択肢 | 不正解の理由 |
|---|---|
| DynamoDB格納 | Bedrock評価ジョブはS3のJSONLのみ対応。DynamoDBから直接読めない |
| SageMaker Processing + カスタムコンテナ + OpenSearch | インフラ管理コストが大幅増。「運用負荷最小化」に反する |
| Step Functions + InvokeModel + Comprehend + QuickSight | Comprehendは品質スコアリングツールではない。QuickSightはCWより重い |

### 概要
Amazon Bedrock のネイティブ機能として、プロンプトテンプレートを一元管理する仕組み。コード変更なしにプロンプトを更新・バージョン管理できる。

### プロンプト管理でバージョン管理される内容（AIP-67 補足）

```
プロンプト管理でバージョン管理されるもの：

① プロンプトテンプレート本文（文字列）
② 推論パラメータ（！← ここが重要）
   - temperature（出力の多様性。0に近いほど決定論的 = 再現性UP）
   - top_p（確率的サンプリングの範囲）
   - max_tokens / top_k なども含む
③ 使用モデルID
④ バージョン番号・タイムスタンプ・変更者

承認ワークフロー：
  プロンプトの変更 → レビュー・承認 → 本番反映
  → 意図しない変更を防止。全環境で同一プロンプト+パラメータを保証

出力の再現性（AIP-67 要件）との関係：
  temperature = 0 に近い値をバージョン管理 → 同一入力に対して同一出力が安定して得られる
  ≠ DynamoDBキャッシュ（表記ゆれのある自然言語には対応不可）
```

### WAF との違い（AIP-67 引っかけ）

```
AWS WAF：
  - HTTP/HTTPSリクエストのフィルタリング（Webアプリ向け）
  - SQLインジェクション・XSS・IP制限などに対応
  - 「モデルが生成した出力の内容」は検査できない
  - 入力のHTTPリクエスト層でしかブロックできない

Bedrock ガードレール（セマンティック拒否ルール）：
  - 入力プロンプト + モデル出力の両方を評価
  - 意味レベルで危険なコンテンツを検知・ブロック
  - 「アレルギー誘発食品を含む食事プラン」のような
    文章の意味を理解してブロックできる（WAFにはできない）
```

### バリアント（Variant）とは

**共通のベーステンプレートから、条件に応じて異なるバージョンのプロンプトを定義する機能。**

```
共通テンプレート（全部門共通フォーマット）
  ├─ バリアント：営業部門用（親しみやすいトーン）
  ├─ バリアント：コンプライアンス部門用（厳格・法的トーン）
  └─ バリアント：経理部門用（数値重視・簡潔なトーン）
```

- 各バリアントはトーン・言い回し・制約条件などを個別に定義できる
- 部門IDなどのコンテキストを渡すことで、対応するバリアントが選択される
- テンプレート自体はBedrock側で管理されるため、コード変更不要で更新可能

### AIP-15の正解構成

**Bedrock プロンプト管理（バリアント） + Bedrock ガードレール**

| 要件 | 対応する機能 |
|---|---|
| 部門ごとに異なるトーン | プロンプト管理の**バリアント**で各部門向けを定義 |
| 統一フォーマット（要点・リスク評価・注意事項） | ベーステンプレートに統一フォーマットを定義 |
| 暴力的・差別的表現の排除 | ガードレールの**コンテンツカテゴリフィルター** |
| PIIが回答に含まれない | ガードレールの**機密情報フィルター** |
| 運用中に柔軟に変更できる | ガードレールのポリシーはコンソールから随時更新可能 |
| 運用チームの負担を最小化 | Bedrockネイティブ機能のみで完結（外部サービス不要） |

### ガードレールのフィルタータイプ整理

| フィルター | 用途 |
|---|---|
| **コンテンツカテゴリフィルター** | 暴力・ヘイト・性的表現などの有害カテゴリを強度設定でブロック |
| **単語フィルター** | 特定の禁止ワードリストでブロック |
| **機密情報フィルター（PIIフィルター）** | 氏名・電話番号・クレジットカード番号などのPIIをマスクまたはブロック |
| **トピック拒否ポリシー** | ビジネス上で扱いたくないトピック（例：競合他社の話題）を拒否 |
| **グラウンディングチェック** | ハルシネーション検出（ソース文書との一致度チェック） |

### 引っかけ選択肢との比較

| 選択肢 | 不正解の理由 |
|---|---|
| S3にJSONテンプレート + Lambda + Comprehend | 応答生成**後**にフィルタリング → PIIや有害表現が一度生成される。Comprehendは汎用NLPで専用フィルタではない |
| SageMaker部門別エンドポイント + Lambda + Macie | Macieはデータレイク・S3上のPII検出用。リアルタイムのLLM出力検査には向かない。インフラ管理コスト大 |
| プロンプト管理 + Step Functions + Comprehend併用 | Step Functionsは不要な複雑さを追加。ComprehendはBedrockガードレールと比べて機能が限定的 |

## PromptOps / Prompt as Code（AIP-69）

### PromptOps とは

**MLOps（機械学習の運用管理プラクティス）をプロンプトエンジニアリングに適用**したもの。

従来のソフトウェア開発では「コードをGitで管理→テスト→CI/CDでデプロイ」するのが当たり前。  
LLMアプリでは**プロンプト自体がシステムの挙動を決める核心**であるため、プロンプトにも同じ規律を適用しようという考え方。

```
【従来のソフトウェア開発】
コード変更 → Git管理 → 自動テスト → CI/CDデプロイ → 品質保証

【PromptOps】
プロンプト変更 → Git管理 → 自動評価テスト → CI/CDデプロイ → 品質保証
```

### なぜ必要か

LLMアプリの品質はプロンプトで決まる。しかしプロンプトは：

- 少し変えるだけで出力が大きく変わる（非決定性）
- 「なんとなく良くなった」では再現性が担保できない
- 誰がいつ何を変えたか追跡できないと障害調査ができない
- 新バージョンが旧バージョンより本当に良いか定量的に証明できない

→ これらを解決するのが PromptOps

### PromptOps の4要素

| 要素 | 内容 |
|---|---|
| **バージョン管理** | Gitでプロンプト・パラメータ・モデルIDを管理。差分・履歴・ロールバック可能 |
| **評価データセット** | 問題が起きるケース（エッジケース）を含むテストデータを整備 |
| **定量的評価指標** | F1スコア・ROUGEなど客観的な数値で品質を測る |
| **自動化パイプライン** | CI/CDでプロンプト変更のたびに自動評価→劣化検知→マージ制御 |

### Bedrock プロンプト管理との使い分け

| | Bedrock プロンプト管理 | Git + CI/CD（PromptOps） |
|---|---|---|
| **バージョン管理** | ○（Bedrock内で管理） | ◎（完全な差分・履歴） |
| **承認ワークフロー** | ◎（組み込み済み） | △（別途実装が必要） |
| **自動評価テスト** | △（手動プレイグラウンド） | ◎（CI/CDで自動化） |
| **継続的品質追跡** | △ | ◎ |
| **向いている場面** | 本番プロンプトの安全な変更管理 | 品質の継続的測定・原因分析が必要な場合 |

> 「泥臭い」が、品質・再現性・変更追跡の全要件を最も確実に満たす方法。
> Bedrock プロンプト管理は便利だが、**継続的な自動テストと完全な変更履歴**の両立には Git + CI/CD が優れる。

### なぜ Git + CI/CD が選ばれるか（AIP-69 の3要件との対応）

| 要件 | Git + CI/CD での対応 |
|---|---|
| **精度低下の原因特定** | 複雑なテストケースを含むデータセットで自動テスト → 失敗パターンを自動抽出 |
| **定量的指標で継続追跡** | CI/CDがプロンプト変更のたびにスコアを自動記録・過去バージョンと比較 |
| **変更履歴の完全保持** | Gitのコミット履歴（誰が・いつ・何を・なぜ変えたか）が永続記録される |

### CI/CDパイプラインのイメージ

```
プロンプト変更 → git commit → CI/CD起動（CodePipeline / GitHub Actions）
                                   ↓
                  問題が起きる複雑なサンプルも含むデータセットで自動テスト
                                   ↓
                  分類精度・F1スコアなどを自動記録・過去バージョンと比較
                                   ↓
                  スコアが劣化 → マージブロック / スコア維持 → マージ許可
```

### ⚠️ 引っかけ選択肢の落とし穴

| 選択肢 | 何が足りないか |
|---|---|
| **Bedrock モデル評価ジョブ + QuickSight** | モデル評価ジョブは「FM同士の比較」用（Claude vs Titan等）。プロンプトの継続追跡ではない。目視判断で自動化なし |
| **Bedrock プロンプト管理 + 手動検証** | テストが「単純サンプル」のみ→問題の原因（複雑ケース）を特定できない。変更履歴が不完全 |
| **SageMaker エンドポイント + Athena** | プロンプト変更ごとにエンドポイント構築は過剰。手動クエリ集計で継続自動追跡なし |

### ⚠️ AIP-69 vs AIP-74 の判断分岐（よく似た問題）（AIP-74）

```
「プロンプト管理の問題」が出たら、まず要件を確認：

AIP-69 パターン（Git + CI/CD が正解）：
  ✅ 「継続的な自動テスト」が必要
  ✅ 「完全な変更履歴（誰が・いつ・何を・なぜ）」が必要
  ✅ 品質の定量的継続追跡が必要
  → Bedrock プロンプト管理は自動テスト・継続追跡が弱い

AIP-74 パターン（Bedrock プロンプト管理 + CloudTrail + IAM が正解）：
  ✅ 「バージョン管理・監査証跡・承認制御・変数化」がすべて必要
  ✅ 「運用オーバーヘッド最小」が明示的な要件
  ✅ カスタム開発なしで済ませたい
  → Bedrock プロンプト管理のネイティブ機能で一括解決できる
```

**キーワード判断**：
- 「継続的自動テスト」「定量的品質追跡」→ Git + CI/CD
- 「運用オーバーヘッド最小」「承認統制」「変数化」→ Bedrock プロンプト管理

**AIP-74 の承認統制**: IAM ポリシーで `bedrock:CreatePrompt`（作成）と `bedrock:CreatePromptVersion`（公開）を別ロールに分離 → 作成者は公開できず、承認者のみ公開可能。追加実装不要。

### ⚠️ CodeCommit は2024年新規アカウントへの提供停止

選択肢に `AWS CodeCommit` が出てきたら不正解サインになりうる。長期運用が前提の問題では特に。

### 評価メトリクスの例

- **分類タスク**: カテゴリ別 精度・再現率・F1スコア
- **要約タスク**: ROUGE（網羅性）、BERTScore（意味的類似度）
- **RAGタスク**: 事実整合性スコア（ソース文書との矛盾検出）

# Bedrock ガードレール / モニタリング

## ハルシネーション検出・監査構成

### 全体像

```
入力コンテキスト（ソース文書）
        ↓
  Bedrock ガードレール（文脈的グラウンディングチェック）
        ↓ グラウンディングスコアがしきい値未満 → ブロック or フラグ付け
  モデル呼び出し
        ↓
  出力ログ → S3（監査・事後分析）
        ↓
  CloudWatch メトリクス（InputTokenCount / OutputTokenCount）
        ↓
  CloudWatch 異常検出アラーム
```

---

## 各コンポーネント詳細

### 1. モデル呼び出しログ（S3保存）

- すべての入出力を監査・事後分析できる基盤
- **テキスト入出力の記録はオプション設定**で、明示的に有効化が必要
- 保存先は S3 または CloudWatch Logs

---

### 2. Bedrock ガードレール：文脈的グラウンディングチェック

ハルシネーション（根拠のない生成）をリアルタイムで検出・対処する機能。

- モデルへの入力コンテキスト（ソース文書）と生成された出力を比較し、出力がソースに根拠を持つかどうかを評価
- **グラウンディングスコア**（0〜1）を設定し、しきい値を下回ると応答をブロックまたはフラグ付け
- リクエスト処理パイプラインに組み込まれるためリアルタイムで機能する

#### ガードレールで設定できるポリシータイプ（組み合わせ可）

| ポリシー | 内容 |
|---|---|
| 文脈的グラウンディング | ハルシネーション検出 |
| コンテンツフィルタリング | 有害コンテンツのブロック |
| トピック拒否 | 特定トピックへの応答を拒否 |
| PII検出 | 個人情報の検出・マスク |
| 機密情報フィルター | PII等の検出・ブロック・マスク |

---

### 3. CloudWatch 異常検出アラーム（トークン使用量）

Bedrock は以下のメトリクスを CloudWatch に自動発行する：

- `InputTokenCount`
- `OutputTokenCount`

**異常検出アラーム**の特徴：
- 過去のメトリクスデータに機械学習アルゴリズムを適用
- 時間帯・曜日・トレンドを考慮した「正常な変動帯」を自動学習
- 変動帯から逸脱した場合にアラームが発火
- 静的しきい値では捉えられない**動的な異常パターン**も検出できる

#### 注意点

- 異常検出は**最低2週間分のメトリクスデータ**があると精度が向上
- 導入直後はモデルの学習が不十分なため誤検知が発生する可能性あり
- Bedrockの料金はトークン数ベースのため、トークン監視は**コスト管理にも直結**

## マルチモーダルデータ × RAG + ガードレール構成（AIP-10）

### 要件

- 異なる形式のデータ（映像・音響・テキスト）を活用した自然言語QA
- 観測事実に基づかない推測的な回答を防止
- 操作履歴を監査目的で記録

### 正解構成

```
映像 / 音響 / 観察記録
  ↓
SageMaker Processing（データ前処理・正規化・構造化）
  ↓
Bedrock ナレッジベース（RAG）
  ↓ ベクトル検索 → 関連情報をコンテキストに挿入
Bedrock FM（回答生成）
  ↓
Bedrock ガードレール（コンテキストグラウンディングチェック）

監査：CloudTrail
設定管理：AWS AppConfig（プロンプトテンプレート）
```

### 各コンポーネントの役割

| コンポーネント | 役割 |
|---|---|
| **SageMaker Processing** | カスタムスクリプト/コンテナで大規模データ前処理を実行するマネージドジョブ環境 |
| **Bedrock ナレッジベース（RAG）** | 観測データをベクトル化し、クエリに意味的に近い情報を検索してコンテキストとしてFMに渡す |
| **ガードレール（コンテキストグラウンディングチェック）** | 回答が提供されたコンテキスト（検索結果）に基づいているかを検証。根拠のない推論・憶測をリアルタイムで抑止 |
| **AWS AppConfig** | アプリ設定の管理・デプロイサービス。プロンプトテンプレートのバージョン管理・段階的ロールアウトに使う |
| **CloudTrail** | `InvokeModel` / `Retrieve` / `RetrieveAndGenerate` などのAPI呼び出しを記録。誰がいつどのモデルにリクエストを送ったかを追跡 |

### ハルシネーション抑制の二重構造

RAGだけでは完全に防げないため、ガードレールのグラウンディングチェックを併用するのが推奨。

### 試験頻出の引っかけ：SageMaker Model Monitor との混同

| サービス | 用途 |
|---|---|
| **SageMaker Model Monitor** | SageMakerにデプロイされたMLモデルのデータドリフト・品質劣化を**統計的に監視**する |
| **Bedrock ガードレール** | 生成AIの回答が推測的かどうかを**リアルタイムに判定・制御**する |

→ 「生成AIの出力が推測的かどうかを制御したい」ならガードレール。Model Monitor は生成AIの出力フィルタリングには使えない。

### CloudTrail vs Bedrock モデル呼び出しログの使い分け

| | CloudTrail | モデル呼び出しログ |
|---|---|---|
| 記録内容 | 誰がいつどのAPIを呼んだか（メタデータ） | 入力プロンプトと出力レスポンスの全文 |
| 用途 | 操作監査・ガバナンス | 詳細な監査・デバッグ |
| 保存先 | S3 / CloudWatch Logs | S3 / CloudWatch Logs |

# Amazon Bedrock Evaluations（AIP-21 / AIP-29）

## サービス概要

**Amazon Bedrock Evaluations** は、Bedrock 上に構築した生成AIシステムの品質を定量的に評価するためのフルマネージドサービス。モデル評価ジョブを作成するだけで、評価データセットに対するスコアリングを自動実行できる。

```
評価の2大モード
  ├─ 自動評価（LLM-as-a-judge）← 試験頻出
  │     評価者モデル（Claude Sonnet等）がスコアリング
  │     大量・高速・一貫した基準で実施可能
  └─ 人間による評価
        評価者チームが直接レビュー
        ニュアンス判断に優れるが自動化不可
```

---

## RAG 評価ジョブの2種類（重要）

RAG システムを評価する場合、**ジョブタイプが2種類**あり、評価できる範囲が根本的に異なる。

### retrieve-only（取得のみ評価）

```
クエリ → ナレッジベース検索 → [取得されたチャンク] ← ここまでを評価
                                             ↓
                                         （FM の回答生成は評価対象外）
```

- **評価できるもの**：検索フェーズの品質のみ
- **主な用途**：チャンキング戦略・埋め込みモデルのチューニング段階
- **評価できないもの**：FM が生成した回答の正確性・完全性・忠実性

### retrieve-and-generate（取得 + 生成を一括評価） ← 本番前の総合評価に使う

```
クエリ → ナレッジベース検索 → FM による回答生成 → [検索品質 + 回答品質] を両方評価
```

- **評価できるもの**：検索フェーズ + 生成フェーズの両方
- **主な用途**：本番リリース前の総合品質評価・複数 FM / チャンキング戦略の比較
- **評価できるもの**（詳細）：

| フェーズ | 組み込みメトリクス | 意味 |
|---|---|---|
| 検索（Retrieval） | Context Relevance（コンテキスト関連性） | 取得チャンクがクエリに関連しているか |
| 検索（Retrieval） | Context Coverage（コンテキストカバレッジ） | 回答に必要な情報が取得できているか |
| 生成（Generation） | Correctness（正確性） | 回答が事実に基づいているか |
| 生成（Generation） | Completeness（完全性） | 回答が質問に対して十分か |
| 生成（Generation） | Faithfulness（忠実性） | 回答が取得コンテキストを逸脱していないか ← RAG で最重要 |
| 生成（Generation） | **Harmfulness（有害性）** | 有害・不適切な内容が含まれていないか |

### RAG評価指標のポイント（AIP-79）

```
Faithfulness（根拠整合性）：
  「生成された回答が参照文書に忠実か」を測定
  → ハルシネーション検知に直接対応する最重要指標
  → 参照文書に載っていない情報を回答に含めていないか

Relevance（検索関連性）：
  「取得された文書がクエリに関連しているか」を測定
  → 検索フェーズの品質指標

Correctness（正確性）：
  「回答が事実に基づいているか」を測定

Harmfulness（有害性）：
  「有害・不適切な内容が含まれていないか」を測定
  → コンテンツ安全性の指標
```

### SageMaker Feature Store と Bedrock 評価の組み合わせ（AIP-79）

```
SageMaker Feature Store とは：
  機械学習の「特徴量（Feature）」を一元管理するストア
  オンラインストア（低レイテンシー）+ オフラインストア（S3ベース）の2層構造
  
Bedrock 評価との接続：
  評価用データセット（プロンプト・参照文書・期待応答のセット）を
  Feature Store に保存して管理できる
  
  ① Feature Store に評価セットを格納（バージョン管理付き）
      ↓
  ② Bedrock model evaluation job が Feature Store から評価データを読み込む
      ↓
  ③ Faithfulness / Relevance / Correctness 等のメトリクスを自動算出
  
メリット：
  ・評価データセットのバージョン管理・再現性を確保
  ・複数の評価実行間でデータセットを一元管理
  ・評価の差分比較が容易（モデル更新前後のスコア比較等）
```

## Amazon Kendra とその関連性スコア（AIP-79 引っかけ）

### Kendra とは

**ML ベースのエンタープライズ検索サービス。** 自然言語の質問に対して関連する文書や回答を返す。

### Kendra の関連性スコアの意味

```
Kendra が算出する関連性スコア：
  「この文書/回答がクエリにどれだけ関連しているか」を示すスコア
  
  つまり：
  「検索クエリと文書の一致度」を測定するもの
  
  ⚠️ 「LLMが生成した回答がハルシネーションを含んでいるか」は判定できない
  ⚠️ 「参照文書に忠実かどうか（Faithfulness）」も判定できない
  
  検索の質 ≠ 生成の質
```

### 試験での引っかけパターン（AIP-79）

```
❌ 「Kendra で関連性スコアを算出してハルシネーション検知」
   → Kendra のスコアは検索関連性のみ
   → 生成フェーズのFaithfulness評価はできない
   
✅ 「Bedrock 組み込み評価の Faithfulness メトリクスでハルシネーション検知」
   → 生成された応答が参照文書に忠実かを直接測定
```

## Amazon CloudWatch Synthetics（AIP-79 引っかけ）

### 一言で

> **Canary（カナリア）スクリプトを定期実行してAPI・Webアプリの可用性・パフォーマンスを合成監視するサービス。**

### 何をするか

```
Canary スクリプト（Node.js / Python）を定期実行：
  ├─ HTTP エンドポイントの死活監視
  ├─ API の正常動作確認（事前定義のレスポンスと比較）
  ├─ Webサイトのページ読み込みチェック
  └─ レイテンシー・エラー率の測定

→ 結果をCloudWatch メトリクスに記録
→ しきい値超過でアラーム発報
```

### RAG品質評価に使えない理由（AIP-79）

```
CloudWatch Synthetics が得意なこと：
  ✅ APIの可用性・レスポンス確認
  ✅ 事前定義した正解との一致チェック（固定パターン）

CloudWatch Synthetics が苦手なこと：
  ❌ 動的に変化するLLM応答のハルシネーション検知
  ❌ 意味的な品質評価（根拠整合性・正確性等）
  ❌ RAG特有の品質指標（Faithfulness等）の算出
  ❌ 新しいクエリパターンへの対応（固定テストケースのみ）
  
→ 「固定テストケース × 事前定義正解」との一致率しか測れない
  本番ユーザーの多様なクエリに対するハルシネーションは検知不可
```

## LLM-as-a-judge（評価者モデルによる自動採点）

人間評価者の代わりに LLM が採点する手法。

```
評価対象 FM の回答 ─────────────────────┐
評価データセット（質問 + 正解）─────────┼→ 評価者モデル（Claude Sonnet） → スコア
採点基準（ルーブリック）────────────────┘
```

- **評価者モデルとして指定できるのは特定モデルのみ**（Claude Sonnet 等）
- 評価対象 FM と評価者モデルは別々に指定（自己評価ではない）
- **カスタムメトリクス**：1〜5 スケール等、ドメイン固有の採点基準を自然言語で定義できる
  - 例：「規制文書への準拠度を 1〜5 でスコアリングせよ」

---

## 品質ゲート（Quality Gate）

メトリクスのしきい値を設定し、スコアが基準を下回る構成を自動排除する仕組み。

```
評価ジョブ実行 → 各メトリクスのスコア算出
  ├─ 全メトリクスがしきい値以上 → 本番デプロイ候補
  └─ いずれかがしきい値未満   → 自動排除（デプロイしない）
```

- CI/CD パイプラインと統合することで、デプロイ前の自動品質チェックとして機能
- **EventBridge + Lambda + CloudWatch** でジョブ完了を検知し、スコアを通知・記録（AIP-16 の正解パターン）

## AIP-21 正解構成まとめ

| 要件 | 対応する仕組み |
|---|---|
| 異なるドキュメント分割方式の検索精度比較 | retrieve-and-generate ジョブに各チャンキング結果を評価データセットとして含める |
| 2 つの FM の生成品質を定量比較 | 同一評価設定で 2 つの FM を個別に実行 |
| 検索 + 生成の両メトリクスを測定 | retrieve-and-generate（取得・生成両フェーズを一括評価） |
| カスタムスコアリング基準 | LLM-as-a-judge + カスタムメトリクス（1〜5 スケール） |
| 基準未満の構成を自動排除 | **メトリクスのしきい値 = 品質ゲート** |
| 評価者モデル | Anthropic Claude Sonnet（評価者として指定） |

---

## 不正解パターン（AIP-21）

| 選択肢 | 不正解の理由 |
|---|---|
| retrieve-only ジョブ | 生成フェーズを評価できない。「FM の回答品質を定量測定」要件を満たせない |
| 人間による評価ワークフロー | 「自動的に排除する」要件に対応不可。しきい値による自動判定ができない |
| SageMaker Clarify + ROUGE/BERTScore | RAG 固有の取得メトリクスを測定できない。ROUGE/BERTScore は表面的類似度で忠実性評価に不十分 |

### ROUGE / BERTScore vs LLM-as-a-judge の比較

| 指標 | 測定するもの | 問題点（RAG 評価において） |
|---|---|---|
| ROUGE | n-gram の一致率（単語の重なり） | 「正解テキスト」との比較が前提。忠実性は測れない |
| BERTScore | 文の意味的類似度 | 参照テキストとの比較。「原文コンテキストに基づいているか」は測れない |
| LLM-as-a-judge | 採点基準（ルーブリック）に基づく多面的評価 | ドメイン固有基準・忠実性・完全性を柔軟に評価できる |

## Amazon A2I（Augmented AI）と Bedrock 評価の組み合わせ（AIP-29）

### サービス概要：Amazon Augmented AI（A2I）

**人間によるレビューワークフローをマネージドで提供するサービス。**

```
通常の自動評価
  ↓ スコアが低い / ガードレールでフラグ付き
Amazon A2I ワークフロー
  ↓ 専門家へタスクを割り当て
人間レビュー（承認 / 差し戻し / 修正）
  ↓
監査証跡として記録
```

- **タスク管理**：専門家にレビュータスクを割り当て・追跡できる
- **監査証跡**：全レビュー結果が永続的に記録（メール通知と異なり後追いが可能）
- **ルーティング条件**：スコアしきい値やガードレールのフラグに基づいて自動振り分け

### AIP-29 正解構成

**Bedrock model evaluation（LLM-as-a-judge）+ Guardrails + Amazon A2I**

| 要件 | 対応する仕組み |
|---|---|
| 日々数万件を効率的に自動評価 | Bedrock model evaluation（Claude Sonnet が judge） |
| 保険業法コンプライアンス違反をリアルタイム検出・ブロック | Guardrails for Amazon Bedrock（ポリシー違反フィルター） |
| リスクの高い対話に絞った専門家レビュー | Amazon A2I ワークフロー（スコア低下 / ガードレールフラグの対話のみ送付） |

### 不正解パターン（AIP-29）

| 選択肢 | 不正解の理由 |
|---|---|
| Comprehend センチメント分析 → SNS 通知 | センチメント（感情極性）は内容の正確性・法令準拠を評価できない |
| モデル自身による自己評価スコア | 評価の客観性ゼロ。自分の回答を自分で採点 → バイアスが強い |
| SageMaker 異常検知モデル + SES | 統計的異常検知は意味的な品質評価に不向き。メール通知は監査証跡・タスク管理が弱い |

### A2I vs その他の「人間に通知する」手段

| 手段 | 特徴 | 試験での位置づけ |
|---|---|---|
| **Amazon A2I** | タスク管理・監査証跡・専門家ルーティングをマネージドで提供 | ✅ 人間レビューワークフローの正解 |
| Amazon SES（メール） | 単純な通知のみ。タスク追跡・監査証跡なし | ❌ 規制業界のレビュー管理には不十分 |
| Amazon Connect（コンタクトセンター） | コールセンター向けサービス。ドキュメントレビューには不適 | ❌ 用途のミスマッチ |

---

### ⚠️ Amazon A2I の動作タイミング・非同期性・リアルタイム判定の要否

```
タイミング（いつ人間に依頼するか）
  → スコアがしきい値未満 / ガードレールがフラグ付け → 自動でレビュータスクを生成
  → 「その場でリアルタイムに判定」するわけではない

非同期か？
  → ✅ 非同期（Async）が前提
  → ユーザーへの応答はいったん保留 or 別途提供し、
    人間レビュー完了後に結果を反映するフローを設計する
  → 人間がレビューするため、秒単位での返却は不可能

リアルタイム判定は必須か？
  → ❌ 必須ではない
  → A2I は「高リスク対話を後から専門家が確認する」設計
  → リアルタイムでブロックしたい場合は Guardrails の役割
```

**試験での判断軸**
- 「即時ブロック・リアルタイム制御」→ Guardrails
- 「専門家が精査・後から確認・監査証跡」→ A2I

---

### ⚠️ モデル自己評価の根本的問題：自己バイアス（Self-bias）

```
LLM が自分の出力を評価させる
  → モデルは自分の生成物を客観的に採点できない
  → 自分のスタイル・語彙・構造を「良い」と判断しやすい（自己バイアス）
  → 実際の正確性・法令準拠とは無関係に高スコアを付ける傾向がある

特にコンプライアンス要件がある領域では
  → 自己評価スコアは信頼性がなく、規制の証拠として使えない
  → 第三者モデル（LLM-as-a-judge）または人間レビューが必要
```

**自己評価が不正解になるパターン**：「モデル自身に自己採点させる」→ 自己バイアスにより客観性ゼロ → コンプライアンス・規制業界では不適切

---

### Amazon Connect とは（混同注意）

```
Amazon Connect
  = AWS のクラウドコンタクトセンターサービス
  = 音声（電話）・チャットを通じた顧客対応ルーティング
  = コールセンター業務の管理（担当者スキルルーティング・待ち時間管理・録音等）

用途の核心
  → 「顧客が電話・チャットで問い合わせてきたとき、どのエージェントに繋ぐか」
  → カスタマーサービス部門向けのインフラ

AI 出力レビューワークフローとの関係
  → ❌ 全く関係ない
  → AI の回答品質管理・コンプライアンス審査には使えない
  → 人間レビューワークフロー = Amazon A2I
```

**試験での引っかけ**：「人間に回す」という文脈で Connect が選択肢に出ることがある → Connect はコールセンター。AI 出力の審査・レビューワークフロー管理は A2I。

# ガードレール統合設計・フィルター強度（AIP-35）

> ⚠️ ガードレール設定の間違えやすいポイントは [exam/traps.md](../exam/traps.md) に集約。

## 1ガードレールに統合できるポリシータイプ（重要）

**1つのガードレール設定に以下をすべてまとめられる：**

| ポリシータイプ | 用途 |
|---|---|
| コンテンツフィルター | 有害カテゴリ（暴力・ヘイト等）の強度設定 |
| 拒否トピック | 業務固有の禁止テーマ（診断・処方、競合他社等） |
| 機密情報フィルター | PII検出 + 入力/出力で別アクション設定 |
| 単語フィルター | 特定ワードのブロック |
| グラウンディングチェック | ハルシネーション検出 |

→ **複数のガードレールに分割してLambdaで順次呼び出す構成はオーバーエンジニアリング**
→ 1回の推論呼び出しに適用できるガードレールは1つだけなので、分割するメリットがない

## AIP-35 正解構成まとめ

```
1つのガードレール内に統合：
  ├─ コンテンツフィルター「中」（過剰ブロック抑制）
  ├─ 拒否トピック：診断・処方（定義文 + サンプルフレーズ付き）
  ├─ 機密情報フィルター：
  │     入力側 → BLOCK（医療保険番号等をモデルに渡さない）
  │     出力側 → MASK（患者情報を伏せ字に置換）
  └─ カスタムブロックメッセージ（ユーザーへのフィードバック）

評価対象：入力・出力の両方（片方だけでは要件を満たせない）
ログ記録：モデル呼び出しログを有効化 → CloudWatch Logs
```

# マルチアカウント Bedrock ガバナンス（AIP-34）

## CloudFormation StackSets とは

**同じ CloudFormation テンプレートを、複数の AWS アカウント・複数のリージョンに一括デプロイできる機能。**

```
通常の CloudFormation スタック
  → 1つのアカウント × 1つのリージョン にのみデプロイ

CloudFormation StackSets
  → 複数アカウント × 複数リージョン に同じテンプレートを一括展開
  → Organizations と統合すると新規アカウント追加時に自動デプロイ
```

### StackSets の2つのモード

| モード | 特徴 |
|---|---|
| **サービスマネージドモード** | AWS Organizations と統合。新規アカウントに自動デプロイ。管理アカウントから操作 |
| **セルフマネージドモード** | 対象アカウントを手動で指定。Organizations 統合なし |

**試験でのポイント：「組織全体」「新規アカウントにも自動」→ StackSets のサービスマネージドモード**

---

## SCP の Bedrock 専用条件キー

SCP（Service Control Policy）= Organizations で全アカウントに強制するポリシー。  
特定の AWS サービス・アクションを**組織全体で強制・禁止**するための仕組み。

### Bedrock ガードレール関連の条件キー

| 条件キー | 意味 |
|---|---|
| `bedrock:GuardrailIdentifier` | 使用するガードレールの ID を指定（このガードレール以外は拒否）|
| `bedrock:GuardrailVersion` | 使用するガードレールのバージョンを指定（特定バージョンのみ許可）|

### SCP の使用例（ガードレール強制）

```json
{
  "Effect": "Deny",
  "Action": "bedrock:InvokeModel",
  "Resource": "*",
  "Condition": {
    "StringNotEquals": {
      "bedrock:GuardrailIdentifier": "arn:aws:bedrock:us-east-1:123456789012:guardrail/abcdef"
    }
  }
}
```

→ **指定したガードレールを適用しないと InvokeModel を拒否する** = ガードレール利用を強制できる

---

## AIP-34 正解パターン：マルチアカウント Bedrock ガバナンス

### 要件（想定）
- 全組織アカウントで特定ガードレールの使用を必須にする
- 個人情報を含む入力を**ブロック**（匿名化ではなく拒否）する
- 新規アカウントにも自動適用する

### 正解構成

```
① CloudFormation StackSets（サービスマネージドモード）
     → Bedrock ガードレール（BLOCK アクション）を全アカウントに一括デプロイ
     → Organizations 統合で新規アカウントにも自動デプロイ

② SCP（条件キー：bedrock:GuardrailIdentifier）
     → 指定ガードレールを適用しない InvokeModel を組織全体で拒否
     → ガードレールのバイパスを防止
```

### なぜこの組み合わせか？

| 役割 | 担当サービス |
|---|---|
| ガードレールを全アカウントに**展開**する | CloudFormation **StackSets** |
| ガードレールの使用を全アカウントに**強制**する | SCP（`bedrock:GuardrailIdentifier` 条件）|

**StackSets だけではガードレールを「作る」だけ。SCP がないと使わない選択肢もある。両方セットで完全なガバナンスになる。**

### IAM ポリシー vs Lambda プロキシ（公式模擬 Q17）

ガードレール強制の手段として Lambda プロキシは**不正解**になる。

| 手段 | オーバーヘッド | 問題点 |
|---|---|---|
| **IAM 条件キー（`bedrock:GuardrailIdentifier`）** | 最小 | なし。組み込み制御 |
| Lambda プロキシ | 大 | 障害点・レイテンシー・カスタムコード管理が発生 |
| Parameter Store + Lambda | 大 | API コールごとに取得処理が走る → レイテンシー悪化 |

### bedrock:PromptRouterArn とは（混同注意）

| 条件キー | 用途 |
|---|---|
| `bedrock:GuardrailIdentifier` | **ガードレールの強制**（これが正解）|
| `bedrock:PromptRouterArn` | 指定したプロンプトルーター経由のアクセスのみ許可 |

**プロンプトルーター**＝プロンプトテンプレートと設定を管理するコンポーネント。ガードレールとは無関係。両方を条件に加えても意味がなく、複雑さが増すだけ。

---

## コンテンツフィルターの動作モード：ブロックモード vs 検出モード（AIP-73）

ガードレールのコンテンツフィルター（性的・暴力・ヘイト等）には2つの**動作モード**がある。

| | ブロックモード | 検出モード |
|---|---|---|
| **違反検出時の動作** | 応答を**遮断**し定型メッセージに置き換える | 応答はそのまま**通過** |
| **記録** | ブロックイベントとして記録 | トレース情報（`trace`オブジェクト）に記録のみ |
| **使うとき** | 「その内容を絶対にユーザーに届けたくない」 | 「検知・記録はしたいが止めたくない」 |

```
ブロックモード：
  性的表現を検出 → 応答を遮断 → 「適切な内容のみ返答できます」等の定型文を返す
  ユーザーはFMの実際の応答を受け取れない

検出モード：
  性的表現を検出 → 応答はそのままユーザーへ届く
  → APIレスポンスの trace オブジェクトに検出カテゴリ・信頼度スコアが記録される
  → アプリ側で CloudWatch Logs や SNS と連携して記録・通知を実装できる
```

**⚠️ 試験ポイント（AIP-73）**：
「FM の応答に性的な表現が含まれる場合は**検知して記録したい**が、**応答の配信自体は停止したくない**」
→ **検出モード** が正解（ブロックモードは配信を止めてしまうので❌）

---

## BLOCK vs MASK（ガードレールのアクション）

ガードレールで PII 等が検出されたときの動作は2種類：

| アクション | 動作 | ユースケース |
|---|---|---|
| **BLOCK** | リクエスト全体を**拒否**（エラー返却）| 個人情報を含む入力を一切通過させたくない |
| **MASK** | 個人情報を `****` 等に**置換**してリクエストを継続 | 個人情報を匿名化して処理を続けたい |

**AIP 試験での識別ポイント**：
- 「入力をブロックする」「拒否する」→ **BLOCK**
- 「匿名化して処理する」「マスクして続ける」→ **MASK**
- 「PIIを含む入力を止めたい」→ MASK ではなく **BLOCK**（MASK は通過させてしまう）

---

## ガードレール機能の要件別使い分け（AIP-36）

保険会社チャットボット要件を例に、ガードレール機能の正しいマッピングを整理。

### 要件 → 機能 対応表

| 要件タイプ | 正しい機能 | 理由 |
|---|---|---|
| 業務固有テーマのブロック（詐欺方法・制度抜け穴など） | **拒否されたトピック（Denied Topics）** | セマンティック分類でテーマを判定。自然言語で定義可能。最大30トピック |
| 特定の文字列・競合他社名のブロック | **カスタム単語フィルター（Word Filter）** | 最大10,000語、正規表現対応、入出力両方に適用可能 |
| 承認済み資料への忠実性（RAG回答の品質保証） | **グラウンディングチェック（高しきい値）** | 参照情報に基づくスコア（0〜0.99）。高しきい値 = 逸脱を厳格にブロック |

---

### コンテンツフィルターを使ってはいけない場面

```
コンテンツフィルター = AWSが定義した汎用カテゴリのみ
  → 暴力・ヘイト・性的・差別・プロンプト攻撃
  → 業務固有トピック（詐欺方法・競合他社名 etc.）は検出できない

業務固有テーマのブロック → 拒否されたトピック を使う
```

**「コンテンツフィルター最大設定」でビジネス特有のリスクをカバーしようとする選択肢は引っかけ。**

---

### グラウンディングチェックのしきい値

```
低しきい値  → 参照情報から逸脱した回答を許容 ← 忠実性要件があるならNG
高しきい値  → 参照情報に忠実でない回答を厳格にブロック ← これが正解
```

- 前提：RAGアーキテクチャ（参照情報が必要）
- スコアが高しきい値を下回る回答はブロックされる

---

### オーバーエンジニアリングの引っかけ

| やりがちな過剰設計 | 正解 |
|---|---|
| Comprehend カスタムエンティティ + Lambda で競合他社名を検出 | **カスタム単語フィルター**で十分（単語リストをベタ登録） |
| 複数ガードレールに分割して管理 | **1ガードレール**に拒否トピック＋単語フィルター＋グラウンディングをまとめる |

**AIP試験のパターン：要件が明示されている場合、Bedrockネイティブ機能で完結する最シンプルな構成が正解。**

## コンテンツモデレーション・PII検出サービスの使い分け（AIP-46）

### 正解構成（ユーザー投稿コンテンツ＋AI生成コンテンツの安全管理）

**Step Functions + Rekognition + Comprehend + Bedrock Guardrails**

```
ユーザー投稿画像
  → Rekognition DetectModerationLabels API
  → 暴力・アダルトなど不適切カテゴリを検出 → 拒否 or フラグ

ユーザー投稿テキスト / AIへのプロンプト
  → Comprehend DetectPiiEntities API
  → NLPベースでPIIを文脈ごと検出 → マスキング or 除去

AIへの入力 / AIの出力
  → Bedrock Guardrails
  → 有害カテゴリフィルタ + PII保護 + トピックブロック（入出力両方に適用）
```

---

### Amazon Rekognition 画像モデレーション

| 項目 | 内容 |
|---|---|
| **主要API** | `DetectModerationLabels` |
| **検出対象** | 暴力・アダルトコンテンツ・過激な表現・グラフィック描写など |
| **仕組み** | 事前学習済みMLモデル。APIを呼ぶだけ。カスタムモデル不要 |
| **出力** | ラベル名 + 信頼スコア（0〜100%）の一覧 |
| **階層構造** | 親カテゴリ（例: "Violence"） → 子カテゴリ（例: "Graphic Violence"） |

```python
# 使用例（呼ぶだけでOK）
response = rekognition.detect_moderation_labels(
    Image={'S3Object': {'Bucket': 'my-bucket', 'Name': 'photo.jpg'}},
    MinConfidence=70  # 信頼度しきい値
)
# → ModerationLabels: [{"Name": "Violence", "Confidence": 92.3, ...}, ...]
```

**カスタムモデル（SageMaker）は不要**。標準的なモデレーション用途なら事前学習済みAPIで十分。

---

### Amazon Comprehend PII検出

| 項目 | 内容 |
|---|---|
| **主要API** | `DetectPiiEntities` |
| **検出対象** | 氏名・メールアドレス・電話番号・クレジットカード番号・住所 etc. |
| **仕組み** | NLPベース（文脈を理解して検出） |
| **強み** | 「田中太郎さんの番号は090-XXXX-XXXXです」の自然文中のPIIを文脈ごと検出 |
| **正規表現との差** | 正規表現は形式が決まっている場合のみ。文脈に埋め込まれた変則的なPIIを見逃す |

---

### サービス目的の識別表（試験頻出）

| サービス | 目的 | コンテンツモデレーションに使える？ |
|---|---|---|
| **Rekognition** | 画像・動画の視覚コンテンツ分析 | ✅ **画像モデレーションの主役** |
| **Comprehend** | テキストのNLP分析・PII検出 | ✅ **テキストPII検出の主役** |
| **Bedrock Guardrails** | LLM入出力の安全制御 | ✅ **AI生成コンテンツの制御** |
| **Macie** | S3内の機密データ分類（セキュリティ） | ❌ リアルタイムモデレーション不可。バッチスキャン向け |
| **GuardDuty** | ネットワーク・APIレベルの脅威検知 | ❌ 画像/テキストの内容判定は不可 |
| **SageMaker カスタムモデル** | 独自分類の学習・推論 | ⚠️ 標準用途はオーバーエンジニアリング |

---

### 試験での判断軸

```
「画像の不適切コンテンツ検出」
  → Rekognition DetectModerationLabels（カスタムモデル不要）

「テキスト内のPII検出・マスキング」
  → Comprehend DetectPiiEntities（正規表現より文脈理解が優れる）

「AIへの入力・AIの出力を安全に制御」
  → Bedrock Guardrails（入出力両方に適用、複数FMに横断適用可）

「S3内の機密データを定期スキャン」
  → Macie（セキュリティ目的、リアルタイム向きではない）

「Macie / GuardDuty = コンテンツモデレーションではない」
  → セキュリティ監視ツール。不適切な画像や文章の判定には使わない
```

## データレジデンシー・不変ログ・機密データ分類（AIP-48）

### 正解構成
**S3 バケットポリシー + S3 Object Lock + 地域内 Bedrock 処理 + Macie + CloudTrail 不変ログ**

---

### データレジデンシー（Data Residency）

```
= データが「どのリージョン（国・地域）に存在するか」を制御・保証すること

EU GDPR などの規制で「EUのデータはEU内でのみ処理・保存」が義務付けられる場合がある
AWS での実現方法：
  ・リージョン固有の S3 バケット（デフォルトでデータはリージョン内に留まる）
  ・S3 バケットポリシーで他リージョンへのアクセスや複製を明示的にブロック
  ・Bedrock もリージョン指定で呼び出す（クロスリージョン推論機能は使わない）

引っかけ：
  「Amazon Bedrock のクロスリージョン推論機能」
  → データが最適なリージョンにルーティングされる = データレジデンシー違反！
```

---

### S3 Object Lock と WORM

```
WORM = Write Once, Read Many（一度書いたら変更・削除できない）

S3 Object Lock の2つのモード：

┌─────────────────┬────────────────────────────────────────────────────┐
│ Governance モード │ 特定の IAM 権限があれば上書き・削除可能              │
│                  │ → 通常のユーザーからは保護するが、管理者は変更できる  │
├─────────────────┼────────────────────────────────────────────────────┤
│ Compliance モード │ 保持期間中は誰も（root含む）上書き・削除できない     │
│                  │ → 完全に改ざん不可能。規制要件の「不変ログ」に最適    │
└─────────────────┴────────────────────────────────────────────────────┘

「改ざん不可能な記録を残すこと」→ Compliance モードが該当
```

---

### CloudTrail Lake のイベントデータストア（追加専用）

```
CloudTrail Lake
  = CloudTrail ログを SQL で検索・分析できるマネージドデータレイク

イベントデータストアの特徴：
  ・追加専用（append-only）= 一度記録したイベントは変更・削除できない
  ・これにより「改ざん不可能な監査ログ」として機能する
  ・保持期間は 90日〜7年

CloudTrail の「不変ログ」= イベントデータストアの append-only 特性のこと
→ AI の判断プロセスの追跡可能性 + 改ざん不可能性 の両方を満たす
```

---

### AWS Control Tower

```
= マルチアカウント AWS 環境を安全に構築・管理するサービス

主な機能：
  ・ランディングゾーン（Landing Zone）の自動セットアップ
  ・ガードレール（予防的・検出的コントロール）の適用
  ・組織全体のアカウント管理（AWS Organizations）

「データレジデンシー」との関係：
  Control Tower はアカウント・ガバナンスの管理が主目的
  各リージョンへのデータ配置制御には S3 バケットポリシーが直接的
  → Control Tower は「おまけで使える」が直接の解決手段ではない
```

---

### Amazon Inspector

```
= EC2・Lambda・コンテナイメージの「脆弱性診断」サービス

検出対象：
  ・OS やソフトウェアの既知の CVE（脆弱性）
  ・ネットワーク露出（意図しない公開ポートなど）

S3 内のデータの機密性識別には使えない！
→ 機密データの分類 = Amazon Macie の役割

引っかけパターン：
  「Inspector でデータの機密性スキャンを実行する」→ ❌ 完全に別用途
```

---

### Amazon Macie ― 正確な役割

```
= S3 バケット内の「機密データを自動検出・分類」するサービス

検出対象：
  ・PII（氏名・メールアドレス・電話番号・クレジットカード番号 etc.）
  ・医療情報、財務情報、認証情報（APIキー等）
  ・カスタムデータ識別子で独自の機密パターンも設定可能

重要ポイント：
  「S3 に保存されたデータの機密性レベルを自動的に識別する」→ Macie が正解
  バッチスキャン向け（リアルタイムモデレーションには不向き）

混乱しやすい組み合わせ：
  Macie = S3の機密データ分類（セキュリティ）

  Inspector = EC2/Lambda の脆弱性診断
  GuardDuty = ネットワーク・APIレベルの脅威検知
```

## データカタログ・ガバナンスサービスの使い分け（AIP-41）

### Amazon DataZone

**目的**：組織全体でのデータの**発見・共有・ガバナンス**を実現するデータマネジメントサービス。

```
主なユーザー：ビジネスアナリスト、データサイエンティスト、データエンジニア

**内部的に Glue Data Catalog を利用している**（DataZone はその上のガバナンスレイヤー）。

- DataZone は「**組織をまたいだデータ共有・承認フロー**」に特化したサービス
- AIP 試験では**正解になりにくい**（ビジネス共有用途に限定されるため）

### Glue Data Catalog vs DataZone vs Lake Formation の使い分け

| サービス | 主な目的 | 使うとき |
|---|---|---|
| **Glue Data Catalog** | 技術メタデータの一元管理（スキーマ・場所・パーティション） | ETL・Athena クエリ・データソースの登録・出典管理 |

```
試験での判断：

「データソースを一元登録・技術メタデータを管理・出典情報を付与」

「列/行レベルのきめ細かいアクセス制御」
  → Lake Formation

「ビジネスユーザーがデータを探して申請・部署間で共有」
  → DataZone（ただし AIP 試験ではほぼ引っかけ候補）

DataZone が出てきたら「過剰構成」として疑う
```

## 多層防御（Defense in Depth）の4層セット（Task 3.1）

### 基本構成

```
① Amazon Comprehend（前処理フィルター）
   → 入力テキストの感情・毒性・PII・エンティティを検出してブロック/マスク

② Amazon Bedrock ガードレール（モデル層）
   → プロンプト攻撃・禁止トピック・有害カテゴリをMLで判定

③ Lambda 関数（後処理検証）
   → 業界固有ルール・出力フォーマットチェック・先行レイヤーをすり抜けた問題を検出

④ API Gateway（出口フィルタリング）
   → マッピングテンプレートで内部情報（デバッグトレース・モデルID等）を除去
```

**1層が突破されても次の層で止める。各層で役割が違う。**

### 試験での判断ポイント

- 「包括的な保護」「悪意のあるプロンプト」→ **Bedrock ガードレール + Step Functions + Lambda**
- 「多層防御」「複数レイヤー」→ **Comprehend（入口）+ Guardrails + Lambda（後処理）+ API Gateway（出口）** の4層セット
- WAF = HTTP/HTTPSリクエストの入口の門番 / API Gateway フィルタ = レスポンスの出口の検閲（役割が違う）

|---|---|

## CloudWatch Metric Math（メトリクス数式）（AIP-84）

### Metric Math とは

**複数のメトリクスを組み合わせて新しいメトリクスを計算する機能。コード不要。**

```
使用例（グループ間差異の計算）：

メトリクスA：属性グループA の公平性スコア（CloudWatch カスタムメトリクス）
メトリクスB：属性グループB の公平性スコア（CloudWatch カスタムメトリクス）

Metric Math で差異を計算：
  差異 = ABS(A - B)          ← 絶対値差
  差異率 = (A - B) / B * 100 ← 百分率差

→ この「差異」メトリクスに対してアラームを設定
→ 差異が所定のしきい値を超えたらSNSで通知
```

### AIP-84 での役割

```
Bedrock ログ記録
  ↓
CloudWatch Logs
  ↓ メトリクスフィルター（設定のみ）
属性グループ別の公平性指標（カスタムメトリクス）
  ↓ Metric Math（設定のみ）
グループ間差異メトリクス
  ↓ CloudWatch アラーム
差異がしきい値超過 → SNS通知

すべて「設定ベース」でカスタムコード不要

## ガードレール介入の詳細分析（公式模擬 Q13）

### 重要前提：モデル呼び出しログとガードレールトレースは別物

| ログ種別 | 記録内容 | ガードレール詳細 |
|---|---|---|
| **モデル呼び出しログ** | リクエスト・レスポンス・メタデータ | **❌ 記録しない** |
| **ガードレールトレース** | どのポリシーが発動したか、理由 | ✅ 詳細あり |

### ガードレールトレースの有効化

```json
// Converse API の guardrailConfig に設定
{
  "guardrailConfig": {
    "guardrailIdentifier": "xxxx",
    "guardrailVersion": "1",
    "trace": "enabled"
  }
}
```

### CloudWatch メトリクス：InvocationsIntervened

ガードレールが介入（ブロック）した回数を計測するメトリクス。  
**ディメンション**で絞り込むことで詳細分析が可能。

| ディメンション | 用途 | 取得できる情報 |
|---|---|---|
| **GuardrailPolicyType** | **どのポリシーが発動したか** | `ContentPolicy` / `TopicPolicy` / `SensitiveInformationPolicy` |
| **GuardrailContentSource** | 入力と出力のどちらで発動したか | `INPUT`（プロンプト） / `OUTPUT`（レスポンス） |

### GuardrailPolicyType の値

| 値 | 対応するガードレール機能 |
|---|---|
| `ContentPolicy` | コンテンツフィルター（暴力・ヘイト等） |
| `TopicPolicy` | 拒否トピック（禁止テーマのブロック） |
| `SensitiveInformationPolicy` | 機密情報フィルター（PII等） |
| `WordPolicy` | 禁止ワードフィルター |
| `PromptAttackPolicy` | プロンプトインジェクション検出 |

### 正解パターン（「どのポリシーが発動したか詳しく知りたい」）

```
guardrailConfig: trace=enabled
  ↓ トレース有効化
InvocationsIntervened メトリクス
  + GuardrailPolicyType ディメンションでフィルタ
  ↓
ContentPolicy / TopicPolicy / SensitiveInformationPolicy ごとに介入数を把握
  ↓
ガードレール設定のファインチューニング
```

### 引っかけポイント

| 選択肢 | 問題点 |
|---|---|
| モデル呼び出しログ + CloudWatch | **ログにガードレール介入詳細は含まれない** |
| `GuardrailContentSource` ディメンション | 入力/出力の区別はできるが**どのポリシーかはわからない** |
| モデル評価（Bedrock Evaluation） | 推論中ではなく評価ジョブ用。リアルタイム介入分析には使えない |

## Bedrock FM 評価ワークフローの正しい順序（公式模擬 Q10）

### 正解の順序（必ず覚える）

```
1. 評価メトリクスを定義（関連性・事実の正確性・流暢さ）
       ↓ 承認
2. テストデータセットを作成（メトリクスに合わせて設計）
       ↓ 承認
3. A/B テストを実施（新旧モデルを同一入力で比較）
       ↓ 承認
4. Step Functions で自動品質ゲートを実装（結果をしきい値と自動検証）
       ↓ 承認
5. 結果を分析・評価レポート作成 → 本番切替を判断
```

### 引っかけポイント（2つ）

| 間違えやすい順序 | 正しい理由 |
|---|---|
| データセット → メトリクス | **メトリクスが先**。何で測るかを決めてからテストデータを設計する |
| Step Functions → A/Bテスト | **A/Bテストが先**。Step Functions はA/Bテスト結果をしきい値チェックするゲート。テスト前に存在できない |

### Step Functions の役割（AIPでの位置づけ）
- ML評価ワークフローで「**ステージ間の承認ゲート**」として機能
- 各ステップの結果を自動検証し、しきい値を満たさなければ次ステップをブロック
- 「手順通りの検証・承認が必要」という要件に対応するサービス

## FM 品質保証（QA）システム構成

### 3サービスの役割分担

| サービス | 担当 | 具体的な処理 |
|---|---|---|
| **Lambda** | 個々の出力を検証 | 期待値との一致確認・スコアリング・フォーマット検証 |
| **Step Functions** | エッジケースを網羅的にテスト | テストケース一覧を順番に実行するオーケストレーション |
| **CloudWatch** | プロンプトリグレッションを監視 | スコアを時系列記録・劣化検知アラーム |

### エッジケース（Edge Cases）とは

**通常の使い方の「端っこ」にある想定外・極端な入力パターン。**

| カテゴリ | 例 |
|---|---|
| 極端な長さ | 空文字列 / 10万字の入力 |
| 言語の境界 | 日英混在 / 記号のみ / 絵文字のみ |
| 悪意ある入力 | プロンプトインジェクション（「前の指示を無視して…」） |
| 曖昧な質問 | 「あれって何？」（文脈なし） |
| 数値の境界 | 金額が 0 円 / マイナス / 極端に大きい値 |

### プロンプトリグレッション（Prompt Regression）とは

**プロンプトやモデルを変更したとき、以前は正しく答えられていた質問が劣化していないかを確認するテスト。**
ソフトウェアの「回帰テスト（リグレッションテスト）」をプロンプトに適用したもの。

```
① ゴールデンセットを用意（質問 + 期待される回答）
② プロンプト変更 or モデルバージョンアップ
③ 同じ質問を新バージョンで実行 → Lambda で出力を検証
④ CloudWatch にスコアを記録・監視
   ├─ スコアが下がった → アラーム → ロールバック
   └─ 維持・改善    → デプロイ続行
```

### インテント認識（Intent Recognition）とは

**ユーザーの発言から「何をしたいのか（目的・意図）」を分類する処理。**

```
「予約キャンセルしたい」  →  cancel_reservation
「明日の空きある？」     →  check_availability
「返金してほしい」       →  request_refund
← 言い方が違っても同じ意図なら同じカテゴリに分類
```

- **Amazon Comprehend のカスタム分類器**が担当
- 分類結果をもとに Step Functions でワークフロー分岐、DynamoDB から情報取得

## モデル信頼度のキャリブレーション（Task 3.4）

### キャリブレーションとは

「モデルが90%の確信と言ったとき、実際に90%の確率で正解しているか」を検証・補正する手法。

- **過信（Over-confident）**：モデルが「確信度95%」と言うのに実際の正解率が60%にとどまる状態
- **過小評価（Under-confident）**：正解率80%なのに「確信度50%」と言う状態

### Bedrock でのキャリブレーションの位置づけ

| 項目 | 内容 |
|---|---|
| ネイティブ機能か | ❌ なし。外部実装が必要 |
| 実装パターン | 評価データセットで「自己申告確信度 vs 実際の正解率」を測定・補正 |
| 確信度の取得方法 | モデルに `{"answer": "...", "confidence": 0.85}` 形式で JSON 出力させる（プロンプトで指示）|
| logprobs（トークン確率）| Bedrock は非対応。OpenAI 等の直接 API のみ |

### 実装フロー例

1. モデルに「確信度を0〜1で出力せよ」とプロンプト指示
2. テストデータ100問で回答＋確信度を収集
3. 「確信度0.8〜0.9 帯の問題の実際の正解率」を計算
4. ずれがあれば確信度にスケーリング補正を加える
5. 補正後の確信度を CloudWatch カスタムメトリクスで継続追跡

### 試験の判断ルール

- 「信頼度スコアをモデルの実際の正解率と一致させる」→ **キャリブレーション**
- Bedrock でキャリブレーションを行う手段 → **プロンプトで JSON 出力 + 外部評価パイプライン**（ネイティブ機能ではない）
