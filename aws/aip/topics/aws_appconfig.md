# AWS AppConfig

## 概要

AWS Systems Manager の一機能。**アプリケーション設定をコード変更・Lambda再デプロイなしに動的管理**するためのマネージドサービス。

---

## 主な機能

| 機能 | 内容 |
|---|---|
| **機能フラグ（Feature Flags）** | アプリの機能をON/OFFする。属性ベースのターゲティングで条件に応じた設定の出し分けが可能 |
| **設定プロファイル** | JSON/YAMLで設定値を管理。バージョン管理も組み込み済み |
| **JSONスキーマバリデーション** | デプロイ前に値の範囲チェックを自動実行。不正値が本番に入るのを防ぐゲートキーパー |
| **デプロイ戦略** | Linear（線形）・Exponential（指数）など段階的ロールアウトを設定だけで実現。CloudWatchアラームと連携して異常時は自動ロールバック |
| **AppConfig Agent（Lambda拡張機能）** | `localhost:2772` 経由で設定をローカルキャッシュから取得。API呼び出しコスト・レイテンシーを最小化。TTLに基づき自動更新 |

---

## 属性ベースのターゲティング

Lambdaがリクエストから属性（例：優先度レベル）を抽出し、AppConfig Agentに渡すことで、条件に応じた設定値が返ってくる仕組み。

```
リクエスト到着
  ↓
Lambda が属性を抽出（例：priority=urgent）
  ↓
AppConfig Agent に属性を渡して設定を問い合わせ
  ↓
AppConfig が条件に合う設定を返す（例：urgentならモデルA）
  ↓
Lambda がその設定でBedrockを呼び出す
```

→ Lambda自体は変わらず、**AppConfig側のルール定義を変えるだけ**でモデルの切り替えができる

---

## スキーマバリデーター

**設定値をデプロイする前に不正な値を弾く**ゲートキーパー的機能。LambdaやコンシューマがデータをSubscribeする際の検証ではない点に注意。

```
開発者が設定値を更新・デプロイしようとする
  ↓
AppConfig がスキーマバリデーターを自動実行
  ├─ 正常（範囲内） → デプロイOK → Lambdaに反映される
  └─ 異常（範囲外） → デプロイ拒否 → Lambdaには届かない
```

スキーマ定義例：
```json
{
  "type": "object",
  "properties": {
    "temperature": {
      "type": "number",
      "minimum": 0.0,
      "maximum": 2.0
    },
    "max_tokens": {
      "type": "integer",
      "minimum": 1,
      "maximum": 4096
    }
  }
}
```

---

## 「デプロイなし」の正確な意味

| 対象 | 要否 |
|---|---|
| **Lambdaのコード変更・関数更新** | 不要（←これが「デプロイなし」の意味） |
| **AppConfig側の設定変更** | 必要（ただしLambdaデプロイより圧倒的に軽い） |

AppConfigへの設定変更のデプロイ方法：
- コンソールから手動
- AWS CLI / SDK経由でスクリプト化
- EventBridge + Lambda で自動化

## AWS AppConfig：設定のデプロイライフサイクル管理（AIP-63）

AWS Systems Manager のサブサービス。アプリケーション設定をバージョン管理しながら安全に配布する。  
**単なるキーバリューストア（Parameter Store等）との違い**：値の保存・取得だけでなく「デプロイライフサイクル（検証→段階的配布→監視→ロールバック）」を一体管理する点。

### 主要機能
| 機能 | 内容 |
|---|---|
| **デプロイ戦略** | Canary（少数先行→全体）/ Linear（一定割合ずつ段階的展開）。GrowthFactor・DeploymentDurationInMinutes・FinalBakeTimeInMinutes で制御 |
| **バリデータ** | JSON スキーマバリデータ または Lambda バリデータ を事前設定。不正な値はデプロイ前に弾かれる |
| **自動ロールバック** | CloudWatch アラームを監視し、アラーム発火時に自動で前バージョンに戻す。人手不要 |
| **Lambda Extension** | サイドカーとしてキャッシュ。毎回 API 呼び出し不要 → レイテンシー影響最小 |

### AppConfig のユースケース（Bedrock文脈）
- Lambda の呼び出す Bedrock モデル ID を設定として管理（モデル切り替えをデプロイレスで実現）
- フィーチャーフラグ（A/B テスト、カナリアリリースのトグル）
- プロンプトテンプレートの動的切り替え
- 複雑なルーティングルール（ユーザー属性・コストしきい値・地域規制・A/Bテスト比率）の外部管理（AIP-64）

### AppConfig エージェント（Lambda Extension）の詳細（AIP-64補足）
- **ローカルキャッシュ**：バックグラウンドでポーリング（デフォルト45秒間隔）し設定を取得・キャッシュ
- 毎リクエストのネットワーク往復が不要 → DynamoDB等への毎回読み取りより低レイテンシー
- **Lambda 環境変数との違い**：環境変数はサイズ上限4KB・更新時に関数再設定が必要（実質デプロイ）。AppConfigは大きなJSON設定も格納可能
- `AllAtOnce` デプロイ戦略：即時全体反映（段階的適用が不要な場合に使用）
