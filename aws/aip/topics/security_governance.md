# AWS セキュリティ・ガバナンス サービス

## AWS Service Catalog（AIP-76）

### 一言で

> **IT管理者が承認済みのAWSリソース構成を「製品」としてカタログ化し、部門（エンドユーザー）がセルフサービスでデプロイできるようにするサービス。**

### 基本構造

```
IT管理者（ガバナンスチーム）
  ↓ 承認済みCloudFormationテンプレートを「製品」として登録
ポートフォリオ（製品のグループ）
  ↓ アカウント・部門に共有
エンドユーザー（各部門）
  → Service Catalog コンソールからクリックでプロビジョニング
```

### ① バージョン管理

- 各製品（Product）に**複数バージョン**を保持できる
- 管理者は特定バージョンを**有効 / 無効**に切り替え可能
- `「最新の承認済みバージョンのみ使用可能にする」` → Service Catalog の組み込み機能で実現
- 部門が古いバージョンを使い続けるリスクを排除できる

```
製品: 「Bedrock AIチャットボット基盤」
  └─ v1.0（無効化済み）
  └─ v2.0（無効化済み）
  └─ v3.0（✅ 現在有効・部門が使用可能なのはこれだけ）
```

### ② セルフサービスポータル

- エンドユーザーは **Service Catalog コンソール**にアクセス
- 許可されたポートフォリオの製品一覧から選んで「起動」するだけ
- CloudFormation の知識不要。**技術的なハードルが低い**
- 部門ごとにアクセスできるポートフォリオを IAM で制御

### ③ 起動制約（Launch Constraint）

```
通常のデプロイ：
  エンドユーザーのIAM権限 → リソース作成（広い権限が必要）

Service Catalog（起動制約あり）：
  エンドユーザーのIAM権限 → 最小限（Service Catalogの使用権限のみ）
  Service Catalogが指定の「起動ロール」でリソース作成
  → 部門ユーザーに過剰な権限を与えずに済む
```

| 制約の種類 | 内容 |
|---|---|
| **起動制約（Launch Constraint）** | 製品起動時に使用するIAMロールを指定。エンドユーザーの権限を最小化 |
| **テンプレート制約** | CloudFormationパラメータの値を制限 |
| **通知制約** | 製品起動・終了時にSNSで通知 |
| **TagOptions** | タグの標準化を強制（コスト管理・リソース追跡に有効） |

### Service Catalog vs 似たサービスの違い

| | Service Catalog | CodePipeline | Control Tower |
|---|---|---|---|
| **対象ユーザー** | 非技術者の部門担当者 | 開発者・DevOps | 管理者（アカウント単位） |
| **主な目的** | 承認済みテンプレートのセルフサービス配布 | CI/CDによるデプロイ自動化 | マルチアカウントのガバナンス |
| **バージョン管理** | ✅ 製品レベルで組み込み | Gitリポジトリで管理 | ❌ テンプレート管理なし |
| **セルフサービスUI** | ✅ カタログポータル | ❌ | ❌ |

## AWS Config（AIP-76）

### 基本機能（インフラ構成の記録 + ルール評価）

**「AWSリソースの構成変更を記録する」だけでなく、「ルールに基づいてコンプライアンスを評価する」機能を持つ。**

```
① 記録（Recording）
   AWSリソースの設定変更を継続的に記録
   「いつ・誰が・何をどう変えたか」の履歴を保持

② ルール評価（Rules）
   定義したルールに照らしてリソースの準拠状況を評価
   組み込みルールの例：
   ・s3-bucket-server-side-encryption-enabled
   ・restricted-ssh

③ コンプライアンスレポート
   準拠（Compliant）/ 非準拠（Non-Compliant）をダッシュボードで可視化
```

### 統制の種類（重要な概念）

| 種類 | タイミング | サービス例 |
|---|---|---|
| **予防的統制（Preventive）** | 違反操作を事前に防ぐ | SCP、IAM、Service Catalog起動制約 |
| **検出的統制（Detective）** | 違反が起きた後に検出する | AWS Config、CloudTrail、GuardDuty |
| **修正的統制（Corrective）** | 違反を検出して自動修正する | AWS Config + SSM Automation |

→ Config は「**検出的統制**」。デプロイされた後に問題を発見する。

```
AWS Config が得意：
  ✅ リソース設定変更の追跡
  ✅ コンプライアンスルールによる継続的評価

AWS Config が苦手（引っかけ）：
  ❌ 非準拠を事前に防ぐ → SCP・IAM が担当
  ❌ テンプレートのバージョン管理・配布 → Service Catalog が担当
```

## AWS Control Tower（AIP-76）

### 一言で

> **マルチアカウントのAWS環境（ランディングゾーン）を自動セットアップし、組織全体のガバナンスを提供するサービス。**

### 何をするか

```
① ランディングゾーン（Landing Zone）の自動構築
   ・ログアーカイブアカウント
   ・監査アカウント
   ・組織構造（OU）の標準化セットアップ

② ガードレールの適用（予防的 / 検出的）
   予防的：SCP で特定操作を禁止
   検出的：AWS Config ルールで非準拠を検出

③ Account Factory
   標準化された新規アカウントをテンプレートから作成
```

### Control Tower vs Service Catalog の住み分け

```
Control Tower：アカウント単位のガバナンス基盤
               「このアカウントでは〇〇を禁止」

Service Catalog：製品（テンプレート）の配布・管理基盤
                「この部門には〇〇の製品を使わせる」
```

## AWS Inspector

**Amazon Inspector** はEC2・Lambda の脆弱性スキャンだけでなく、**ECR（Elastic Container Registry）に push されたコンテナイメージも自動スキャン**できる。

| スキャン対象 | 内容 |
|---|---|
| EC2インスタンス | OS・ソフトウェアの脆弱性 |
| Lambda関数 | コードの依存パッケージの脆弱性 |
| **ECRコンテナイメージ** | **イメージ内のOS・パッケージの脆弱性** |

- ECRへのpush時に自動でスキャンが走る（継続スキャンも可能）
- 検出結果はInspectorコンソール・Security Hubに集約される

## AWS Artifact

**AWS Artifact** は、AWSのコンプライアンスレポートおよびセキュリティ証明書を取得するためのセルフサービスポータル。

```
AWS Artifact でできること：
  ① コンプライアンスレポートのダウンロード
      例：SOC 1/2/3、PCI DSS、ISO 27001 など
  ② AWS との契約（BAA）の締結・管理
      例：HIPAA対応のためのBAA（Business Associate Agreement）
```

| 用途 | 内容 |
|---|---|
| 監査対応 | AWSのISO/SOC/PCI等の証明書を提出 |
| 法的契約 | HIPAAのBAAなど規制対応の契約を締結 |

## SAML フェデレーション

### SAML とは

**SAML（Security Assertion Markup Language）** = 異なるシステム間で「このユーザーは認証済みです」という情報を安全に受け渡す標準規格。

```
イメージ：
  社内の Active Directory / Okta（会社の ID 管理）
         ↓ SAML assertion（XML形式の「認証済み証明書」）
       AWS STS
         ↓ 一時的な AWS 認証情報に変換
       AWS リソース（Bedrock 等）にアクセス

→ ユーザーは会社の既存 ID/パスワードでそのまま AWS にアクセスできる
→ AWS 側に新たな IAM ユーザーを作る必要がない
→ 認証情報は一時的（長期 Access Key を持たない）
```

### SAML vs OIDC

| | SAML | OIDC |
|---|---|---|
| 形式 | XML ベース | JSON/JWT ベース |
| 主な用途 | 企業向けSSO（Active Directory等） | モバイル/Webアプリ向け |
| AWSでの使い方 | IAM Identity Center との連携が多い | Amazon Cognito との連携が多い |

### AWS での SAML フェデレーション構成

```
企業の IdP（Okta/Azure AD/ADFS 等）
  ↓ SAML 2.0
IAM Identity Center
  ↓ 権限セット（カスタムアクセス許可）
一時的な STS 認証情報
  ↓
Amazon Bedrock / AWS サービス
```

### 正解の構成

```
S3（メールデータ）

  → Amazon Kendra（マスキング済みデータをインデックス化・自然言語検索）
  → Amazon Bedrock FM（会話型AI応答）
```

### サービスの役割比較

| サービス | 役割 | PII処理 |
|---|---|---|

## AWS Well-Architected 3点セット

ベストプラクティス系の問題で混同しやすい3つを整理。

### ① Well-Architected Framework（フレームワーク）

**= AWSの設計思想・ベストプラクティス集（ドキュメント）**

6つの柱：
1. 運用上の優秀性
2. セキュリティ
3. 信頼性
4. パフォーマンス効率
5. コスト最適化
6. 持続可能性

→ ホワイトペーパーで読む「教科書」。

### ② Well-Architected Lens（レンズ）

**= 特定分野向けの追加ガイドライン（ドキュメント）**

- **Generative AI Lens** ← AIP頻出
- Machine Learning Lens
- SaaS Lens / Financial Services Lens / IoT Lens など

→ Frameworkに「メガネをかけて分野特化の観点を追加」するイメージ。中身はやはり設計思想。

### ③ Well-Architected Tool（WA Tool）

**= 自分のシステムを上記基準で自己診断するAWSコンソール上のツール**

- マネジメントコンソールから利用
- ワークロード（自分のシステム）を登録 → 質問票に回答
- 「高リスク項目」「中リスク項目」を自動レポート
- **Lensを適用すると追加質問が増える**（例: Generative AI Lens適用 → AI特化の質問追加）

→ 設計思想を「やってるか？」と質問してくれる**自動レビュー官**。リソースは作らない。

### 関係性

```
Framework（教科書）         ─┐
                            ├─→ WA Tool（試験官）が「やってる？」と質問
Generative AI Lens（追加教科書）─┘
```

| ツール | 役割 |
|---|---|
| Framework / Lens | 何をすべきか（What） |
| **WA Tool** | やってるか診断（Check） |
| CloudFormation / CDK | リソースを作る（Build） |
| Trusted Advisor | 既存リソースの運用状況チェック |
| AWS Config | リソースの設定変更を監視 |

## サーキットブレーカー × Step Functions

**「障害が続いたら一時的にリクエストを遮断する」パターン。**

```
CLOSED（通常）→ 連続失敗がしきい値を超えたら → OPEN（遮断）→ 一定時間後 → HALF-OPEN（試験）
```

Step FunctionsでのImplementation：
- **Choice ステート** × エラーカウント × タイムアウト
- 複数実行をまたいでエラー数を保持 → しきい値超えで別ルートへ自動遷移

| 選択肢 | ダメな理由 |
|---|---|
| Exponential Backoff | リトライするだけ。「遮断」しない |
| API Gateway再試行 | HTTPレベル。状態を持てない |
| DynamoDB追跡 | 追跡はできるが即時作動できない |
| EventBridge | 事後対応（リアクティブ）。予防的に遮断できない |

## ID フェデレーション：IAM Identity Center vs Cognito

**IDフェデレーション：「既存の認証基盤のIDをAWSでも使い回す」**

| | IAM Identity Center | Amazon Cognito |
|---|---|---|
| 主な対象 | **社員**（内部ユーザー） | **アプリのエンドユーザー**（外部） |
| 典型例 | 社員がAWSコンソールにログイン | アプリの会員登録・ログイン |
| AD連携 | ネイティブに得意 | できるが設定が複雑 |
| スケール | 数百〜数千人 | 数百万人対応 |

```
社内Active Directory（AD）
    ↓ フェデレーション
IAM Identity Center
    ↓
Bedrock / S3等にアクセス（社員が社内IDでそのままAWSにアクセス）
```

## Cognito プレトークン生成 Lambda トリガー

「**既存システムのユーザーIDとCognitoを紐づけ、JWTにカスタム情報を追加する**」パターン。
AWSアカウントを別途作らせずに、既存の社内ユーザーIDのままAI機能を使わせたい場合。

```
ユーザーログイン → Cognito（認証）
    ↓ トークン生成直前にLambdaトリガー発火
Lambda → 社内DBでロール情報を取得
    ↓ JWTにカスタムクレームを追加
JWT発行 → Bedrock API呼び出し時に提示
```

**Cognitoを挟む理由：** レガシー認証システムはAWS標準の認証（IAM/JWT）を直接受け付けない。CognitoがAWS標準JWTに変換する「通訳」の役割を果たす。

## Amazon Verified Permissions + Cedar ポリシー

「**アプリの認可ロジックを外出しして一元管理する**サービス」。

**認証 vs 認可：**
- 認証（Authentication）：あなたは誰？ → Cognito
- 認可（Authorization）：あなたは何ができる？ → Verified Permissions

```
アプリ → 「このユーザーはこの操作していい？」と問い合わせ
    ↓
Verified Permissions が Cedar ポリシーで判定
    ↓
permit / deny を返す
```

**Cedarポリシーの形式：**
```
permit(
  principal in 製造部門,
  action == "BedrockInvoke",
  resource.dataLevel == "一般"
);
```
「誰が・何を・どのリソースに」という形式。

## AWS RAM（Resource Access Manager）

「**AWSリソースを複数アカウント間で共有する**サービス」。

```
アカウントA（AI開発チーム）でカスタムモデル作成
    ↓ RAMで共有
アカウントB（製造部門）→ そのモデルで推論
アカウントC（営業部門）→ そのモデルで推論
```

**共有できるリソース例：** Bedrockカスタムモデル、Knowledge Base、VPCサブネット、Transit Gateway

**SCPとの違い（重要）：**

| | SCP | RAM |
|---|---|---|
| 目的 | **何をできないか**を制限 | **リソースを別アカウントに渡す** |
| 方向 | 制限・禁止 | 共有・許可 |

SCPだけではリソースを別アカウントに渡せない。RAMだけでは権限の制限ができない。**両方必要**。

## KMS マルチリージョンキー

「**複数リージョンで同じデータを暗号化・復号できる**KMSキー」。

```
通常のKMSキー：リージョン固有 → 別リージョンでは復号不可

マルチリージョンキー：
東京（mrk-abc123）とバージニア（mrk-abc123）に同じキーマテリアルが存在
→ どのリージョンでも暗号化・復号できる
```

**なぜAIシステムで必要か：**
日本のユーザーデータ（東京で暗号化）をグローバルなBedrock推論（米国リージョン）で処理する場合。

**自動キーローテーション：** 定期的にキーの暗号素材を自動更新。古いキーで暗号化されたデータも引き続き復号可能。

## Transit Gateway + ブラックホールルーティング

**Transit Gateway：** 複数VPCをハブ1つで接続するサービス（VPC Peeringの代替）。

```
開発VPC ─┐
テストVPC ┼─ Transit Gateway（ハブ）
本番VPC  ─┘
```

**ドメイン分離：** 環境ごとにルートテーブルを分けて相互アクセスを制限。
```
開発用ルートテーブル  → 開発VPCのみ到達可
本番用ルートテーブル  → 本番VPCのみ到達可
```

**ブラックホールルーティング：** 特定の宛先へのトラフィックを無条件に破棄するルート。
```
192.168.0.0/16 → blackhole（破棄）
→ このアドレス宛のパケットは届かず消える
```
セキュリティインシデント時の即時経路遮断、環境間の完全分離に使う。

## AWS Network Firewall

「**VPC内のトラフィックをパケットの中身まで検査**できるマネージドファイアウォール」。

| | セキュリティグループ | Network Firewall |
|---|---|---|
| 検査内容 | IPアドレス・ポート番号のみ | パケットの中身・ドメインまで |
| ドメインフィルタリング | ❌ | ✅ |

**ステートフルルールグループ：** 通信の往復（リクエスト+レスポンス）を追跡して判断。単純なIPフィルタより賢い。

**ドメインフィルタリング：** 承認済みエンドポイントのみ許可（アウトバウンド通信の制限）。
```
✅ bedrock.ap-northeast-1.amazonaws.com
✅ s3.amazonaws.com
❌ それ以外のドメイン → 全部ブロック
```

## IAM Condition 時刻指定の制約

IAMのConditionで時刻は指定できるが**「毎日の繰り返し時間帯」はIAM単体では不可**。

```json
// これは「特定期間」の指定（毎日繰り返しではない）
"DateGreaterThan": {"aws:CurrentTime": "2024-01-01T09:00:00Z"},
"DateLessThan":   {"aws:CurrentTime": "2025-12-31T18:00:00Z"}
```

**「毎日9〜18時のみ」を実現するには：**
```
EventBridge（9時）→ Lambda → IAMポリシーをAllow
EventBridge（18時）→ Lambda → IAMポリシーをDeny
```

## IAM 段階的アクセスレベル（動的リスク評価）

「**運用の重要度と自動リスク評価に基づいた段階的アクセス制御**」。

```
低リスク操作  → 自動実行OK
中リスク操作  → 条件付きで実行（リスク評価通過時のみ）
高リスク操作  → 制限 or ブロック
```

固定的なルールではなく運用状況に応じてスケールする点がポイント。
「リアルタイム保護」「不必要な遅延なし」要件で、サーキットブレーカー（Step Functions）と組み合わせて使う。

## IAM ABAC と Bedrock 条件キー（AIP-51）

### IAM ABAC（属性ベースアクセスコントロール）

**RBAC vs ABAC 比較**：

| 観点 | RBAC（ロールベース） | ABAC（属性ベース） |
|---|---|---|
| 制御単位 | ロール（職種・役職） | タグ（属性キー＝値） |
| ポリシー数 | リソース種類 × ロール数 分必要 | 少数の汎用ポリシーで済む |
| リソース追加時 | ポリシー更新が必要 | タグ設定だけで自動適用 |
| スケーラビリティ | 組織拡大でポリシーが爆発的に増加 | タグ付けルールを守ればスケール |
| 典型ユースケース | 小規模・シンプルな権限管理 | 診療科・部署など多部門組織 |

```
ABAC の動作イメージ（診療科タグで制御）：
  IAM ユーザー/ロール に department=cardiology タグ
        ↓
  ポリシー: department タグが一致するリソースのみ許可
        ↓
  新しい診療科が増えても ポリシー変更不要、タグ設定だけで OK
```

```
ABAC 例（診療科タグで制御）：
  IAM ユーザー/ロール に department=cardiology タグ
        ↓
  ポリシー: department タグが一致するリソースのみ許可
        ↓
  新しい診療科が増えても ポリシー変更不要、タグ設定だけで OK
```

### Bedrock 専用の IAM 条件キー

| 条件キー | 用途 |
|---|---|
| `bedrock:ModelId` | 利用可能な基盤モデルを制限 |
| `bedrock:GuardrailIdentifier` | 使用するガードレールを強制 |

```json
{
  "Condition": {
    "StringEquals": {
      "bedrock:ModelId": "anthropic.claude-3-sonnet-20240229-v1:0",
      "bedrock:GuardrailIdentifier": "arn:aws:bedrock:...:guardrail/cardiology-guardrail"
    }
  }
}
```

**試験での判断軸**：
```
「診療科ごとに使えるモデル・ガードレールを制限」
  → IAM ABAC + bedrock:ModelId + bedrock:GuardrailIdentifier 条件キー

「SCP でアカウント単位に制限」
  → ❌ 粒度が粗い（同一アカウント内の診療科分離には不十分）

「Bedrock Guardrails でコンテンツフィルター」
  → ❌ それは入出力の安全制御。モデル利用の制限ではない
```

### 予防的統制 vs 検出的統制（AIP-51）

```
予防的統制（preventive）：不正を事前に防ぐ
  → IAM ポリシー + 条件キー、VPC エンドポイント

検出的統制（detective）：発生後に検知・記録する
  → CloudTrail, GuardDuty, CloudWatch アラーム

「厳密に制限する」= 予防的統制が必要
ガードレールや Insights だけでは予防的統制にならない ← 試験の罠
```

## Bedrock リソースベースポリシー（Task 3.2）

IAM ポリシーには2種類ある：

| 種類 | どこに貼る？ | 主な用途 |
|---|---|---|
| **ID ベース** | ユーザー / ロール側 | 「自分はこれを呼べる」 |
| **リソースベース** | KB 等のリソース側 | 「自分にアクセスできるのは誰か」 |

```json
{

  "Principal": { "AWS": "arn:aws:iam::111122223333:role/AppRole" },

```
→ KB を持つアカウントと使うアプリのアカウントが別々の場合に必要（ID ベースだけではアカウントをまたげない）

## Bedrock プライベート接続 × データレイク列単位制御（AIP-58）

### Bedrock へのプライベート接続：インターフェイス VPC エンドポイント
- **bedrock-runtime** エンドポイント = InvokeModel などの推論 API 用（bedrock エンドポイントとは別）
- プライベート DNS を有効化すれば **SDK のコードを変更せず** にエンドポイント経由の通信に切り替わる
- NAT ゲートウェイ経由は **インターネットを通る** → 「インターネット経由なし」要件を満たさない
- Transit Gateway で Bedrock の「サービス VPC」にピアリングは **技術的に不可**（Bedrock にユーザーがアクセスできる VPC は存在しない）

#### aws:SourceVpce 条件キー
```json
{
  "Condition": {
    "StringEquals": {
      "aws:SourceVpce": "vpce-xxxxxxxxxx"
    }
  }
}
```
- IAM ポリシーの Condition 要素でこれを指定すると、**特定の VPC エンドポイント経由のリクエストのみ**を許可できる
- エンドポイントポリシーと組み合わせて二重防御が可能

### データレイクの列単位アクセス制御

#### Lake Formation → Glue Data Catalog → Athena の連携
```
S3（生データ）
  ↓ Glue Crawler
Glue Data Catalog（テーブル定義）
  ↓ Lake Formation（権限管理レイヤー）
Athena（クエリ実行）
  → Lake Formation のフィルターが適用された結果のみ返る
```

#### アクセス制御粒度の比較
| 手段 | 粒度 | クロスアカウント列制御 |
|---|---|---|
| **Lake Formation LF-Tag ベース（TBAC）** | **列単位** | **✅ 対応** |
| Glue Data Catalog リソースポリシー | DB / テーブル単位 | ❌ 列制御不可 |
| Athena ワークグループ + IAM | ワークグループ単位 | ❌ 列フィルタリング不可 |
| S3 バケットポリシー / ACL | オブジェクト（ファイル）単位 | ❌ 列制御不可 |

#### Lake Formation LF-Tag ベースアクセスコントロール（TBAC）の特徴
- Data Catalog のリソース（DB・テーブル・**列**）にキーバリュータグを付与
- タグに基づいてアクセス権限を付与 → ポリシーの更新頻度が大幅に減少
- **AWS RAM** 経由で他アカウントに Data Catalog リソースを共有 → クロスアカウント列制御が実現

## ログ・コンプライアンス保持の設計パターン

### CloudTrail vs Bedrock モデル呼び出しログ（最重要区別）

| | CloudTrail | Bedrock モデル呼び出しログ |
|---|---|---|
| 記録内容 | **APIメタデータのみ**（誰が・いつ・何のAPIを呼んだか） | **実際のプロンプト内容・モデル応答** |
| プロンプト内容 | ❌ 記録しない | ✅ 記録する |
| 規制コンプライアンス用途 | △ 監査証跡にはなるが内容再現不可 | ✅ 内容まで再現可能 |
| 保持設定 | CloudTrail Lake（最大7年） | S3 + Object Lock |

→ **「規制コンプライアンスで呼び出し内容を保持」= 必ず Bedrock モデル呼び出しログ + S3**

### S3 Object Lock（コンプライアンスモード）

```
Bedrock モデル呼び出しログ記録
  → 宛先: S3バケット
  → S3 Object Lock: コンプライアンスモード
  → 保持期間: 7年
  → 効果: 管理者を含む誰もログを削除できない（イミュータブル）
  → ビジネスユニット分離: S3プレフィックスで区別
```

### Bedrock Prompt Management（プロンプトテンプレート管理）

- プロンプトテンプレートの**作成・バージョン管理・承認ワークフロー**を内蔵
- IAMポリシーでマルチパーティ承認を強制可能
- DynamoDBなどカスタム実装より運用オーバーヘッドが低い

#### プロンプトガバナンス（Task 5.2）

複数アプリ・チームをまたいでプロンプトの品質・一貫性を統制する仕組み。

| 要素 | 実現方法 |
|------|---------|
| **一元管理** | Bedrock Prompt Management で全プロンプトを集約 |
| **承認ワークフロー** | IAM ポリシーでマルチパーティ承認を強制 |
| **バージョン固定** | 本番は特定バージョンを参照（意図しないドリフト防止） |
| **アクセス制御** | IAM でチームごとに編集権限を制限 |
| **監査証跡** | CloudTrail で誰がいつ変更したかを記録 |

→ 試験で「複数チーム・アプリで統一されたプロンプト品質管理」文脈 → **Bedrock Prompt Management ＋ IAM ＋ CloudTrail** が正解構成

## データ匿名化・仮名化の手法（Task 3.2）

### 仮名化 + セキュアなマッピングテーブル
- 実IDを偽IDに置き換えるが、対応表（マッピングテーブル）を保持して元に戻せる
- マッピングテーブル自体を暗号化・厳格なアクセス制御・別環境で保管することが必須
- テーブルが漏れると再特定されるため、テーブルの保護がセキュリティの核心

```
患者ID: 12345 → 仮名: PTN-A7X2   ← データ本体（分析に使う）
マッピングテーブル: 12345 ↔ PTN-A7X2  ← 別VPC・暗号化・最小権限で厳格管理
```

### フォーマット保持暗号化（FPE: Format-Preserving Encryption）
- 暗号化後も元データと同じ形式（桁数・文字種）を維持する暗号化方式
- 既存システムのバリデーションロジックを変えずに暗号化できる

```
患者ID（元）: MED-12345  →  暗号化後: MED-87X2K  （形式そのまま）
```

### プライバシー予算（差分プライバシー）
- 差分プライバシーで「どこまでノイズを使ってよいか」を管理する上限値（ε: イプシロン）
- クエリを重ねるごとに予算を消費し、使い切ると追加クエリを拒否
- プライバシー保護とデータ有用性のバランスを数学的に管理する仕組み

### VPC エンドポイント種別（ゲートウェイ vs インターフェイス）

| 種別 | 対応サービス | 料金 |
|---|---|---|
| **ゲートウェイエンドポイント** | S3・DynamoDB のみ | **無料** |
| **インターフェイスエンドポイント（PrivateLink）** | Bedrock・その他大多数 | 有料 |

- Bedrock へのプライベート接続 → インターフェイスエンドポイント
- S3・DynamoDB を VPC 内に閉じる → ゲートウェイエンドポイント（コスト最適）

## 責務分離（Separation of Duties）パターン（Task 3.3）

モデルのライフサイクルで役割ごとに IAM ロールを分離する。「学習した人が本番デプロイもできる」状態を防ぐのが目的。金融・医療等の規制業界で必須。

```
DataScientistRole  → 学習ジョブの実行のみ可
ModelEvaluatorRole → 評価ジョブ・Clarify の実行のみ可
DeployerRole       → 本番エンドポイントへのデプロイのみ可（承認後）
MonitorRole        → Model Monitor の閲覧のみ可（変更不可）
```

## SageMaker Model Monitor 4種類 + Clarify 設定（Task 3.3）

### Model Monitor の4種類

| 種類 | 何を検出 | 使うもの |
|---|---|---|
| データ品質 | 入力データの欠損・分布の変化 | Model Monitor のみ |
| モデル品質 | 精度・F1スコアの劣化 | Model Monitor のみ |
| **バイアスドリフト** | 属性間の公平性の変化 | **Clarify モニター（別途設定必要）** |
| **Feature Attribution ドリフト** | 特徴量の重要度の変化 | **Clarify モニター（別途設定必要）** |

- バイアスドリフト・Feature Attribution ドリフトは **Model Monitor だけでは検出できない**
- Clarify を使ったモニタリングスケジュールを別途作成する必要がある

### Feature Attribution ドリフトとは

「どの特徴量がモデルの判断に影響しているか」の割合が時間とともに変化すること。

```
学習時：年齢(40%) > 収入(35%) > 地域(25%) でローン審査
6ヶ月後：地域(60%) > 年齢(25%) > 収入(15%) に変化
→ 地域への依存が急増 = 差別的判断のリスク → アラート
```

### 自動テストフレームワーク（継続的コンプライアンス評価）

```
Step Functions スケジュール実行：
  ① 既知の有害プロンプトセットでモデルをテスト
  ② Guardrails が正しく機能するか確認
  ③ バイアス指標が閾値内か確認
  → 全通過 → コンプライアンス証跡として記録
```

### 不正使用モニタリング

| サービス | 検出パターン |
|---|---|
| **CloudWatch Anomaly Detection** | 通常パターンを ML で学習 → 急激なリクエスト数増加等を自動検出 |
| **GuardDuty** | 深夜に普段アクセスしない IAM ロールが Bedrock を呼び出した等の脅威検知 |

## CloudWatch カスタムメトリクス 名前空間ルール（Task 3.4 セルフチェック）

- `AWS/Bedrock`・`AWS/SageMaker`・`AWS/Lambda` は AWS が予約済みの名前空間
- カスタムメトリクスは **`AWS/` で始まらない**独自名前空間を使う（例：`MyApp/FairnessMetrics`）
- 信頼度スコア・引用頻度・バイアス指標などは全てカスタム名前空間で `PutMetricData` で送る

## IAM Identity Center + SAMLフェデレーション + OU/SCP（公式模試2週目）

- 「既存のエンタープライズIdPと統合し、部署別RBAC、一元ガバナンス」→ **IAM Identity Center + SAMLフェデレーション**
  - SAMLフェデレーション＝外部IdP（AD/Entra ID/Okta等）が署名付きアサーションでユーザーを保証 → AWSはIAMユーザーを作らず一時権限を付与。二重管理なし・退職はIdP側無効化で即失効＝最もセキュア
  - 権限セットをOUごとに割当＝RBAC、監査＝CloudTrail
- **OU** = AWS Organizations内でアカウントを束ねるフォルダ（部署単位）。入れ子可
- **SCP** = OU/アカウントに適用する「禁止」の上限ガードレール（許可はしない、禁止のみ）。「組織全体で未承認モデル呼び出しを禁止」等
- 補足：Cognitoの "user" は「ログインする人間」全般（顧客だけでなく評価者・社内ユーザー）。ただし組織のAWSリソース制御はIAM Identity Center側

## AWS Config：既存リソースの構成コンプライアンス（公式模試2週目）

- 「**既存の**S3バケット等が基準（例：SSE-KMS＋カスタマーマネージドキー）に準拠しているか検証し、非準拠を是正」→ **AWS Config カスタムルール（RDK）＋自動修復**
- 混同注意：Macie=機密データの発見/分類（構成準拠チェックではない）／CloudTrail+Athenaで自作監査=overhead＆脆い／EventBridge+Step Functions=**新規PutObject時のゲート**で「既存リソースの検証」にならない
