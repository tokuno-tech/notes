# AWS セキュリティ・ガバナンス サービス

## AWS Service Catalog（AIP-76）

### 一言で

> **IT管理者が承認済みのAWSリソース構成を「製品」としてカタログ化し、部門（エンドユーザー）がセルフサービスでデプロイできるようにするサービス。**

### 基本構造

```
IT管理者（ガバナンスチーム）
  ↓ 承認済みCloudFormationテンプレートを「製品」として登録
ポートフォリオ（製品のグループ）
  ↓ アカウント・部門に共有
エンドユーザー（各部門）
  → Service Catalog コンソールからクリックでプロビジョニング
```

### ① バージョン管理

- 各製品（Product）に**複数バージョン**を保持できる
- 管理者は特定バージョンを**有効 / 無効**に切り替え可能
- `「最新の承認済みバージョンのみ使用可能にする」` → Service Catalog の組み込み機能で実現
- 部門が古いバージョンを使い続けるリスクを排除できる

```
製品: 「Bedrock AIチャットボット基盤」
  └─ v1.0（無効化済み）
  └─ v2.0（無効化済み）
  └─ v3.0（✅ 現在有効・部門が使用可能なのはこれだけ）
```

### ② セルフサービスポータル

- エンドユーザーは **Service Catalog コンソール**にアクセス
- 許可されたポートフォリオの製品一覧から選んで「起動」するだけ
- CloudFormation の知識不要。**技術的なハードルが低い**
- 部門ごとにアクセスできるポートフォリオを IAM で制御

### ③ 起動制約（Launch Constraint）

```
通常のデプロイ：
  エンドユーザーのIAM権限 → リソース作成（広い権限が必要）

Service Catalog（起動制約あり）：
  エンドユーザーのIAM権限 → 最小限（Service Catalogの使用権限のみ）
  Service Catalogが指定の「起動ロール」でリソース作成
  → 部門ユーザーに過剰な権限を与えずに済む
```

| 制約の種類                        | 内容                                                              |
| --------------------------------- | ----------------------------------------------------------------- |
| **起動制約（Launch Constraint）** | 製品起動時に使用するIAMロールを指定。エンドユーザーの権限を最小化 |
| **テンプレート制約**              | CloudFormationパラメータの値を制限                                |
| **通知制約**                      | 製品起動・終了時にSNSで通知                                       |
| **TagOptions**                    | タグの標準化を強制（コスト管理・リソース追跡に有効）              |

### Service Catalog vs 似たサービスの違い

|                      | Service Catalog                          | CodePipeline              | Control Tower                |
| -------------------- | ---------------------------------------- | ------------------------- | ---------------------------- |
| **対象ユーザー**     | 非技術者の部門担当者                     | 開発者・DevOps            | 管理者（アカウント単位）     |
| **主な目的**         | 承認済みテンプレートのセルフサービス配布 | CI/CDによるデプロイ自動化 | マルチアカウントのガバナンス |
| **バージョン管理**   | ✅ 製品レベルで組み込み                  | Gitリポジトリで管理       | ❌ テンプレート管理なし      |
| **セルフサービスUI** | ✅ カタログポータル                      | ❌                        | ❌                           |

## AWS Config（AIP-76）

### 基本機能（インフラ構成の記録 + ルール評価）

**「AWSリソースの構成変更を記録する」だけでなく、「ルールに基づいてコンプライアンスを評価する」機能を持つ。**

```
① 記録（Recording）
   AWSリソースの設定変更を継続的に記録
   「いつ・誰が・何をどう変えたか」の履歴を保持

② ルール評価（Rules）
   定義したルールに照らしてリソースの準拠状況を評価
   組み込みルールの例：
   ・s3-bucket-server-side-encryption-enabled
   ・restricted-ssh

③ コンプライアンスレポート
   準拠（Compliant）/ 非準拠（Non-Compliant）をダッシュボードで可視化
```

### 統制の種類（重要な概念）

| 種類                         | タイミング                 | サービス例                        |
| ---------------------------- | -------------------------- | --------------------------------- |
| **予防的統制（Preventive）** | 違反操作を事前に防ぐ       | SCP、IAM、Service Catalog起動制約 |
| **検出的統制（Detective）**  | 違反が起きた後に検出する   | AWS Config、CloudTrail、GuardDuty |
| **修正的統制（Corrective）** | 違反を検出して自動修正する | AWS Config + SSM Automation       |

→ Config は「**検出的統制**」。デプロイされた後に問題を発見する。

```
AWS Config が得意：
  ✅ リソース設定変更の追跡
  ✅ コンプライアンスルールによる継続的評価

AWS Config が苦手（引っかけ）：
  ❌ 非準拠を事前に防ぐ → SCP・IAM が担当
  ❌ テンプレートのバージョン管理・配布 → Service Catalog が担当
```

## AWS Control Tower（AIP-76）

### 一言で

> **マルチアカウントのAWS環境（ランディングゾーン）を自動セットアップし、組織全体のガバナンスを提供するサービス。**

### 何をするか

```
① ランディングゾーン（Landing Zone）の自動構築
   ・ログアーカイブアカウント
   ・監査アカウント
   ・組織構造（OU）の標準化セットアップ

② ガードレールの適用（予防的 / 検出的）
   予防的：SCP で特定操作を禁止
   検出的：AWS Config ルールで非準拠を検出

③ Account Factory
   標準化された新規アカウントをテンプレートから作成
```

### Control Tower vs Service Catalog の住み分け

```
Control Tower：アカウント単位のガバナンス基盤
               「このアカウントでは〇〇を禁止」

Service Catalog：製品（テンプレート）の配布・管理基盤
                「この部門には〇〇の製品を使わせる」
```

## AWS Inspector

**Amazon Inspector** はEC2・Lambda の脆弱性スキャンだけでなく、**ECR（Elastic Container Registry）に push されたコンテナイメージも自動スキャン**できる。

| スキャン対象            | 内容                                   |
| ----------------------- | -------------------------------------- |
| EC2インスタンス         | OS・ソフトウェアの脆弱性               |
| Lambda関数              | コードの依存パッケージの脆弱性         |
| **ECRコンテナイメージ** | **イメージ内のOS・パッケージの脆弱性** |

- ECRへのpush時に自動でスキャンが走る（継続スキャンも可能）
- 検出結果はInspectorコンソール・Security Hubに集約される

## SAML フェデレーション

### SAML とは

**SAML（Security Assertion Markup Language）** = 異なるシステム間で「このユーザーは認証済みです」という情報を安全に受け渡す標準規格。

```
イメージ：
  社内の Active Directory / Okta（会社の ID 管理）
         ↓ SAML assertion（XML形式の「認証済み証明書」）
       AWS STS
         ↓ 一時的な AWS 認証情報に変換
       AWS リソース（Bedrock 等）にアクセス

→ ユーザーは会社の既存 ID/パスワードでそのまま AWS にアクセスできる
→ AWS 側に新たな IAM ユーザーを作る必要がない
→ 認証情報は一時的（長期 Access Key を持たない）
```

### SAML vs OIDC

|               | SAML                               | OIDC                          |
| ------------- | ---------------------------------- | ----------------------------- |
| 形式          | XML ベース                         | JSON/JWT ベース               |
| 主な用途      | 企業向けSSO（Active Directory等）  | モバイル/Webアプリ向け        |
| AWSでの使い方 | IAM Identity Center との連携が多い | Amazon Cognito との連携が多い |

### AWS での SAML フェデレーション構成

```
企業の IdP（Okta/Azure AD/ADFS 等）
  ↓ SAML 2.0
IAM Identity Center
  ↓ 権限セット（カスタムアクセス許可）
一時的な STS 認証情報
  ↓
Amazon Bedrock / AWS サービス
```

### Active Directory（AD）とは

Microsoft製の社内向けディレクトリサービス。社員・PC・部署グループなどの情報を一元管理し、社内ネットワークでの認証（ログイン）と認可（どのリソースにアクセスできるか）を担う。オンプレミス版が「Active Directory」、クラウド版が「Microsoft Entra ID（旧Azure AD）」。試験で「既存のIdPがEntra ID/Active Directory」と出たら、SAMLフェデレーションでAWSと連携するのが定石（AD FSがオンプレADのSAML変換役）。

### AssumeRole（素の版）vs AssumeRoleWithSAML

同じ「一時認証情報を発行するSTS API」でも、呼び出す前提が違う。ここを混同すると「IAMユーザー不要のはずが実は必要」という罠にはまる。

| API                       | 呼び出す前に必要なもの                                                       | IAMユーザーは要る?                        |
| ------------------------- | ---------------------------------------------------------------------------- | ----------------------------------------- |
| **AssumeRole**（素）      | 呼び出し元が**既にIAMプリンシパル**（IAMユーザーまたは既存ロール）であること | ✅ 要る（長期クレデンシャルの管理が発生） |
| **AssumeRoleWithSAML**    | 外部IdPが発行した**SAMLアサーション**                                        | ❌ 不要（IdP側の認証をそのまま信頼）      |
| AssumeRoleWithWebIdentity | OIDCトークン（Cognito/Google等）                                             | ❌ 不要                                   |

→「オンプレアプリからAssumeRole APIコールでクロスアカウントアクセス」という選択肢は、素のAssumeRoleを指す＝**呼び出し元アプリ自身がIAMユーザー/ロールの長期クレデンシャルを持つ必要がある**＝既存IdP（Entra ID等）を一切使わない構成。既存のSAML対応IdPがあるのにこれを選ぶと「IdP連携なし」「認証情報の二重管理」で不正解になりやすい。
→ 素のAssumeRoleが妥当なのは「SAMLフェデレーションが使えない/組めない」場合の代替手段であり、既存IdPがSAML対応済みなら基本的に選ばない。

## AWS Well-Architected 3点セット

ベストプラクティス系の問題で混同しやすい3つを整理。

### ① Well-Architected Framework（フレームワーク）

**= AWSの設計思想・ベストプラクティス集（ドキュメント）**

6つの柱：

1. 運用上の優秀性
2. セキュリティ
3. 信頼性
4. パフォーマンス効率
5. コスト最適化
6. 持続可能性

→ ホワイトペーパーで読む「教科書」。

### ② Well-Architected Lens（レンズ）

**= 特定分野向けの追加ガイドライン（ドキュメント）**

- **Generative AI Lens** ← AIP頻出
- Machine Learning Lens
- SaaS Lens / Financial Services Lens / IoT Lens など

→ Frameworkに「メガネをかけて分野特化の観点を追加」するイメージ。中身はやはり設計思想。

### ③ Well-Architected Tool（WA Tool）

**= 自分のシステムを上記基準で自己診断するAWSコンソール上のツール**

- マネジメントコンソールから利用
- ワークロード（自分のシステム）を登録 → 質問票に回答
- 「高リスク項目」「中リスク項目」を自動レポート
- **Lensを適用すると追加質問が増える**（例: Generative AI Lens適用 → AI特化の質問追加）

→ 設計思想を「やってるか？」と質問してくれる**自動レビュー官**。リソースは作らない。

### 関係性

```
Framework（教科書）         ─┐
                            ├─→ WA Tool（試験官）が「やってる？」と質問
Generative AI Lens（追加教科書）─┘
```

| ツール               | 役割                           |
| -------------------- | ------------------------------ |
| Framework / Lens     | 何をすべきか（What）           |
| **WA Tool**          | やってるか診断（Check）        |
| CloudFormation / CDK | リソースを作る（Build）        |
| Trusted Advisor      | 既存リソースの運用状況チェック |
| AWS Config           | リソースの設定変更を監視       |

## サーキットブレーカー × Step Functions

**「障害が続いたら一時的にリクエストを遮断する」パターン。**

```
CLOSED（通常）→ 連続失敗がしきい値を超えたら → OPEN（遮断）→ 一定時間後 → HALF-OPEN（試験）
```

Step FunctionsでのImplementation：

- **Choice ステート** × エラーカウント × タイムアウト
- 複数実行をまたいでエラー数を保持 → しきい値超えで別ルートへ自動遷移

| 選択肢              | ダメな理由                                     |
| ------------------- | ---------------------------------------------- |
| Exponential Backoff | リトライするだけ。「遮断」しない               |
| API Gateway再試行   | HTTPレベル。状態を持てない                     |
| DynamoDB追跡        | 追跡はできるが即時作動できない                 |
| EventBridge         | 事後対応（リアクティブ）。予防的に遮断できない |

## IAM Identity Center とは

「AWS Organizations配下の複数アカウントに、1つのIDでシングルサインオンする仕組み」。identityはOrganizations全体で1箇所（管理アカウント／委任アカウント）に置かれ、外部IdP（AD/Entra ID/Okta等）と連携するか、Identity Center自前のディレクトリを使う。ユーザー自身は長期認証情報（パスワード／アクセスキー）をアカウント側に持たない。

ユーザーにアクセスさせたいアカウントへは、**パーミッションセット**を割り当てるだけで良い（アカウントごとに個別のID作成は不要）。パーミッションセットを割り当てると、Identity Centerが裏側で対象アカウントに**IAMロールを自動作成**し、ログイン時にそのロールへSTSで一時的にスイッチする。つまりログインしても「そのアカウントのIAMユーザーになる」わけではない。

退職・異動時はIdP側（AD等）で1回無効化すれば、割り当てられている全アカウントのアクセスが即座に失効する。パーミッションセットの実体については[パーミッションセット（Permission Set）の正体](#パーミッションセットpermission-setの正体identity-center管理下のiamロール)を参照。

**IAMポリシー条件はどこにアタッチするか（グループ単位ではない）**：条件付きIAMポリシーは「グループ」に直接アタッチするのではなく、**パーミッションセットの中身**として定義する。ここでの「グループ」はclassicなIAMグループ（単一アカウント内のIAMリソース）とは別物で、Identity Centerに接続されたディレクトリ（外部IdPやIdentity Center自前ディレクトリ）側のグループを指す。実際の割り当ては「①ユーザー/グループを選ぶ→②アカウントを選ぶ→③パーミッションセットを選ぶ」という3点セットの組み合わせで行い、その結果として対象グループのメンバーが対象アカウントで（条件込みの）パーミッションセットのポリシーを持つIAMロールを使えるようになる。

## ID フェデレーション：IAM Identity Center vs 直接SAMLフェデレーション vs Cognito

**IDフェデレーション：「既存の認証基盤のIDをAWSでも使い回す」**

|                          | IAM Identity Center                             | 直接SAMLフェデレーション（IAMのSAMLプロバイダー） | Amazon Cognito                           |
| ------------------------ | ----------------------------------------------- | ------------------------------------------------- | ---------------------------------------- |
| 主な対象                 | **社員**（ワークフォース）                      | **社員**（ワークフォース）                        | **アプリのエンドユーザー**（顧客・CIAM） |
| 典型例                   | 社員がAWSコンソール/複数アカウントにログイン    | 社員が個別アカウントにSAMLでログイン              | アプリの会員登録・ログイン               |
| マルチアカウント一元管理 | ✅ 許可セットを組織全体に一括配布               | ❌ アカウントごとに個別設定が必要                 | ❌ Identity Poolは1アカウント単位        |
| AD/Entra ID連携          | ネイティブ・一元的                              | 可能（アカウントごとに設定）                      | 可能だが本来用途外・設定が複雑           |
| 運用オーバーヘッド       | 最小（AWS公式ベストプラクティスが名指しで推奨） | アカウント数に比例して増加                        | 目的外使用のため増加                     |
| スケール                 | 数百〜数千人（組織のワークフォース）            | 同左                                              | 数百万人（消費者向け）                   |

```
社内Active Directory（AD）
    ↓ フェデレーション
IAM Identity Center（複数アカウントを横断するSSOハブ）
    ↓ 許可セット（Permission Set）
Bedrock / S3等にアクセス（社員が社内IDでそのままAWSにアクセス）
```

IAM公式ベストプラクティス："For centralized access management, we recommend that you use AWS IAM Identity Center to manage access to your accounts and permissions within those accounts."（人間ユーザーの集中アクセス管理にはIAM Identity Centerを推奨）

### パーミッションセット（Permission Set）の正体＝Identity Center管理下のIAMロール

パーミッションセットは単なる「条件付きポリシーの入れ物」ではなく、**アカウントに割り当てると裏側でIAM Identity Centerが自動的にIAMロールを各アカウントに作成・管理する**仕組み。パーミッションセットの内容を編集すれば、対象の全アカウントのロール・ポリシーが自動で追従する（公式：「As you modify the permission set, IAM Identity Center ensures that the corresponding IAM policies and roles are updated accordingly」）。

- **パーミッションセット vs カスタムIAMロール＋StackSets**：どちらも最終的には「特定モデルARNのみ許可」のような条件付きIAMポリシーを使うという点は同じ。違いは管理主体——パーミッションセットは既存のIdentity Center基盤内でロールの作成・更新・同期をAWSに任せられるのに対し、カスタムロール＋StackSetsは自前でCloudFormationテンプレートを書きデプロイ・ドリフト管理をする必要がある
- 会社が既にOrganizations＋Identity Centerでアクセス管理している前提がある場合、StackSetsで別のロール管理系統を新設するのは**既存のSSO基盤と競合する冗長な仕組み**になり運用オーバーヘッド増で不正解になりやすい

### SAML federationとSCIM provisioningは別物（重要）

外部IdP（Okta、Entra ID等）とIAM Identity Centerを連携する際、**SAMLだけではユーザー・グループの情報は得られない**。

- **SAML**：認証（サインイン）のプロトコル。「このユーザーは本人である」を証明するだけで、IdP側のユーザー・グループ一覧をIAM Identity Centerが**問い合わせる手段を持たない**
- **SCIM（System for Cross-Domain Identity Management）**：ユーザー・グループの情報をIdPからIAM Identity Centerに**プロビジョニング（同期）**するプロトコル。手動プロビジョニングでも代替可能
- 外部IdP統合では**SAML（認証）＋SCIM（ユーザー・グループ同期）の両方**を設定するのが標準構成。SCIM（または手動プロビジョニング）でIAM Identity Center側にユーザー・グループを事前に登録しておかないと、AWSアカウントやアプリケーションへの権限割り当てができない

AWS公式ドキュメントの明言："The SAML protocol does not provide a way to query the IdP to learn about users and groups. Therefore, you must make IAM Identity Center aware of those users and groups by provisioning them into IAM Identity Center."

## Amazon Verified Permissions + Cedar ポリシー

「**アプリの認可ロジックを外出しして一元管理する**サービス」。

**認証 vs 認可：**

- 認証（Authentication）：あなたは誰？ → Cognito
- 認可（Authorization）：あなたは何ができる？ → Verified Permissions

```
アプリ → 「このユーザーはこの操作していい？」と問い合わせ
    ↓
Verified Permissions が Cedar ポリシーで判定
    ↓
permit / deny を返す
```

**Cedarポリシーの形式：**

```
permit(
  principal in 製造部門,
  action == "BedrockInvoke",
  resource.dataLevel == "一般"
);
```

「誰が・何を・どのリソースに」という形式。

## AWS RAM（Resource Access Manager）

「**AWSリソースを複数アカウント間で共有する**サービス」。

```
アカウントA（AI開発チーム）でカスタムモデル作成
    ↓ RAMで共有
アカウントB（製造部門）→ そのモデルで推論
アカウントC（営業部門）→ そのモデルで推論
```

**共有できるリソース例：** Bedrockカスタムモデル、Knowledge Base、VPCサブネット、Transit Gateway

**SCPとの違い（重要）：**

|      | SCP                      | RAM                              |
| ---- | ------------------------ | -------------------------------- |
| 目的 | **何をできないか**を制限 | **リソースを別アカウントに渡す** |
| 方向 | 制限・禁止               | 共有・許可                       |

SCPだけではリソースを別アカウントに渡せない。RAMだけでは権限の制限ができない。**両方必要**。

## PrivateLink エンドポイントポリシー

「**インターネットを経由せずにBedrockに接続**する構成」。

```
オンプレ / 社内ネットワーク
    ↓ Direct Connect or VPN
VPC
    ↓ PrivateLink（VPCエンドポイント）
Bedrock
```

**エンドポイントポリシー：** PrivateLink経由で何ができるかをIAMポリシーで制限（特定モデルのみ呼び出し可等）。

**セキュリティグループのCIDR制限：** 社内ネットワーク（10.0.0.0/8等）のみ許可。

### エンドポイントポリシーの限界と強制力の序列

エンドポイントポリシーは**そのVPCエンドポイントを経由したリクエストの中身**（プリンシパル・アクション・リソース）しか制御できない。**エンドポイントを経由しない通信（パブリックエンドポイントへの直接アクセス）は制御対象外**——エンドポイントを使わせること自体を強制する仕組みではない。

「IAMロールの設定やアプリケーションの動作に関係なく、パブリックインターネット経由のBedrockアクセスを禁止したい」という要件では、強制力の強さの序列で選ぶ：

| 仕組み                     | 強制力                                                                | 弱点                                               |
| -------------------------- | --------------------------------------------------------------------- | -------------------------------------------------- |
| **SCP**（OUにアタッチ）    | アカウント管理者のIAM設定変更に関係なく強制。組織ガードレールの最上位 | ─                                                  |
| **IAMロールのポリシー**    | ロール単位。アカウント管理者が変更・誤設定できてしまう                | 組織全体を強制できない                             |
| **エンドポイントポリシー** | エンドポイント経由の通信の中身は絞れる                                | 経路自体（パブリックエンドポイント迂回）は防げない |

→ 「〇〇の設定に関係なく強制」「アカウント管理者の裁量を排除」というキーワードが出たら**SCP**を最優先で検討する。

## Transit Gateway + ブラックホールルーティング

**Transit Gateway：** 複数VPCをハブ1つで接続するサービス（VPC Peeringの代替）。

```
開発VPC ─┐
テストVPC ┼─ Transit Gateway（ハブ）
本番VPC  ─┘
```

**ドメイン分離：** 環境ごとにルートテーブルを分けて相互アクセスを制限。

```
開発用ルートテーブル  → 開発VPCのみ到達可
本番用ルートテーブル  → 本番VPCのみ到達可
```

**ブラックホールルーティング：** 特定の宛先へのトラフィックを無条件に破棄するルート。

```
192.168.0.0/16 → blackhole（破棄）
→ このアドレス宛のパケットは届かず消える
```

セキュリティインシデント時の即時経路遮断、環境間の完全分離に使う。

「**VPC内のトラフィックをパケットの中身まで検査**できるマネージドファイアウォール」。

| ---------------------- | -------------------------- | ---------------------------- |

## IAM 段階的アクセスレベル（動的リスク評価）

「**運用の重要度と自動リスク評価に基づいた段階的アクセス制御**」。

```
低リスク操作  → 自動実行OK
中リスク操作  → 条件付きで実行（リスク評価通過時のみ）
高リスク操作  → 制限 or ブロック
```

固定的なルールではなく運用状況に応じてスケールする点がポイント。
「リアルタイム保護」「不必要な遅延なし」要件で、サーキットブレーカー（Step Functions）と組み合わせて使う。

## IAM ABAC と Bedrock 条件キー（AIP-51）

### IAM ABAC（属性ベースアクセスコントロール）

**RBAC vs ABAC 比較**：

| 観点             | RBAC（ロールベース）             | ABAC（属性ベース）             |
| ---------------- | -------------------------------- | ------------------------------ |
| 制御単位         | ロール（職種・役職）             | タグ（属性キー＝値）           |
| ポリシー数       | リソース種類 × ロール数 分必要   | 少数の汎用ポリシーで済む       |
| リソース追加時   | ポリシー更新が必要               | タグ設定だけで自動適用         |
| スケーラビリティ | 組織拡大でポリシーが爆発的に増加 | タグ付けルールを守ればスケール |
| 典型ユースケース | 小規模・シンプルな権限管理       | 診療科・部署など多部門組織     |

```
ABAC の動作イメージ（診療科タグで制御）：
  IAM ユーザー/ロール に department=cardiology タグ
        ↓
  ポリシー: department タグが一致するリソースのみ許可
        ↓
  新しい診療科が増えても ポリシー変更不要、タグ設定だけで OK
```

```
ABAC 例（診療科タグで制御）：
  IAM ユーザー/ロール に department=cardiology タグ
        ↓
  ポリシー: department タグが一致するリソースのみ許可
        ↓
  新しい診療科が増えても ポリシー変更不要、タグ設定だけで OK
```

### Bedrock 専用の IAM 条件キー

| 条件キー                      | 用途                       |
| ----------------------------- | -------------------------- |
| `bedrock:ModelId`             | 利用可能な基盤モデルを制限 |
| `bedrock:GuardrailIdentifier` | 使用するガードレールを強制 |

```json
{
  "Condition": {
    "StringEquals": {
      "bedrock:ModelId": "anthropic.claude-3-sonnet-20240229-v1:0",
      "bedrock:GuardrailIdentifier": "arn:aws:bedrock:...:guardrail/cardiology-guardrail"
    }
  }
}
```

**試験での判断軸**：

```
「診療科ごとに使えるモデル・ガードレールを制限」
  → IAM ABAC + bedrock:ModelId + bedrock:GuardrailIdentifier 条件キー

「SCP でアカウント単位に制限」
  → ❌ 粒度が粗い（同一アカウント内の診療科分離には不十分）

「Bedrock Guardrails でコンテンツフィルター」
  → ❌ それは入出力の安全制御。モデル利用の制限ではない
```

### 予防的統制 vs 検出的統制（AIP-51）

```
予防的統制（preventive）：不正を事前に防ぐ
  → IAM ポリシー + 条件キー、VPC エンドポイント

検出的統制（detective）：発生後に検知・記録する
  → CloudTrail, GuardDuty, CloudWatch アラーム

「厳密に制限する」= 予防的統制が必要
ガードレールや Insights だけでは予防的統制にならない ← 試験の罠
```

IAM ポリシーには2種類ある：

## Bedrock プライベート接続 × データレイク列単位制御（AIP-58）

### Bedrock へのプライベート接続：インターフェイス VPC エンドポイント

- **bedrock-runtime** エンドポイント = InvokeModel などの推論 API 用（bedrock エンドポイントとは別）
- プライベート DNS を有効化すれば **SDK のコードを変更せず** にエンドポイント経由の通信に切り替わる
- NAT ゲートウェイ経由は **インターネットを通る** → 「インターネット経由なし」要件を満たさない
- Transit Gateway で Bedrock の「サービス VPC」にピアリングは **技術的に不可**（Bedrock にユーザーがアクセスできる VPC は存在しない）

#### aws:SourceVpce 条件キー

```json
{
  "Condition": {
    "StringEquals": {
      "aws:SourceVpce": "vpce-xxxxxxxxxx"
    }
  }
}
```

- IAM ポリシーの Condition 要素でこれを指定すると、**特定の VPC エンドポイント経由のリクエストのみ**を許可できる
- エンドポイントポリシーと組み合わせて二重防御が可能

### データレイクの列単位アクセス制御

Lake Formation / Glue Data Catalog の詳細は [data_pipeline_integration.md](./data_pipeline_integration.md) に集約。ここでは Bedrock への PrivateLink 接続と、データレイク側の列単位アクセス制御を組み合わせる設計パターンとして扱う。

#### アクセス制御粒度の比較

| 手段                                     | 粒度                         | クロスアカウント列制御  |
| ---------------------------------------- | ---------------------------- | ----------------------- |
| **Lake Formation LF-Tag ベース（TBAC）** | **列単位**                   | **✅ 対応**             |
| Glue Data Catalog リソースポリシー       | DB / テーブル単位            | ❌ 列制御不可           |
| Athena ワークグループ + IAM              | ワークグループ単位           | ❌ 列フィルタリング不可 |
| S3 バケットポリシー / ACL                | オブジェクト（ファイル）単位 | ❌ 列制御不可           |

#### Lake Formation LF-Tag ベースアクセスコントロール（TBAC）の特徴

- Data Catalog のリソース（DB・テーブル・**列**）にキーバリュータグを付与
- タグに基づいてアクセス権限を付与 → ポリシーの更新頻度が大幅に減少
- **AWS RAM** 経由で他アカウントに Data Catalog リソースを共有 → クロスアカウント列制御が実現

## ログ・コンプライアンス保持の設計パターン

### CloudTrail vs Bedrock モデル呼び出しログ（最重要区別）

|                          | CloudTrail                                             | Bedrock モデル呼び出しログ           |
| ------------------------ | ------------------------------------------------------ | ------------------------------------ |
| 記録内容                 | **APIメタデータのみ**（誰が・いつ・何のAPIを呼んだか） | **実際のプロンプト内容・モデル応答** |
| プロンプト内容           | ❌ 記録しない                                          | ✅ 記録する                          |
| 規制コンプライアンス用途 | △ 監査証跡にはなるが内容再現不可                       | ✅ 内容まで再現可能                  |
| 保持設定                 | CloudTrail Lake（最大7年）                             | S3 + Object Lock                     |

→ **「規制コンプライアンスで呼び出し内容を保持」= 必ず Bedrock モデル呼び出しログ + S3**

### CloudTrail：management event vs data event（Bedrock API別）

| 区分                 | 対象API                                                                                                                           | デフォルト記録                                                                 |
| -------------------- | --------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| **management event** | `InvokeModel` / `InvokeModelWithResponseStream` / `Converse` / `ConverseStream`、その他大半のBedrock制御系API                     | ✅ 追加設定不要で自動記録                                                      |
| **data event**       | `InvokeAgent` / `InvokeInlineAgent` / `Retrieve` / `RetrieveAndGenerate` / `InvokeFlow` / `InvokeModelWithBidirectionalStream` 等 | ❌ advanced event selectorで明示的に有効化しないと記録されない（追加課金あり） |

- `InvokeModel`系のログ本文は `requestParameters: {modelId: ...}` のみ。**プロンプト本文・レスポンス本文は含まれない**（`responseElements: null`）→ 上表の「APIメタデータのみ」の裏付け
- 「エージェント呼び出し／KB検索／Flow実行の監査証跡が必要」という問題で「CloudTrailを有効にするだけ」は罠になりうる → **data eventの明示設定**が必要な場合がある

### S3 サーバーアクセスログ記録

- S3バケットへの**全リクエストの詳細ログ**（誰が・いつ・どのオブジェクトに・どんな操作をしたか）を別バケットに配信する機能
- CloudTrail（API呼び出しの監査）や Bedrockモデル呼び出しログ（プロンプト内容）とは対象が異なる：**S3に保存済みのオブジェクト自体へのアクセス証跡**（読み取り・書き込みされたか）を残す
- 「S3に保存したプロンプト/成果物データソースの監査可能な証跡」という要件が出たら、CloudWatchやモデル呼び出しログだけでは埋まらず**S3サーバーアクセスログ記録**が要る（バケットレベルのアクセス監査という別レイヤーのため）

### S3 Object Lock（コンプライアンスモード）

```
Bedrock モデル呼び出しログ記録
  → 宛先: S3バケット
  → S3 Object Lock: コンプライアンスモード
  → 保持期間: 7年
  → 効果: 管理者を含む誰もログを削除できない（イミュータブル）
  → ビジネスユニット分離: S3プレフィックスで区別
```

### Bedrock Prompt Management（プロンプトテンプレート管理）

- プロンプトテンプレートの**作成・バージョン管理・承認ワークフロー**を内蔵
- IAMポリシーでマルチパーティ承認を強制可能
- DynamoDBなどカスタム実装より運用オーバーヘッドが低い

#### プロンプトガバナンス（Task 5.2）

複数アプリ・チームをまたいでプロンプトの品質・一貫性を統制する仕組み。

| 要素                 | 実現方法                                             |
| -------------------- | ---------------------------------------------------- |
| **一元管理**         | Bedrock Prompt Management で全プロンプトを集約       |
| **承認ワークフロー** | IAM ポリシーでマルチパーティ承認を強制               |
| **バージョン固定**   | 本番は特定バージョンを参照（意図しないドリフト防止） |
| **アクセス制御**     | IAM でチームごとに編集権限を制限                     |
| **監査証跡**         | CloudTrail で誰がいつ変更したかを記録                |

→ 試験で「複数チーム・アプリで統一されたプロンプト品質管理」文脈 → **Bedrock Prompt Management ＋ IAM ＋ CloudTrail** が正解構成

## Bedrock ネイティブプライバシー + AWS AI サービス オプトアウトポリシー（Task 3.2）

### Bedrock のデフォルト保証

- 顧客のプロンプト・レスポンスは**基盤モデルの再学習に使われない**（契約レベルの保証）
- 転送中・保管中の暗号化もデフォルト適用済み

### AWS AI Services Opt-Out Policy

- **AWS Organizations** のポリシーで設定
- Comprehend・Rekognition・Transcribe 等の AI サービス全般に対して「コンテンツをサービス改善に使用しない」を**組織単位で強制**できる
- Bedrock は元々使わない設計だが、他の AI サービスへの適用が主な用途

## データ匿名化・仮名化の手法（Task 3.2）

### 仮名化 + セキュアなマッピングテーブル

- 実IDを偽IDに置き換えるが、対応表（マッピングテーブル）を保持して元に戻せる
- マッピングテーブル自体を暗号化・厳格なアクセス制御・別環境で保管することが必須
- テーブルが漏れると再特定されるため、テーブルの保護がセキュリティの核心

```
患者ID: 12345 → 仮名: PTN-A7X2   ← データ本体（分析に使う）
マッピングテーブル: 12345 ↔ PTN-A7X2  ← 別VPC・暗号化・最小権限で厳格管理
```

### フォーマット保持暗号化（FPE: Format-Preserving Encryption）

- 暗号化後も元データと同じ形式（桁数・文字種）を維持する暗号化方式
- 既存システムのバリデーションロジックを変えずに暗号化できる

```
患者ID（元）: MED-12345  →  暗号化後: MED-87X2K  （形式そのまま）
```

### プライバシー予算（差分プライバシー）

- 差分プライバシーで「どこまでノイズを使ってよいか」を管理する上限値（ε: イプシロン）
- クエリを重ねるごとに予算を消費し、使い切ると追加クエリを拒否
- プライバシー保護とデータ有用性のバランスを数学的に管理する仕組み

### VPC エンドポイント種別（ゲートウェイ vs インターフェイス）

| 種別                                              | 対応サービス          | 料金     |
| ------------------------------------------------- | --------------------- | -------- |
| **ゲートウェイエンドポイント**                    | S3・DynamoDB のみ     | **無料** |
| **インターフェイスエンドポイント（PrivateLink）** | Bedrock・その他大多数 | 有料     |

- Bedrock へのプライベート接続 → インターフェイスエンドポイント
- S3・DynamoDB を VPC 内に閉じる → ゲートウェイエンドポイント（コスト最適）

**設定の仕組みも別物**（模試の解説がここを誤記していることがあるので要注意、2026-07-08公式ドキュメント裏取り済み）：

|                    | インターフェイスエンドポイント                                                                                                             | ゲートウェイエンドポイント                                                                   |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------- |
| 実体               | サブネットに**ENI**（プライベートIP）を作成                                                                                                | ENIなし                                                                                      |
| アクセス制御       | **セキュリティグループ**でポート443のHTTPS許可                                                                                             | **エンドポイントポリシー**（リソースベースポリシー）                                         |
| 経路の解決         | **プライベートDNS**を有効化すると、通常の公開ホスト名が裏の非表示ホストゾーン経由でENIのプライベートIPに解決される。アプリのコード変更不要 | **ルートテーブルに専用ルートを追加**（プレフィックスリスト宛のトラフィックをゲートウェイへ） |
| ルートテーブル変更 | **不要**                                                                                                                                   | **必要**                                                                                     |

「インターフェイスエンドポイントのためにルートテーブルを設定する」という記述は不正確——ルートテーブル変更はゲートウェイエンドポイント側の話。


## IAM Identity Center + SAMLフェデレーション + OU/SCP（公式模試2週目）

- 「既存のエンタープライズIdPと統合し、部署別RBAC、一元ガバナンス」→ **IAM Identity Center + SAMLフェデレーション**
  - SAMLフェデレーション＝外部IdP（AD/Entra ID/Okta等）が署名付きアサーションでユーザーを保証 → AWSはIAMユーザーを作らず一時権限を付与。二重管理なし・退職はIdP側無効化で即失効＝最もセキュア
  - 権限セットをOUごとに割当＝RBAC、監査＝CloudTrail
- **OU** = AWS Organizations内でアカウントを束ねるフォルダ（部署単位）。入れ子可
- **SCP** = OU/アカウントに適用する「禁止」の上限ガードレール（許可はしない、禁止のみ）。「組織全体で未承認モデル呼び出しを禁止」等
- **サービスリンクロール（Service-Linked Role）** = 特定のAWSサービスに紐づく専用IAMロールで、そのサービス自身が裏側でAssumeして処理を行うためのもの（例：AWS Organizations用のサービスリンクロールは、Organizationsが他サービスとの信頼関係設定等を代行するためのもの）。**人間やチームにBedrockモデルの呼び出し権限を割り当てる目的の仕組みではない**。IAM管理者は閲覧のみ可能で編集不可、削除も関連リソース削除後でないとできない
- **粒度の注意（OUがあれば十分ではない）**：各部門にOUがあっても、OUレベルの制御は「部門単位」までしか絞れない。部門内に複数チームがいて、その一部だけが承認済みという場合、OUだけでは「その部門の中のこのチームだけ」を表現できない。承認セットを**チーム/グループ単位**で割り当て、SCPで未承認モデルの呼び出し自体を明示的にブロックする、という2段階が揃って初めて「承認済み開発チームのみ」の最小権限が成立する
- 補足：Cognitoの "user" は「ログインする人間」全般（顧客だけでなく評価者・社内ユーザー）。ただし組織のAWSリソース制御はIAM Identity Center側

## ⚠️ 弱点：IAM Identity Center設定 と SAMLフェデレーション設定は「競合する2択」ではない（航空会社CMMS問題・2択で正解）

- 問題：Entra ID既存IdP＋複数AWSアカウント＋職務ベースでBedrockアクセスを一元制御したい（2つ選択）
- **正解の組み合わせ = A + D**（一見別々の設定に見えて、実は1本のパイプラインの前半・後半）
  - **A**：「IAM Identity CenterをEntra ID外部IdPとして設定＋カスタム許可セット」＝**接続の確立**（Identity CenterというSSOハブを作る）
  - **D**：「Entra ID⇔IAM間にSAMLフェデレーションを設定＋Entra IDグループにマッピングしたIAMロール作成」＝**認可の紐付け**（渡ってきたグループ情報をロールにマッピングする）
  - A単独＝繋がってはいるが「誰がどこまでアクセスできるか」の職務ベース制御が未定義。D単独＝マッピング方式はあってもSSOハブ（Identity Center）がなく複数アカウント一元管理にならない
- **誤答のIAMロール+AssumeRole**が悪い理由：素のAssumeRole＝呼び出し元アプリ自身がIAMユーザー/ロールの長期クレデンシャルを持つ必要がある＝Entra ID連携を一切使わない構成（[SAML フェデレーション](#saml-フェデレーション)のAssumeRole vs AssumeRoleWithSAML表を参照）
- **判断ルール**：「一元アクセス制御」＋「既存IdP統合」の2要件が出たら、①SSOハブ設定（IAM Identity Center外部IdP接続）②グループ→ロールのマッピング設定、の**両方が必要ステップ**として2択に分かれて出題されることがある。片方だけで「もう要件は満たした」と早合点しない

## S3 ライフサイクルポリシー：Transition と Expiration（公式模試2週目）

S3ライフサイクルルールのアクションは2種類あり、混同すると保持要件の判定を誤る。

| アクション                 | 内容                                                                                            | データ保持要件への効果                                                   |
| -------------------------- | ----------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| **Transition**（移行）     | 一定期間後に安価なストレージクラス（Standard-IA/Glacier等）へ移動。**データ自体は削除されない** | 「保持期限を超えても残り続ける」ため、単独では保持ポリシー違反になりうる |
| **Expiration**（期限切れ） | 一定期間後にオブジェクトを**自動削除**                                                          | 「必要以上に長く保持しない」を機械的に強制する                           |

→ 「保持期限の強制」「必要以上に保持しない」という要件には**Expiration**アクションが対応する。「アーカイブストレージへの移行」だけの選択肢はTransitionに過ぎず、削除を保証しない（罠）。

**ライフサイクルルールはバケットポリシーより優先される**：公式ドキュメント曰く「バケットポリシーでライフサイクルルールによる削除・移行を阻止することはできない」。全プリンシパルを拒否するバケットポリシーがあっても、ライフサイクルの削除は通常通り実行される。この特性により「保持ポリシーを確実に強制する」手段としてライフサイクルポリシーが選ばれる。
