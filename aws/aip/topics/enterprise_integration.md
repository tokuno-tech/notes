## Lambda：Bedrock Agent のアダプター・防波堤

Bedrock Agent（Action Group）はLambdaを経由して外部システムを呼び出す。

```
Bedrock Agent
    ↓ Action Group（ツール呼び出し）
Lambda関数  ← エラー処理・パラメータ検証
    ↓ 検証済みリクエスト
外部API / DB / サービス
```

**Lambdaを挟む理由：**

| 役割 | 内容 |
|---|---|
| パラメータ検証 | モデルが生成した引数の型・範囲チェック |
| エラーハンドリング | 外部APIが落ちてもエージェントに構造化エラーを返す |
| IF安定化 | 外部サービスの変更をLambda内で吸収 |
| セキュリティ | 認証情報をLambda内に閉じる |

## Lambda vs ECS：MCP サーバーの使い分け

**軽い処理（Lambda）：**
- 短い・ステートレス・イベント駆動
- 例：センサー値取得、アラート判定、ステータスチェック

**重い処理（ECS）：**
- 長い・専門環境・常駐型・GPU必要
- 例：波形解析、画像検査、ML推論、コード実行エンジン

```
Bedrock Agent / Claude
    ↓ MCPプロトコル（統一IF）
ECS上のMCPサーバー（重い処理）
Lambda上のMCPサーバー（軽い処理）
```

AIから見ると同じMCPプロトコルで呼び出せるので、バックエンドの違いを意識しなくて良い。

## Amazon AppFlow

「**既知SaaSとAWSのデータ連携専用**ノーコードサービス」。

**対応SaaS（代表例）：**Salesforce / ServiceNow / SAP / Slack / Google Analytics / Zendesk

```
Salesforce（等のSaaS）
    ↓ AppFlow（フィルタリング・マッピング・暗号化）
S3 / Redshift / Bedrock Knowledge Base等
```

**⚠️ 重要な制約：**
- 対応リスト外の自社独自システムとは連携不可
- 自社基幹システム → API Gateway + Lambda / DataSync / DMS を使う

| キーワード | サービス |
|---|---|
| 既知SaaSとの連携 | AppFlow |
| 自社システム・独自API | Lambda + API Gateway |
| オンプレDB同期 | DMS / DataSync |

## Amazon MQ

「**レガシーメッセージングプロトコル（ActiveMQ/RabbitMQ）をAWSで動かす**サービス」。

```
背景：
新しいシステム → SQS/SNS（AWSネイティブ）で十分
古い企業システム → ActiveMQ/RabbitMQプロトコルで動いている
                    → SQSに移行できない（プロトコルが違う）
                    → Amazon MQで既存プロトコルのまま動かす
```

**アクティブ/スタンバイ構成：** 高可用性のためにブローカーA（稼働）とB（待機）を構成。Aが落ちたら自動切替。

## EventBridge 入力トランスフォーマー

「複数ソースのイベントを**統一フォーマットに整形**するための静的マッピング機能」。

```
イベントソースA：{"t": 23, "sensor_id": "A001"}
イベントソースB：{"temperature": 23.5, "device": "B002"}
    ↓ EventBridge入力トランスフォーマー（JSONPathで静的設定）
統一フォーマット：{"temperature": 23, "sensor": "A001"}
    ↓
Bedrock（常に同じ形式で受け取れる）
```

**できること・できないこと：**

| 操作 | 可否 |
|---|---|
| フィールド名の変換・値の抽出 | ✅ |
| 文字列の結合・固定値の埋め込み | ✅ |
| 四則演算・条件分岐・型変換 | ❌ |

複雑な変換が必要なら Lambda を挟む。

## AppSync + VTL マッピングテンプレート

**AppSync（AWS AppSync）：** フルマネージドのGraphQL APIサービス。
- Query（データ取得）/ Mutation（更新）/ Subscription（リアルタイム購読）

**VTL（Apache Velocity Template Language）：**
AppSyncがGraphQLリクエストをバックエンドに渡す前に変換するテンプレート言語。
条件分岐・ループが使えるスクリプト的な言語（YAMLよりプログラムに近い）。

```
GraphQLリクエスト { generateText(prompt: "要約して") }
    ↓ VTLテンプレートで変換
Lambdaへの入力 { "inputText": "要約して", "modelId": "..." }
    ↓
Bedrock
```

## Amplify DataStore

「**スマホ/PCがオフラインでも動くアプリのためのローカルDB同期サービス**」。

```
オフラインファースト：
操作 → まずローカルに保存（Amplify DataStore）
     → オンラインになったらAppSync → DynamoDBに同期
```

**選択的同期：** 全データでなく「自分のデータだけ」端末に同期。
**競合解決：** オフライン中に複数人が同じデータを編集した場合の戦略（例：Last Write Wins）。

## AWS Outposts vs Wavelength vs Local Zones

| サービス | 一言説明 | 使いどころ |
|---|---|---|
| **AWS Outposts** | AWSのラックを自社DCに設置 | データを国外に出せない法規制・オンプレ統合 |
| **AWS Wavelength** | 5G基地局内にAWSを配置（キャリアと提携） | 5G・エッジ・超低レイテンシー（自動運転・AR/VR） |
| **Local Zones** | 特定都市にAWSインフラを拡張 | 遠隔地・小規模拠点 |

**Wavelengthの補足：** 一般企業は基地局を所有不要。AWSがVerizon/KDDIなどと提携し、企業はWavelength ZoneをAWSリージョンの一部として使うだけ。

## Direct Connect + BGP ルーティングポリシー

**BGP：** 複数の経路がある時に「どの経路を優先するか」を制御する仕組み。

```
オンプレ
    ├── Direct Connect（専用線）← 高品質・低レイテンシー
    └── VPN（インターネット）  ← バックアップ

BGPポリシーで：
推論トラフィック → Direct Connect優先
バッチ処理     → VPN（安い方）でOK
```

**Direct Connectが必要なケース：** 大量データ転送・リアルタイム推論SLA・法規制でインターネット不可  
**VPNで十分なケース：** データ量少・レイテンシー要件緩い

## 3層キャッシュ戦略

```
ユーザーリクエスト
    ↓
【API Gatewayキャッシュ】完全一致リクエストを即返す（マネージド・設定のみ）
    ↓ キャッシュミス
【ElastiCache（Redis）】プロンプト→回答をアプリ層でキャッシュ
    ↓ キャッシュミス
【DynamoDB DAX】メタデータ・履歴の高速取得
    ↓ キャッシュミス
Bedrock（実際の推論）← コストが発生するのはここだけ
```

**API Gatewayキャッシュ：** 設定のみでOK、ElastiCacheクラスター不要。REST APIのみ対応（HTTP API不可）。  
**キャッシュ無効化：** データ更新時にTTL切れを待たずに即削除する仕組み。

## DLQ（デッドレターキュー）+ フォールバックモデル

「**失敗したメッセージを捨てずに保存して後で処理**」する仕組み。

```
SQS → Lambda → Bedrock（失敗・リトライ上限）
    ↓ 自動でDLQに移動
DLQ → フォールバックLambda → 別モデルで再推論
```

**サーキットブレーカーとの違い：**

| | DLQ | サーキットブレーカー |
|---|---|---|
| 何を守るか | **メッセージ（データ）** | **システム全体** |
| 粒度 | メッセージ単位 | システム単位 |
| 目的 | データを捨てずに後で処理 | 障害の連鎖を防ぐ |

組み合わせて使うことも多い（サーキットブレーカーで遮断中のメッセージをDLQに退避）。

## リアルタイム取引シグナル生成パターン

「取引シグナル」= 買え・売れ・待て という判断指示。鮮度が命（1秒遅れると無意味）。

```
取引データ（大量・高頻度）
    ↓
Kinesis Data Streams（高スループット取り込み）
    ↓
EventBridge（イベント駆動トリガー）
    ↓
Lambda → Bedrock（シグナル生成）
    ↓
WebSocket API（クライアントへリアルタイムプッシュ）
```

**なぜWebSocketか：** チャート画面がリアルタイムで動くイメージ。RESTのポーリングだと遅延が発生する。

## ミリ秒未満レイテンシーの認証：IAM + VPCエンドポイント

**「ミリ秒未満」という厳しいSLAがある場合、認証方式の選択が重要。**

| 認証方式 | レイテンシー | 問題点 |
|---|---|---|
| Lambda オーソライザー | ❌ Lambda起動時間が乗る | 変動あり |
| Cognito JWT検証 | ❌ トークン検証処理が乗る | オーバーヘッド |
| DBベースのカスタム認証 | ❌ DB検索時間が乗る | ボトルネック |
| **IAM + VPCエンドポイント** | ✅ AWSインフラレベルで完結 | 追加レイテンシーほぼゼロ |

```
VPC内部で直接通信
    ↓
認証処理がAWSインフラレベルで完結
    ↓
追加レイテンシーがほぼゼロ
```

## AWS Amplify / AppSync / Amplify AI Kit（AIP-31）

### AWS Amplify とは

```
AWS Amplify
  = Web・モバイルアプリのフロントエンド開発を加速するフルスタックプラットフォーム
  = React・Next.js・Vue などのフレームワークと統合
  = バックエンド（認証・API・ストレージ）をコードから宣言的に構築できる
  = Amplify Hosting でフロントエンドを S3 + CloudFront でホスト

イメージ：「AWSのFirebase」的なポジション
  → バックエンドを意識せずにフロントエンドから書けるようにするツール群
```

---

### AWS AppSync とは

```
AWS AppSync
  = フルマネージドの GraphQL API サービス
  = REST API（API Gateway）の GraphQL 版に相当

GraphQL の3種類の操作
  Query      → データ取得（REST の GET）
  Mutation   → データ更新（REST の POST/PUT/DELETE）
  Subscription → リアルタイム更新の購読（REST には相当なし）

Subscription の仕組み
  → WebSocket で接続を確立
  → バックエンドで Mutation が発行されるたびに、購読中のクライアントへ自動でプッシュ
  → クライアント側はポーリング不要でリアルタイムにデータを受け取れる
```

---

### AWS Amplify AI Kit とは

```
Amplify AI Kit
  = Amplify Gen 2 に組み込まれた AI 機能セット
  = Amazon Bedrock との接続・ストリーミングを宣言的に設定できる

できること
  ① Bedrock との接続設定（モデル指定・プロンプト定義）
  ② AppSync Subscription を通じたストリーミングレスポンスの有効化
  ③ React 向けフック（useAIConversation）でストリーミング状態管理を自動化

開発者がやること
  → amplify/ai/ に conversation ルートを定義するだけ
  → Bedrock との接続・WebSocket の管理・チャンク受信はフレームワークが自動処理
```

---

### AppSync の Resolver とは（補足）

```
GraphQL のフィールドごとに「どこからデータを取ってくるか」を定義した関数。
REST で言う Controller に相当。

クライアント → GraphQL Query → AppSync → Resolver 実行 → DB/API → データ返却
```

**Resolver の実態**
- DB 接続・SQL・API 呼び出しは**自分で書く**（フレームワークが自動でやるわけではない）
- バリデーション・ビジネスロジックも Resolver 内、または Resolver から呼ぶ Service 層に書く
- 認証情報は `context`（リクエスト情報）を通じて Resolver に渡される

**AppSync がやってくれること（Resolver の自動生成）**

| データソース | Resolver の扱い |
|---|---|
| **DynamoDB** | AppSync が CRUD Resolver を**自動生成**。コード不要 |
| **Lambda** | Lambda 関数自体が Resolver 扱い。ロジックは Lambda に書く |
| **RDS（Aurora）** | SQL は自分で書く。自動生成なし |
| **Bedrock（Amplify AI Kit）** | スキーマ定義だけで Resolver を自動生成 |

**DynamoDB と自動生成の相性が良い理由**
- DynamoDB = キーバリュー型 → 操作パターンが「キーで GET / PUT / DELETE」のみ
- パターンが有限なので AppSync が網羅的に自動生成できる
- SQL の JOIN や複雑な WHERE 条件は無限にあり得るため RDS では自動生成が成立しない

## GenAI UI パターン（Task 2.5）

### プログレッシブレンダリングパターン

Bedrock のストリーミング応答をUIに逐次反映するパターン。

```
Bedrock: "こ"→"ん"→"に"→"ち"→"は"（トークン単位で届く）
UI: こんにちは（50〜200ms ごとにまとめてDOMに反映）
```

- **なぜバッファリングするか**: 毎トークンごとにDOMを更新するとレンダリングコストが高い
- **バッファ間隔**: 50〜200ms ごとにまとめて1回描画

### プログレッシブエンハンスメント

クライアントの機能レベルに応じて体験が段階的に向上する設計思想。

```
低機能ブラウザ: テキストのみ表示（最低限動く）
普通のブラウザ: スケルトンスクリーン（生成中の骨格表示）
高機能ブラウザ: アニメーション付きスケルトン
```

GenAI的意味: 生成中はスケルトン表示、失敗時は適切なエラーメッセージを表示。

### アクセシビリティ（ARIA / スクリーンリーダー対応）

**ARIA属性**: スクリーンリーダー（視覚障害者向け読み上げソフト）に要素の役割・状態を伝える属性。

```html
<div role="status" aria-live="polite" aria-label="AI応答生成中">
  こんにちは...
</div>
```

- `aria-live="polite"`: 現在の読み上げが終わってから更新を通知（ストリーミング中に連続割り込みしない）
- `role="status"`: 「状態表示エリア」であることをスクリーンリーダーに伝える

**キーボードナビゲーション**: マウスなしでUIを操作できること。

```
Tab       → 次の要素（入力欄→送信ボタン→履歴）へ移動
Shift+Tab → 前の要素へ戻る
Enter     → 送信
Esc       → 生成中止・モーダルを閉じる
```

### クライアント側キャッシュ戦略

```
入力類似性 85〜95% → キャッシュ応答を返す（モデル呼び出しなし）
入力類似性 <85%   → 新規モデル呼び出し
時間ベース有効期限 → 古いキャッシュは自動無効化
```

例: 「AWSとは？」と「AWSって何？」は90%類似 → キャッシュヒット。

## AI強化システムと既存業務アプリの双方向同期（Task 2.5）

AI がCRMにインサイト（解約リスク等）を書き込み、人間も同じレコードを編集する構成での設計ポイント。

**双方向同期の実現：** AppFlow（SaaS⇔AWS の双方向転送）+ EventBridge（変更イベントの即時伝搬）。
AI→CRM はインサイト書き込み、CRM→AI は人間の最新編集の取得（古いデータで分析しないため）。

**競合解決戦略（同一レコードへの同時更新）：**

| 戦略 | 内容 | AI文脈での評価 |
|---|---|---|
| Last Write Wins | タイムスタンプが新しい方 | AIが人間の修正を上書きする事故リスク |
| **人間優先** | 人間の編集は常にAIに勝つ | ✅ AI統合の定番。AIは「提案」、人間が「確定」 |
| フィールドレベル分離 | AIは `risk_score` のみ、人間は連絡先のみ | そもそも衝突しない設計。最も堅い |

**AIによる変更の監査ログ：** いつ・どのモデル（バージョン）が・どのレコードを・変更前後の値・信頼度。
目的 = ①ロールバック ②説明責任（責任あるAI） ③コンプライアンス（人間/AIどちらの変更か証明）。
※ CloudTrail は AWS API コールの監査。**ビジネスデータの変更履歴はアプリ側で記録**する（DynamoDB/S3）。
