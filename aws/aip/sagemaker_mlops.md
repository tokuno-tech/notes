# SageMaker / MLOps パターン

## MLモデル再学習 → 自動デプロイ構成（AIP-13）

### 要件
- S3にデータが追加されるたびに再学習・デプロイを自動化
- モデル学習に数時間かかる
- 運用オーバーヘッドを最小化

### 正解構成

**Step Functions Standard + SageMaker Pipelines**

```
S3にバッチファイルがアップロード
  ↓ S3イベント通知
Lambda（ファイル情報を取得・整形）
  ↓
Step Functions Standard ワークフロー
  ↓ StartPipelineExecution（.syncパターンで完了を待機）
SageMaker Pipelines
  ├─ データ前処理
  ├─ モデルトレーニング
  ├─ モデル評価
  ├─ モデル登録
  └─ エンドポイントへのデプロイ
```

### 各コンポーネントの役割

| コンポーネント | 役割 |
|---|---|
| **Step Functions Standard** | 最大1年間実行可能。状態管理・エラーハンドリング・リトライを組み込みで提供。数時間かかるML学習のオーケストレーションに最適 |
| **SageMaker Pipelines** | ML専用のオーケストレーションサービス。前処理→学習→評価→登録→デプロイをDAGとして定義。パラメータ管理・成果物管理・再現性を実現 |
| `.sync` パターン | Step FunctionsがSageMaker Pipelinesの完了を待機できる同期統合パターン |

### SageMaker Pipelinesの利点
- パイプライン定義をバージョン管理できる
- 各実行のパラメータ・成果物・メトリクスを追跡できる
- SageMaker Studio と統合されGUI でパイプラインを可視化・モニタリング可能
- MLOps のベストプラクティスに沿った構成を実現

---

## Step Functions Standard vs Express（重要な区別）

| | Standard | Express |
|---|---|---|
| **最大実行時間** | **1年** | **5分** |
| 用途 | 長時間処理・ML学習・カナリアデプロイ | 高頻度・短時間処理（IoT・ストリーミング） |
| 料金 | 状態遷移ごとに課金 | 実行時間・リクエスト数で課金 |
| 実行履歴保持 | 90日間 | なし（CloudWatch Logsに送信） |

→ **「数時間かかる」という記述があれば必ずStandard**。Expressは不正解になる。

---

## 不正解パターンとの比較

### Lambda + SNS構成が不正解な理由

```
Lambda の最大実行時間 = 15分
数時間の学習を Lambda 内で待機 → 不可能
```

さらに：
- ワークフロー全体の状態管理が欠如
- どのステップまで完了したか、エラー時のリトライ先の制御を自前実装が必要
- 「運用オーバーヘッド最小化」に反する

### Kinesis Data Firehose + SageMaker推論エンドポイントが不正解な理由

- Kinesis Data Firehoseはデータ配信サービス（S3・Redshiftへの配信）
- SageMakerの推論エンドポイントはリクエストに推論結果を返すもの
- 「エンドポイントでデータを蓄積して再学習をトリガー」する仕組みは存在しない
- **技術的に成立しない構成**

### Step Functions Express が不正解な理由

- 最大実行時間が5分 → 数時間の学習に対応不可

---

## Lambda の制約（試験頻出）

| 制約 | 値 |
|---|---|
| 最大実行時間 | **15分** |
| 最大メモリ | 10GB |
| GPU | **使用不可** |
| 同時実行数（デフォルト） | 1,000（引き上げ可） |

→ LLM推論をLambdaで実行 = GPUなし = 150ms以内の応答は不可能（AIP-12の引っかけ）

## Provisioned Throughput（バッチ安定性） + SageMaker Neo（オンプレ推論）（AIP-26）

### 要件

- 大量の医療画像を定期バッチ処理。**安定したパフォーマンス**が必要（→ Provisioned Throughput）
- 患者データの規制によりオンプレミスで推論を実行（→ SageMaker Neo）
- クラウドとオンプレミスの**ハイブリッド構成**

### 正解構成（2つ選択）

#### ① Amazon Bedrock Provisioned Throughput（バッチ安定性）

```
オンデマンド推論（共有リソース）
  → スロットリング発生リスクあり。ピーク時にレイテンシが変動する

Provisioned Throughput（専用キャパシティ）
  → 予約した処理能力が常に確保される
  → 大規模バッチジョブで「安定したパフォーマンス」要件に合致
```

| 項目 | Provisioned Throughput | オンデマンド |
|---|---|---|
| 処理能力 | 専用確保（MU単位で予約） | 共有リソース |
| スロットリングリスク | **なし**（予約分まで保証） | あり |
| 費用モデル | 時間課金（使用量に関わらず） | リクエスト数 × トークン数 |
| 適用シーン | 大量バッチ・低レイテンシ保証 | 小〜中規模・散発的なリクエスト |

#### ② Amazon SageMaker Neo（オンプレ推論）

```
クラウド上のモデル（PyTorch / TensorFlow 等）
  ↓ SageMaker Neo でコンパイル
        ├─ 対象ハードウェアを指定（ARM / x86 / NVIDIA GPU / Inferentia 等）
        └─ そのハードウェア向けに最適化された実行バイナリを生成
  ↓
オンプレミスサーバー / エッジデバイスにデプロイ
  → クラウドAPIを呼ばずにローカルで推論実行
  → 患者データがクラウドに送信されない
```

- **コンパイル = モデルを特定のハードウェア用に変換**
- AWS IoT Greengrass と組み合わせることでエッジデバイス管理も可能

### 不正解パターン（AIP-26）

| 選択肢 | 不正解の理由 |
|---|---|
| AWS Outposts 上の SageMaker エンドポイント | Outposts は AWS インフラをオンプレに持ち込む。SageMaker エンドポイントのセットアップが複雑でコスト大。Neo のほうがシンプル |
| SageMaker リアルタイムエンドポイント → オンプレから API 呼び出し | **データがクラウドに送信される**。患者データの規制要件「オンプレミスで実行」に違反 |
| Bedrock オンデマンド推論でバッチ順次送信 | 共有リソース。大量バッチ時のスロットリング・レイテンシ変動が発生。「安定したパフォーマンス」を保証できない |

### 試験での識別キーワード

| キーワード | 示すもの |
|---|---|
| 「安定したパフォーマンス」「スロットリング防止」「バッチ推論」 | Provisioned Throughput |
| 「オンプレミスで推論」「データをクラウドに出せない」「エッジ」 | SageMaker Neo（コンパイル → ローカルデプロイ） |
| 「クロスリージョン推論」「地理的冗長性」「別リージョンへフェイルオーバー」 | Cross-Region Inference（AIP-14参照） |
