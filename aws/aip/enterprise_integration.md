# エンタープライズ統合アーキテクチャ（Task 2.3）

## 概要：なぜエンタープライズ統合が必要か

AIPの文脈では「FMをゼロから作る」ではなく「**既存企業システムにFMをどう賢く組み込むか**」が主題。

```
既存CRM ─┐
既存ERP ─┼─ Bedrock（FM）に繋いでAI化する
既存DB  ─┘
```

## Lambda：Bedrock Agent のアダプター・防波堤

Bedrock Agent（Action Group）はLambdaを経由して外部システムを呼び出す。

```
Bedrock Agent
    ↓ Action Group（ツール呼び出し）
Lambda関数  ← エラー処理・パラメータ検証
    ↓ 検証済みリクエスト
外部API / DB / サービス
```

**Lambdaを挟む理由：**

| 役割 | 内容 |
|---|---|
| パラメータ検証 | モデルが生成した引数の型・範囲チェック |
| エラーハンドリング | 外部APIが落ちてもエージェントに構造化エラーを返す |
| IF安定化 | 外部サービスの変更をLambda内で吸収 |
| セキュリティ | 認証情報をLambda内に閉じる |

**試験的なポイント：**
- 「信頼性が高く保守しやすいツール統合」→ Bedrock Agentの標準化された関数定義 + Lambdaでエラー処理・パラメータ検証

## Lambda vs ECS：MCP サーバーの使い分け

**軽い処理（Lambda）：**
- 短い・ステートレス・イベント駆動
- 例：センサー値取得、アラート判定、ステータスチェック

**重い処理（ECS）：**
- 長い・専門環境・常駐型・GPU必要
- 例：波形解析、画像検査、ML推論、コード実行エンジン

```
Bedrock Agent / Claude
    ↓ MCPプロトコル（統一IF）
ECS上のMCPサーバー（重い処理）
Lambda上のMCPサーバー（軽い処理）
```

AIから見ると同じMCPプロトコルで呼び出せるので、バックエンドの違いを意識しなくて良い。

## 動的ツール検出メカニズム

「拡張性」「柔軟性」キーワード → ツールレジストリによる動的検出。

```
通常（静的）：ツールがハードコード → 追加のたびに設定変更必要

動的検出あり：
エージェント起動 → ツールレジストリに問い合わせ
    ↓
現在登録済みのツール一覧＋説明を受け取る
    ↓
タスクに応じて動的にツールを選択・呼び出し
```

MCPの `tools/list` がまさにこれに該当。Bedrock AgentのAction Groupも動的更新が可能。

## エンタープライズ接続の3パターン

| パターン | サービス例 | 特徴 |
|---|---|---|
| APIベース統合 | API Gateway + Lambda | 同期・リアルタイム |
| イベント駆動（疎結合） | SQS / EventBridge | 非同期・スケーラブル |
| データ同期 | S3 + Knowledge Base | バッチ・RAG向け |

**試験での使い分け：**
- 「疎結合」→ SQS/EventBridge
- 「リアルタイム」→ API Gateway + Lambda
- 「既存データを検索させたい」→ S3 + Knowledge Base

## Amazon AppFlow

「**既知SaaSとAWSのデータ連携専用**ノーコードサービス」。

**対応SaaS（代表例）：**Salesforce / ServiceNow / SAP / Slack / Google Analytics / Zendesk

```
Salesforce（等のSaaS）
    ↓ AppFlow（フィルタリング・マッピング・暗号化）
S3 / Redshift / Bedrock Knowledge Base等
```

**⚠️ 重要な制約：**
- 対応リスト外の自社独自システムとは連携不可
- 自社基幹システム → API Gateway + Lambda / DataSync / DMS を使う

**試験での識別キーワード：** 「SaaS名（Salesforce等）が問題文に出たら AppFlow を疑う」

| キーワード | サービス |
|---|---|
| 既知SaaSとの連携 | AppFlow |
| 自社システム・独自API | Lambda + API Gateway |
| オンプレDB同期 | DMS / DataSync |

## Amazon MQ

「**レガシーメッセージングプロトコル（ActiveMQ/RabbitMQ）をAWSで動かす**サービス」。

```
背景：
新しいシステム → SQS/SNS（AWSネイティブ）で十分
古い企業システム → ActiveMQ/RabbitMQプロトコルで動いている
                    → SQSに移行できない（プロトコルが違う）
                    → Amazon MQで既存プロトコルのまま動かす
```

**アクティブ/スタンバイ構成：** 高可用性のためにブローカーA（稼働）とB（待機）を構成。Aが落ちたら自動切替。

**試験での識別：**
- 「レガシー・既存メッセージング・移行」→ Amazon MQ
- 「新規構築・AWSネイティブ」→ SQS / SNS

## EventBridge 入力トランスフォーマー

「複数ソースのイベントを**統一フォーマットに整形**するための静的マッピング機能」。

```
イベントソースA：{"t": 23, "sensor_id": "A001"}
イベントソースB：{"temperature": 23.5, "device": "B002"}
    ↓ EventBridge入力トランスフォーマー（JSONPathで静的設定）
統一フォーマット：{"temperature": 23, "sensor": "A001"}
    ↓
Bedrock（常に同じ形式で受け取れる）
```

**できること・できないこと：**

| 操作 | 可否 |
|---|---|
| フィールド名の変換・値の抽出 | ✅ |
| 文字列の結合・固定値の埋め込み | ✅ |
| 四則演算・条件分岐・型変換 | ❌ |

複雑な変換が必要なら Lambda を挟む。
**試験：** 「シンプルな正規化はEventBridge、複雑な処理はLambda」

## AppSync + VTL マッピングテンプレート

**AppSync（AWS AppSync）：** フルマネージドのGraphQL APIサービス。
- Query（データ取得）/ Mutation（更新）/ Subscription（リアルタイム購読）

**VTL（Apache Velocity Template Language）：**
AppSyncがGraphQLリクエストをバックエンドに渡す前に変換するテンプレート言語。
条件分岐・ループが使えるスクリプト的な言語（YAMLよりプログラムに近い）。

```
GraphQLリクエスト { generateText(prompt: "要約して") }
    ↓ VTLテンプレートで変換
Lambdaへの入力 { "inputText": "要約して", "modelId": "..." }
    ↓
Bedrock
```

**試験：** VTLの構文詳細は問われない。「AppSyncがGraphQLリクエストを変換する仕組み」という認識でOK。
「GraphQL・リアルタイム・フロントエンドからBedrock呼び出し」→ AppSync

## Amplify DataStore

「**スマホ/PCがオフラインでも動くアプリのためのローカルDB同期サービス**」。

```
オフラインファースト：
操作 → まずローカルに保存（Amplify DataStore）
     → オンラインになったらAppSync → DynamoDBに同期
```

**選択的同期：** 全データでなく「自分のデータだけ」端末に同期。
**競合解決：** オフライン中に複数人が同じデータを編集した場合の戦略（例：Last Write Wins）。

**試験：** 「オフラインファースト・モバイルアプリ」→ Amplify DataStore

## AWS Outposts vs Wavelength vs Local Zones

| サービス | 一言説明 | 使いどころ |
|---|---|---|
| **AWS Outposts** | AWSのラックを自社DCに設置 | データを国外に出せない法規制・オンプレ統合 |
| **AWS Wavelength** | 5G基地局内にAWSを配置（キャリアと提携） | 5G・エッジ・超低レイテンシー（自動運転・AR/VR） |
| **Local Zones** | 特定都市にAWSインフラを拡張 | 遠隔地・小規模拠点 |

**Wavelengthの補足：** 一般企業は基地局を所有不要。AWSがVerizon/KDDIなどと提携し、企業はWavelength ZoneをAWSリージョンの一部として使うだけ。

**試験での使い分け：**
- 「データを国外に出せない・オンプレ統合」→ Outposts
- 「5G・エッジ・超低レイテンシー」→ Wavelength

## Amazon A2I（Augmented AI）

「**必要な場合だけ人間のレビューを挟む**マネージドサービス」。

```
通常業務 → Step Functionsが自動処理
    ↓ 信頼スコアが閾値以下 or 例外ケース
A2I → 人間にレビュータスクを送る
    ↓ 承認/修正
処理再開
```

**試験での使いどころ：**
- 「自律的な運用と人間の介入の両方を効率的に処理」→ A2I + Step Functions
- 「全件手動承認」→ ボトルネックになるので不正解
- 「完全自律」→ 高リスクなので不正解

## Outposts vs 抽象化レイヤー：使い分けの判断基準

```
「機密データを外部AIに送信しない」
    → 抽象化レイヤー（PII除去して送信）で十分
    → Outpostsは不要・過剰

「データをオンプレミスの外に出してはいけない」（法規制）
    → Outposts一択（物理的に施設内で処理）
```

**暗号化・VPN・Direct Connectでも「施設外に出る」ので不正解になる。**

## Direct Connect + BGP ルーティングポリシー

**BGP：** 複数の経路がある時に「どの経路を優先するか」を制御する仕組み。

```
オンプレ
    ├── Direct Connect（専用線）← 高品質・低レイテンシー
    └── VPN（インターネット）  ← バックアップ

BGPポリシーで：
推論トラフィック → Direct Connect優先
バッチ処理     → VPN（安い方）でOK
```

**Direct Connectが必要なケース：** 大量データ転送・リアルタイム推論SLA・法規制でインターネット不可  
**VPNで十分なケース：** データ量少・レイテンシー要件緩い

## 3層キャッシュ戦略

```
ユーザーリクエスト
    ↓
【API Gatewayキャッシュ】完全一致リクエストを即返す（マネージド・設定のみ）
    ↓ キャッシュミス
【ElastiCache（Redis）】プロンプト→回答をアプリ層でキャッシュ
    ↓ キャッシュミス
【DynamoDB DAX】メタデータ・履歴の高速取得
    ↓ キャッシュミス
Bedrock（実際の推論）← コストが発生するのはここだけ
```

**API Gatewayキャッシュ：** 設定のみでOK、ElastiCacheクラスター不要。REST APIのみ対応（HTTP API不可）。  
**キャッシュ無効化：** データ更新時にTTL切れを待たずに即削除する仕組み。

## DLQ（デッドレターキュー）+ フォールバックモデル

「**失敗したメッセージを捨てずに保存して後で処理**」する仕組み。

```
SQS → Lambda → Bedrock（失敗・リトライ上限）
    ↓ 自動でDLQに移動
DLQ → フォールバックLambda → 別モデルで再推論
```

**サーキットブレーカーとの違い：**

| | DLQ | サーキットブレーカー |
|---|---|---|
| 何を守るか | **メッセージ（データ）** | **システム全体** |
| 粒度 | メッセージ単位 | システム単位 |
| 目的 | データを捨てずに後で処理 | 障害の連鎖を防ぐ |

組み合わせて使うことも多い（サーキットブレーカーで遮断中のメッセージをDLQに退避）。

## リアルタイム取引シグナル生成パターン

「取引シグナル」= 買え・売れ・待て という判断指示。鮮度が命（1秒遅れると無意味）。

```
取引データ（大量・高頻度）
    ↓
Kinesis Data Streams（高スループット取り込み）
    ↓
EventBridge（イベント駆動トリガー）
    ↓
Lambda → Bedrock（シグナル生成）
    ↓
WebSocket API（クライアントへリアルタイムプッシュ）
```

**なぜWebSocketか：** チャート画面がリアルタイムで動くイメージ。RESTのポーリングだと遅延が発生する。

**試験キーワード：** 「リアルタイム・高スループット・双方向通信・取引・金融・センサー」→ Kinesis + WebSocket API

## ミリ秒未満レイテンシーの認証：IAM + VPCエンドポイント

**「ミリ秒未満」という厳しいSLAがある場合、認証方式の選択が重要。**

| 認証方式 | レイテンシー | 問題点 |
|---|---|---|
| Lambda オーソライザー | ❌ Lambda起動時間が乗る | 変動あり |
| Cognito JWT検証 | ❌ トークン検証処理が乗る | オーバーヘッド |
| DBベースのカスタム認証 | ❌ DB検索時間が乗る | ボトルネック |
| **IAM + VPCエンドポイント** | ✅ AWSインフラレベルで完結 | 追加レイテンシーほぼゼロ |

```
VPC内部で直接通信
    ↓
認証処理がAWSインフラレベルで完結
    ↓
追加レイテンシーがほぼゼロ
```

**試験キーワード：** 「ミリ秒未満・超低レイテンシー + 認証」→ IAM + VPCエンドポイント一択

## ReAct パターン

「**推論（Reasoning）と実行（Action）を交互に繰り返す**エージェントパターン」。

```
Reasoning → Action → Observation（結果確認）→ Reasoning → ...
```

外部ツール・APIを使いながら推論するのがポイント。一発で終わりではなく観察結果を次の推論に使う。

**CoTとの違い：**

| | CoT（思考連鎖） | ReAct |
|---|---|---|
| 外部情報 | 使わない | 使う（API・DBを叩く） |
| 特徴 | 思考を段階的に展開 | 思考＋外部ツール実行を交互に |

**試験でのセット：**
「複数の解決策を実装前に評価」→ ReAct（検証ステップあり）+ CoT（複数戦略評価）の組み合わせ。

## マルチエージェント：スーパーバイザーパターン（AWS Agent Squad）

「**明確な制御階層**」キーワード → スーパーバイザーパターン一択。

```
スーパーバイザーエージェント（調整・タスク振り分け）
    ├── ワーカーエージェントA（品質検査専門）
    ├── ワーカーエージェントB（在庫管理専門）
    └── ワーカーエージェントC（設備モニタリング専門）
```

**他パターンの問題点：**

| パターン | 問題点 |
|---|---|
| シングルエージェント | モノリシック、スケールしない |
| P2P直接通信 | エージェント数が増えると調整が指数的に複雑化 |
| Lambdaのみ | ステート管理・高度なオーケストレーションが欠如 |
