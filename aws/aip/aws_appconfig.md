# AWS AppConfig

## 概要

AWS Systems Manager の一機能。**アプリケーション設定をコード変更・Lambda再デプロイなしに動的管理**するためのマネージドサービス。

---

## 主な機能

| 機能 | 内容 |
|---|---|
| **機能フラグ（Feature Flags）** | アプリの機能をON/OFFする。属性ベースのターゲティングで条件に応じた設定の出し分けが可能 |
| **設定プロファイル** | JSON/YAMLで設定値を管理。バージョン管理も組み込み済み |
| **JSONスキーマバリデーション** | デプロイ前に値の範囲チェックを自動実行。不正値が本番に入るのを防ぐゲートキーパー |
| **デプロイ戦略** | Linear（線形）・Exponential（指数）など段階的ロールアウトを設定だけで実現。CloudWatchアラームと連携して異常時は自動ロールバック |
| **AppConfig Agent（Lambda拡張機能）** | `localhost:2772` 経由で設定をローカルキャッシュから取得。API呼び出しコスト・レイテンシーを最小化。TTLに基づき自動更新 |

---

## 属性ベースのターゲティング

Lambdaがリクエストから属性（例：優先度レベル）を抽出し、AppConfig Agentに渡すことで、条件に応じた設定値が返ってくる仕組み。

```
リクエスト到着
  ↓
Lambda が属性を抽出（例：priority=urgent）
  ↓
AppConfig Agent に属性を渡して設定を問い合わせ
  ↓
AppConfig が条件に合う設定を返す（例：urgentならモデルA）
  ↓
Lambda がその設定でBedrockを呼び出す
```

→ Lambda自体は変わらず、**AppConfig側のルール定義を変えるだけ**でモデルの切り替えができる

---

## スキーマバリデーター

**設定値をデプロイする前に不正な値を弾く**ゲートキーパー的機能。LambdaやコンシューマがデータをSubscribeする際の検証ではない点に注意。

```
開発者が設定値を更新・デプロイしようとする
  ↓
AppConfig がスキーマバリデーターを自動実行
  ├─ 正常（範囲内） → デプロイOK → Lambdaに反映される
  └─ 異常（範囲外） → デプロイ拒否 → Lambdaには届かない
```

スキーマ定義例：
```json
{
  "type": "object",
  "properties": {
    "temperature": {
      "type": "number",
      "minimum": 0.0,
      "maximum": 2.0
    },
    "max_tokens": {
      "type": "integer",
      "minimum": 1,
      "maximum": 4096
    }
  }
}
```

---

## 「デプロイなし」の正確な意味

| 対象 | 要否 |
|---|---|
| **Lambdaのコード変更・関数更新** | 不要（←これが「デプロイなし」の意味） |
| **AppConfig側の設定変更** | 必要（ただしLambdaデプロイより圧倒的に軽い） |

AppConfigへの設定変更のデプロイ方法：
- コンソールから手動
- AWS CLI / SDK経由でスクリプト化
- EventBridge + Lambda で自動化

---

## AIPで正解になるキーワード

- 「デプロイなしで」「コード変更なしに」動的に設定を切り替えたい
- 「段階的ロールアウト」をしたい
- 「パラメータの事前検証（バリデーション）」が必要
- 「機能フラグ」でユーザー属性に応じて出し分けたい

## 引っかけとして登場する選択肢

| 選択肢 | 不正解の理由 |
|---|---|
| Lambda環境変数 | 更新に関数設定の更新（実質再デプロイ）が必要 |
| Lambda加重ルーティング | ランダムな比率分割のみ。属性ベースの出し分け不可 |
| S3 + GetObject毎回 | レイテンシー増・コスト増・バリデーション機能なし |
| DynamoDB + カスタムロジック | バリデーション・ロールアウト管理をすべて自前実装 |
| CodePipeline + CodeBuild | CI/CDパイプライン自体の管理コストが増える。リアルタイム切替に不向き |
