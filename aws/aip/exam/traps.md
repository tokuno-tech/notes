## よくある引っかけ（1行サマリー + 詳細リンク）

- **API Gateway リクエスト検証ツール（JSON スキーマ）** は「門番」（必須フィールド・型・サイズを検証して弾く）。**マッピングテンプレート**は「翻訳者」（形を変えるだけ、検証しない）→ 詳細: [topics/bedrock_core.md](../topics/bedrock_core.md)（Task 2.4）
- **Kendra の関連性スコア** は検索結果の確信度であって、RAG の評価メトリクスではない → 詳細: [topics/bedrock_guardrails.md](../topics/bedrock_guardrails.md)（AIP-79）
- **CloudWatch Synthetics** は外形監視（カナリア）であって、LLM 出力の品質評価ではない → 詳細: [topics/bedrock_guardrails.md](../topics/bedrock_guardrails.md)（AIP-79）
- **Cognito** は「人間のユーザー向け認証」。サービス間（M2M）の API 認証は IAM ロール + SigV4 → 詳細: [mistakes.md](./mistakes.md)（AIP-25）
- **SQS + KB 同期はリアルタイムではない**。リアルタイムデータはエージェントのアクショングループで API 直呼び → 詳細: [mistakes.md](./mistakes.md)（AIP-27）
- **Knowledge Bases のクエリ分解は内蔵機能**。別途 Agent + 複数 FM を使わなくていい。「クエリ分解を有効にする」だけで OK → 詳細: [topics/bedrock_agents.md](../topics/bedrock_agents.md)（Q4）
- **Bedrock Flows の正式名称は「Amazon Bedrock Flows」**。「Prompt Flows」「プロンプトフロー」は別名・旧称。試験では「Bedrockフロー」「Bedrock Flows」で出てくる → 詳細: [topics/bedrock_agents.md](../topics/bedrock_agents.md)（AIP-73）
- **Lambda 用 Interface VPC Endpoint は「Lambdaを呼ぶ側」の入口**（`lambda:Invoke` inbound）であって、「Lambdaが他サービスを呼ぶための出口」ではない。VPC内Lambda→Bedrockを private に呼ぶには **bedrock-runtime** エンドポイントが必要。「プロキシLambdaを挟む」も同じ問題を移すだけで解決しない → 詳細: [topics/bedrock_core.md](../topics/bedrock_core.md)（Bedrockサービスエンドポイント4種）
- **Client VPN / Site-to-Site VPN / Direct Connect は AWSサービス（Bedrock等）への private access 用途では正解にならない**。Client VPN＝リモートユーザ→VPC、Site-to-Site VPN＝拠点→VPC、DX＝専用線オンプレ→AWS、いずれも「AWSサービスAPIへの private path」を作る仕組みではない。正解は **PrivateLink（Interface VPC Endpoint）** → 詳細: [topics/bedrock_core.md](../topics/bedrock_core.md)

## ⚠️ 間違えやすいポイント3つ

### ① コンテンツフィルターの強度（低・中・高）

| 強度 | 特徴 | 向いているケース |
|---|---|---|
| **低** | ゆるい。有害コンテンツを見逃しやすい | ほぼ使わない |
| **中** | バランスが良い。医療・法律用語の誤検知を抑えつつ有害コンテンツを検出 | 医療・専門分野のチャットボット |
| **高** | 厳格。医療・法律の専門用語が誤検知されやすい。過剰ブロックが増える | 汎用的な消費者向けアプリ |

**AIP試験の識別ポイント**：
- 「過剰なブロックを避けつつ」「ユーザー体験の低下を防ぐ」→ **中** が正解
- 「高」はセキュリティ最優先でUXを犠牲にするケース

---

### ② 入力側・出力側で別々のアクションを設定できる

**機密情報フィルター（PIIフィルター）は入力と出力で異なるアクションを設定可能。**

```
入力（ユーザーの質問）
  → 医療保険番号が含まれていたら → BLOCK（モデルに渡さず拒否）
  → 理由：入力段階でモデルに渡したくない情報はBLOCKで止める

出力（モデルの回答）
  → 患者氏名・住所が含まれていたら → MASK（**** に置換して返す）
  → 理由：回答の中に含まれてしまった個人情報は伏せ字にして安全に返す
```

**よくある誤解**：
「回答に含まれる患者情報はマスクし」= 出力側のMASK設定（モデルが生成した回答の中の情報を伏せる）
→ 「モデルに渡す前にマスク」ではない。入力側はBLOCKで止めるのが正解。

---

### ③ ガードレールのログはデフォルトでは記録されない

```
モデル呼び出しログ記録
  → デフォルト: OFF
  → 明示的に有効化が必要（コンソール or API）
  → 有効化すると入力プロンプト・出力・ガードレール評価詳細が記録される
  → 保存先: S3 または CloudWatch Logs

CloudWatch Logs に出力される内容
  → どのポリシーがトリガーされたか
  → ブロックの理由
  → カスタムブロックメッセージ
  → 規制対応の監査証跡として利用可能
```

**DynamoDBへの独自記録（不正解パターン）は冗長**：Bedrockネイティブのログ機能（CloudWatch Logs）で要件を満たせる。

## 略語「SSE」の2つの意味（Task 2.5）

ストリーミングの文脈と暗号化の文脈で同じ略語が出る。Skill Builder の和訳でも混同があった。

- **Server-Sent Events**：ストリーミング応答の配信方式。`Content-Type: text/event-stream`
- **Server-Side Encryption**：S3/DynamoDB のサーバー側暗号化（SSE-S3 / SSE-KMS）

「ストリーミング・チャット・リアルタイム」の文脈 → Server-Sent Events / 「保存データ・暗号化」の文脈 → Server-Side Encryption

## 「合成モニタリング」と「複合アラーム」（Task 2.5）

- 合成（Synthetic）= **人工ユーザー**が定期的に叩く外形監視（CloudWatch Synthetics / Canary）。「ユーザー影響前に検知」
- 複合（Composite）= 複数アラームの **AND/OR 結合**（誤検知削減）

「合成＝複数を合わせる」と誤読しない。合わせるのは複合アラームの方。

## Amazon Pinpoint vs Bedrock（メール生成）（Task 2.5）

同じ「メール送信」文脈で両方出るが目的が全く違う。

- **Pinpoint** = セグメント × テンプレートの**一斉配信**。「30代・東京在住・購入履歴あり」への定型メール。内容の動的生成はできない
- **Bedrock + Lambda（+ SES）** = 「この顧客の通話メモを読んで文章を生成して送る」。**個別・動的・パーソナライズ**

「顧客とのやり取りのメモに基づいてパーソナライズ」→ Bedrock。「セグメントにテンプレート配信」→ Pinpoint。

## `StartAsyncInvoke` ≠ 汎用非同期推論（Domain 2 Practice）

「非同期」という名前に引っ張られて、テキスト系モデルのバッチ処理に使おうとする罠。

- **`StartAsyncInvoke`** = **Amazon Nova Reel（動画生成）専用**。他のモデルでは使えない
- テキスト系モデルの非同期・バッチ推論 = **`CreateModelInvocationJob`**

```
❌ よくある誤解
「Lambda から非同期に Bedrock を呼びたい → StartAsyncInvoke」

✅ 正解
「テキスト系モデルのバッチ推論 → CreateModelInvocationJob」
「動画生成（Nova Reel）→ StartAsyncInvoke」
```

## Lambda プロキシ統合の「非同期呼び出し」は低レイテンシー要件 NG（Domain 2 Practice 問題2.3）

AWS Secrets Manager から OAuth 認証情報を取得して bearer token を生成する構成（選択肢C）は **セキュリティ的に正しい**。
唯一の問題は「**Lambda プロキシ統合で Bedrock Agent を非同期に呼び出す**」こと。

- 会話型・低レイテンシー要件 → **同期呼び出し**が必要
- 非同期呼び出しはリアルタイム会話に対してレスポンスを返せない

**見落としパターン**：「認証情報の保存方法（Secrets Manager）に引っかかり、本当の問題（非同期）を見逃す」

## Step Functions / Bedrock Flows は「疎結合」要件を満たさない（Domain 2 Practice）

| 構成 | 疎結合か | 理由 |
|------|---------|------|
| Step Functions | ❌ | ステートマシン定義にツール呼び出しを直書き → ツール変更のたびに定義書き換え |
| Bedrock Flows | ❌ | ノードを GUI で事前設計 → 動的なツール選択不可 |
| Strands + MCP | ✅ | エージェントとツールが MCP 標準 I/F で接続 → ツール追加時にエージェント無変更 |

「疎結合」「動的ツール選択」「新ツールを追加しやすい」→ **Strands + MCP**

## Bedrock Prompt Flows vs Bedrock Agents（Task 2.5）

同じ「Bedrockで複数ステップを処理」に見えるが、自律性が根本的に異なる。

| | Prompt Flows | Bedrock Agents |
|---|---|---|
| フローの決定者 | **人間が事前に GUI でノードを設計** | **FMが実行時に自律的にツールを選択** |
| 対象ユーザー | 非技術者・ノーコード | 開発者 |
| 複雑さ | 固定フロー（条件分岐・Lambda呼び出しは可） | 動的推論・マルチステップ |
| キーワード | 「ノーコード・非技術者が自分で設計」「固定ステップ」 | 「自律的・複雑なタスク・マルチステップ推論」 |

Prompt Flows も Lambda ノード経由で複雑な処理を呼べるが「どのツールをいつ呼ぶかをFMが決める」のは Agents のみ。

### 「ドキュメント分析システムなのにRAG?」の思い込み（Domain 5 ボーナス1）

長文（コンテキストウィンドウ超過）の処理で、システムの呼び名(分析/要約/チャット)に引っ張られて RAG を選択肢から外す罠。

- **RAG = 外部知識追加だけではない**。「ウィンドウに入りきらない長文を分割して必要箇所を取り出す」汎用手段。**1つの長文内から関連チャンクを引くのもRAG**
- 「コンテキストウィンドウを超える」キーワードが出たら、システム名に関係なく **セマンティック境界＋重複チャンキング＋RAG** を第一候補に
- **要約系（再帰的要約）は情報を捨てる方向** → 「包括的な分析を維持」「条項間の関係を失わない」要件では必ず脱落（法的契約の相互参照が消える）
- チャンキングが複数選択肢に出たら **overlap / セマンティック境界 / RAG取得** が付いた方を選ぶ（固定サイズ＋関係維持なし＋連結は脱落）

## bedrock_resilience（traps）

出典: topics/bedrock_resilience.md

### フォールバック ≠ スロットリング対処（Domain 5 Practice 5.3）

`ThrottlingException`(429) の問題で「代替モデル/エンドポイントへのフォールバック」を選ばせる引っかけ。

| 用語 | 何への対処か | 429 に効くか |
|---|---|---|
| **フォールバック**（代替モデル/エンドポイント） | プライマリリソースが**見つからない/利用不可**（可用性障害） | ❌ 代替先にも同じアカウントクォータが適用される |
| **指数バックオフ＋リトライ** | 一時的なレート制限超過 | ✅ 吸収する |
| **プロビジョンドスループット** | スロットリングの発生源（容量不足） | ✅ 専用容量で断つ |
| マルチリージョンデプロイ | 可用性・負荷分散 | ❌ クォータは**アカウント単位**でリージョン単位ではない |

**判断**：429 が出たら **指数バックオフ＋リトライ ＋ プロビジョンドスループット**。フォールバックは「モデルが見つからない」エラー用で、レート制限には効かない（代替先も同じクォータ）。

**プロビジョンドスループットに time-of-day スケジュール機能はない**。「特定の時間帯だけ容量を増やす」ネイティブ機能は存在せず、PT は安定・予測可能な持続トラフィック向け。時間帯で変動させたいならオンデマンド。

### CRI × SCP のトラブルシューティング（AIP-9 不正解ポイント）

**エラーの原因**

CRIはリクエストを複数のEUリージョン（eu-west-1、eu-west-3、eu-north-1、eu-central-1など）にルーティングする。SCPが eu-west-1 以外を拒否していると、ルーティング先のリージョンで `SCP Deny` が発生する。

**解決に必要な対応（2つ両方必要）**

| 対応 | 内容 |
|---|---|
| ① SCP に例外ルールを追加 | `eu.amazon.nova-pro-v1:0` という**特定の推論プロファイル**に限定して、CRIルーティング先となる全EUリージョンのBedrock推論アクションを許可する例外を追加 |
| ② IAM ポリシーを確認 | 推論プロファイルARN **および** 関連する全EUリージョンの基盤モデルARNへの `bedrock:InvokeModel*` 権限が許可されていることを確認 |

→ SCP と IAM は**独立したレイヤー**。どちらか一方だけでは解決しない。

**不正解になるパターン（試験頻出）**

| 選択肢 | 不正解の理由 |
|---|---|
| eu-north-1でモデルアクセスを個別有効化 | CRIではルーティング先でのモデルアクセス個別有効化は不要。SCPのDenyも解消されない |
| PowerUserAccessをアタッチ | SCPのDenyはIAMのAllowより常に優先される。IAMレイヤーだけでは解決不可 |
| `eu.*` で全推論プロファイルを一括許可 | 技術的には動くが、意図しないモデルへのアクセスも許可されガバナンスが弱まる。最小権限の原則に反する |

**権限評価の優先順位**
```
SCP（Deny） > IAM（Allow）
```
SCPで明示的Denyがある場合、IAMでいくら許可しても無効。

### 不正解パターンとの比較：ALB加重ターゲットグループ

ALBの加重ルーティングでLambdaに振り分ける構成。**ALBはLambdaをターゲットに指定できる**（知っておくべきポイント）。

ただしこの構成が不正解になる理由：

- メトリクス判定後の重み調整が「SNS通知を受けた運用担当者が手動で調整」する流れ
- **「人手を介さず即座に切り戻す」という要件を満たさない**
- 段階的な比率引き上げを自動で進めるオーケストレーション機構が欠如

→ ALBの加重ルーティング自体は有用だが、**Step Functionsのような自動ワークフロー制御と組み合わせなければ**段階的デプロイの自動化は実現できない

## bedrock_guardrails（traps）

出典: topics/bedrock_guardrails.md

### 「ガードレール＝有害フィルタだけ」の即外し罠（Domain 5 ボーナス2）

RAG評価/ハルシネーション防止の問題で、「ガードレール」という単語だけで選択肢を即外すと正解を落とす。

- ガードレールには **コンテキストグラウンディングチェック** があり、RAG応答がソース文書に事実上基づくか（＋クエリに関連するか）をリアルタイム検証できる → ハルシネーション防止・ソース一致に使える正解候補
- 同じ「ガードレール」でも **有害応答ブロック用フィルタ（ヘイト/侮辱等）は事実正確性・ハルシネーション・関連性を評価しない** → こちらは不正解。**A vs B の差は「どのガードレール機能か」**
- 2フィルタ：**Grounding**（ソースに事実上基づくか）/ **Relevance**（クエリに応答してるか）。どちらか低ければハルシネーション扱い
- 対比：RAG評価で **ROUGE/BLEU（語彙類似度）は不適**（単語重複のみ、意味/ハルシネーション測れない）。体系的評価は**ナレッジベースの評価ジョブ**。**SageMaker Clarify** は従来MLバイアス用で生成LLM出力評価には非対応

### 不正解パターン整理

| 選択肢 | 不正解の理由 |
|---|---|
| Lambda + 正規表現 + ガードレール + CloudTrail + Global Accelerator | 正規表現は高度な攻撃を検出できない。CloudTrailはガードレール介入詳細を記録しない |
| ガードレール（拒否トピック・単語フィルターのみ）+ CloudTrail + Route 53 | プロンプト攻撃フィルター未使用。CloudTrailでは介入理由不十分。Route 53はガードレール継続を保証しない |
| ガードレール + SageMakerカスタム分類 + 各リージョンSMエンドポイント + S3 | 過剰なカスタム実装。Bedrockネイティブ機能で要件を満たせるのにオーバーエンジニアリング |

---

### 各選択肢の比較と間違えた理由

| 選択肢 | 結果 | 理由 |
|---|---|---|
| **OpenSearch k-NN** | ✅ 正解 | ANN専用アルゴリズム（HNSW・IVF）でミリ秒検索。意味ベクトル検索に最適化 |
| **Aurora pgvector** | ❌ 不正解（選んだもの） | ベクトル検索はできるが、RDB上でのSQL ORDER BYは大規模ANNに最適化されていない。「応答時間の短縮」要件でOpenSearchに劣る |
| **ElastiCache Redis** | ❌ 不正解（選びそうになった） | ハッシュ値の完全一致のみ。「表現が違う同じ意図」を検出できない |
| **DynamoDB + Comprehend** | ❌ 不正解 | Comprehendのキーフレーズ抽出は構文レベル。意味全体をベクトルで捉えるのとは別物 |

### 不正解パターン

| 選択肢 | 不正解の理由 |
|---|---|
| DynamoDB格納 | Bedrock評価ジョブはS3のJSONLのみ対応。DynamoDBから直接読めない |
| SageMaker Processing + カスタムコンテナ + OpenSearch | インフラ管理コストが大幅増。「運用負荷最小化」に反する |
| Step Functions + InvokeModel + Comprehend + QuickSight | Comprehendは品質スコアリングツールではない。QuickSightはCWより重い |

```
✅ 「Bedrock 組み込み評価の Faithfulness メトリクスでハルシネーション検知」
   → 生成された応答が参照文書に忠実かを直接測定
```

### RAG品質評価に使えない理由（AIP-79）

```
CloudWatch Synthetics が得意なこと：
  ✅ APIの可用性・レスポンス確認
  ✅ 事前定義した正解との一致チェック（固定パターン）

CloudWatch Synthetics が苦手なこと：
  ❌ 動的に変化するLLM応答のハルシネーション検知
  ❌ 意味的な品質評価（根拠整合性・正確性等）
  ❌ RAG特有の品質指標（Faithfulness等）の算出
  ❌ 新しいクエリパターンへの対応（固定テストケースのみ）
  
→ 「固定テストケース × 事前定義正解」との一致率しか測れない
  本番ユーザーの多様なクエリに対するハルシネーションは検知不可
```

### CloudWatch Events との混同

```
CloudWatch Events（= Amazon EventBridge の旧名）
  ✅ AWSリソースの状態変化に反応してアクションを起こす
     例）「EC2が停止したらLambdaを呼ぶ」
  ❌ 異常検出機能はない
  ❌ メトリクスのバーストパターン検知はできない

異常検出が必要なら → CloudWatch 異常検出アラーム（MLベースのバンド）
イベント駆動の自動化が必要なら → CloudWatch Events / EventBridge
```

|---|---|
| Step Functions **Express** + Aurora | Express は最大 5 分。承認待ちが数分〜数時間に及ぶ場合にタイムアウト |
| EventBridge + SQS FIFO + Glue + Redshift | キューでの承認リクエスト管理・状態追跡を自前実装。Glue + Redshift は監査ログ用途に過剰 |
| SNS + Lambda + **ElastiCache** + S3 | ElastiCache は揮発性キャッシュ。「永続保管」要件を満たせない |

→ **ElastiCache が含まれる選択肢は「永続性が必要」要件では必ず不正解**（AIP-20・AIP-28 共通）

---

### なぜ ElastiCache は不正解になるか（AIP-80）

```
❌ ElastiCache（Redis / Memcached）：
  ・Step Functions との直接統合なし
   → 各エージェントにキャッシュ読み書きロジックを追加実装が必要
   → 問題文「エージェントの推論・行動ループを変更しない」に違反
  ・VPCへのクラスタープロビジョニングが必要（運用負荷増）
  ・推論トレースのような大容量非構造化データの保存には不向き
  
✅ S3（正解）：
  ・Step Functions と Bedrock の両方にネイティブ統合あり
  ・問題文に「中間データはすでにS3に格納」と記載
  → 既存インフラを活用するだけ、追加実装不要
  ・S3は大容量データの永続保存に最適
```

## 公式模試2週目（2026-07-01）で判明した罠

### 網羅感トラップ（"しっかり見える"選択肢に釣られる）
- 「監視＋監査ログ」「AD＋セキュリティグループ＋CloudWatch」など、要素が多く"網羅的で安全そう"な選択肢を選んでしまう
- 多要素選択肢は、1個が overhead を足す/検知止まりで要件（運用負荷最小・"保護"）に反していないか、**要素ごとに軸を当てる**
- 特に金融・コンプラ文脈は「監視・監査ログ」で網羅感の罠を仕込んでくる

### 偽パラメータ罠（馴染みのサービス名＋検証できない具体値）
- 例：「SageMaker Ground Truth + `AwsManagedHumanLoopRequestSource: "AWS/Bedrock/Evaluation"`」。知ってるサービス名で"それっぽい"が、文脈違いの設定を盛った引っかけ
- 検証できない妙に具体的なパラメータが付いた選択肢は**緑信号でなく赤信号**

### Cognito即外し（"user"を「顧客」と脳内変換）
- Cognitoユーザープールは「ログインする人間の名簿」。アプリ顧客だけでなく、**Bedrock人手評価/Ground Truthのプライベートワークフォース（評価者）**の管理にも使う
- 「Cognito=アプリ認証」で即外すと、評価ワークフォース管理の正解を落とす
- 補足：既存の「Cognitoは組織のAWSリソース制御には向かない（そこはIAM Identity Center）」と矛盾しない。Cognitoは一貫して"人間の認証"で、評価者も人間

### 有害コンテンツ→トピックフィルターの誤マッピング
- 有害コンテンツ検出は**コンテンツフィルター**の仕事。トピックフィルターは"禁止テーマ"を止める機能。混同注意

### CloudTrail(監査) と CloudWatch(監視) の使い分け（2026-07-01）
- 「誰が何を変更/操作したかの**監査証跡・コンプライアンス・変更追跡**」＝ **CloudTrail** が正答
- 「性能・異常の**監視**」＝ CloudWatch。**"保護"や"監査"の代わりにCloudWatch監視**を出す選択肢は罠（金融ガードレール問題の誤答 A がこれ）
- 対比：Prompt Management監査問題では CloudTrail が正答／金融ガードレール問題では CloudWatch監視が罠、という真逆の出方をする

### 「検索」でpgvector脊髄反射の罠（2026-07-01）
- 「メタデータに基づく検索・フィルタリング」＝構造化クエリ。**DynamoDB GSI** が正答。pgvector(意味/ベクトル検索)はミスマッチ
- 「検索」の語＋pgvectorを見て飛びつくと、会話履歴の大規模ストア問題（正答=DynamoDB）を落とす（Dの網羅感トラップ）
