新しいミスは試験番号順に追記し、導出した判断ルールを [cheatsheet.md](./cheatsheet.md) にも1行転記する。

## ⚠️ 模試ミス回収（AIP-23）：「サブモジュール + @workspace」を選んでしまった

**選んだ理由（誤った思考）**：
「社内資産を別リポジトリとしてサブモジュールでローカルに置けば、プロジェクトリポジトリ自体は変えずに Q Developer が参照できるのでは？」

**なぜ間違いか：2 つの見落とし**

**① サブモジュールの追加 = リポジトリ構成の変更**

```
サブモジュールを追加すると、プロジェクトリポジトリに変更が入る
  ├─ .gitmodules ファイルが新規作成（コミットが必要）
  └─ サブモジュールのマウントポイントディレクトリが追加される
→ 「リポジトリ構成を変更しない」要件に反する
```

「別リポジトリのコードをローカルに置く」行為が、プロジェクトリポジトリへの変更を伴う点を見落とした。

**② @workspace はインライン補完には効かない・毎回手動指定が必要**

```
@workspace が効く場面
  → チャットで「@workspace」と書いたときのみ（手動）

@workspace が効かない場面
  → コード入力中のインライン補完（リアルタイムのサジェスト）
  → 開発者が @workspace を書き忘れたとき
```

→ 組織全体への一貫した適用が保証されず、「全チームへの強制反映」にならない。

**正しい判断軸**

| | サブモジュール + @workspace | カスタマイズ機能 |
|---|---|---|
| リポジトリ変更 | **.gitmodules が追加される** | **なし** |
| インライン補完への反映 | 効かない | **常時反映** |
| 適用の一貫性 | 開発者が毎回手動指定 | 管理者が割り当てれば全員に自動適用 |

**覚え方**：「サービス側で設定 = プロジェクトへの変更なし」。リポジトリ変更ゼロを実現できるのはサービス設定で完結するカスタマイズ機能だけ。

## ⚠️ 模試ミス回収（AIP-25）：「API Gateway HTTP API + Cognito」を選んでしまった

**選んだ理由（誤った思考）**：
「OAuth 2.0 でアクセストークンを発行すれば強固な認証になる。Cognito はエンタープライズでよく使われる認証サービスだから適切では？」

**なぜ間違いか：2 つの見落とし**

**① API Gateway + Cognito = 追加の管理コンポーネント 2 つ**

```
Lambda Function URL（正解）
  → Lambda のみ（管理コンポーネント最小）

API Gateway + Cognito（不正解）
  → API Gateway の設定・監視・バージョン管理
  + Cognito ユーザープールの管理・アプリクライアント発行・トークン有効期限管理
  → 問題文「管理すべきインフラコンポーネントを最小限に」に反する
```

**② API Gateway HTTP API はレスポンスストリーミングに非対応**

```
MCP の要件：「ストリーマブル HTTP トランスポート」= レスポンスを逐次返す
API Gateway HTTP API    → ストリーミング非対応（全レスポンスを一括返却）
Lambda Function URL     → RESPONSE_STREAM モードでネイティブ対応
```

## ⚠️ 模試ミス回収（AIP-27）：「KB + SQS + Cognito」を選んでしまった

**選んだ理由（誤った思考）**：
「ビルごとに KB を分けて、SQS でリアルタイムに更新し、Cognito でアクセス制御すれば全要件を満たせるのでは？」

**なぜ間違いか：3 つの見落とし**

**① SQS + KB 更新はリアルタイムではない**

```
SQS にメッセージが届く
  ↓ Lambda が受信（ポーリング）
  ↓ KB のインジェストジョブを実行（チャンキング・埋め込み・格納）
  ↓ 数分〜数十分後に反映

「空室状況」のようにリアルタイム性が必要なデータには不適切。
正解：エージェントのアクショングループで BMS API を直接呼び出す。
```

**② Cognito は「AWS リソースへの組織的アクセス管理」には向いていない**

```
Cognito：Web/モバイルアプリのユーザー認証が主目的。
IAM Identity Center：組織の SSO・AWS リソースへの権限管理が主目的。
→「管理スタッフが自分のビルの KB だけにアクセス」= IAM Identity Center の許可セットが正解
```

**③ KB の上限（約 10 個）を意識する**

```
ビルが 10 棟を超えると KB のクォータに達する可能性がある。
エージェントが複数 KB をオーケストレーションする構成なら、
クォータ管理やメタデータフィルタとの組み合わせで柔軟に対応できる。
```

**正しい判断軸**

| 判断ポイント | 正解の考え方 |
|---|---|
| リアルタイムデータ | KB 同期ではなく、エージェントのアクショングループで API を直接呼ぶ |
| 組織の AWS リソース制御 | Cognito ではなく IAM Identity Center |
| データ分離 | ビルごとに KB を作成（ただし上限に注意）|

## Domain 2 Practice 問題2.3：C の脱落理由を誤認した

**誤った思考**：「Secrets Manager に OAuth 認証情報を保存することが最小権限原則に反するのでは？」と判断して C を除外した。

**なぜ間違いか**：Secrets Manager への OAuth 認証情報（client_id / client_secret 等）の保存は正しいベストプラクティス。C が不正解な本当の理由は「**Lambda プロキシ統合を使用して Bedrock エージェントを非同期に呼び出す**」こと。

- 会話型・低レイテンシー要件 → 同期呼び出しが必要
- 非同期呼び出し → レスポンスをリアルタイムに返せない

**正しい判断軸**：OAuth 認証情報の保存方法ではなく、「**同期か非同期か**」を確認する。要件に「低レイテンシー」「リアルタイム」があれば非同期呼び出しは即除外。

## Domain 2 Practice ボーナス問題4：Step Functions + StartAsyncInvoke を選んだ

**誤った思考**：「リアルタイムとバッチの2系統をStep Functions の choice ステートで振り分ければ良い。バッチ側は StartAsyncInvoke で非同期呼び出しすれば良い」と判断した。

**なぜ間違いか：2つの問題**

**① StartAsyncInvoke は Nova Reel（動画生成）専用**

```
テキスト系モデルのバッチ推論 → CreateModelInvocationJob
動画生成（Nova Reel）→ StartAsyncInvoke
```

**② Step Functions Standard は同期実行 → 呼び出し元がバッチ完了まで待機**

```
Express ワークフロー → 最大5分（バッチ処理時間を超える可能性）
Standard ワークフロー → 同期実行（呼び出し元がブロックされる）
→ どちらもリアルタイム+バッチ 2系統には不向き
```

**正しい判断軸**：応答時間が異なる2要件は **別アーキテクチャで捌く**。
`API GW → Lambda → InvokeModel`（リアルタイム）+ `SQS → Lambda → CreateModelInvocationJob`（バッチ）

## ⚠️ 模試ミス回収（Task 3.2）：「包括的セキュリティ」でS3暗号化を選んでしまった

**選んだ理由（誤った思考）**：「データ保護 → 暗号化」と直結させて CloudWatch + IAM + S3暗号化 を選んだ。

**なぜ間違いか**：Bedrock はデフォルトで転送中・保管中の暗号化が適用済みのため、S3暗号化を追加設定する話にならない。問題文の「データ保護」は監査ログ（CloudTrail）によるアクセス追跡の文脈で使われていた。

**正しい判断軸**：「Bedrock の包括的セキュリティ（モニタリング・アクセスコントロール・データ保護）」→ IAM + CloudTrail + CloudWatch。CloudTrail が "データ保護（監査）" を担う。暗号化はデフォルト済みなので選択肢に出てきたら罠。

## ⚠️ 模試ミス回収（Task 3.2 ナレッジチェック01）：「包括的セキュリティ2択」でLake Formationを外した

**選んだ理由（誤った思考）**：「セキュリティグループ + IAM ロール」でネットワーク・ID制御は揃っていると判断して選んだ。

**なぜ間違いか**：「包括的」には2軸が必要。①ネットワーク分離×アクセスコントロール②モニタリング×データガバナンス。セキュリティグループ+IAMロールはデータレベルの制御（Lake Formation）とモニタリングが欠落。また「保護された AI 環境」でパブリックサブネットは即アウト。

**正しい判断軸**：「機密データを扱う保護された AI 環境」で「包括的」→ Lake Formation が必ず入る。モニタリング軸には CloudWatch。パブリックサブネットは即除外。2択問題は「何の軸が揃っているか」を確認する。

## ⚠️ 模試ミス回収（Task 3.3 ナレッジチェック01）：「包括的コンプライアンス2択」でModel Cardsを外した

**選んだ理由（誤った思考）**：Glue によるメタデータタグ付けのみ選択。Model Cards はデータリネージュ追跡ではないと判断して除外した。

### AIP-31 の正解構成と不正解パターン

**問題**：AppSync + Lambda（同期）で Bedrock を呼び出すと「表示まで長時間待つ」「複雑な質問でタイムアウト」が発生

**根本原因**：同期一括返却（全文生成完了まで何も返さない）

**正解**：Amplify AI Kit + AppSync Subscription → ストリーミングで逐次描画

```
[Before] 同期方式
  ユーザー → Lambda → Bedrock（全文生成） → 一括返却 → 画面表示
  問題：数十秒間 画面が空のまま / タイムアウト

[After] ストリーミング方式
  ユーザー → AppSync Subscription（WebSocket）
          → Bedrock がトークン生成するたびにチャンクをプッシュ
          → フロントエンドで逐次描画
  効果：TTFT（最初の文字表示まで）が大幅短縮 / タイムアウト回避
```

| 不正解選択肢 | 理由 |
|---|---|
| Lambda のメモリ増強 + プロビジョンドコンカレンシー | Bedrock 側の生成時間は変わらない。同期構造のまま |
| API Gateway REST + Step Functions + ポーリング | アーキテクチャを全面刷新。ポーリングは体感速度改善が限定的 |

```
BOLD データセット          → 属性バイアス（性別・人種・職業・年齢）専用

Bedrock Prompt Management → 制御されたプロンプトバリアントテスト
```

**データセットの目的対応表（試験頻出）：**

| データセット | 目的 | 使うべき場面 |

| **BOLD** | 属性バイアス（性別・人種・職業・年齢）| LLM の公平性評価 |
| **RealToxicityPrompts** | 有害コンテンツ（Toxicity）| 有害性フィルタの評価 |

## ⚠️ 模試ミス回収（Domain 3 Practice 3.5）：ガバナンス要件で CloudTrail + CloudWatch を選んでしまった

**選んだ理由（誤った思考）**：
「ガバナンス = 監査できればOK」と考え、CloudTrail（API操作ログ）+ CloudWatch（アラーム）で十分と判断した。ガードレールを「コンプライアンス違反の検出だけならいらない」と誤解した。

**なぜ間違いか：3つの見落とし**

① **CloudTrail ≠ プロンプト履歴**
```
CloudTrail     = 誰がいつどのAPIを呼んだか（APIメタデータのみ）
Model Invocation Logging = プロンプト内容・完了内容・モデルバージョンの詳細を記録
→ 「プロンプト履歴を記録する」要件 = Model Invocation Logging が必須
```

② **CloudWatch アラーム ≠ コンプライアンス違反のブロック**
```
CloudWatch アラーム = 違反を「検出・通知」するだけ
Bedrock ガードレール = 違反コンテンツを「フィルタリング・ブロック」する
→ 「コンプライアンス違反をモニタリング」だけでなく「防ぐ」要件 = ガードレールが必要
```

③ **「開発作業を最小限」= Bedrock ネイティブ機能を使う**
```
A（誤）: CloudTrail + CloudWatch → プロンプト内容追跡に追加実装が必要
C（正）: Guardrails + Model Invocation Logging + CloudTrail → GenAI ガバナンス専用機能でカバー
```

**正しい判断軸（正解 C の構成）：**

| 要件 | 使うサービス |
|---|---|
| プロンプト・完了・モデルバージョンの履歴記録 | **Model Invocation Logging**（→ CloudWatch Logs に送信） |
| コンプライアンス違反のフィルタリング | **Bedrock ガードレール** |
| アクセス追跡・監査ログ | **CloudTrail** |

## ⚠️ 模試ミス回収（Domain 3 Practice 3.6）：S3のPII検出でバッチ処理（C）を選んでしまった

**選んだ理由（誤った思考）**：
「S3に保存済みのデータ = バッチ処理で十分、リアルタイムAPIはコスト高」と思い込んでBを除外。Cに「修復アクション」の記載があったため選択した。

**なぜ間違いか：**

① **Macie には「機密データの自動検出（継続的監視）」機能がある**
```
Macie 自動検出（B）  = S3バケットを常時監視。新データを継続的に検出
Macie 検出ジョブ（C）= 週次・月次スケジュール実行 → 検出に遅れが生じる
→「自動修復アクション」要件には継続的監視が必要
```

② **C の運用オーバーヘッドが高い**
```
C: Comprehend（月次）+ Macie（週次）の結果を Lambda で比較・統合 → カスタム実装が必要
B: Macie が検出 → EventBridge が拾う → Lambda が修復 → シンプルな連携
→「運用オーバーヘッド最小」要件では B が優位
```

③ **「S3のデータ = バッチでいい」は試験では通じない**
- 要件に「自動修復」「継続的コンプライアンス」があれば、継続的監視（Macie自動検出）が正解
- リアルタイム vs バッチは「コスト」ではなく「要件（遅延許容度）」で判断する

**正しい判断軸（正解 B の構成）：**

| 要件 | 使うサービス |
|---|---|
| テキスト内のPII検出（名前・電話・クレカ） | **Amazon Comprehend** リアルタイム PII 検出 API |
| S3全体の機密データ分類・継続監視 | **Amazon Macie**（自動検出を有効化） |
| PII検出時の自動修復トリガー | **EventBridge** ルール → **Lambda** でS3バケットポリシー更新 |
| 監査用コンプライアンスレポート | Macie のネイティブレポート機能 |

**Comprehend vs Macie の役割整理（混同注意）：**

| サービス | 得意領域 | S3との関係 |
|---|---|---|
| **Comprehend** | テキスト文書内のPII・エンティティ検出（API呼び出し型） | S3のファイルを読んでAPIで解析 |
| **Macie** | S3バケット全体の機密データ管理・分類・コンプライアンスレポート（継続監視型） | S3ネイティブ統合。バケット単位で自動スキャン |

## 公式模試2週目 Q（評価ジョブ開始失敗の原因）：CORS想起 → 正答D（1,000件クォータ超過）

- 問題：5,000プロンプトのカスタムデータセット(JSONL)をS3にアップロードして自動モデル評価ジョブを作成→開始失敗。原因と対処。
- **CORSに飛びついた理由**：直前に学んだ「Bedrock評価ジョブ＋S3＋CORS必須」という記憶が強く残っていて、別の失敗モードに誤って結びつけた。実際はCORSは**人による評価（Human Evaluation）でブラウザUIが評価者にコンテンツを表示する場合**の要件で、今回の自動モデル評価とは無関係
- **正答D**：**1データセットあたり1,000プロンプトのクォータ**を超過するとジョブ開始自体が失敗する。5,000件を1,000件×5データセットに分割し、個別ジョブを実行する
- **誤答Aのダメな点**：gzip圧縮してもプロンプト**件数**は変わらないのでクォータ超過は解消しない（サイズの問題ではなく件数の問題）
- **誤答Bのダメな点**：カスタムプロンプトデータセットは拡張子`.jsonl`のJSONL形式が必須。CSVへの変換はさらに形式エラーを生む
- **判断ルール**：「ジョブ**開始**（作成時点）で失敗」は多くの場合、実行前の**静的な設定検証**（形式・クォータ・権限）に引っかかっている。直前に学んだ別の失敗パターン（CORS等）を反射的に当てはめず、エラーメッセージが指す対象（件数か・形式か・権限か）を先に切り分ける

## 診療所ごとのコスト追跡（S3+Bedrock要約、2つ選択）：B,E → 正答C,E

- **判断ルール**：「複数テナント/顧客/拠点ごとのFM利用コストを追跡」→ **アプリケーション推論プロファイル**（テナントIDごとに作成、追加インフラなし）。EventBridge/CloudWatchメトリクスやS3系の分析ツール（Storage Lens等）はイベント配信・ストレージの話であってBedrockのコスト追跡はできない、と即座に区別する

- **判断ルール**：①「SageMakerで無停止のエンドポイント設定変更」→ 新エンドポイント設定+`UpdateEndpoint` API（エンドポイント自体の削除・再作成は不正解） ②「テキスト/埋め込みのドリフト検出」→ Model Monitor標準4種にもClarifyにも当てはまらない場合は**カスタムECRイメージでのModel Monitor自作**を検討

## Step Functionsで一貫した多段階推論を実装（Bedrock FM、推論パターン問題）：B → 正答A

- 問題：企業がBedrock FMをAPI経由で呼び出し、AWS Step Functionsで構造化された方法でFMが各問題を一連の論理推論ステップに分解できるようにしたい。設問末尾「この推論パターンを**最も一貫性のある方法**で実装できるソリューションはどれか」
- **誤答B（思考連鎖＋Choice状態で中間応答に基づき推論経路を動的調整）のダメな点**：「状況に応じて経路を変えられる＝柔軟＝良さそう」という直感で選んだが、これは軸違い。設問の軸は「一貫性」であって「柔軟性」ではない。動的分岐は問題ごとに異なる経路をたどるため、複数ドメイン・シナリオに適用できる予測可能で構造化された分解フレームワークにならない
- **正答A**：ReActパターンの状態を持つステートマシンを作成し、観察・推論・アクション計画それぞれの状態に**固定の特定プロンプト**を設定、組み込みのStep Functionsエラー処理・再試行ロジックでモデル呼び出しを行う。プロンプトが固定（＝分岐しない）であることで、どんな問題を渡しても同じ構造で体系的に分解される＝一貫性が担保される
- **判断ルール**：設問末尾の軸が「一貫性」「予測可能性」「標準化」の場合、選択肢の「状況に応じて動的に調整」「Choice状態で分岐」という記述は柔軟性の魅力に引っ張られて選ばない——固定ステート＋固定プロンプトの方が一貫性軸では強い。逆に軸が「複雑度に応じた最適振り分け」なら動的分岐が正解になる逆転パターンもあるため、まず設問末尾の軸を確定させてから選択肢を裁く

## メディア企業のレガシーCMS統合（複数AIエージェント、標準化＋再利用）：A → 正答D

- 問題：メディア企業がコンテンツ準備を自動化する複数のAIエージェントを構築中。レガシーCMSのAPIが個々に不整合（エンドポイントごとに挙動が異なる）で、各エージェントが個別に対処している。複数のエージェントワークフローで再利用でき、運用負荷を最小限にする標準化されたインターフェースが欲しい。
- **誤答A（REST APIの変換・プロキシ層を構築しドキュメント化・標準化）のダメな点**：レスポンスの見た目は整うが、**関数呼び出しという意味的（semantic）インターフェースを提供しない**。結局各AIエージェントは整形されたREST APIに対して個別の統合コードを書く必要が残り、「複数エージェントで再利用できる仕組み」が存在しない
- **正答D**：CMS操作用の関数スキーマを持つ**MCPサーバー**を実装し、API個々の不整合はMCPサーバー内部で吸収。**Bedrock AgentCore**をMCPクライアント（エージェント実行環境）として構成し、MCP関数呼び出し経由でCMSとやり取りさせる。AgentCoreを使う全エージェントワークフローが同じMCP関数群をそのまま再利用できる
- **選択肢B（サイドカーコンテナとしてカスタムミドルウェアを各エージェントに配置）が不正解な理由**：ECSサイドカーはメインコンテナと同一タスクで並走する補助コンテナだが、エージェントの数だけミドルウェアを重複デプロイ・保守することになりスケールしない。標準化された関数呼び出しI/Fを1箇所に集約する発想がない
- **判断ルール**：「レガシーAPIの不整合を吸収しつつ複数エージェントで再利用・運用負荷最小」という軸が出たら → **MCPサーバー（関数呼び出しスキーマ）＋AgentCore**一択。「REST/OpenAPIをきれいにする・ドキュメント化する」系の選択肢は、関数呼び出しの抽象化＝AIネイティブな再利用機構を提供しない限り不正解。「サイドカー」「エージェントごとに個別配置」という記述はスケーラビリティ面のトラップ

## Bedrock Guardrailsのプロンプト攻撃対策（InvokeModel API、多層防御）：Classicティア案 → 正答Standardティア案

- 問題：InvokeModel APIで動く保険会社のカスタマーサポートAIが「前の指示を無視して」型のプロンプトインジェクションを受けた。プロンプトインジェクション・ジェイルブレイク・**プロンプトリーケージ（システムプロンプト内容の抜き出し）**の3種すべてを検知・ブロックしたい。
- **誤答の原因**：**Guardrailsに「Standard」と「Classic」の2つのセーフガードティアが存在すること自体を知らなかった**。プロンプト攻撃フィルター＋XMLタグ付けまでは正しく想起できても、ティアの選択で脱落する設計
- **正答の構成**：Guardrailsを**Standardティア**で作成し、プロンプト攻撃フィルターの入力感度を最高（HIGH）に設定。InvokeModel使用時はユーザー入力を`<amazon-bedrock-guardrails-guardContent_{ランダムサフィックス}>`タグで囲んでシステムプロンプトと区別する
- **なぜClassicティアが不正解か**：**プロンプトリーケージ検出はStandardティアのみ対応、Classicティアは非対応**（公式ドキュメントで明示）。プロンプト攻撃フィルター自体はClassicにもあるが、3要件のうち1つを構造的に満たせない
- **他の誤答パターン**：AWS WAF（HTTPレイヤの文字列マッチ。プロンプト攻撃は正常なHTTPリクエストに見えるため検出不可）、Amazon Comprehendの有害コンテンツスコアリング（プロンプト攻撃は「有害」カテゴリで高スコアにならない）、Guardrailsの拒否トピック機能（開発者定義の話題誘導防止であり、プロンプトリーケージ専用の検出ではない）
- **判断ルール**：「プロンプトリーケージ」というキーワードが要件に出たら即座に**Standardティア**（Classicは検討候補から外れる）。InvokeModel系APIを使う設問では「XMLタグ付け必須」も必ずセットで確認する。詳細は [bedrock_guardrails.md](../topics/bedrock_guardrails.md) のティア比較表を参照

## セマンティックキャッシュ（クエリ表現の揺れを吸収）：誤答 → 正答 OpenSearch Service + k-NN

- 問題：顧客クエリの40%が表現違いの同一質問。冗長なBedrock呼び出しを削減し、意味的に同等な質問には一貫した回答を低レイテンシーで返したい。
- **誤答の原因（本人談）**：①正解の選択肢が「クエリと応答の**テキストペア**を格納する」という書き方だったため「ベクトル化していないのでは」と勘ぐって除外してしまった。②OpenSearch Serviceは**コストが高いという先入観**で他の選択肢に流れた
- **①への訂正**：「テキストペアを格納」は生テキストをそのまま保存するという意味ではなく、**埋め込みベクトルと元テキストを両方格納する**（`knn_vector`フィールド＋通常フィールドの併用）という意味。選択肢文中に「k-NN機能」「近似k-NN」と明記されている時点で、内部的にベクトル化している前提で読む必要があった

- **他の誤答選択肢の構造**：MemoryDB for Valkey案は「ベクトル検索機能はあるが**RANGEクエリ**という誤ったクエリ手法を指定」（正しくは`FT.SEARCH`+KNN構文）。DynamoDB DAX案は「**LIKE演算子**を使うと書いてあるが、DynamoDBにLIKEは存在しない」（文字列関数は`begins_with`のみ）。DynamoDB GSI+ステミング案は「語幹抽出だけでは同義語・言い回し違いを吸収できない」
- **判断ルール**：「表現・言い回しが違うが意味は同じ」→ **ベクトル埋め込み＋近似k-NN検索が必須**（OpenSearch Service k-NNが定番）。「テキストペアを格納」という表現だけでベクトル化なしと判断しない。ベクトル検索対応をうたうサービスでも、選択肢中の**具体的なクエリコマンド**（RANGE/LIKE/begins_with等）が本当にベクトル近傍探索に対応しているか確認する。コストが問われていない設問でコストを理由に機能要件から逃げない。詳細は [bedrock_guardrails.md](../topics/bedrock_guardrails.md) のセマンティックキャッシュ節参照

## 静的システムプロンプトの入力トークンコスト削減：Prompt Managementと迷って正答 → プロンプトキャッシュ

- 問題：8,000トークンの静的システムプロンプトを毎回送信しているチャットボット。入力トークンコスト削減＋リアルタイム応答維持＋API変更最小限が要件。
- **危なかった理由（本人談）**：**プロンプトキャッシュを「応答（回答）自体をキャッシュするもの」と誤解**していた。実際は応答のキャッシュではなく、**入力プロンプトの"計算結果"（KVキャッシュ相当）を再利用してトークン再処理を省く**仕組み。名前の印象から意味的キャッシュ（セマンティックキャッシュ）と混同しやすい
- **Prompt Managementとの違い**：Prompt Managementは**プロンプトの保存・バージョン管理・承認ワークフロー**が目的で、リソース識別子で参照する構成に変えても**推論時は毎回全トークンを再処理する**ため、入力トークンコストは削減されない。「プロンプトキャッシュを明示的に有効化」しない限り無関係
- **プロビジョンドスループット・バッチ推論が不正解な理由**：プロビジョンドスループットはMU単位のスループット容量予約であり、**リクエストごとの入力トークン課金構造は変わらない**（8,000トークンは毎回再処理される）。バッチ推論は非同期でS3経由の結果取得となるため、リアルタイム応答要件に反する
- **正答の構成**：Converse APIの`system`フィールド末尾に`cachePoint`を1箇所追加するだけ。8,000トークンは全対応モデルの最小チェックポイントトークン数（1,024〜4,096）を超えるため確実にキャッシュ書き込みが発生し、以降のリクエストはTTL内（多くのモデルで5分、一部1時間オプションあり）でキャッシュ読み取りレート（標準入力より低い）が適用される
- **判断ルール**：「リクエスト間で変わらない静的プロンプト」＋「入力トークンコスト削減」＋「API変更最小限」→ **プロンプトキャッシュ（cachePoint）一択**。プロンプトキャッシュ＝応答キャッシュではなく**入力の再計算省略**という理解を優先する。「プロンプト管理」「バージョン管理」という言葉に釣られてPrompt Managementを選ばない。詳細比較は [bedrock_resilience.md](../topics/bedrock_resilience.md) 参照

## 長文技術文書の要約ハルシネーション削減（2つ選択）：「Guardrailsパターンマッチ」で速攻選択して誤答 → 正答 ゼロショットCoT + RAG

- 問題：長い技術文書の要約でハルシネーション急増。事実精度改善＋1時間1,000件以上のスループット＋3秒以内のレイテンシーを維持する構成を2つ選ぶ。**この問題は既存メモ（bedrock_rag.md「ハルシネーション削減アプローチ AIP-30」）にほぼ同一の問題が既に記録済みだった**——知識の欠落ではなく既知パターンの取りこぼし
- **誤答の原因（本人談）**：選択肢に「Guardrails」「グラウンディング」の語を見た瞬間に**「ハルシネーション対策＝Guardrails」という連想だけで速攻選択**してしまい、その選択肢が「**パターンマッチ**でハルシネーションをブロックする」という誤った仕組みで説明されている点を最後まで読んでいなかった
- **なぜ不正解か**：コンテキストグラウンディングチェック・自動推論チェックは、どちらも「既知のハルシネーションパターンを文字列マッチでブロックする」仕組みでは**ない**。実際はGrounding Source（参照ソース）を別途渡し、応答がそのソースに基づいているかをスコアリング／形式論理で検証する仕組み。**参照ソースを提供せずパターンマッチだけでハルシネーションを検出する、という選択肢の説明自体が構造的に成立しない**。加えてGuardrailsは生成後の出力をブロックする事後レイヤーであり、生成品質そのものを改善する根本対策（RAG・CoT）にはなり得ない
- **正答**：①ゼロショットCoT（「ステップバイステップで考え、根拠を明示してから要約」という指示をプロンプトに追加。追加インフラ不要でレイテンシー要件を維持）②RAG（Bedrock Knowledge Base + セマンティックチャンキングで、ソースコンテンツに基づいた要約を生成させグラウンディングを確保）
- **判断ルール**：**「Guardrails＝ハルシネーション対策」という機能名の連想だけで即選択しない**。選択肢中に書かれた**検出メカニズムの記述**（パターンマッチなのか、ソース照合・スコアリングなのか）を必ず最後まで読む。メカニズムの記述が実際の仕組みと矛盾していれば、機能名が正しくても不正解。長文要約のハルシネーション対策は**ゼロショットCoT＋RAG（Bedrock Knowledge Base）の組み合わせ**が定番構成——同型の問題が既に[bedrock_rag.md](../topics/bedrock_rag.md)のAIP-30節に記録済みなので、次回は思い出すこと

## 多ターン会話の入力トークン超過エラー：Lambdaタイムアウト延長+リトライを選んで誤答 → 正答 CountTokens API + 古いターンの要約置き換え

- 問題：Converse APIで多ターンチャット運用中、会話が長いユーザーほど推論リクエストがエラーになる。エラーを恒久的に解消しつつ会話継続性を維持したい。Lambda上で動作、現行タイムアウト30秒。
- **誤答の原因**：「エラー」「タイムアウト30秒」という表層情報に反応して**Lambdaのタイムアウト延長＋指数バックオフリトライ**を選択。これは「処理が重くて時間切れ」という別種の障害（ダウンストリームの応答待ち遅延）への対処法であり、今回の症状とは原因が違う
- **なぜ不正解か**：エラーの根本原因は**Bedrockがステートレスなため会話ターンを重ねるたびに送信メッセージ列の総トークン数が蓄積し、モデル固有のコンテキストウィンドウ上限（固定・API変更不可）を超過した**こと。タイムアウトを延長してもリトライを重ねても、**同じ過大な入力を送り続ける限り何度でも同じエラーが再発する**——「会話が長いユーザーほど発生」「ユーザーごとにタイミングが違う」という記述が、時間切れ型障害ではなく蓄積型（データ量起因）の障害であることを示す明確なシグナルだった
- **正答**：**CountTokens API**で推論前に入力トークン数を計測（推論しないので無課金）→ 閾値に近づいたら会話の先頭（古いターン）を**要約した短いメッセージに置き換えて**から送信。コンテキストサイズを上限内に維持しつつ文脈も失わない
- **他の不正解選択肢の構造**：`maxTokens`引き上げは**出力**トークン上限の話であり**入力**側のコンテキストウィンドウとは無関係（頻出の引っかけ）。Guardrails+CloudWatch強制再起動案は、Guardrailsがコンテンツ安全性目的でコンテキスト超過検知機能を持たない上、Lambda再起動は次回同じ入力を送れば再発するため無意味
- **判断ルール**：「エラー」というワードを見たら反射的にタイムアウト/リトライ系を疑う前に、**症状の発生パターン**（時間経過で悪化？特定ユーザーで悪化？データ量に比例？）から根本原因（時間切れ型 vs 蓄積型）を先に切り分ける。「会話が長い/データ量が多いユーザーほど発生」は蓄積型（コンテキスト超過）のシグナル。詳細は [bedrock_core.md](../topics/bedrock_core.md) のトークンウィンドウ管理節参照

## Bedrock支出の急増を機械学習で異常検知：CloudWatch異常検知（推定請求額）を選び誤答 → 正答 AWS Cost Anomaly Detection

## Lambda+EKS、Python+JS SDK、統一認証・マルチターン維持：既存メモ（AIP-52）と同型で誤答

- 問題：Lambda(Python)とEKS(JavaScript)の2環境で同じFMを使い、マルチターン会話の文脈維持＋一貫した認証を実現したい。**この問題も既存メモ（bedrock_core.md「Converse API vs InvokeModel API」AIP-52）にほぼ同一の要件セットが既に記録済みだった**——当日4件目の既知パターン再現ミス

- **DynamoDBパターンとの使い分け**：「同時セッション・任意地点から再開・メタデータ検索・保存ポリリシー」等の**追加要件**が明示された場合はDynamoDBが正解になる（bedrock_core.md 824行目付近の別パターン）。今回はそうした追加要件がなく、単純な「マルチターン維持」だけだったのでmessages[]で十分だった
- **判断ルール**：「マルチターン継続」のみ→**messages[]（外部ストア不要）**。「検索・再開・監査保持」まで要求→**DynamoDB**。この軸で読み分ける

## S3+OpenSearch ServiceをCMKで暗号化強制：バケットポリシーによる強制が抜けて誤答

- 問題：Bedrock KB（S3ソース＋OpenSearch Serviceベクトルインデックス）を顧客管理キー（CMK）で保存時暗号化。追加コードなしでAWS標準機能のみで実現したい
- **誤答の原因**：「S3のデフォルト暗号化をSSE-KMS(CMK)に設定する」だけで暗号化が保証されると思い込んだ。実際は**デフォルト暗号化はPutObjectリクエスト側で別の暗号化方式（SSE-S3等）を明示指定されると簡単に上書き・回避できる**フォールバックにすぎない
- **正答に必要な追加要素**：**バケットポリシーに条件キー（`s3:x-amz-server-side-encryption`・`s3:x-amz-server-side-encryption-aws-kms-key-id`）を設定し、指定CMK以外でのPutObjectをDenyする**ことで初めて暗号化が漏れなく強制される
- **もう1つの誤答選択肢（非対称CMK案）**：**S3のSSE-KMSとOpenSearch Serviceの保存時暗号化はどちらも対称型CMKのみ対応**。非対称キー（RSA等）は署名・TLS用途でありデータ暗号化には使えない
- **判断ルール**：「S3のデフォルト暗号化＝CMK暗号化の保証」ではない。**「すべての保存データに強制」「漏れなく」というニュアンスが出たらバケットポリシーによるDeny条件の追加を疑う**。KMSキーの種別（対称/非対称）も要件に応じた制約があるので毎回確認する。詳細は [security_governance.md](../topics/security_governance.md) 参照

## Strands Agents SDKのツール定義：Bedrock Agentsのアクショングループ/OpenAPIスキーマと混同して誤答

- 問題：Strands Agents SDKで構築中のエージェントに、実行環境を分離したカスタムツール（REST API呼び出し＋DynamoDB書き込み）を追加し、不正パラメータ時は後続処理を止めてエラーを返したい
- **誤答の原因**：**Strands Agents SDK（OSSのPythonフレームワーク）とAmazon Bedrock Agents（AWSマネージドサービス）を混同**し、Bedrock Agents側の概念（アクショングループ・OpenAPIスキーマ）を答えに使ってしまった。2つは製品レイヤーが根本的に別物
- **正答**：Strands Agents SDKのツール定義は**Python関数に`@tool`デコレータを付与するだけ**（関数シグネチャ＋docstringから自動でツール説明が生成される）。パラメータ検証は**SDK側に自動検証機構が無い**ため、ツール関数内で自前実装し不正時は例外を送出する。ツール実行環境はLambda等オーケストレーションループとは別の場所に明示的に配置する
- **判断ルール**：「OpenAPIスキーマ」「アクショングループ」＝Bedrock Agents側の語彙／「`@tool`デコレータ」＝Strands SDK側の語彙。問題文がどちらの製品を使っていると明示しているかを最初に確認し、語彙を取り違えない。Step FunctionsもStrandsのモデル駆動ループ自体の代替にはならない点も併せて注意。詳細は [bedrock_agents.md](../topics/bedrock_agents.md) 参照

## 3エージェント逐次連携（調査→要約→レポート作成）：「コード最小限」でStep Functionsを選び誤答 → 正答 Strands Agents SDK

## Udemy問4：「Glue ETL + SageMaker Processing（HuggingFaceProcessor）」を選んでしまった

**選んだ理由（誤った思考）**：「データパイプライン＝ETL系サービス（Glue）を使うのが定石」という連想で選んだ。

## Udemy問7：「API Gateway HTTP API + Lambda」を選び誤答 → 正答「API Gateway REST API + Lambda」

**なぜ誤答か**：既存の理解では「API Gatewayはストリーミング非対応、ストリーミングが要件なら常にLambda Function URL」という判断ルールを持っていた。この問題ではHTTP APIを避けた選択肢が複数あったが、REST API側の選択肢を「REST APIもストリーミング不可のはず」と誤って除外した。

**公式ドキュメントで確認した事実（2025-11〜の仕様変更）**：
- **API Gateway REST APIは`response transfer mode`を`STREAM`に設定することでネイティブにレスポンスストリーミング対応**（Lambda proxy統合＝`AWS_PROXY`、または`HTTP_PROXY`統合が対象）

- REST APIのストリーミングは最大15分、10MBペイロード制限・29秒タイムアウトも回避可能。ただしキャッシュ・コンテンツエンコーディング・VTLレスポンス変換とは併用不可

**正しい判断ルール**：「ストリーミング＝Lambda Function URLしかない」という理解は2025-11時点で古くなった。**REST API + Lambda proxy統合（STREAM転送モード）も正解になり得る**。HTTP APIのみ引き続き非対応。ECS/コンテナ化推論サーバーは「Bedrockがマネージドで推論基盤を提供している時点で不要」という理由で運用オーバーヘッド最小の問題では基本的に不正解。

## Udemy問4：「インテリジェントプロンプトルーティング」を選んでしまった

- 問題：大学のAIアプリで、プロンプトのバージョン管理・バリアント比較・テスト・変更と使用状況の監査証跡・メインアプリへの一貫した統合を運用オーバーヘッド最小で実現したい（2つ選択）
- **誤答の原因（本人談）**：要件を見て一度は正しい方向（Prompt Management系）が浮かんでいたが、「インテリジェントプロンプトルーティングというサービスもあった気がする」と自信を失い、名前の響き（「プロンプト」）に釣られて選び直した
- **なぜ不正解か**：インテリジェントプロンプトルーティングは「プロンプト」という名前が付いているが、実際に振り分けている対象は**モデルへのリクエスト**（同一モデルファミリー内の異なるFM間で品質とコストに基づき動的ルーティング）。プロンプト自体のバージョン管理・バリアント比較・監査証跡・アプリ統合とは無関係
- **正答**：Amazon Bedrock Prompt Management（バージョン管理・バリアント比較・パラメータ変数・CloudTrail連携）＋ Amazon Bedrock Flows（プロンプトをメインアプリのワークフローに統合、バージョン/エイリアスでデプロイ管理）
- **判断ルール**：サービス名に含まれる単語だけで機能を連想しない。「ルーティング」なら**何を**振り分けるのか（プロンプトかリクエスト/モデルか）を必ず確認する。一度選んだ判断を、名前が紛らわしい別サービスの存在を思い出しただけで根拠なく差し替えない（[traps.md](./traps.md)の造語トラップとは別種の「実在するが対象を誤認するトラップ」）
- **判断ルール**：「カスタムオーケストレーションコード最小化」という要件を見たら、「宣言的サービス＝コード少ない」という直感を疑う。**エージェントの推論ループ自体を組み込みで提供するフレームワーク（Strands等）かどうか**で判断する。ECS/Lambda+SQSによる自前疎結合構成も同様に「配線ロジック全部自作」で不正解。詳細は [bedrock_agents.md](../topics/bedrock_agents.md) のStrands組み込みパターン（Workflow/Graph/Swarm）節参照

## 演習2 問14：技術＋ビジネスメトリクス統合監視で「QuickSight＋EventBridge」を選んでしまった

- 問題：FMの技術メトリクス（トークン/レイテンシ/エラー率）とビジネスメトリクス（コンバージョン率/収益影響）を相関させ、劣化時にステークホルダーへ自動アラート＋包括的可視性
- **誤答の原因（本人談）**：「BIデータ＝QuickSight」と反射的に結びつけ、CloudWatchメトリクスをS3へストリーム→QuickSightで可視化→EventBridgeで通知、という選択肢を選んだ
- **なぜ不正解か（2点）**：
  1. **QuickSightはBIダッシュボードでリアルタイム運用監視・アラートには不向き**。S3経由の可視化はリアルタイム性に欠ける
  2. **EventBridgeにはCloudWatchメトリクスの変動を直接監視して閾値超過でアラームを出す機能がない**。EventBridgeは「イベント（状態変化）が来たらルーティングする」サービスであり、メトリクスの数値を自分で監視する仕組みを持たない。メトリクスのしきい値・変動監視でアラームを出すのは**CloudWatchアラームの役割**
- **正答**：CloudWatchカスタムダッシュボード（ビジネスメトリクスは`PutMetricData`でインポートし技術メトリクスと同一ビュー）＋**異常検出（Anomaly Detection）付き複合アラーム**＋SNS通知。CloudWatchネイティブで追加サービス不要
- **判断ルール**：
  - 「メトリクスの**劣化を動的に検出**（トラフィック変動で正常範囲が変わる）」→ 静的しきい値ではなく**CloudWatch異常検出**（MLでベースライン学習）
  - 「複数メトリクスの**相関**（技術×ビジネスが同時に悪化した時だけ）」→ **複合アラーム**（AND/OR/NOT）
  - **QuickSight/Managed Grafana＝可視化（BI）ツールであって運用監視・アラート基盤ではない**。可視化と監視・アラートを混同しない
  - Managed Grafana＋Lambda「自動修復」は要件（ステークホルダーへの通知）とズレている（自動修復≠通知）

## 演習2 問18：トークン使用量の可観測性で再び「QuickSight」を選んでしまった（問14と同型の再発）

- 問題：複数Bedrockアプリのトークン使用パターンをリアルタイム可視化＋複数ステークホルダー向けカスタムダッシュボード＋全FMのトークン消費アラート、を運用オーバーヘッド最小で（2つ選択）
- **誤答の原因**：問14と同じ「BI/可視化＝QuickSight」の反射。加えて「zero-ETL統合＋Managed Grafana」という具体的っぽい選択肢にも注意が必要（**BedrockとManaged Grafanaの間にネイティブなzero-ETL統合は存在しない**）
- **なぜ不正解か**：**QuickSightはCloudWatchメトリクスを直接データソースにできない**（S3/Athena経由のETLパイプラインが別途必要＝運用オーバーヘッド増）
- **正答（2つ）**：①**CloudWatchカスタムダッシュボード（ネイティブのトークン/呼び出しメトリクス）＋CloudWatchアラーム** ②**CloudWatch Logs Insights**（Bedrockモデル呼び出しログに対するアドホッククエリでアプリ別/高使用量パターンを分析）→ **クエリ結果はログウィジェットとしてダッシュボードに追加できる**（メトリクスウィジェットと同じダッシュボードに混在可能）
- **判断ルール**：「BI＝QuickSight」は**QuickSightがCloudWatchを直接読めない**という技術的制約を忘れた早合点。CloudWatch関連の可観測性問題では、**メトリクス系はCloudWatchダッシュボード+アラーム、ログ系（呼び出し内容の詳細分析）はLogs Insights+ログウィジェット**の2本柱で運用オーバーヘッド最小を実現できることが多い。EventBridge+Firehose+OpenSearchのような多段パイプラインも同様に「具体的だが自作コンポーネント過多」の罠

## 演習2 問29：「プロンプトルーター」と「プロンプト管理」を混同して誤答

- 問題：4つの必須入力変数＋一貫したテキスト出力形式＋いじめ表現の検出(ブロックしない)、Bedrock Flowsで実現（2つ選択）
- **誤答の原因**：正答の一つ（ガードレール:侮辱カテゴリ+検出モード）は選べたが、もう一つを「プロンプトルーター」で埋めてしまい、正しくは「プロンプト管理（Prompt Management）」だった
- **なぜ混同したか**：名前が似ている上、既存メモ（ai_governance.md）に**「プロンプトルーター＝プロンプトテンプレート管理」という誤記**があり、この誤解を助長していた（今回の問題文の解説で誤りと判明し訂正済み）
- **正しい役割分担**：
  - **プロンプトルーター（Prompt Router）＝インテリジェントプロンプトルーティングの実体**。複数モデル間でコスト/性能バランスを見てルーティングする機能。**変数定義・出力形式制御の機能はない**
  - **プロンプト管理（Prompt Management）＝プロンプト内に変数（プレースホルダー）を定義し、モデル選択・出力形式指定ができる機能**。フローのプロンプトノードに統合可能
- **判断ルール**：「入力変数を定義したい」「出力形式を指定したい」→ **Prompt Management**。「モデル間の振り分け・コスト最適化」→ **Prompt Router（Intelligent Prompt Routing）**。名前の類似（どちらも"プロンプト"）で機能を取り違えない
- **ガードレールの侮辱(Insults)カテゴリ**：いじめ・見下し・嘲笑等はHateではなくInsultsに分類（Hateはアイデンティティに基づく差別）。「通知はするがブロックしない」→ フィルターアクションは**検出（Detect/NONE）**、BLOCKにすると遮断されてしまい要件（ブロックしない）に反する

## 演習2 問35：「API Gateway HTTP APIはストリーミング非対応」を理由に正答を外して誤答

- 問題：Lambda上のMCPサーバーをリモートから認可済みユーザーのみアクセス可能にして公開したい
- **誤答の原因**：「ストリーミングが要件に絡む→HTTP APIは非対応(既存ルール)」と機械的に適用し、API Gateway HTTP API+Streamable HTTPの正答選択肢を除外した
- **なぜ不正解か**：既存ルール（REST APIのみSTREAM転送モード対応・HTTP APIは非対応）自体は正しいが、**MCPのStreamable HTTPトランスポートは「ストリーミングが任意・必須ではない」仕様**であり、HTTP APIの標準的なバッファリング（Lambda完了後にまとめて返す）でもMCP通信としては成立してしまう
- **正しい判断ルール**：「ストリーミング＝HTTP API不可」は**API Gatewayのネイティブなレスポンスストリーミング機能の話**であって、**MCPプロトコルレベルでのSSE表現**とは別レイヤー。MCP文脈でHTTP APIが選択肢にあっても即除外しない。真に低TTFB・逐次配信が要件なら別途REST API STREAMモード/Lambda Function URLが必要 → [bedrock_agents.md](../topics/bedrock_agents.md)
- 副産物：STDIOトランスポート（同一マシン上の子プロセス通信、stdin/stdoutパイプ、リモート通信不可）を初見から新規理解

## 演習2 問37：OpenSearchの検索レイテンシ問題（完全初見・不正解）

- 問題：大量ドキュメント一括投入後に検索レイテンシが悪化。CPU/JVMヒープは正常。インデックス削除・再作成せず低負荷で根本改善したい
- **誤答の原因**：OpenSearch管理ドメインのデータノード/シャード/セグメント/ISMという内部構造が完全初見で、選択肢の技術的な妥当性を判断できなかった
- **正答**：シャード数をデータノード数の倍数に均等化＋セグメント統合(force_merge)をオフピーク実行＋ISMポリシーで定期スケジュール化
- **なぜ他の選択肢が不正解か**：①インスタンスのスケールアップ＝リソース不足が否定されている前提と矛盾 ②全件再インデックス＝運用負荷が大きく「削除・再作成しない」制約に反する ③OpenSearch Serverlessへの移行＝データ全件移行＋IAM再設計を伴う過大な手段
- **新規理解した用語**：データノード（サーバー実体）／シャード（インデックスの分割区画、後から変更不可）／セグメント（書き込みのたびに増える不変ファイル、k-NNはセグメントごとにHNSWグラフを構築するため断片化が検索コスト増に直結）／ISM（インデックスライフサイクル自動化機能）
- 詳細は [bedrock_rag.md](../topics/bedrock_rag.md) の「クラスターの構造」節参照
