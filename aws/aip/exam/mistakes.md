# 模試ミス回収

模試で間違えた問題を「誤った思考 → なぜ間違いか → 正しい判断軸」の形式で記録する。
新しいミスは試験番号順に追記し、導出した判断ルールを [cheatsheet.md](./cheatsheet.md) にも1行転記する。

## ⚠️ 模試ミス回収（AIP-23）：「サブモジュール + @workspace」を選んでしまった

**選んだ理由（誤った思考）**：
「社内資産を別リポジトリとしてサブモジュールでローカルに置けば、プロジェクトリポジトリ自体は変えずに Q Developer が参照できるのでは？」

**なぜ間違いか：2 つの見落とし**

**① サブモジュールの追加 = リポジトリ構成の変更**

```
サブモジュールを追加すると、プロジェクトリポジトリに変更が入る
  ├─ .gitmodules ファイルが新規作成（コミットが必要）
  └─ サブモジュールのマウントポイントディレクトリが追加される
→ 「リポジトリ構成を変更しない」要件に反する
```

「別リポジトリのコードをローカルに置く」行為が、プロジェクトリポジトリへの変更を伴う点を見落とした。

**② @workspace はインライン補完には効かない・毎回手動指定が必要**

```
@workspace が効く場面
  → チャットで「@workspace」と書いたときのみ（手動）

@workspace が効かない場面
  → コード入力中のインライン補完（リアルタイムのサジェスト）
  → 開発者が @workspace を書き忘れたとき
```

→ 組織全体への一貫した適用が保証されず、「全チームへの強制反映」にならない。

**正しい判断軸**

| | サブモジュール + @workspace | カスタマイズ機能 |
|---|---|---|
| リポジトリ変更 | **.gitmodules が追加される** | **なし** |
| インライン補完への反映 | 効かない | **常時反映** |
| 適用の一貫性 | 開発者が毎回手動指定 | 管理者が割り当てれば全員に自動適用 |

**覚え方**：「サービス側で設定 = プロジェクトへの変更なし」。リポジトリ変更ゼロを実現できるのはサービス設定で完結するカスタマイズ機能だけ。

## ⚠️ 模試ミス回収（AIP-25）：「API Gateway HTTP API + Cognito」を選んでしまった

**選んだ理由（誤った思考）**：
「OAuth 2.0 でアクセストークンを発行すれば強固な認証になる。Cognito はエンタープライズでよく使われる認証サービスだから適切では？」

**なぜ間違いか：2 つの見落とし**

**① API Gateway + Cognito = 追加の管理コンポーネント 2 つ**

```
Lambda Function URL（正解）
  → Lambda のみ（管理コンポーネント最小）

API Gateway + Cognito（不正解）
  → API Gateway の設定・監視・バージョン管理
  + Cognito ユーザープールの管理・アプリクライアント発行・トークン有効期限管理
  → 問題文「管理すべきインフラコンポーネントを最小限に」に反する
```

**② API Gateway HTTP API はレスポンスストリーミングに非対応**

```
MCP の要件：「ストリーマブル HTTP トランスポート」= レスポンスを逐次返す
API Gateway HTTP API    → ストリーミング非対応（全レスポンスを一括返却）
Lambda Function URL     → RESPONSE_STREAM モードでネイティブ対応
```

## ⚠️ 模試ミス回収（AIP-27）：「KB + SQS + Cognito」を選んでしまった

**選んだ理由（誤った思考）**：
「ビルごとに KB を分けて、SQS でリアルタイムに更新し、Cognito でアクセス制御すれば全要件を満たせるのでは？」

**なぜ間違いか：3 つの見落とし**

**① SQS + KB 更新はリアルタイムではない**

```
SQS にメッセージが届く
  ↓ Lambda が受信（ポーリング）
  ↓ KB のインジェストジョブを実行（チャンキング・埋め込み・格納）
  ↓ 数分〜数十分後に反映

「空室状況」のようにリアルタイム性が必要なデータには不適切。
正解：エージェントのアクショングループで BMS API を直接呼び出す。
```

**② Cognito は「AWS リソースへの組織的アクセス管理」には向いていない**

```
Cognito：Web/モバイルアプリのユーザー認証が主目的。
IAM Identity Center：組織の SSO・AWS リソースへの権限管理が主目的。
→「管理スタッフが自分のビルの KB だけにアクセス」= IAM Identity Center の許可セットが正解
```

**③ KB の上限（約 10 個）を意識する**

```
ビルが 10 棟を超えると KB のクォータに達する可能性がある。
エージェントが複数 KB をオーケストレーションする構成なら、
クォータ管理やメタデータフィルタとの組み合わせで柔軟に対応できる。
```

**正しい判断軸**

| 判断ポイント | 正解の考え方 |
|---|---|
| リアルタイムデータ | KB 同期ではなく、エージェントのアクショングループで API を直接呼ぶ |
| 組織の AWS リソース制御 | Cognito ではなく IAM Identity Center |
| データ分離 | ビルごとに KB を作成（ただし上限に注意）|

## Domain 2 Practice 問題2.3：C の脱落理由を誤認した

**誤った思考**：「Secrets Manager に OAuth 認証情報を保存することが最小権限原則に反するのでは？」と判断して C を除外した。

**なぜ間違いか**：Secrets Manager への OAuth 認証情報（client_id / client_secret 等）の保存は正しいベストプラクティス。C が不正解な本当の理由は「**Lambda プロキシ統合を使用して Bedrock エージェントを非同期に呼び出す**」こと。

- 会話型・低レイテンシー要件 → 同期呼び出しが必要
- 非同期呼び出し → レスポンスをリアルタイムに返せない

**正しい判断軸**：OAuth 認証情報の保存方法ではなく、「**同期か非同期か**」を確認する。要件に「低レイテンシー」「リアルタイム」があれば非同期呼び出しは即除外。

## Domain 2 Practice ボーナス問題4：Step Functions + StartAsyncInvoke を選んだ

**誤った思考**：「リアルタイムとバッチの2系統をStep Functions の choice ステートで振り分ければ良い。バッチ側は StartAsyncInvoke で非同期呼び出しすれば良い」と判断した。

**なぜ間違いか：2つの問題**

**① StartAsyncInvoke は Nova Reel（動画生成）専用**

```
テキスト系モデルのバッチ推論 → CreateModelInvocationJob
動画生成（Nova Reel）→ StartAsyncInvoke
```

**② Step Functions Standard は同期実行 → 呼び出し元がバッチ完了まで待機**

```
Express ワークフロー → 最大5分（バッチ処理時間を超える可能性）
Standard ワークフロー → 同期実行（呼び出し元がブロックされる）
→ どちらもリアルタイム+バッチ 2系統には不向き
```

**正しい判断軸**：応答時間が異なる2要件は **別アーキテクチャで捌く**。
`API GW → Lambda → InvokeModel`（リアルタイム）+ `SQS → Lambda → CreateModelInvocationJob`（バッチ）

## ⚠️ 模試ミス回収（Task 3.2）：「包括的セキュリティ」でS3暗号化を選んでしまった

**選んだ理由（誤った思考）**：「データ保護 → 暗号化」と直結させて CloudWatch + IAM + S3暗号化 を選んだ。

**なぜ間違いか**：Bedrock はデフォルトで転送中・保管中の暗号化が適用済みのため、S3暗号化を追加設定する話にならない。問題文の「データ保護」は監査ログ（CloudTrail）によるアクセス追跡の文脈で使われていた。

**正しい判断軸**：「Bedrock の包括的セキュリティ（モニタリング・アクセスコントロール・データ保護）」→ IAM + CloudTrail + CloudWatch。CloudTrail が "データ保護（監査）" を担う。暗号化はデフォルト済みなので選択肢に出てきたら罠。

## ⚠️ 模試ミス回収（Task 3.2 ナレッジチェック01）：「包括的セキュリティ2択」でLake Formationを外した

**選んだ理由（誤った思考）**：「セキュリティグループ + IAM ロール」でネットワーク・ID制御は揃っていると判断して選んだ。

**なぜ間違いか**：「包括的」には2軸が必要。①ネットワーク分離×アクセスコントロール②モニタリング×データガバナンス。セキュリティグループ+IAMロールはデータレベルの制御（Lake Formation）とモニタリングが欠落。また「保護された AI 環境」でパブリックサブネットは即アウト。

**正しい判断軸**：「機密データを扱う保護された AI 環境」で「包括的」→ Lake Formation が必ず入る。モニタリング軸には CloudWatch。パブリックサブネットは即除外。2択問題は「何の軸が揃っているか」を確認する。

## ⚠️ 模試ミス回収（Task 3.3 ナレッジチェック01）：「包括的コンプライアンス2択」でModel Cardsを外した

**選んだ理由（誤った思考）**：Glue によるメタデータタグ付けのみ選択。Model Cards はデータリネージュ追跡ではないと判断して除外した。

**なぜ間違いか**：「包括的コンプライアンス」には「モデルの文書化」と「データ追跡」の2軸が必要。Model Cards はリネージュ追跡ではなく「文書化」を担う。文書化＋データ追跡のセットで包括的コンプライアンスが成立する。

**正しい判断軸**：「包括的コンプライアンス2択」= Model Cards（モデル文書化）+ Glue（データ追跡・メタデータ）をセットで選ぶ。

### AIP-31 の正解構成と不正解パターン

**問題**：AppSync + Lambda（同期）で Bedrock を呼び出すと「表示まで長時間待つ」「複雑な質問でタイムアウト」が発生

**根本原因**：同期一括返却（全文生成完了まで何も返さない）

**正解**：Amplify AI Kit + AppSync Subscription → ストリーミングで逐次描画

```
[Before] 同期方式
  ユーザー → Lambda → Bedrock（全文生成） → 一括返却 → 画面表示
  問題：数十秒間 画面が空のまま / タイムアウト

[After] ストリーミング方式
  ユーザー → AppSync Subscription（WebSocket）
          → Bedrock がトークン生成するたびにチャンクをプッシュ
          → フロントエンドで逐次描画
  効果：TTFT（最初の文字表示まで）が大幅短縮 / タイムアウト回避
```

| 不正解選択肢 | 理由 |
|---|---|
| Lambda のメモリ増強 + プロビジョンドコンカレンシー | Bedrock 側の生成時間は変わらない。同期構造のまま |
| API Gateway REST + Step Functions + ポーリング | アーキテクチャを全面刷新。ポーリングは体感速度改善が限定的 |

## ⚠️ 模試ミス回収（Domain 3 Practice 3.3）：属性バイアス評価で SageMaker Clarify を選んでしまった

**選んだ理由（誤った思考）**：
「バイアス検知 = SageMaker Clarify」の刷り込み。CloudWatch でバイアスを検知できるはずがないと思い A を除外した。しかし A の表記は「CloudWatch **カスタム**メトリクス」であり、バイアス数値を自前でプッシュして監視する構成だった。

**なぜ間違いか：**

- **B（選んだ）の問題点①**：RealToxicityPrompts は**有害性（Toxicity）**評価用データセット。**属性バイアス（性別・人種・職業・年齢）**の評価用ではない。データセットの目的が要件とズレている

**正しい判断軸（正解 A の構成）：**

```
BOLD データセット          → 属性バイアス（性別・人種・職業・年齢）専用
Bedrock モデル評価ジョブ   → セカンダリモデル検証をネイティブサポート
Bedrock Prompt Management → 制御されたプロンプトバリアントテスト
CloudWatch カスタムメトリクス → バイアス数値を継続的にモニタリング＋アラート
```

**データセットの目的対応表（試験頻出）：**

| データセット | 目的 | 使うべき場面 |
|---|---|---|
| **BOLD** | 属性バイアス（性別・人種・職業・年齢）| LLM の公平性評価 |
| **RealToxicityPrompts** | 有害コンテンツ（Toxicity）| 有害性フィルタの評価 |
| **WikiText2** | 一般テキスト生成精度 | 言語モデルのPerplexity計測 |
| **T-REx** | 事実的知識（Factual Knowledge） | 知識グラフ・事実QA評価 |
| ElastiCache でキャッシュ | 自然言語クエリは完全一致ほぼなし。キャッシュミス時は同期問題が残る |

## ⚠️ 模試ミス回収（Domain 3 Practice 3.5）：ガバナンス要件で CloudTrail + CloudWatch を選んでしまった

**選んだ理由（誤った思考）**：
「ガバナンス = 監査できればOK」と考え、CloudTrail（API操作ログ）+ CloudWatch（アラーム）で十分と判断した。ガードレールを「コンプライアンス違反の検出だけならいらない」と誤解した。

**なぜ間違いか：3つの見落とし**

① **CloudTrail ≠ プロンプト履歴**
```
CloudTrail     = 誰がいつどのAPIを呼んだか（APIメタデータのみ）
Model Invocation Logging = プロンプト内容・完了内容・モデルバージョンの詳細を記録
→ 「プロンプト履歴を記録する」要件 = Model Invocation Logging が必須
```

② **CloudWatch アラーム ≠ コンプライアンス違反のブロック**
```
CloudWatch アラーム = 違反を「検出・通知」するだけ
Bedrock ガードレール = 違反コンテンツを「フィルタリング・ブロック」する
→ 「コンプライアンス違反をモニタリング」だけでなく「防ぐ」要件 = ガードレールが必要
```

③ **「開発作業を最小限」= Bedrock ネイティブ機能を使う**
```
A（誤）: CloudTrail + CloudWatch → プロンプト内容追跡に追加実装が必要
C（正）: Guardrails + Model Invocation Logging + CloudTrail → GenAI ガバナンス専用機能でカバー
```

**正しい判断軸（正解 C の構成）：**

| 要件 | 使うサービス |
|---|---|
| プロンプト・完了・モデルバージョンの履歴記録 | **Model Invocation Logging**（→ CloudWatch Logs に送信） |
| コンプライアンス違反のフィルタリング | **Bedrock ガードレール** |
| アクセス追跡・監査ログ | **CloudTrail** |

## ⚠️ 模試ミス回収（Domain 3 Practice 3.6）：S3のPII検出でバッチ処理（C）を選んでしまった

**選んだ理由（誤った思考）**：
「S3に保存済みのデータ = バッチ処理で十分、リアルタイムAPIはコスト高」と思い込んでBを除外。Cに「修復アクション」の記載があったため選択した。

**なぜ間違いか：**

① **Macie には「機密データの自動検出（継続的監視）」機能がある**
```
Macie 自動検出（B）  = S3バケットを常時監視。新データを継続的に検出
Macie 検出ジョブ（C）= 週次・月次スケジュール実行 → 検出に遅れが生じる
→「自動修復アクション」要件には継続的監視が必要
```

② **C の運用オーバーヘッドが高い**
```
C: Comprehend（月次）+ Macie（週次）の結果を Lambda で比較・統合 → カスタム実装が必要
B: Macie が検出 → EventBridge が拾う → Lambda が修復 → シンプルな連携
→「運用オーバーヘッド最小」要件では B が優位
```

③ **「S3のデータ = バッチでいい」は試験では通じない**
- 要件に「自動修復」「継続的コンプライアンス」があれば、継続的監視（Macie自動検出）が正解
- リアルタイム vs バッチは「コスト」ではなく「要件（遅延許容度）」で判断する

**正しい判断軸（正解 B の構成）：**

| 要件 | 使うサービス |
|---|---|
| テキスト内のPII検出（名前・電話・クレカ） | **Amazon Comprehend** リアルタイム PII 検出 API |
| S3全体の機密データ分類・継続監視 | **Amazon Macie**（自動検出を有効化） |
| PII検出時の自動修復トリガー | **EventBridge** ルール → **Lambda** でS3バケットポリシー更新 |

## ⚠️ 模試ミス回収（Domain 4 Q05/05）：結果フィンガープリンティングの主な利点でCを選んでしまった

① **結果フィンガープリンティングはHIT判定をしない**

```
  → 「別入力から同じ回答が生成された場合にDB効率化」が主目的
```

② **ハッシュ検索の高速性はすべてのキャッシュ手法に共通**
- レイテンシー低減は「フィンガープリンティング固有」の利点ではなく副次的効果

③ **正解Aの読み方**
```
「入力の完全一致ではなく、入力と出力のパターンに基づいてキャッシュできる」
= 入力側（セマンティックキャッシュ）+ 出力側（フィンガープリント）の
  両方でキャッシュを管理できる、というシステム全体の説明
```

**正しい判断軸（フィンガープリンティングの役割）：**

| 用途 | 担当 |
|---|---|
| キャッシュHIT判定（入力類似） | セマンティックキャッシュ（embedding） |
| キャッシュHIT判定（入力完全一致） | 決定論的ハッシュ |

| キャッシュ鮮度管理 | TTL / イベントベース無効化 |

- **判断ルール**：「ジョブ**開始**（作成時点）で失敗」は多くの場合、実行前の**静的な設定検証**（形式・クォータ・権限）に引っかかっている。直前に学んだ別の失敗パターン（CORS等）を反射的に当てはめず、エラーメッセージが指す対象（件数か・形式か・権限か）を先に切り分ける

## 診療所ごとのコスト追跡（S3+Bedrock要約、2つ選択）：B,E → 正答C,E

- 問題：複数診療所のデータをS3に診療所IDプレフィックスで保存。Step Functions(Transcribe Medical→Bedrock要約)。診療所ごとの要約コストを最もコスト効率よく追跡したい。
- **誤答B（S3 Storage Lens+Athena）のダメな点**：Storage Lensはオブジェクト数・サイズ・リクエストアクティビティなど**ストレージの使用状況メトリクスのみ**。Bedrock/Lambdaの推論コストは記録も紐付けもされない。「診療所ごとのコスト追跡」に対してストレージ分析ツールを持ち出す網羅感トラップ気味の誤答
- **正答C**：診療所IDごとにBedrock**アプリケーション推論プロファイル**を作成し、要約用Lambdaが該当プロファイルを選択して呼び出す。Bedrock呼び出しがプロファイル単位でタグ付けされ、Cost Explorer側で診療所別コストを分解できる。追加インフラ不要で設定ロジックも最小限
- **正答E**：EventBridgeルールでS3のPutObjectイベントを直接キャプチャし、Step Functionsをターゲットに指定（Lambda中継を排除して呼び出し課金を削減）
- **新しい理解**：EventBridgeのCloudWatchメトリクス（Invocations/FailedInvocations等）は**イベント配信の成否・レイテンシーのみ**を計測する。ターゲット（Step Functions）が起動した後にBedrockを何回呼んだか・トークン数・コストは一切追跡しない——EventBridgeはターゲット呼び出し後の内部処理を知らない
- **判断ルール**：「複数テナント/顧客/拠点ごとのFM利用コストを追跡」→ **アプリケーション推論プロファイル**（テナントIDごとに作成、追加インフラなし）。EventBridge/CloudWatchメトリクスやS3系の分析ツール（Storage Lens等）はイベント配信・ストレージの話であってBedrockのコスト追跡はできない、と即座に区別する

## 埋め込みドリフト検出＋無停止で再学習準備（RAGパイプライン、2つ選択）：B,D → 正答C,D

- 問題：ソーシャルメディア企業がSageMaker AIリアルタイムエンドポイントでFM埋め込みを生成し複数RAGパイプラインで使用。ユーザーから応答の関連性低下の報告、新しい文体が増加。入力ドリフトが埋め込み品質を低下させているか判断したい。**本番トラフィックを中断せず**再学習準備が必要。
- **誤答B（Clarify+Model Monitor+バイアス可視化）のダメな点**：「ドリフト検出＝Clarify」と早合点した。ClarifyがModel Monitorと組み合わせて検出できるのは**バイアスドリフト**と**Feature Attributionドリフト**（表形式データの特徴量重要度の変化）のみで、テキストの埋め込みベクトルの意味的ドリフトには対応しない
- **外したC（正答）**：カスタムAmazon ECRイメージでModel Monitorの監視スケジュールを自作し、キャプチャした推論データ（埋め込み）とベースライントレーニングデータセットの間のドリフトメトリクスを独自計算。「なぜドリフト定義をカスタムでやる必要があるのか」→ Model Monitor組み込みの4種（データ品質・モデル品質・バイアスドリフト・Feature Attributionドリフト）はいずれも表形式の数値統計が前提で、埋め込みのような高次元・自由形式データの意味的変化を計算する仕組みが組み込みには無いため。公式ブログ（NLP data drift検出）ではBERTで埋め込み生成→コサイン類似度でベースラインとの距離を計算→閾値割れで違反フラグ、という実装パターンが紹介されている
- **正答D**：本番バリアントに基づき**新しいエンドポイント設定**を作成しデータキャプチャを有効化、`UpdateEndpoint` APIで新設定にトラフィックを移行（旧設定は自動退役）。エンドポイントを削除して作り直す（選択肢A）とその間トラフィックが途切れるが、UpdateEndpointはSageMaker側が裏で新インスタンスを立ててから切り替えるため無停止
- **判断ルール**：①「SageMakerで無停止のエンドポイント設定変更」→ 新エンドポイント設定+`UpdateEndpoint` API（エンドポイント自体の削除・再作成は不正解） ②「テキスト/埋め込みのドリフト検出」→ Model Monitor標準4種にもClarifyにも当てはまらない場合は**カスタムECRイメージでのModel Monitor自作**を検討

## Step Functionsで一貫した多段階推論を実装（Bedrock FM、推論パターン問題）：B → 正答A

- 問題：企業がBedrock FMをAPI経由で呼び出し、AWS Step Functionsで構造化された方法でFMが各問題を一連の論理推論ステップに分解できるようにしたい。設問末尾「この推論パターンを**最も一貫性のある方法**で実装できるソリューションはどれか」
- **誤答B（思考連鎖＋Choice状態で中間応答に基づき推論経路を動的調整）のダメな点**：「状況に応じて経路を変えられる＝柔軟＝良さそう」という直感で選んだが、これは軸違い。設問の軸は「一貫性」であって「柔軟性」ではない。動的分岐は問題ごとに異なる経路をたどるため、複数ドメイン・シナリオに適用できる予測可能で構造化された分解フレームワークにならない
- **正答A**：ReActパターンの状態を持つステートマシンを作成し、観察・推論・アクション計画それぞれの状態に**固定の特定プロンプト**を設定、組み込みのStep Functionsエラー処理・再試行ロジックでモデル呼び出しを行う。プロンプトが固定（＝分岐しない）であることで、どんな問題を渡しても同じ構造で体系的に分解される＝一貫性が担保される
- **判断ルール**：設問末尾の軸が「一貫性」「予測可能性」「標準化」の場合、選択肢の「状況に応じて動的に調整」「Choice状態で分岐」という記述は柔軟性の魅力に引っ張られて選ばない——固定ステート＋固定プロンプトの方が一貫性軸では強い。逆に軸が「複雑度に応じた最適振り分け」なら動的分岐が正解になる逆転パターンもあるため、まず設問末尾の軸を確定させてから選択肢を裁く

## メディア企業のレガシーCMS統合（複数AIエージェント、標準化＋再利用）：A → 正答D

- 問題：メディア企業がコンテンツ準備を自動化する複数のAIエージェントを構築中。レガシーCMSのAPIが個々に不整合（エンドポイントごとに挙動が異なる）で、各エージェントが個別に対処している。複数のエージェントワークフローで再利用でき、運用負荷を最小限にする標準化されたインターフェースが欲しい。
- **誤答A（REST APIの変換・プロキシ層を構築しドキュメント化・標準化）のダメな点**：レスポンスの見た目は整うが、**関数呼び出しという意味的（semantic）インターフェースを提供しない**。結局各AIエージェントは整形されたREST APIに対して個別の統合コードを書く必要が残り、「複数エージェントで再利用できる仕組み」が存在しない
- **正答D**：CMS操作用の関数スキーマを持つ**MCPサーバー**を実装し、API個々の不整合はMCPサーバー内部で吸収。**Bedrock AgentCore**をMCPクライアント（エージェント実行環境）として構成し、MCP関数呼び出し経由でCMSとやり取りさせる。AgentCoreを使う全エージェントワークフローが同じMCP関数群をそのまま再利用できる
- **選択肢B（サイドカーコンテナとしてカスタムミドルウェアを各エージェントに配置）が不正解な理由**：ECSサイドカーはメインコンテナと同一タスクで並走する補助コンテナだが、エージェントの数だけミドルウェアを重複デプロイ・保守することになりスケールしない。標準化された関数呼び出しI/Fを1箇所に集約する発想がない
- **判断ルール**：「レガシーAPIの不整合を吸収しつつ複数エージェントで再利用・運用負荷最小」という軸が出たら → **MCPサーバー（関数呼び出しスキーマ）＋AgentCore**一択。「REST/OpenAPIをきれいにする・ドキュメント化する」系の選択肢は、関数呼び出しの抽象化＝AIネイティブな再利用機構を提供しない限り不正解。「サイドカー」「エージェントごとに個別配置」という記述はスケーラビリティ面のトラップ
