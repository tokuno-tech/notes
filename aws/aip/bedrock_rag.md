# Bedrock RAG / ベクトルストア選定

## RAG構成の全体像

```
ドキュメント（S3 等）
  ↓ チャンク分割・埋め込み生成
ベクトルストア（OpenSearch / Aurora pgvector / Pinecone 等）
  ↓ クエリ → 類似検索（k-NN/ANN）+ メタデータフィルタ
取得した関連チャンクをコンテキストとしてプロンプトに挿入
  ↓
Bedrock FM（Claude など）→ 回答生成
```

Bedrock Knowledge Bases はこの一連を**マネージドで自動化**してくれる仕組み。

---

## インテリジェント検索とは（基本概念）

### 従来の検索 vs インテリジェント検索

| | 従来の検索（キーワード検索） | インテリジェント検索 |
|---|---|---|
| **仕組み** | 入力した単語と一致する文書を探す | 質問の**意図・意味**を理解して答えを探す |
| **質問例** | "有給 申請 方法" | "有給を取るにはどうすればいいですか？" |
| **返すもの** | 関連文書のリスト | **直接的な回答**（該当箇所を抜粋） |
| **同義語** | "有給"と"年次休暇"は別扱い | 同じ意味として認識 |
| **AWSサービス例** | ElasticSearch（旧）, BM25 | **Amazon Kendra** |

```
例：「育児休業は何日取れますか？」という質問に対して

キーワード検索 → "育児休業" "何日" にマッチする文書を列挙（自分で読んで探す）

インテリジェント検索（Kendra）→ 社内規則ドキュメントから
                               「育児休業は最大〇〇日取得可能です」という
                               答えの文章を直接抜き出して提示
```

- **Amazon Kendra** = AWSにおけるインテリジェント検索の代表サービス
- MLベースのセマンティック＋キーワードのハイブリッド検索（純粋なキーワード検索ではない）
- 自然言語の質問に対して**直接回答を抜粋して返す**のが特徴

## Kendra vs OpenSearch（RAGでの使い分け）

両方ともBedrockに接続できるが、**設計思想と適用シーンが根本的に異なる**。

### サービスの本質

| | Amazon Kendra | Amazon OpenSearch（Service / Serverless） |
|---|---|---|
| 本来の用途 | **エンタープライズ検索**（社内ドキュメント検索エンジン） |  全文検索 + 分析 + ベクトル検索の汎用エンジン |
| 検索方式 | ML組み込みの**意味検索**（埋め込みは内部で自動生成） | 全文検索（BM25）+ **k-NN/ANN によるベクトル検索** |
| 埋め込みの扱い | **ユーザーが意識しなくてよい**（Kendraが裏で生成） | **ユーザーが埋め込みを生成して投入**する必要あり |
| データソースコネクタ | S3・SharePoint・Confluence・Salesforce 等の**公式コネクタ多数** | 基本はS3 + 自前のETL（Bedrock Knowledge Bases経由なら自動化） |
| インフラ管理 | フルマネージド（インデックス管理のみ） | OpenSearch Service：ノード管理あり / **Serverless：自動スケール** |

### Bedrock との接続方式

| | Kendra | OpenSearch Serverless |
|---|---|---|
| Bedrock Knowledge Bases のベクトルストアとして使えるか | **✕（公式サポートなし）** | **◯（公式サポートあり）** |
| Bedrock からの利用方法 | RetrieveAndGenerate API で Kendra をデータソース指定 | Knowledge Bases 経由で透過的に利用 |
| カスタム埋め込みモデルの選択 | **不可**（Kendra内蔵） | **可**（Titan・Cohere Multilingual 等を選択） |

### 適用シーンの判断軸

| シーン | 推奨 | 理由 |
|---|---|---|
| 社内文書を自然言語で検索したい（埋め込みを管理したくない） | **Kendra** | コネクタが多く、埋め込みを意識せず使える |
| 自前で生成した埋め込みを大量に管理したい | **OpenSearch** | カスタム埋め込みモデルに対応 |
| Bedrock Knowledge Bases で RAG パイプラインを構築したい | **OpenSearch Serverless** | Knowledge Bases の公式ベクトルストア |
| 多言語（日本語・英語・中国語等）を**単一空間**で扱う | **OpenSearch + Cohere Embed Multilingual** | 多言語埋め込みモデルを明示的に選べる |
| メタデータ（カテゴリ・日付・地域）で絞り込み + 類似検索 | **OpenSearch** | k-NN + keyword/date フィルタの同時実行が可能 |
| 数百万件以上のベクトルを低レイテンシで検索 | **OpenSearch（HNSW/IVF）** | ANN専用アルゴリズムで最適化済み |
| インフラ管理を完全に避けたい | **OpenSearch Serverless** | OCU課金で自動スケール、ノード管理不要 |

### AIPでの引っかけパターン

| 選択肢 | 不正解になる理由 |
|---|---|
| Kendra + カスタムドキュメントエンリッチメントで埋め込み生成 | Kendra は**カスタムベクトルを直接格納・検索する設計ではない**。Knowledge Bases のベクトルストアにもなれない |
| EC2上にElasticsearch + k-NNプラグイン | 機能的にはOpenSearchと同等だが、**ノード・シャード・パッチ等の管理が自前**。「インフラ管理を最小化」要件に反する |
| DynamoDB + Lambdaで類似度計算 | DynamoDB に ANN 機能なし。**全件スキャン**になりレイテンシ・コスト共に破綻 |
| Aurora pgvector + SQL ORDER BY | ベクトル検索自体は可能だが、**大規模ANNに最適化されていない**。低レイテンシ要件に劣る |

## OpenSearch Serverless ベクトルコレクション（AIP-19）

### 構成（正解パターン）

**Bedrock Knowledge Bases + OpenSearch Serverless（ベクトルコレクション）+ Bedrock FM（Claude）**

```
ドキュメント（S3）
  ↓ Bedrock Knowledge Bases が同期
チャンク分割 → 埋め込み生成（Titan / Cohere Multilingual）
  ↓
OpenSearch Serverless（ベクトル + メタデータ）
  ↓ k-NN（HNSW）+ メタデータフィルタ
取得チャンク → Claude にコンテキスト挿入
  ↓
回答生成
```

### 各要件との対応

| 要件 | 対応する仕組み |
|---|---|
| 約800万件のベクトル埋め込み | OpenSearch Serverless（HNSWでミリ秒検索） |
| 製品カテゴリ・作成年・対象地域でフィルタ | `keyword` / `date` 型メタデータフィールドで k-NN と**同時クエリ** |
| 日本語・英語・中国語の多言語対応 | **Cohere Embed Multilingual**で単一ベクトル空間に統合 |
| インフラ管理の最小化 | Serverless：ノード・シャード・スケーリング自動 |
| 低レイテンシ | HNSW + OCU自動スケール |
| RAGパイプライン構築 | Bedrock Knowledge Bases が同期〜回答生成まで一貫提供 |

### OpenSearch Serverless の料金モデル

- **OCU（OpenSearch Compute Units）単位**で課金
- 検索処理量に応じて自動スケール → 固定インスタンス料金なし
- アクセス変動の大きいチャットボット用途と相性がよい

## セマンティックキャッシュ（AIP-17 再掲・参照）

「表現は違うが同じ意図」の問い合わせを意味ベクトルで検出し、Bedrock呼び出しを削減するパターン。詳細は [bedrock_guardrails.md](./bedrock_guardrails.md) のセマンティックキャッシュセクション参照。

### 要点

- **完全一致系（ElastiCache のハッシュキー等）では意味の類似は検出不可**
- **キーフレーズ抽出（Comprehend）は構文レベル**で、意味レベルではない
- **OpenSearch k-NN + 埋め込み**が唯一意味ベースのキャッシュを実現できる組み合わせ

## ベクトルストア選定フローチャート

```
Bedrock Knowledge Bases を使いたい？
  ├─ Yes → 公式サポートされるストアから選択
  │         （OpenSearch Serverless / Aurora pgvector / Pinecone / Redis Enterprise 等）
  │         インフラ管理を避けたい → OpenSearch Serverless
  │
  └─ No  → 社内文書検索が主目的？
            ├─ Yes（埋め込みを意識したくない）→ Kendra（RetrieveAndGenerate）
            └─ No（カスタム埋め込みを大規模管理）→ OpenSearch Service / Serverless
```

## Knowledge Bases の仕組み：S3 のベクトル化はサービスに含まれるか？

**はい、含まれる。** S3 にドキュメントを置いて同期を実行するだけで、以下が全自動で動く。

```
S3 のドキュメント（PDF / Word / HTML / CSV など）
  ↓ ① 読み込み
  ↓ ② チャンク分割（固定サイズ / セマンティック / 階層的 から選択）
  ↓ ③ 埋め込みモデルで各チャンクをベクトル化
       （Titan Embeddings / Cohere Multilingual 等を選択）
  ↓ ④ ベクトルストアへ格納
       （OpenSearch Serverless / Aurora pgvector / Pinecone 等）
```

**ユーザーがやること**：S3 にドキュメントを置く、Knowledge Bases を設定する、更新時に同期を走らせる

**Knowledge Bases が自動でやること**：埋め込みモデルの呼び出し・チャンキング・ベクトルストアへの書き込み

→ 「Bedrock Knowledge Bases はマネージド RAG パイプライン」という一文が指しているのは、この一連の処理がフルマネージドである点。

## Knowledge Bases ソース帰属 + Converse API（AIP-22）

### 要件キーワードと対応機能

| 要件 | 対応機能 |
|---|---|
| 社内ドキュメントから関連情報を検索して回答に使用 | **Knowledge Bases for Amazon Bedrock**（RAG） |
| 回答の根拠となった参照元ドキュメントを明示 | **ソース帰属（Source Attribution）機能** |
| 結論までの判断過程を段階的に提示 | **Converse API** + Claude のチェーン推論 |
| 応答時間 3 秒以内 | Knowledge Bases + Converse API（軽量構成） |
| インフラ管理負担を最小化 | Knowledge Bases + Bedrock（フルマネージド） |

### ソース帰属（Source Attribution）とは

「**回答生成に使ったドキュメントの出典情報を自動付与する機能**」。論文の脚注に相当するものが API レスポンスに自動で入ってくるイメージ。

```
質問：「社員の有給休暇は何日ですか？」

通常の LLM 回答（Knowledge Bases なし）
  → 「有給休暇は 20 日です」
  ← どこに書いてあったか不明。ハルシネーションの可能性あり。

ソース帰属あり（Knowledge Bases）の回答
  → 「有給休暇は 20 日です」
     + 引用元：s3://my-bucket/HR-規定2024.pdf
     + 引用箇所テキスト：「正社員の年次有給休暇は 20 日とする（第 14 条）」
```

Knowledge Bases を RetrieveAndGenerate API または Converse API 経由で使うと、レスポンスに **引用情報（citations）が自動付与**される。

```json
{
  "output": "税率は 10% です。",
  "citations": [
    {
      "retrievedReferences": [
        {
          "content": { "text": "消費税率は 10% です。" },
          "location": { "s3Location": { "uri": "s3://bucket/doc.pdf" } }
        }
      ]
    }
  ]
}
```

- 参照元 S3 URI・ドキュメント名・引用チャンクのテキストがレスポンスに含まれる
- **別途記録するコードを書く必要はない**（引用情報はAPIレスポンスに含まれる）
- 引用できる根拠がある＝ハルシネーションのリスクが下がる（根拠なき生成が難しくなる）

### Converse API とは

複数のモデルを**統一インターフェース**で呼び出せる Bedrock の API。

```
InvokeModel API（旧来）
  Claude → { "anthropic_version": ..., "messages": [...] }
  Titan  → { "inputText": ..., "textGenerationConfig": {...} }
  Llama  → { "prompt": ..., "max_gen_len": ... }
  ← モデルを変えるたびにコードを書き直す必要がある

Converse API
  全モデル共通フォーマット → modelId を変えるだけで切り替え可能
```

**マルチターン対話の仕組み**：会話履歴（messages 配列）をそのまま次のリクエストに渡すことで文脈を維持する。

```python
messages = [
    {"role": "user",      "content": [{"text": "AWS とは？"}]},
    {"role": "assistant", "content": [{"text": "AWS は..."}]},
    {"role": "user",      "content": [{"text": "その中で S3 とは？"}]},
    # ← 「その中で」が何を指すかを理解できる
]
```

**試験での判断軸**：「複数モデルを比較・切り替えたい」「マルチターン対話」→ Converse API

### RetrieveAndGenerate API とは

Knowledge Bases を使った RAG を**1回の API 呼び出しで完結**させる API。

```
自前実装の RAG（3ステップ必要）
  ① Retrieve API でベクトル検索 → 関連チャンクを取得
  ② 取得チャンクをプロンプトに組み込む（自前コード）
  ③ InvokeModel / Converse API でモデルを呼ぶ

RetrieveAndGenerate API（1ステップ）
  質問を渡すだけで ①②③ を内部でまとめて実行
  → 回答 + citations（ソース帰属）が自動で返ってくる
```

**Retrieve API との違い**

| API | 何をするか | 使いどころ |
|---|---|---|
| **Retrieve API** | ベクトル検索のみ（モデル呼び出しなし） | 検索結果を自前でハンドリングしたい場合 |
| **RetrieveAndGenerate API** | 検索 + 生成 + citations 付与を1回で完結 | 標準的な RAG ユースケース |

**マルチターン対話**：`sessionId` を引き継ぐことで会話の文脈を維持できる。

### Converse API vs RetrieveAndGenerate API（使い分け）

| | Converse API | RetrieveAndGenerate API |
|---|---|---|
| 主な用途 | 汎用モデル対話（RAG なしも含む） | Knowledge Bases を使った RAG |
| RAG の組み込み | 自前で Retrieve してプロンプトに挿入 | **自動**（1回の呼び出しで完結） |
| ソース帰属（citations） | 自前実装が必要 | **自動付与** |
| モデル切り替え | modelId を変えるだけ | KB 設定で指定 |
| マルチターン管理 | messages 配列 | sessionId |

- Knowledge Bases と組み合わせることで、RAG + マルチターン + ソース帰属を一体で実現

### 不正解パターン（AIP-22）

| 選択肢 | 不正解の理由 |
|---|---|
| Extended Thinking（拡張思考） | 内部推論トークンが大量に生成される → **3 秒以内**の応答要件に反する可能性が高い |
| SageMaker + OpenSearch + Lambda | 自前構築でインフラ管理コスト大。「管理負担の最小化」に反する |
| Bedrock Agents + プロンプトエンジニアリング | エージェントのオーケストレーション（ReAct ループ）でレイテンシが増加。3 秒以内の達成が困難 |

## マルチテナント Knowledge Base 構成（AIP-27）

### 要件

- 複数テナント（ビルごと）のデータを**厳密に分離**
- 空室状況の問い合わせに**ほぼリアルタイムの応答**
- 管理スタッフのアクセスを**テナント単位で制御**
- 繁忙期でも安定した応答速度

### 正解構成

**中央集権的な Bedrock Agent（ビルごとの KB をオーケストレーション）+ IAM Identity Center**

```
管理スタッフ
  ↓ IAM Identity Center（許可セット）でアクセス制御
     → 自分のビルの KB のみ参照可能
  ↓
Bedrock Agent（中央集権的に複数 KB を管理）
  ├─ ビル A の Knowledge Base（静的な運用ルール・テナント情報）
  ├─ ビル B の Knowledge Base
  └─ アクショングループ → BMS API を直接呼び出す（リアルタイム空室情報）
```

### ポイント①：リアルタイム性はエージェントのアクショングループで実現

```
【間違いパターン】
BMS → SQS → Lambda → KB を更新 → 検索
         ↑ KB のインジェスト処理に時間がかかる → リアルタイム不可

【正解パターン】
BMS → エージェントのアクショングループが API を直接呼び出す
         ↑ 最新データをその場で取得 → リアルタイム対応
```

**「頻繁に変わるデータ（空室状況）」は KB に同期するのではなく、Agent が直接 API 呼び出し。**
KB はあくまで「静的に近いドキュメント（運用ルール・テナント契約等）」に使う。

### ⚠️ OpenSearch のメタデータフィルタ ≠ 厳密なデータ分離

OpenSearch では k-NN クエリに `filter`（メタデータフィルタ）を付与することでテナントごとにデータを絞り込める。
ただし、これは**論理的なフィルタリング**であり**物理的・厳密な分離ではない**。

```
論理的フィルタリング（OpenSearch メタデータフィルタ）
  → 同一インデックス内でクエリ時に条件を絞るだけ
  → フィルタの設定ミスやバグがあると他テナントのデータが漏洩するリスクがある
  → ストレージレベルでは全テナントのデータが混在している

厳密な分離
  → テナントごとに別インデックス or 別 KB
  → 物理的にデータが混在しないため漏洩リスクがない
```

**試験での判断軸**：「厳密な分離（strict isolation）」が要件 → 単一インデックス + メタデータフィルタは**不正解**。
テナントごとに KB（＝別インデックス）を作成する構成が正解。

---

### ポイント②：KB は 1 アカウントあたり約 10 個の上限あり

ビルごとに KB を作成する方針自体は有効だが、ビル数が増えるとクォータに達する。
大規模展開ではメタデータフィルタリングとの組み合わせや、クォータ引き上げリクエストが必要。

### ポイント③：アクセス制御は IAM Identity Center（Cognito ではない）

| | Amazon Cognito | AWS IAM Identity Center |
|---|---|---|
| 主な用途 | **Web/モバイルアプリのユーザー認証** | **組織の AWS リソースへのアクセス管理（SSO）** |
| 向いているケース | サインアップ・ログイン・ソーシャルログイン | ビル管理スタッフが特定の KB だけにアクセスできるよう制御 |
| AWS リソース制御 | Identity Pool 経由で間接的に | **許可セット（Permission Set）で直接 IAM ロールを割り当て** |

→「AWS リソース（Bedrock KB）へのアクセスをテナント単位で制御」→ **IAM Identity Center が正解**

### 不正解パターン（AIP-27）

| 選択肢 | 不正解の理由 |
|---|---|
| 全ビルを単一 OpenSearch インデックスに統合 | 論理的な分離のみ（メタデータフィルタ）。フィルタのミスでデータ漏洩リスク。「厳密な分離」要件に反する |
| CloudWatch Logs でアクセス監査 | 事後的な記録であり事前の認可制御ではない |
| ビルごとに KB + SQS + Cognito | SQS 経由の KB 更新はリアルタイム性が低い。Cognito は AWS リソース制御より Web 認証向き |
| ビルごとに専用 AWS アカウント + Kendra | アカウント管理コストが爆発的に増大。Kendra は KB のベクトルストアとして使えない。日次バッチではリアルタイム不可 |

### ⚠️ 模試ミス回収（AIP-27）：「KB + SQS + Cognito」を選んでしまった

**選んだ理由（誤った思考）**：
「ビルごとに KB を分けて、SQS でリアルタイムに更新し、Cognito でアクセス制御すれば全要件を満たせるのでは？」

**なぜ間違いか：3 つの見落とし**

**① SQS + KB 更新はリアルタイムではない**

```
SQS にメッセージが届く
  ↓ Lambda が受信（ポーリング）
  ↓ KB のインジェストジョブを実行（チャンキング・埋め込み・格納）
  ↓ 数分〜数十分後に反映

「空室状況」のようにリアルタイム性が必要なデータには不適切。
正解：エージェントのアクショングループで BMS API を直接呼び出す。
```

**② Cognito は「AWS リソースへの組織的アクセス管理」には向いていない**

```
Cognito：Web/モバイルアプリのユーザー認証が主目的。
IAM Identity Center：組織の SSO・AWS リソースへの権限管理が主目的。
→「管理スタッフが自分のビルの KB だけにアクセス」= IAM Identity Center の許可セットが正解
```

**③ KB の上限（約 10 個）を意識する**

```
ビルが 10 棟を超えると KB のクォータに達する可能性がある。
エージェントが複数 KB をオーケストレーションする構成なら、
クォータ管理やメタデータフィルタとの組み合わせで柔軟に対応できる。
```

**正しい判断軸**

| 判断ポイント | 正解の考え方 |
|---|---|
| リアルタイムデータ | KB 同期ではなく、エージェントのアクショングループで API を直接呼ぶ |
| 組織の AWS リソース制御 | Cognito ではなく IAM Identity Center |
| データ分離 | ビルごとに KB を作成（ただし上限に注意）|

## ハルシネーション削減アプローチ（AIP-30）

### 問題：ハルシネーション（原文にない情報を生成）

現象：文書全体をそのままプロンプトに入力するだけでは、FM が原文にない統計データ・研究結果を生成する。

### 正解：2 つの組み合わせ（複数選択）

#### ① ゼロショット Chain-of-Thought（CoT）プロンプティング

```
プロンプトの設計：
「要約を生成する前に、以下のステップを実行すること。
 ①ソース文書のどの部分に根拠があるかを特定する
 ②その根拠と結論の論理的なつながりを説明する
 ③根拠のない情報は含めないこと」
```

- **追加インフラ不要**（プロンプト変更のみ）
- モデルが明示的に「ソース文書との照合」を行うため、ハルシネーションが減少
- 「ゼロショット」= Few-shot の例示なしで、指示だけで推論プロセスを誘導

#### ② RAG（Bedrock Knowledge Bases + セマンティックチャンキング）

```
ドキュメント
  ↓ セマンティックチャンキング（意味のまとまりで分割）
  ↓ 埋め込みモデルでベクトル化
ベクトルストア
  ↓ クエリに最も関連するチャンクを取得
FM（「提供されたコンテキストのみを使って要約すること」と指示）
  ↓
コンテキストに基づいた回答
```

- FM に「取得したコンテキストに根拠を置く」という制約が生まれ、ハルシネーションが減少
- セマンティックチャンキングにより文脈の切れ目が少なく、検索精度が向上

### 不正解パターン（AIP-30）

| 選択肢 | 不正解の理由 |
|---|---|
| SageMaker フルファインチューニング | ハルシネーション解決策として不十分。実装・学習コスト大。「3 秒以内」「毎時 1500 件」という運用要件にも影響 |
| Comprehend Medical で前処理しエンティティのみ入力 | エンティティ抽出は文脈を失う。重要な数値や因果関係が落ちる可能性があり、かえって品質が下がる |
| temperature=1.0 / top_p 最大値 | **逆効果**。temperature が高いほど多様性（ランダム性）が増し、ハルシネーションが増加する |

### temperature とハルシネーションの関係（試験頻出）

```
temperature = 0    → 最も確実な（高確率の）トークンを選択 → ハルシネーション少ない
temperature = 1.0  → 確率分布の多様性を最大化 → ハルシネーション増加
```

→ 「事実精度を高めたい」要件で temperature を上げる選択肢は**必ず不正解**。

---

### ⚠️ CoT（Chain-of-Thought）≠ Extended Thinking（拡張思考）

混同しやすいが別物。

```
Chain-of-Thought（CoT）プロンプティング
  = プロンプトの設計技法
  = 「ステップバイステップで考えてください」という指示をプロンプトに加える
  = モデルの推論プロセスを出力に含めさせて、照合・確認ステップを強制
  = 追加インフラ不要・プロンプト変更のみで即導入可能
  = ハルシネーション防止に有効

Extended Thinking（拡張思考）
  = Anthropic Claude の特定機能
  = モデルが内部で長い推論トークンを消費してから回答を生成する
  = 主に数学・コード等の複雑推論タスク向け
  = トークン増加により大幅なレイテンシー悪化（3秒以内の要件を満たしにくい）
  = RAG の代替にはならない
```

**試験での誤解パターン**：「モデルが推論する」と聞いてストリーミングや Extended Thinking を連想しない → CoT はあくまで**プロンプトへの指示**。

---

### ⚠️ 「要約なのになぜ RAG？」の疑問への回答

「文書を要約するならその文書を直接渡せばいい」という直感は正しい。ただし、長大な臨床試験レポートを丸ごとプロンプトに入力することには問題がある。

```
文書全体をそのまま入力する問題
  → コンテキストウィンドウの圧迫（モデルが全体を均等に注意できない）
  → 入力が長すぎると FM が情報を正確にトレースしにくくなる
  → ハルシネーションが発生しやすい

RAG（セマンティックチャンキング）を使う意味
  → 文書を意味のまとまりで分割し、必要な部分だけを取得してコンテキストに渡す
  → FM に「提供されたチャンクのみを使って要約すること」と制約をかけられる
  → モデルが「取得されたコンテキストに根拠を置く」形になりハルシネーション減少
  → 入力量が減り、レイテンシーも改善しやすい
```

**整理**：RAG は「知識ベースから情報を引っ張ってくる」だけでなく、「対象文書を分割して根拠付き生成を強制する」グラウンディング手法としても使う。

## チャンキング戦略（Bedrock Knowledge Bases）

文書をベクトルストアに格納する際に、どう分割するかを決める設定。分割単位が検索精度・文脈保持・レイテンシーに直接影響する。

### 種類と比較

| 戦略 | 分割の基準 | 向いているケース |
|---|---|---|
| **固定サイズチャンキング** | 文字数・トークン数で機械的に分割 | 構造が均質なドキュメント / シンプルな設定で良い場合 |
| **セマンティックチャンキング** | 意味のまとまり（文章の区切れ目）で分割 | 臨床試験レポート・法令文書など構造化された長文 |
| **階層的チャンキング** | 親チャンク（大きい単位）と子チャンク（小さい単位）の二層構造で分割 | 概要レベルの検索と詳細レベルの検索を両立したい場合 |

---

### 固定サイズチャンキング

```
文書を N トークン（or N 文字）ごとに機械的に切る
オーバーラップ（重複）を設定することで文脈の切れ目を緩和できる

例：512 トークンで分割、オーバーラップ 50 トークン
  [...文章A | 文章B | 文章C...]
         [重複]   [重複]
```

- シンプルで実装コストが低い
- 文章の途中で切れて文脈が失われるリスクがある
- 「意味の切れ目」を考慮しないため、同じ話題が複数チャンクに分断されやすい

---

### セマンティックチャンキング

```
文の埋め込みベクトルを比較し、意味的に大きく変化する箇所で分割する
→ 段落・節の区切りに近い単位でチャンクが作られる

例：臨床試験レポート
  [患者背景セクション] [投与量・手順セクション] [結果・統計セクション] ...
  → 各セクションが意味的なまとまりとして 1 チャンクになる
```

- 固定サイズより検索精度が高い（関連する情報がチャンク内に収まりやすい）
- 構造化された長文ドキュメントに特に有効
- **試験頻出**：ハルシネーション削減 + 高精度 RAG のキーワードとして出題

---

### 階層的チャンキング（Hierarchical Chunking）

```
文書を 2 つの粒度で保持する

親チャンク（大きい）：章・節レベルのまとまり → 文脈の把握に使う
子チャンク（小さい）：文・段落レベルの細粒度 → 精密な検索に使う

検索フロー
  クエリ → 子チャンクで類似検索 → ヒットした子チャンクの「親」を返す
```

- 検索は細かい粒度で行い、回答生成は広い文脈で行うという二段構え
- 「どの段落が関連するか」を見つけつつ「その段落が属するセクション全体」を FM に渡せる
- 長文で細部の検索と広い文脈の保持を両立したいケースに向く

---

### 試験での判断軸

- 「意味のまとまりを保ちたい」「ハルシネーション削減」「高精度な要約」→ **セマンティックチャンキング**
- 「概要と詳細の両方を検索に活かしたい」→ **階層的チャンキング**
- 「シンプルに実装したい」→ **固定サイズチャンキング**（ただし文脈切断リスクあり）

## ⚠️ Bedrock は埋め込み（ベクトル化）もできる（AIP-32 不正解ポイント）

### Bedrock の埋め込みモデル

```
Amazon Bedrock = 生成AIだけでなく、埋め込み（Embedding）生成もできる

主な埋め込みモデル
  Amazon Titan Embeddings V2        ← AWS純正
  Cohere Embed（英語 / 多言語）      ← 多言語対応

呼び出し方：InvokeModel API（生成と同じ）
  → テキスト → float配列（ベクトル）が返ってくる
  → モデルのホスティング・GPU・管理は一切不要
```

### HuggingFace とは（混同注意）

```
HuggingFace（ハギングフェイス）
  = AIモデルの GitHub 的なプラットフォーム＋Pythonライブラリ群

  ① Hub       → 数十万のAIモデルが公開（BERT / LLaMA / Stable Diffusion etc.）
  ② transformers ライブラリ → Hubのモデルを Python で使うためのライブラリ
  ③ sentence-transformers  → テキスト→ベクトル変換に特化したライブラリ
```

```python
# 使い方のイメージ
from sentence_transformers import SentenceTransformer
model = SentenceTransformer('all-MiniLM-L6-v2')  # Hub からモデルをDL
vector = model.encode("こんにちは")               # → float配列（ベクトル）
```

ローカル・研究環境では便利だが、**モデルを自分の環境で動かす**設計。

### HuggingFace を Glue で動かすと不正解になる理由

| 問題点 | 内容 |
|---|---|
| GPU サポート | Glue（Apache Spark基盤）はGPUサポートに制約あり → 推論が遅い |
| モデル管理 | 数百MB〜数GBのモデルファイルのキャッシュ・ライブラリ依存関係を自前管理 |
| 運用負荷 | 「運用負荷最小」要件に反する |
| コスト | Bedrock API より高くなるケースがある |

| | HuggingFace（自前実行） | Bedrock 埋め込み API |
|---|---|---|
| 実態 | モデルを自分の環境で実行 | API を叩くだけ |
| 管理 | モデル・GPU・ライブラリを自前管理 | AWS 任せ |
| 向いている場面 | ローカル開発・研究・カスタム学習 | 本番・サーバーレス構成 |
| AIP 試験での扱い | 「運用負荷が高い」**不正解パターン** | **正解パターン** |

→ 試験で HuggingFace が選択肢に出てきたら**ほぼ不正解**。  
→ 埋め込みが必要なら **Bedrock の InvokeModel を呼ぶだけ**が最もシンプル。

### AIP-32 の正解構成（RAG取り込みパイプライン・サーバーレス）

```
S3（約40GB の JSON）
  ↓
Step Functions Distributed Map（最大10,000並列でS3ファイルを処理）
  ↓ 各ファイルに対して
Comprehend DetectPiiEntities API → PIIをマスク（コード不要）
  ↓
Bedrock InvokeModel（Titan Embeddings V2）→ ベクトル化
  ↓
OpenSearch Serverless（ベクトル格納・k-NN検索）
```

全工程がマネージドAPIの呼び出しのみ → 自前コードほぼゼロ。

### サーバーレス要件でのストレージ選定

| サービス | サーバーレス? | 備考 |
|---|---|---|
| **OpenSearch Serverless** | ◯ | OCU自動スケール、クラスター管理不要 |
| OpenSearch Service（マネージドクラスター） | ✗ | ノード数・インスタンスタイプの選定・管理が必要 |
| RDS for PostgreSQL（pgvector） | ✗ | インスタンス管理・パッチ適用が必要 |
| DocumentDB | ✗ | ベクトル検索が限定的、クラスター管理あり |

→ **「サーバーレス＋最小運用負荷」と来たら OpenSearch Serverless** 一択。

## OpenSearch 重要用語

### BM25（Best Match 25）

OpenSearch / Elasticsearch のデフォルト**全文検索スコアリングアルゴリズム**。

- **TF（Term Frequency）**：文書内での単語出現頻度
- **IDF（Inverse Document Frequency）**：全文書中での単語の希少さ
- 2つを組み合わせて「この文書にとってこの単語がどれだけ重要か」をスコア化

| 検索方式 | アルゴリズム | 得意なこと |
|---|---|---|
| 全文検索 | **BM25** | キーワードが含まれる文書を優先 |
| セマンティック検索 | **k-NN（HNSW/IVF）** | 意味的に近いベクトルを探す |

> 例：「子供向けビーチホテル」→ BM25はこの単語を含む文書を優先、k-NNは「ファミリー向け海辺リゾート」も意味的に近いとして引っかける

### JVM ヒープサイズ

OpenSearch は JVM（Java VM）上で動作。ヒープ = JVM が使えるメモリ領域。

| 設定 | 影響 |
|---|---|
| 小さすぎる | GC が頻繁に発生 → レイテンシースパイク |
| 大きすぎる | GC 1回の停止時間が長くなる（Stop-the-World） |
| **推奨** | 物理メモリの50%以下、かつ上限31GB（Compressed OOPs が効く境界） |

試験文脈：「検索レイテンシーが不安定」の原因としてヒープ設定ミスが登場しがち。

### Auto-Tune

OpenSearch の**自動パフォーマンス最適化機能**。

- JVM ヒープサイズ、スレッド数、キューサイズ等を自動調整
- `_cluster/settings` で有効化
- 手動チューニング不要でレイテンシーを安定化

試験文脈：「インフラ管理を最小化しつつ安定したレイテンシーを維持」という要件でAuto-Tuneが正解になる。

### Step Functions Distributed Map とは

```
S3上のオブジェクトを自動列挙 → 最大10,000並列で各ファイルを処理

通常の Map ステート     → 配列データのイテレート
Distributed Map ステート → S3インベントリ・S3リストから直接取得して大規模並列
```

- リトライ・エラーハンドリングは Step Functions の組み込み機能で対応
- 個別のエラー処理コードを書く必要なし
- 大量ファイルのバッチ処理オーケストレーションとして最適
